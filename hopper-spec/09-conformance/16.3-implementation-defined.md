# 16.3 Implementation-Defined Behavior

## 16.3.1 Definition

Implementation-defined behavior is behavior for which this specification requires the
implementation to document its choice, but does not prescribe which choice MUST be made.
Conforming implementations MAY differ from each other in these areas. Programs that
depend on implementation-defined behavior are conforming but not portable.

## 16.3.2 Stack Size

The initial stack size of the main thread is implementation-defined. On Linux, the default
is controlled by the kernel's `RLIMIT_STACK` resource limit, which is typically 8 MiB.
The reference implementation does not set any stack-size limit; programs that require a
specific stack size MUST set it themselves via `setrlimit` or by spawning threads with an
explicit stack.

## 16.3.3 Register Allocation

The assignment of Hopper variables to machine registers or stack slots is
implementation-defined. The reference implementation delegates register allocation to
LLVM's register allocator, which varies by optimization level (`-O0` in dev builds,
`-O2` in production builds). Programs MUST NOT assume that a given variable resides in a
specific register at any point in execution.

## 16.3.4 Class Instance Memory Layout

The memory layout of a class instance — the offset of each field within the allocated
block — is implementation-defined, subject to the following normative constraint: fields
MUST appear in the order they are declared in the class body, with no field reordering.
Alignment padding between fields is implementation-defined. Programs MUST NOT depend on
the byte offset of any field. Programs that require a specific layout SHOULD use
`struct` declarations, which provide a fully specified layout.

## 16.3.5 Debug Information Format

The format and completeness of debug information emitted by `hopper build` (without
`--production`) is implementation-defined. The reference implementation passes `-g -O0`
to clang, producing DWARF debug information in the LLVM IR. The presence, version, and
contents of DWARF sections in the final binary are not specified by this document.

## 16.3.6 Dead Code Elimination

Whether unreachable functions, unused class methods, or dead branches are eliminated from
the compiled output is implementation-defined. The reference implementation passes the
generated LLVM IR to clang, which applies LLVM's standard optimization passes at the
selected optimization level. Programs MUST NOT depend on dead code being present or
absent in the binary.

## 16.3.7 Diagnostic Message Text

The exact text of a diagnostic message (error or warning) is implementation-defined.
Only the presence of a diagnostic and its source location are required (§16.2.2).
Conformance tests MUST NOT match exact error message text; they MUST check only that a
diagnostic was produced for the expected source location.

## 16.3.8 Inline Assembly Constraint Strings

The exact LLVM inline assembly constraint strings used for `asm {}` register operands are
implementation-defined. The reference implementation uses LLVM's constraint string syntax
(e.g. `"r"` for a general-purpose register, `"={rax}"` for a specific named register).
An alternative implementation that targets a different backend MAY use different
constraint encodings, provided the assembly semantics are preserved.

## 16.3.9 String Constant Deduplication

Whether two identical string literals in the same compilation unit are stored at the same
address or different addresses is implementation-defined. Programs MUST NOT compare
string literals by pointer identity; they MUST compare by content.

## 16.3.10 LLVM IR Label Names

The names of LLVM IR basic block labels (e.g. `%requires.ok.1`, `%while.cond.3`) are
implementation-defined. They are not part of the ABI and will differ across compiler
versions and optimization levels.
