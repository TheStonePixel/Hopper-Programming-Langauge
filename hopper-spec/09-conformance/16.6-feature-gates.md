# 16.6 Feature Gates

## 16.6.1 No Feature Gates

**Not yet implemented.**

Hopper does not currently have a feature-gate system. There is no mechanism to
conditionally enable or disable language features at the project or file level (analogous
to Rust's `#![feature(...)]` or C's `#define _POSIX_C_SOURCE`).

All language features described in this specification (excluding those marked
"Not yet implemented.") are available unconditionally in every Hopper source file.

## 16.6.2 The --strict Flag Is Not a Feature Gate

The `--strict` compiler flag is sometimes confused with a feature gate. It is not.

`--strict` enables a compile-time constant-folding pre-pass (§16.2.6, item 21) that
promotes certain contract violations from runtime failures to compile-time errors.
It does not enable or disable language features. Every program that compiles under
`--strict` also compiles without it; `--strict` only changes whether some errors are
caught earlier.

The `--strict` flag is invoked as:

```bash
hopper build --strict
```

It is intended for development and CI use where early detection of provable contract
violations is valuable.

## 16.6.3 Build Profiles Are Not Feature Gates

The three build profiles — `dev` (default), `test`, and `production` — affect
optimization level and whether contract checks are emitted (contracts are elided in
`production`), but they do not change which language features are available. A program
that uses a given syntactic feature MUST compile successfully under all three profiles,
though the runtime behavior of contract violations differs.

## 16.6.4 Future Direction

A future version of Hopper MAY introduce a feature-gate mechanism to allow
experimentation with unstable language features before they are stabilized. Any such
mechanism would be declared in `hopper.json` and would not require source-file
annotations. The design of this mechanism is not yet specified.
