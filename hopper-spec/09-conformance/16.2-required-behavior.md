# 16.2 Required Behavior

## 16.2.1 Overview

This section lists behaviors that a conforming implementation MUST exhibit. "Required"
means the behavior is observable and testable; an implementation that does not exhibit
it is not conforming.

## 16.2.2 Parsing

1. A conforming implementation MUST accept every program that conforms to the Hopper
   grammar (§2) without producing a diagnostic.
2. A conforming implementation MUST reject every program that violates the grammar with
   at least one diagnostic message. The diagnostic MUST identify the source location
   (file, line, column) of the first detected error.
3. A conforming implementation MUST NOT produce executable output for a program that has
   a parse error.

## 16.2.3 Type Checking

4. A conforming implementation MUST reject any program where a variable is used before
   it is declared in the enclosing scope.
5. A conforming implementation MUST reject any assignment where the right-hand-side type
   is not assignment-compatible with the declared type of the left-hand-side variable,
   unless a `cast<T>` is present.
6. A conforming implementation MUST reject any function call where the number of
   arguments does not match the number of declared parameters (excluding variadic
   functions).
7. A conforming implementation MUST verify that a class declared `satisfies Interface`
   provides a concrete definition for every function declared in that contract, with
   identical parameter types and return type. Failure MUST produce a diagnostic.

## 16.2.4 Code Generation

8. `allocate N` MUST compile to a call to `malloc(N)` on hosted targets. The result MUST
   be of type `address` (i8*).
9. `deallocate ptr` MUST compile to a call to `free(ptr)`.
10. `entry` blocks MUST compile to an LLVM function returning `i64`, consistent with the
    C ABI `int main(...)` convention.
11. Inline `asm {}` blocks MUST compile to LLVM inline assembly using the `asm` keyword
    with appropriate constraint strings for the named registers.
12. The compiler MUST emit `declare i8* @malloc(i64)`, `declare void @free(i8*)`, and
    (in non-release builds) `declare void @abort()` in every hosted compilation unit.
13. The compiler MUST NOT emit the above declarations for bare-metal targets.

## 16.2.5 Interface Conformance Dispatch

14. A free function from an contract implementation file MUST come into scope in the
    compiling file when the build system resolves the contract binding. Calls to such
    functions MUST compile to direct calls to the implementation's LLVM function symbol.
15. There are no vtable pointers. Method dispatch MUST be resolved statically at compile
    time based on the declared type of the receiver variable.

## 16.2.6 Contract System

16. A `requires` clause on a function MUST produce a runtime check at the entry of that
    function that calls `@abort()` if the condition is false (in non-release builds).
17. An `ensures` clause on a function MUST produce a runtime check before every `return`
    from that function (in non-release builds).
18. An `invariant` clause on a loop MUST produce a runtime check at the loop condition
    label on every iteration (in non-release builds).
19. A `constrain` annotation on a variable declaration MUST produce a runtime range check
    on the assigned value (in non-release builds).
20. In release builds (`--production`), all contract checks MUST be elided entirely — no
    `@abort()` call, no branch, no dead code.
21. In strict mode (`--strict`), the compiler MUST perform a compile-time constant-folding
    pass and report a diagnostic (compile error) for any contract violation that is
    provable at compile time.

## 16.2.7 Module System

22. The build system MUST resolve contract imports by reading the `targets` section of the
    nearest `hopper.json` in the ancestor directories of the source file.
23. The build system MUST compile both the contract file and the implementation file for
    each binding and include both in the linked binary.
24. `hopper install <name>` MUST copy the named module and all of its transitive
    dependencies from the global store into the project's `modules/` directory, nested to
    mirror the dependency graph.

## 16.2.8 Entry and Exit

25. A Hopper program MUST exit with the value returned by the `entry` block, truncated to
    8 bits, as the process exit status.
26. If the `entry` block does not contain a `return` statement, the compiler MUST emit a
    `ret i64 0` at the end of the entry function.
