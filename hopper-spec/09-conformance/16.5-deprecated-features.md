# 16.5 Deprecated Features

## 16.5.1 Definition

A deprecated feature is one that the current compiler accepts but that the language
design intends to remove in a future version. Programs that use deprecated features will
continue to compile today but SHOULD be updated. A future version of this specification
will mark deprecated features as removed, at which point conforming implementations
MUST reject them.

The reference implementation does not currently emit warnings for deprecated features.
It SHOULD do so in a future release.

## 16.5.2 Top-Level const Declarations

**Deprecated. Use `enum` instead.**

The grammar accepts top-level constant declarations of the form:

```hopper
const STDIN  = 0
const STDOUT = 1
const NAME   = "hello"
const NEG    = -1
```

These are inlined at compile time and generate no data in the binary. They work correctly
in the current compiler.

**Why deprecated:** The language design intent is to eliminate module-level constants as
a category. Named integer sets belong in `enum` declarations, which have explicit scope,
are grouped logically, and prevent accidental name collisions. String constants belong
inside classes or functions, or should be passed as parameters.

**Migration:**

Replace a block of related integer constants with an enum:

```hopper
// Before (deprecated)
const STDIN  = 0
const STDOUT = 1
const STDERR = 2

// After (preferred)
enum FileDescriptor { STDIN = 0  STDOUT  STDERR }
```

Access as `FileDescriptor.STDIN`. For unrelated constants that do not form a logical
group, scope them inside the class or function that uses them, or pass them as parameters.

**Timeline:** Top-level `const` will be removed once `enum` covers all use cases and
the grammar supports constants inside contract bodies (§Open Design Questions, item 2).

## 16.5.3 Lowercase Module Import Form

**Deprecated. Use the contract import form instead.**

The grammar accepts a legacy import form:

```hopper
import io from arch
import tty from linux
```

This form performs file-based resolution: it looks for
`modules/<module>/src/<name>.hop` without consulting `hopper.json`. It was used by
early Hopper programs before the contract system existed.

**Why deprecated:** The file-based import form does not support the hardware-abstraction
model. It bypasses `hopper.json` target bindings, making programs that use it impossible
to retarget without source changes. It couples program source code to specific file paths
rather than named interfaces.

**Migration:**

```hopper
// Before (deprecated)
import io from arch

// After (preferred)
import IO from linux
```

Add the corresponding binding to `hopper.json`:

```json
"IO": {
  "from":           "linux",
  "contract":      "modules/linux/interfaces/IO.hop",
  "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
}
```

**Timeline:** The lowercase file-based import form will be removed when all modules in
`hopper/nonconform/` have been updated and moved to `hopper/modules/`. Until then, the
form remains in the grammar to support legacy modules.

## 16.5.4 Direct Invocation of hopperc.js

**Deprecated. Use `hopper build` instead.**

The reference compiler (`seed/compiler/hopperc.js`) is an internal tool. Invoking it
directly — rather than through the `hopper` build system CLI — bypasses:

- `hopper.json` parsing and target binding resolution.
- Dependency installation and path localization.
- Deduplication of `declare` stubs across compilation units.
- Profile-based optimization flags.

**Migration:** Use `hopper build` for all compilation. Use `hopper ir <file.hop>` to
inspect generated LLVM IR. Use `hopper ast <file.hop>` to inspect the AST.

**Timeline:** `hopperc.js` will not be removed (it is the compiler), but it will no
longer be documented as a user-facing tool. Its internal API is subject to change without
notice.
