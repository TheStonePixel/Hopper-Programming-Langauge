# 16.1 Compliance Levels

## 16.1.1 Two Levels

A conforming Hopper implementation MUST achieve compliance at two levels:

1. **Core conformance** — correct implementation of the Hopper language: syntax, type
   system, memory model, execution model, module system, and contract system.
2. **Platform conformance** — correct implementation of the ABI for at least one supported
   platform.

An implementation that achieves core conformance but targets no conforming platform is
not a conforming Hopper implementation.

## 16.1.2 Core Conformance

An implementation is core-conformant if it:

1. Accepts all syntactically valid Hopper source programs and rejects all syntactically
   invalid programs with at least one diagnostic message.
2. Implements the type system as specified in §§4–7: primitive types, class types,
   contract types, enum types, function types, and the cast system.
3. Generates code that satisfies the memory model as specified in §12.4: `allocate`
   maps to `malloc`, `deallocate` maps to `free`, stack variables are automatically
   managed.
4. Generates code that satisfies the execution model as specified in §12: no managed
   runtime, no GC, no scheduler.
5. Implements the contract system: `requires`, `ensures`, `invariant`, and `constrain`
   clauses, with runtime checks in non-release builds and elision in release builds.
6. Implements the module system: contract imports, contract conformance checking,
   `hopper.json` target binding resolution.
7. Implements `defer` with function-scoped LIFO semantics.

## 16.1.3 Platform Conformance

### x86-64 Linux Hosted

An implementation is x86-64 Linux hosted conformant if it:

1. Generates ELF-64 executables that run on a Linux kernel (version 4.4 or later).
2. Follows the System V AMD64 ABI for all function calls: integer arguments in
   `rdi`, `rsi`, `rdx`, `rcx`, `r8`, `r9`; return value in `rax`; callee-saves
   `rbx`, `rbp`, `r12`–`r15`.
3. Uses `i64` for `int`, `i8` for `byte`/`char`, `i1` for `bool`, `double` for `float`,
   and `i8*` for `address`/`string`, consistent with the type mappings in §4.
4. Links against `libc` (dynamically or statically) and `-lpthread` only, unless the
   program declares additional `extern function` dependencies.
5. Passes the toolchain-level tests in `hopper/tests/`.

### ARMv6 Bare-Metal

An implementation is ARMv6 bare-metal conformant if it:

1. Generates a raw binary image loadable at the board's load address (e.g. `0x8000`
   for Pi Zero).
2. Targets `arm-none-eabi` with `-mcpu=arm1176jzf-s -mfloat-abi=soft`.
3. Uses 32-bit word size for `int` and pointer types when targeting bare-metal
   (`wordBits = 32`).
4. Links with a board-specific boot stub that initializes the stack pointer and zeroes
   `.bss` before calling the Hopper entry function.
5. Does not link `libc` or any hosted library.

## 16.1.4 Achieving Both Levels for One Platform

A conforming implementation MUST be compliant at both levels for at least one supported
platform. An implementation that is core-conformant but generates incorrect ABI code for
every platform it targets is not conforming.

## 16.1.5 Partial Implementations

An implementation MAY document that it supports only a subset of the language (e.g.
no inline assembly, no bare-metal target). Such an implementation MUST clearly state
which features are unsupported and MUST reject programs that use unsupported features
with a diagnostic rather than silently producing incorrect output.
