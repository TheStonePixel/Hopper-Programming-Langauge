# 16.4 Unspecified Behavior

## 16.4.1 Definition

Unspecified behavior is behavior where this specification does not prescribe a specific
outcome, but also does not require the implementation to document its choice.
Implementations are free to choose any behavior in these areas, and the choice MAY vary
between invocations, compiler versions, or optimization levels, provided that the behavior
is consistent within a single compilation.

Unspecified behavior differs from undefined behavior: the program remains well-defined;
the specification simply does not constrain the outcome. Programs that depend on
unspecified behavior are not incorrect but are not portable.

## 16.4.2 Order of Top-Level Declarations in Output IR

The order in which functions, type definitions, global constants, and `declare` stubs
appear in the emitted LLVM IR is unspecified, except where order affects semantics (for
example, a `define` MUST appear before any `call` to it in the same function body, which
LLVM enforces by the SSA form of the IR — this is not an ordering requirement on the
module level).

Programs MUST NOT depend on the order of symbols in the compiled binary.

## 16.4.3 String Constant Placement within .rodata

The byte offset at which a given string constant is placed within the `.rodata` section
of the binary is unspecified. The compiler MAY place string constants in any order and
MAY merge identical constants from different compilation units. The only guarantee is that
each string constant is null-terminated and resides in a read-only segment.

## 16.4.4 Overloaded Function Name Mangling

When two or more functions share the same name (overload set), the compiler generates
mangled names to distinguish them. The mangling scheme is unspecified and may change
between compiler versions. External code that needs to call overloaded Hopper functions
MUST NOT depend on the mangled symbol name; it MUST use an `extern function` declaration
with the exact mangled name as it appears in the compiled object, which is a
version-specific implementation detail.

## 16.4.5 Template Instantiation Naming

The LLVM symbol name produced for an instantiated template class or function is
unspecified. The reference implementation uses the form
`__tmpl_<function>__<typeParam>__<params>`, but this is not normative.

## 16.4.6 Class Destructor Invocation Order

When a program calls `destroy` on multiple class instances in the same function,
the relative order of side effects from their destructors is determined by the order
of the `destroy` calls in the source, which is well-defined. However, if a program uses
`defer destroy a` and `defer destroy b`, the deferred destructor calls execute in LIFO
order relative to the `defer` statements — `b` before `a` if `b` was deferred later.
This is specified. The unspecified behavior is what happens if a destructor raises a
contract violation via `abort()` while another deferred destructor is still pending:
the remaining deferred calls are not guaranteed to run.

## 16.4.7 malloc Alignment

The alignment of the pointer returned by `allocate N` (i.e. `malloc(N)`) is unspecified
beyond the guarantee provided by the C standard: `malloc` returns a pointer suitably
aligned for any fundamental type. On x86-64 Linux with glibc, this is 16 bytes. Programs
that require larger alignment (e.g. for SIMD loads) MUST use `mmap` with appropriate
flags, or implement their own aligned allocator on top of `allocate`.
