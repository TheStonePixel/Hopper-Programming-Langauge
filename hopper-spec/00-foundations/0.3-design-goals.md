## Design Goals

The following goals are explicit design constraints. Changes to the language, build system, or standard library that violate these goals require revising this document, not the goals.

**G1 — Zero mandatory runtime.** A Hopper program that makes no system calls and performs no heap allocation must be compilable to a binary with no runtime dependencies. No startup code, no exception tables, no garbage collector, no thread-local storage beyond what the target ABI requires.

**G2 — Hardware access without escape hatches.** Programs must be able to read and write memory-mapped hardware registers, issue architecture-specific instructions, and control interrupt behavior using Hopper syntax. The inline assembly facility covers cases the module system cannot express. No C FFI is required for any hardware operation.

**G3 — Source-portable, build-configured hardware.** A program that runs on x86-64 Linux and a program that runs on ARM Cortex-M with no OS must be writable in identical Hopper source. The difference is exclusively in build configuration. No preprocessor conditionals, no platform-specific imports, no `#ifdef`.

**G4 — Casting is source code.** Type conversion rules are defined in Hopper source files, not in compiler internals. Adding a new type conversion does not require modifying the compiler.

**G5 — Module resolution is deterministic.** Given a project manifest and a module registry, the set of source files compiled is fully determined. There is no search path, no implicit discovery, no version resolution algorithm with nondeterministic outcomes.

**G6 — No hidden allocations.** Allocation only occurs at the explicit use of `allocate`. No language construct implicitly allocates heap memory. Standard library types that allocate are explicit about it in their contract.

**G7 — Interface conformance is compile-time verified.** If a class declares that it satisfies an contract, the compiler verifies that every method in the contract is implemented with the correct signature. This check requires no test suite.

**G8 — Inline assembly is bounded.** Inline assembly is available for expressing hardware-specific operations that cannot be expressed in the module system. It is restricted to leaf functions and is not permitted in functions that themselves call non-assembly functions, to prevent optimization-breaking side effects.
