# Versioning

## Current Status

Hopper is at version **0.1.0**. The language is pre-1.0. The specification is a working draft.

During the 0.x period, breaking changes may occur in minor version increments. A change from 0.1 to 0.2 may introduce syntax changes, semantic changes, or ABI-incompatible modifications. Consumers of the language should track the specification version, not assume stability between minor versions.

---

## Version Numbers

The specification version tracks the language version. They are the same number. A specification labeled `0.1.0` describes language version `0.1.0`.

Version numbers follow **semantic versioning** (semver) after 1.0:

- **Patch** (0.1.x → 0.1.x+1): clarifications, corrections to the specification text, and bug fixes in the reference implementation that do not change specified behavior. Conforming programs are unaffected.
- **Minor** (0.1.0 → 0.2.0): additive changes. New syntax, new standard library interfaces, new build system features. Existing conforming programs continue to compile and behave correctly.
- **Major** (0.x → 1.0, 1.x → 2.0): breaking changes. Syntax that previously compiled may no longer compile. Semantics of existing programs may change.

During the 0.x period, minor version increments may include breaking changes. This exception ends at 1.0.

---

## The 1.0 Commitment

Version 1.0 commits to backward compatibility within the 1.x series. A conforming program written for 1.0 will compile and behave correctly on any 1.x implementation. Breaking changes require a major version bump.

The pre-1.0 period exists to allow the design to stabilize without the constraint of backward compatibility. Known areas of design instability are:

- **Top-level `const`:** the intent is to remove it in favor of `enum`. This is a breaking change.
- **`hopper.json` schema:** the `exports` field and `targets` structure may change.
- **Grammar:** new constructs may be added and existing syntax may be adjusted.

These will be resolved before 1.0. The 1.0 release marks the point where the design is stable enough to commit to.

---

## Spec Version in `hopper.json`

A future version of the build system will include a `hopperVersion` field in `hopper.json` to declare which language version a package was written for:

```json
{
  "hopperVersion": "0.1.0"
}
```

> Not yet implemented.

When implemented, the build system will warn or error if a package declares a version incompatible with the installed compiler.
