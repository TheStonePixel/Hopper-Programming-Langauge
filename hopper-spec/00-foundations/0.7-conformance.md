## Conformance

### Conforming Programs

A conforming Hopper program is one that:

- Uses only syntax defined in the Hopper grammar
- Does not invoke undefined behavior
- Declares all external dependencies in its `hopper.json` manifest
- Has exactly one `entry` declaration (if it is a program) or zero `entry` declarations (if it is a library)
- Does not rely on behavior marked implementation-defined without acknowledging that it is not portable

A conforming program is not required to be portable across targets. A program that uses x86-64 inline assembly is conforming. It is simply not portable to ARM targets.

### Conforming Implementations

A conforming Hopper implementation is a compiler and build system that:

- Accepts all conforming programs and produces correct output
- Rejects programs with syntax errors, type errors, and contract conformance failures
- Implements all normative requirements in this specification marked MUST or SHALL
- Documents all implementation-defined behavior
- Produces output that, when executed on the target platform, behaves as this specification requires

An implementation is not required to:

- Accept programs that trigger undefined behavior
- Produce optimal output
- Implement optional features marked MAY
- Support targets beyond those it documents

### Partial Conformance

An implementation that satisfies a strict subset of this specification — for example, one that supports only the language section and not the package system — is a partially conforming implementation. It must document which sections of the specification it satisfies and which it does not.

### The Reference Implementation

The Hopper bootstrap compiler (`seed/compiler/`) is the reference implementation. Where this specification is ambiguous, the reference implementation's behavior is authoritative for the current working draft. Discrepancies between the specification and the reference implementation should be treated as specification defects until otherwise resolved.
