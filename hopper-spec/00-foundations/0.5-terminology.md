## Terminology

The following terms are used throughout this specification with specific meanings.

**Program.** A Hopper project of type `executable`. It has an entry point, produces a binary, and may declare interface bindings in its build manifest.

**Library.** A Hopper project of type `library`. It has no entry point, declares exported interfaces, and is consumed by programs or other libraries as a dependency.

**Module.** A single Hopper source file (`.hop`). Modules are the unit of compilation.

**Package.** A directory containing a `hopper.json` manifest, source files, and optionally nested dependency packages. A package is the unit of distribution and dependency resolution.

**Interface.** A named set of function signatures declared with the `interface` keyword. Interfaces are contracts — they describe what a type can do, not how it does it.

**Implementation.** A class or set of free functions that satisfies an interface contract. An implementation is bound to an interface by the consuming project's build manifest, not by the library itself.

**Binding.** An entry in a project's `hopper.json` targets section that maps an interface name to a concrete implementation file for a specific build target.

**Target.** A named build configuration in `hopper.json` (e.g., `host`, `armv6-bare`). A target specifies which implementation files are used to satisfy interface bindings.

**Template function.** A function with a compile-time type parameter, declared with the `template function` keyword. Each distinct combination of type parameter and argument types is a separate specialization compiled to a distinct function.

**Entry point.** The `entry` declaration in a program. Exactly one entry point is permitted per program. It is compiled to the `main` function in the output binary.

**Inline assembly.** An `asm { }` block containing architecture-specific register assignments and instructions. Inline assembly is valid only in functions that do not call other non-assembly functions.

**Conforming implementation.** A compiler, build system, or runtime that produces behavior consistent with this specification for all normative requirements.

**Undefined behavior.** A condition explicitly marked as undefined by this specification. A conforming implementation may do anything when undefined behavior occurs. Programs that trigger undefined behavior are not conforming programs.
