# Specification Structure

## Organization

The specification is organized into numbered sections, each in a subdirectory of `hopper-spec/`. Files within each section use numeric prefixes for ordering (e.g., `1.1-types.md`, `1.2-expressions.md`).

---

## Sections

| Directory | Section | Status |
|-----------|---------|--------|
| `00-foundations/` | Foundations — purpose, philosophy, terminology, normative language, conformance | Working Draft |
| `01-language/` | Language — types, expressions, statements, declarations, grammar | Working Draft |
| `02-workspace/` | Workspace — project layout, `hopper.json` schema, directory conventions | Working Draft |
| `03-build/` | Build system — compilation model, target resolution, import resolution | Working Draft |
| `04-packages/` | Packages — dependency management, `hopper install`, module registry | Working Draft |
| `05-runtime/` | Runtime model — entry point, stack, heap, execution environment | Working Draft |
| `06-stdlib/` | Standard library — interfaces and reference implementations | Working Draft |
| `07-abi/` | ABI — calling conventions, data layout, name mangling, object format | Working Draft |
| `08-tooling/` | Tooling — compiler CLI, build CLI, IR and AST debug output | Working Draft |
| `09-conformance/` | Conformance — what conforming implementations and programs must satisfy | Working Draft |
| `10-appendices/` | Appendices — grammar reference, keyword list, operator table, error codes | Working Draft |
| `11-rationale/` | Rationale — non-normative design explanations | Non-normative |

---

## Normative vs Non-Normative

Sections 00–10 are **normative**. They define what conforming implementations and programs must do. Normative requirements use RFC 2119 terms (MUST, SHALL, SHOULD, MAY) as defined in `0.6-normative-language.md`.

Section 11 (Rationale) is **non-normative**. It explains why decisions were made but does not impose requirements. When a normative section and a rationale document conflict, the normative section governs.

---

## File Naming

Files within a section use the following naming convention:

```
<section>.<subsection>-<topic>.md
```

Examples:
- `1.1-types.md` — language section, subsection 1, types
- `3.2-target-resolution.md` — build section, subsection 2, target resolution

The `README.md` in each section directory provides an overview and index of the section's contents.

---

## Status Markers

Individual sections may carry status markers:

- **Working Draft** — the content reflects current implementation but may change
- **Stable** — the content reflects a finalized design decision; changes require revision justification
- **Not yet implemented** — the feature is specified but not yet in the reference implementation

Unimplemented features are marked with a blockquote:

> Not yet implemented.
