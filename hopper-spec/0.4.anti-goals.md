## Anti-Goals

The following are explicitly not goals. They are recorded here to prevent design drift and to explain rejections of otherwise reasonable-sounding proposals.

**AG1 — Garbage collection.** Hopper does not have and will not have a garbage collector. Garbage collection introduces latency that is not visible in source code and requires a runtime that may not be present on the target. Manual memory management is a feature, not a deficiency.

**AG2 — Implicit type coercion.** Hopper does not perform automatic type conversion. An `int` does not become a `float` when passed to a function that expects a `float`. Calling `cast<float>(n)` is required. This eliminates a category of bugs that arises from implicit numeric promotion rules.

**AG3 — Closures and lambda expressions.** Hopper does not support closures. Closures imply captured environment, which implies implicit allocation or stack manipulation that is not visible in source. Function pointers are supported through the `callback` type; they carry no captured state.

**AG4 — Exceptions.** Hopper does not have an exception mechanism. Error handling is explicit: functions return error values, or they do not return at all. Exception handling requires stack unwinding tables, a runtime unwinder, and nondeterministic control flow — none of which are acceptable in a zero-runtime target.

**AG5 — A built-in standard library.** There is no standard library baked into the compiler. The `stdlib` section of this specification defines interfaces and reference implementations, but these are Hopper source modules that must be explicitly declared as dependencies. A program with no `stdlib` dependency compiles with no standard library.

**AG6 — Null safety as a language mechanism.** Hopper has a `null` literal for address types. There is no optional type, no null-tracking type system, and no compiler-enforced null safety. These mechanisms add complexity to the type system and have limited value in systems code where null often has a valid meaning (e.g., the zero page, a sentinel pointer, an uninitialized register).

**AG7 — Reflection.** Hopper has no runtime reflection. Types do not carry metadata at runtime. This eliminates the need for type tables, method dictionaries, and runtime type checks, all of which have unpredictable cost and require runtime infrastructure.

**AG8 — A module system that requires network access.** The package manager resolves dependencies from a local store. Builds are reproducible without network access. A build that requires fetching a package at compile time is not a Hopper build.
