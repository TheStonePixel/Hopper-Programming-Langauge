## Build Caching

**Not yet implemented.** The Hopper build system has no build cache. It does not hash inputs,
store derived artifacts between builds, or consult any cache when deciding whether to recompile.

### Current Behavior

Every `hopper build` invocation unconditionally overwrites all files under `build/`:

- `build/ll/<name>.ll` — LLVM IR for each compiled source unit
- `build/o/<name>.o` — object file for each compiled source unit
- `build/<name>` — final linked executable (or `build/kernel.img` for bare-metal)

The build system provides no mechanism to compare the current source against a previously
built output, and it does not preserve artifacts from previous builds for reuse.

### Implications

Because no caching exists:

- Re-running `hopper build` with no source changes produces identical output but incurs the
  full compilation cost every time.
- Switching between build profiles (e.g. from `dev` to `production`) requires a full rebuild;
  no profile-segregated cache exists.
- CI pipelines that want to avoid redundant compilation MUST implement their own caching layer
  at the filesystem level (e.g. by caching `build/o/` between runs keyed on source hashes),
  if desired.

### Future Work

A future caching implementation would require:

- Input hashing: a stable hash over the source file content, all transitive dependency
  content, the binding configuration, the build profile, and the compiler version.
- A cache store mapping input hashes to cached `.ll` and `.o` artifacts.
- Cache validation on each build: compare current input hash against stored hash, skip
  recompilation on a hit.

None of these mechanisms exist today. Until they are implemented, every build is a full
recompilation.
