## Linking

### Hosted Linking

For hosted (non-bare-metal) builds, linking is performed by invoking clang as the linker
driver:

```
clang <objects...> -lpthread -o <outFile>
```

The object files produced by the compilation step are passed as positional arguments.
`-lpthread` is always linked. No other libraries are explicitly requested by the build system.

The resulting executable is a dynamically linked ELF binary on Linux (or a Mach-O binary on
macOS). The dynamic linker resolves symbols at load time against the system's libc and
libpthread.

### Standard Library Dependency

Hopper programs that use `allocate` and `deallocate` depend on the C runtime functions
`malloc`, `free`, and `abort`. These are resolved automatically via the system libc at link
time; no explicit `-lc` flag is needed because clang links libc by default.

No other libc functions are linked unless the program explicitly declares them with `extern`:

```hopper
extern function memset(address dst, int c, int n) address
```

An `extern` declaration causes the compiler to emit an LLVM `declare` for that symbol. The
linker resolves it against whatever library provides it in the link step. Programs SHOULD NOT
rely on libc for application logic; `extern` declarations to libc are acceptable only for
temporary diagnostic use.

### Output File Naming

The output file name for a hosted build is determined in priority order:

1. The `-o <path>` argument to `hopper build`, if provided.
2. The `name` field in `hopper.json` at the project root.
3. The basename of the entry file (without `.hop` extension).

The file is written directly to `build/<name>` (not to a profile subdirectory).

### Bare-Metal Linking

For bare-metal builds, linking is performed by `ld.lld` (LLVM's linker) with a generated
linker script:

```
ld.lld -T build/link.ld <boot.o> <objects...> -o build/<name>.elf
```

The linker script is generated by `generateLinkerScript()` for the target board. For pi-zero,
the script places code at load address `0x8000`, with sections in this order:
`.text.boot`, `.text`, `.rodata`, `.data`, `.bss`. The `.ARM.exidx`, `.ARM.extab`, and
`.note` sections are discarded.

The entry symbol is `_start`, which is defined in the board's boot stub
(`seed/build_system/boards/<board>/boot.s`). The boot stub is compiled separately by clang
with bare-metal flags before linking:

```
clang -target arm-none-eabi -mcpu=arm1176jzf-s -mfloat-abi=soft \
      -ffreestanding -nostdlib -c boot.s -o build/o/boot.o
```

After ELF linking, `llvm-objcopy` strips the ELF down to a raw binary image:

```
llvm-objcopy -O binary build/<name>.elf build/kernel.img
```

The raw binary is the file that the Pi Zero's GPU firmware loads at address `0x8000`.

### Bare-Metal: No libc

Bare-metal builds pass `-ffreestanding -nostdlib` to both the clang compilation step and
implicitly to the linker (via the ld.lld invocation without a C runtime). There is no libc,
no `malloc`, and no `free` available. Programs targeting bare-metal MUST NOT use `allocate` or
`deallocate` unless they provide their own heap implementation.

### Build Metadata (Bare-Metal)

After a successful bare-metal link, the build system writes a `<name>.meta.json` file
alongside the kernel image in `build/`. This file records:

```json
{
  "name":        "<project name>",
  "board":       "pi-zero",
  "outputFile":  "kernel.img",
  "arch":        "armv6",
  "endian":      "little",
  "bits":        32,
  "loadAddress": "0x8000",
  "entrySymbol": "_start",
  "size":        <bytes>,
  "builtAt":     "<ISO 8601 timestamp>"
}
```

The `builtAt` field contains the wall-clock time of the build as an ISO 8601 string. This
field is written by the build system and is NOT reproducible across builds performed at
different times.
