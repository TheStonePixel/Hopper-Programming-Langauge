## Incremental Compilation

**Not yet implemented.** Hopper does not support incremental compilation. Every invocation of
`hopper build` performs a complete rebuild of the program from source.

### Current Behavior

Each `hopper build` invocation:

1. Re-reads every source file in the transitive import closure of the entry file.
2. Re-parses every file through ANTLR.
3. Re-generates LLVM IR for the entire merged AST.
4. Re-compiles the IR to an object file via clang.
5. Re-links the final executable.

No source file is skipped, regardless of whether it has changed since the previous build.
The intermediate files in `build/ll/` and `build/o/` are overwritten on every build.

### Rationale

The whole-program compilation model is the current design choice. For the project sizes the
build system targets today, a full rebuild is fast enough that the absence of incrementality
is not a practical limitation.

### Path to Incrementality

A future incremental compilation system would require at minimum:

- A content hash or modification-time record for each source file and its transitive
  dependencies.
- A persistent build database mapping source file sets to cached object files.
- A mechanism for detecting which cached `.o` files are still valid and which must be
  regenerated.

None of these mechanisms exist in the current build system. Until they are implemented, all
builds MUST be treated as full rebuilds.

### Possible ccache Acceleration

If `ccache` is installed on the host system and configured to intercept `clang` calls, it may
cache the clang compilation step (IR → object file) transparently. The build system does not
invoke `ccache` directly and makes no guarantees about `ccache` compatibility; this is an
unofficial workaround only.
