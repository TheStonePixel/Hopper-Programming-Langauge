## Cross-Compilation

### Supported Boards

The build system supports one bare-metal target board. The `BOARD_CONFIGS` table in
`seed/build_system/hopper` is the authoritative registry of supported boards. Currently it
contains one entry:

| Board name | Architecture | CPU | Float ABI | Load address | Entry symbol |
|------------|-------------|-----|-----------|-------------|--------------|
| `pi-zero` | ARMv6 | `arm1176jzf-s` | `soft` | `0x8000` | `_start` |

### Declaring a Board Target

To target a board, the project's `hopper.json` MUST contain a `"board"` field naming a key in
`BOARD_CONFIGS`:

```json
{
  "name":  "my-firmware",
  "board": "pi-zero",
  "entry": "src/main.hop",
  "targets": {
    "armv6-bare": {
    }
  }
}
```

When `hopper build` detects a `"board"` field, it switches to the bare-metal pipeline for that
board. The `"host"` target is ignored.

### Bare-Metal Compilation Flags

For the `pi-zero` board, each `.hop` source file is compiled with:

```
clang -target arm-none-eabi -mcpu=arm1176jzf-s -mfloat-abi=soft \
      -ffreestanding -nostdlib -O2 -c -Wno-override-module
```

Note that bare-metal builds always compile at `-O2` regardless of the `--production` flag.
There is no bare-metal equivalent of the `dev` or `test` profiles.

### Boot Stub

Each board has a boot stub assembly file at
`seed/build_system/boards/<board>/boot.s`. For pi-zero, the stub
(`seed/build_system/boards/pi-zero/boot.s`) does the following at startup (in ARM state):

1. Zeros the BSS region from `__bss_start` to `__bss_end`.
2. Sets the stack pointer to `0x8000` (below the kernel load address).
3. Branches with link (`bl main`) to the Hopper entry point.

The stub is placed in the `.text.boot` section so the linker script's `KEEP(*(.text.boot))`
directive ensures it is placed first, at the load address `0x8000`, before all other code.

The boot stub is compiled separately by the build system before linking:

```
clang -target arm-none-eabi -mcpu=arm1176jzf-s -mfloat-abi=soft \
      -ffreestanding -nostdlib -c boot.s -o build/o/boot.o
```

Programs MUST NOT include inline assembly in their source files for board initialization.
All hardware-specific startup code belongs exclusively in the board's boot stub.

### Required Toolchain

Cross-compilation for `pi-zero` requires the following tools to be installed and on `PATH`:

- `clang` — with support for the `arm-none-eabi` target triple
- `ld.lld` — LLVM linker (used instead of the GNU ld on bare-metal)
- `llvm-objcopy` — to convert the linked ELF to a raw binary image

The build system does not check for these tools before attempting to build; a missing tool
causes a `command not found` error from the shell.

### Adding a New Board

To add support for a new board, add an entry to `BOARD_CONFIGS` in
`seed/build_system/hopper`. A valid entry MUST supply:

- `llvmTarget` — the string passed to `loadBindings()` and used as the target key in
  `hopper.json`.
- `clangTarget` — the Clang/LLVM target triple (e.g. `"arm-none-eabi"`).
- `mcpu` — the `-mcpu=` value for clang.
- `floatAbi` — the `-mfloat-abi=` value.
- `textBase` — integer load address (used in the generated linker script).
- `linkerEntry` — the entry symbol name for the linker script `ENTRY()` directive.
- `bootStub` — absolute path to the board's assembly boot stub file.
- `meta` — an object with `arch`, `endian`, `bits`, `loadAddress`, `entrySymbol` strings
  (written to `<name>.meta.json` after each build).
- `flash` — a flash configuration object with at minimum `method`, `outputFile`, and
  `requiredFirmware` fields.

A corresponding `boot.s` file MUST be created at the path given in `bootStub`.

### The `flash` Command

`hopper flash` copies the built `kernel.img` (or the board-configured `outputFile`) to a
mounted SD card.

SD card detection on Linux reads `/proc/mounts` for `vfat` filesystems containing the first
entry in `flash.requiredFirmware` (for pi-zero: `bootcode.bin`). On macOS it scans `/Volumes`.
Common user mount locations (`/media`, `/mnt`, `/run/media`) are also checked as a fallback.

The mount point may be specified explicitly with `--mount <path>` or via the
`flash.mount` field in `hopper.json`. If neither is set, auto-detection is attempted.

Before writing, `hopper flash` verifies that all entries in `flash.requiredFirmware` exist on
the SD card. For pi-zero these are `bootcode.bin` and `start.elf`, which must be obtained from
the official Raspberry Pi firmware repository.

`hopper flash` also ensures that every line in `flash.configLines` is present in `config.txt`
on the SD card. For pi-zero the required lines are:

```
init_uart_clock=3000000
dtparam=act_led_trigger=none
```

After writing, `hopper flash` calls `sync` to flush filesystem buffers.

The `--dry-run` flag causes `hopper flash` to print what it would write without making any
changes.
