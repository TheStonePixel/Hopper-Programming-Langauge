## Dependency Resolution

### Two Import Forms

Hopper has two import forms with different resolution strategies.

#### Interface Import (preferred)

```hopper
import IO from linux
```

The contract name (`IO`) and project name (`linux`) are both present. Resolution proceeds
through the `hopper.json` targets binding:

1. `loadBindings()` walks up the directory tree from the importing source file until it finds
   a `hopper.json`. The walk terminates at the filesystem root if no manifest is found, in
   which case an empty binding map is returned.
2. The `targets[target]` object is read from the manifest. The default target is `"host"`.
3. Each entry in `targets[target]` produces a binding with four fields:
   - `from` — the project name the binding belongs to (e.g. `"linux"`)
   - `contract` — absolute path to the interface file (resolved relative to the manifest dir)
   - `implementation` — absolute path to the implementation file
   - `inline` — boolean; `true` if the binding sets `"inline": true` in `hopper.json`
4. `buildAstFromSource()` looks up the interface name in the binding map. If found, it loads
   the interface file and then the implementation file, merging both into the AST.
5. The `from` field in the binding is validated against the module name in the import
   statement. A mismatch is a hard error.

Both the interface file and the implementation file MUST exist. If either is absent, the build
fails with a descriptive error naming the missing path.

#### Module Import (legacy, internal only)

```hopper
import io from arch
import tty from linux
```

The module name is lowercase. Resolution walks the directory tree looking for
`modules/<name>/src/<module>.hop`:

1. Starting from the importing file's directory (`baseDir`), walk up looking for a
   `modules/<moduleName>/` directory.
2. If a `src/` subdirectory exists inside the module directory, source files are loaded from
   there; otherwise from the module directory itself.
3. If specific names are listed (`import io, tty from linux`), only the corresponding
   `.hop` files are loaded. If no names are given, all `.hop` files in the directory are loaded.
4. If no matching `modules/<name>/` directory is found anywhere in the ancestor chain, the
   build fails with a message directing the user to run `hopper install <name>`.

This form MUST NOT be used in new application code. It exists only for legacy files under
`hopper/nonconform/`.

### The `loadBindings()` Function

`loadBindings(sourceFile, target)` is called once per source file compilation. It:

1. Resolves `sourceFile` to an absolute path.
2. Walks up from the file's directory, checking each directory for a `hopper.json`.
3. Reads and parses the first `hopper.json` found.
4. Extracts `config.targets[target]` (defaulting to an empty object if absent).
5. Resolves `contract` and `implementation` paths relative to the manifest's directory using
   `path.resolve(manifestDir, b.contract)`.
6. Returns a `Map<string, binding>`.

If `hopper.json` is malformed (invalid JSON), that manifest is silently skipped and the walk
continues upward.

### Binding Validation

At import time, `buildAstFromSource()` performs these checks before loading any files:

- The `from` field in the binding MUST match the module name in the import statement.
- The resolved `contract` path MUST exist on disk.
- The resolved `implementation` path MUST exist on disk.

Violations produce a fatal error with the source file and binding name in the message.

### Dependency Installation

Before a dependency can be imported, it MUST be installed via `hopper install <name>`. The
install command copies module source from the global store (`hopper/modules/`) into the
project's `modules/` directory and recursively installs that module's own dependencies into
`modules/<name>/modules/`. The result is a nested directory tree that mirrors the dependency
graph exactly.

After installation, `hopper install` rewrites binding paths in `hopper.json` to point at
locally installed copies (paths of the form `modules/<name>/...`) rather than global store
paths.
