## Target Resolution

### The `targets` Section

Every executable project's `hopper.json` MUST contain a `targets` object. Each key within
`targets` is a target name; each value is a map of interface names to binding objects.

```json
{
  "targets": {
    "host": {
      "IO": {
        "from":           "linux",
        "interface":      "modules/linux/interfaces/IO.hop",
        "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
      }
    }
  }
}
```

### The `host` Target

The default target name is `"host"`. When `hopper build` is invoked without specifying a
target, the build system uses `loadBindings(sourceFile, "host")` to resolve all bindings.

The `host` target SHOULD contain bindings for every interface the program imports. Missing
bindings cause a resolution failure at compile time.

### Binding Fields

Each binding object inside a target MUST supply the following fields:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `from` | string | yes | The project name the binding belongs to (must match the `from` clause of the import statement) |
| `interface` | string | yes | Path to the interface `.hop` file, relative to the `hopper.json` |
| `implementation` | string | yes | Path to the implementation `.hop` file, relative to the `hopper.json` |
| `inline` | boolean | no | When `true`, all functions and class methods from the implementation are marked `alwaysinline` in LLVM IR |

All paths are resolved to absolute paths by `path.resolve(manifestDir, path)` at binding load
time.

### The `inline` Flag

When `binding.inline === true`, `buildAstFromSource()` iterates over all free functions and
class methods in the loaded implementation AST and sets the `_inline` flag on each non-extern
node. The code generator then emits the `alwaysinline` attribute on those LLVM function
definitions.

This is intended for thin translation layers (for example, a syscall wrapper that consists
entirely of inline-assembly stubs) where the overhead of a function call is undesirable and
the implementation adds no logic of its own.

### Bare-Metal Target Names

For bare-metal builds, the target name is the board's `llvmTarget` string defined in
`BOARD_CONFIGS`. For the pi-zero board this is `"armv6-bare"`. When the build system detects a
`"board"` field in `hopper.json`, it calls `loadBindings(sourceFile, cfg.llvmTarget)` instead
of `loadBindings(sourceFile, "host")`.

Bare-metal targets MUST have a corresponding entry in `hopper.json targets` using the
`llvmTarget` string as the key, or the binding map will be empty and all imports will fall
through to legacy module resolution.

### Type Aliases from Bindings

When an implementation file is loaded via a binding, the build system inspects the
implementation AST for a class that satisfies one of these conditions (in priority order):

1. A class whose name exactly matches the binding name.
2. A class that already declares `implements <bindingName>` in source.
3. The first class defined in the file.

If such a class is found and its name differs from the binding name, a type alias is
registered: `bindingName â†’ concreteClassName`. This allows program source to refer to the
interface name as a type (e.g. `IO io = IO()`) and have it resolve to the concrete class.

### Library `exports` Field

Library projects MAY declare an `exports` field in `hopper.json`:

```json
{
  "exports": {
    "MyInterface": {
      "interface":      "interfaces/MyInterface.hop",
      "implementation": "src/MyImpl.hop"
    }
  }
}
```

**Not yet implemented.** The `exports` field is documentation and tooling metadata only. The
build system does not currently read it when compiling a consuming program. Consuming programs
MUST still declare the full binding (with `interface` and `implementation` paths) in their own
`hopper.json targets` section.
