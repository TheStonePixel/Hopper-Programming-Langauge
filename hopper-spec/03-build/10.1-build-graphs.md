## Build Graphs

### Whole-Program Compilation

The Hopper compiler is whole-program: all imported files are merged into a single compilation
unit before LLVM IR is emitted. There is no partial or incremental dependency graph. Every
`hopper build` invocation recompiles the entire program from source.

### The Merge Process

Import resolution is performed by `buildAstFromSource()` in `seed/compiler/src/astBuilder.js`.
The function:

1. Reads the entry source file.
2. Strips all `import` directives from the source text before passing it to the ANTLR parser
   (the parser never sees import lines).
3. Collects the stripped import requests.
4. Recursively calls itself for each imported file, merging the resulting sub-ASTs back into
   the top-level AST before returning.

### Prepend Ordering

Imported declarations are merged by **prepending** (`.unshift`) into the consuming AST's
declaration lists, not appending. This applies to all declaration categories: functions, structs,
classes, enums, aliases, templates, template functions, binds, stricts, and interfaces.

The effect is that dependency declarations appear before the declarations of the file that
imports them in the final merged AST. Programs MUST NOT rely on import declaration order for
semantic correctness; the compiler resolves names by lookup, not by textual position.

The `entry` declaration is never inherited from an imported file. Only the top-level entry
source file may define `entry main`. Imported files that happen to contain an `entry` block
have it silently ignored.

### Visited-Set Deduplication

A `visited` set (a JavaScript `Set` of absolute resolved file paths) is threaded through every
recursive call to `buildAstFromSource()`. Before loading any file, its absolute path is checked
against the set. If the path is already present, that file is skipped entirely.

This ensures that a file reached via multiple import paths (a diamond dependency) is compiled
exactly once. The first import encounter wins: the file's declarations appear in the merged AST
at the position of the first resolution.

### Declare-Line Deduplication

After LLVM IR is generated by `genModule()`, `dedupeDeclarations()` performs a second pass over
the emitted IR text. It scans the IR line by line and removes any duplicate `declare` lines —
LLVM external function declarations — keeping only the first occurrence of each unique line.

This step is necessary because multiple source files may each emit `declare` lines for the same
external function (for example, a libc function or a Linux syscall). Without this pass,
duplicate `declare` lines in the IR would cause clang to emit a warning or error. The
deduplication runs for both hosted and bare-metal builds.

### No Dependency Graph Persistence

The build system does not persist a dependency graph between invocations. There is no lock file,
no build database, and no cached artifact for individual translation units. Every build starts
from scratch.

**Not yet implemented.** True incremental compilation — where only changed files and their
dependents are recompiled — is not implemented. The whole-program model is fast enough for
current project sizes.
