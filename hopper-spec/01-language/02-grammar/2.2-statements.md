## Statements

### Statement Grammar

A statement is a syntactic unit that performs an action but does not itself produce a value (with the exception of `ExprStmt`, which evaluates an expression for its side effects). Statements appear inside blocks.

```antlr
statement
    : 'callback' Identifier '=' Identifier '(' (type (',' type)*)? ')' type  # CallbackDeclTyped
    | 'const' type Identifier '=' expression                     # ConstVarDecl
    | type Identifier '[' IntegerLiteral ']' '=' '[' argList ']'  # ArrayDeclInit
    | type Identifier '[' IntegerLiteral ']'                    # ArrayDecl
    | type Identifier '=' 'allocate' expression constrainClause?  # AllocateVarDecl
    | type Identifier '=' expression constrainClause?           # VarDecl
    | type Identifier constrainClause?                          # VarDeclNoInit
    | Identifier '[' expression ']' '=' expression              # ArrayAssign
    | Identifier '=' 'allocate' expression                      # AllocateAssign
    | Identifier '=' expression                                 # Assign
    | Identifier '.' fieldName '.' fieldName '=' expression     # NestedFieldAssign
    | Identifier '.' fieldName '=' 'allocate' expression        # AllocateFieldAssign
    | Identifier '.' fieldName '=' expression                   # FieldAssign
    | Identifier '::' 'value' '=' expression                    # DerefAssign
    | expression                                                # ExprStmt
    | 'if' '(' expression ')' block elseClause?                 # IfStmt
    | 'while' '(' expression ')' (NEWLINE+ invariantClause)* block  # WhileStmt
    | 'for' '(' forInit? ';' expression? ';' forUpdate? ')' block  # ForStmt
    | 'break'                                                   # BreakStmt
    | 'continue'                                                # ContinueStmt
    | 'return' expression?                                      # ReturnStmt
    | 'defer' expression                                        # DeferStmt
    | 'deallocate' expression                                   # DeallocateStmt
    | AsmBlock                                                  # AsmStmt
    ;
```

Alternatives are listed from most-specific to least-specific. The parser MUST attempt alternatives in the order listed and select the first that matches. A conforming implementation MUST resolve the ambiguity between `VarDecl` and `Assign` (both begin with an `Identifier`) by checking whether the leading token is a type keyword or a user-defined type name.

### Variable Declaration with Initializer (`VarDecl`)

```antlr
type Identifier '=' expression constrainClause?
```

Declares a new local variable named `Identifier` with the declared type `type` and initializes it to the value of `expression`. The variable is allocated on the stack (LLVM `alloca`) in the function entry block.

The declared type and the expression type MUST be compatible. If they are not identical, the compiler MAY apply an implicit widening coercion for address types (an `address:T` expression may initialize an `address` variable). For all other type mismatches, the program MUST use an explicit cast.

Example:
```hopper
int x = 42
string name = "hopper"
address buf = allocate 128
```

If a `constrainClause` is present, the compiler MUST emit a runtime range check in debug mode (non-release builds) verifying that the initialized value falls within the range of the constrain type. In release mode the check is omitted.

**Constructor call sugar.** When the expression is a call whose callee name matches the declared type name (or the base name of a template instantiation), the compiler treats this as a constructor call and emits `alloca` + `call TypeName_constructor` in place of the general assignment sequence.

### Variable Declaration Without Initializer (`VarDeclNoInit`)

```antlr
type Identifier constrainClause?
```

Declares a local variable with no initializer. The variable is allocated on the stack. Its initial value is undefined. A program MUST NOT read from a `VarDeclNoInit` variable before assigning to it; doing so is undefined behavior.

Example:
```hopper
int result
address ptr
```

### Constant Variable Declaration (`ConstVarDecl`)

```antlr
'const' type Identifier '=' expression
```

Declares a local variable that is marked immutable. After initialization, any attempt to assign to this variable MUST produce a compile-time error. The value is stored on the stack like any other variable; it is not inlined as a compile-time constant.

Example:
```hopper
const int limit = 100
const string label = "max"
```

### Array Declaration (`ArrayDecl`)

```antlr
type Identifier '[' IntegerLiteral ']'
```

Declares a local array of `IntegerLiteral` elements of type `type`. The array is allocated on the stack as an LLVM array type `[N x T]`. The size MUST be a positive integer literal known at compile time.

Example:
```hopper
byte buf[256]
int values[16]
```

The variable has internal type `array:T:N`. Elements are accessed with `name[i]` in expression position and `name[i] = expr` in assignment position.

### Array Declaration with Initializer (`ArrayDeclInit`)

```antlr
type Identifier '[' IntegerLiteral ']' '=' '[' argList ']'
```

Declares and initializes a local array. The number of elements in `argList` MUST equal `IntegerLiteral`. Each element expression MUST be compatible with `type`.

Example:
```hopper
int primes[5] = [2, 3, 5, 7, 11]
byte header[4] = [0x89, 0x50, 0x4E, 0x47]
```

### Allocate Variable Declaration (`AllocateVarDecl`)

```antlr
type Identifier '=' 'allocate' expression constrainClause?
```

Declares a variable and initializes it with a heap allocation. The `allocate` keyword calls `malloc` with the size given by `expression` (in bytes). The result is an `address`. The declared type MUST be `address`.

Example:
```hopper
address buf = allocate 4096
address node = allocate 64
```

The size expression MUST evaluate to a non-negative integer. Allocation of zero bytes is implementation-defined. The returned address is `null` if allocation fails; the program is responsible for checking.

### Assignment (`Assign`)

```antlr
Identifier '=' expression
```

Assigns the value of `expression` to the existing variable named `Identifier`. The variable MUST have been previously declared in the enclosing scope or an outer scope. The expression type MUST be compatible with the variable's declared type.

A conforming implementation MUST reject assignment to a variable declared `const`.

For MMIO (`strict`) variables, this emits a `volatile store` through the hardware address rather than a stack store.

For `callback` variables, if `expression` is a bare function name, the compiler MUST emit a bitcast of the function pointer to `i8*` and store it in the callback slot.

### Allocate Assignment (`AllocateAssign`)

```antlr
Identifier '=' 'allocate' expression
```

Assigns a heap allocation to an existing variable. The variable MUST have type `address`.

Example:
```hopper
buf = allocate newSize
```

### Field Assignment (`FieldAssign`)

```antlr
Identifier '.' fieldName '=' expression
```

Assigns `expression` to the named field of the object variable. The variable MUST be a class or struct instance. The field MUST exist in that type.

For ordinary class instances, direct field assignment is only permitted from within the class's own methods (self access) or from template instances. Assigning to a field of a foreign class instance from outside the class MUST produce a compile-time error.

For MMIO bitfield variables, field assignment emits a read-modify-write sequence: load the container integer with `volatile load`, clear the field's bit range using a mask, OR in the new value shifted to position, and `volatile store` the result back.

### Allocate Field Assignment (`AllocateFieldAssign`)

```antlr
Identifier '.' fieldName '=' 'allocate' expression
```

Assigns a heap allocation to a field. The field MUST have type `address`.

### Nested Field Assignment (`NestedFieldAssign`)

```antlr
Identifier '.' fieldName '.' fieldName '=' expression
```

Assigns `expression` to a field of a field. `obj.outer.inner = expr` resolves `obj.outer` to a class instance and then assigns to `inner` within that class. Both levels of field access MUST resolve to class types.

### Dereference Assignment (`DerefAssign`)

```antlr
Identifier '::' 'value' '=' expression
```

Writes `expression` to the memory location pointed to by variable `Identifier`. The variable MUST have type `address` or `address:T`. The expression type MUST be compatible with the pointed-to type.

The compiler determines the pointed-to type from the variable's `address:T` tag if present, or infers it from the expression type if the variable has plain `address` type.

In LLVM IR: load the `i8*`, bitcast to `T*`, then `store T val, T* ptr`.

Example:
```hopper
ptr::value = 42
bytePtr::value = byte 0xFF
```

### Array Element Assignment (`ArrayAssign`)

```antlr
Identifier '[' expression ']' '=' expression
```

Assigns a value to element `i` of an array variable. If the variable's type is a class with a `[]=` operator, the assignment dispatches to that operator method (`TypeName_op_setidx`). Otherwise the variable MUST have array type `array:T:N`, and this emits `getelementptr` + `store`.

Example:
```hopper
buf[i] = 0
values[0] = 100
```

### Expression Statement (`ExprStmt`)

```antlr
expression
```

Evaluates an expression for its side effects and discards the result. The most common use is a function or method call:

```hopper
write(1, buf, n)
obj.flush()
```

Any expression may appear as an `ExprStmt`, but expressions with no side effects in this position are useless.

### If Statement (`IfStmt`)

```antlr
'if' '(' expression ')' block elseClause?
```

Conditional execution. The condition MUST be an expression that produces `bool` or a type coercible to `bool`. If the condition is truthy, the `block` executes; otherwise the `elseClause`, if present, executes.

```antlr
elseClause
    : 'else' block           # ElseBlock
    | 'else' 'if' '(' expression ')' block elseClause?   # ElseIf
    ;
```

`else if` chains are supported by the recursive `elseClause` rule. There is no limit on chain depth.

In LLVM IR: emits `br i1 <cond>, label %if.then, label %if.else` (or `%if.end` if no else), followed by the labeled blocks.

### While Statement (`WhileStmt`)

```antlr
'while' '(' expression ')' (NEWLINE+ invariantClause)* block
```

Executes `block` repeatedly while the condition expression is truthy. The condition is evaluated before each iteration, including the first.

Loop invariants MAY be specified between the condition and the opening brace:

```antlr
invariantClause : 'invariant' expression ;
```

Each `invariant` expression is evaluated at the top of each iteration (at the condition-check point). In debug builds, the compiler MUST emit a runtime assertion that the invariant holds. In release builds, invariant checks are omitted.

In LLVM IR: emits `while.cond:`, `while.body:`, and `while.end:` labeled blocks with appropriate branches.

### For Statement (`ForStmt`)

```antlr
'for' '(' forInit? ';' expression? ';' forUpdate? ')' block
```

A C-style counted loop. All three header clauses are optional.

```antlr
forInit   : type Identifier '=' expression   # ForInitDecl
          | Identifier '=' expression         # ForInitAssign
          ;

forUpdate : Identifier '=' expression ;
```

The `forInit` clause either declares and initializes a new variable (`ForInitDecl`) or assigns to an existing variable (`ForInitAssign`). The condition expression is evaluated before each iteration; if absent, the loop runs indefinitely. The `forUpdate` clause is evaluated after each iteration's body.

**Note:** The update clause is a single assignment only. Compound updates are not supported because Hopper has no increment operator. Use `i = i + 1`.

In LLVM IR: emits `for.cond:`, `for.body:`, `for.update:`, and `for.end:` labeled blocks.

### Break Statement (`BreakStmt`)

```antlr
'break'
```

Exits the innermost enclosing `while` or `for` loop immediately. Transfers control to the loop's end label. A `break` MUST NOT appear outside a loop; doing so is a compile-time error.

### Continue Statement (`ContinueStmt`)

```antlr
'continue'
```

Skips the remainder of the current loop iteration and transfers control to the condition check (for `while`) or to the update clause (for `for`). A `continue` MUST NOT appear outside a loop.

### Return Statement (`ReturnStmt`)

```antlr
'return' expression?
```

Returns from the enclosing function or procedure. If the function has a declared return type, `expression` MUST be present and MUST be compatible with the return type. If the function is a procedure (no return type), `expression` MUST be absent.

Before emitting the `ret` instruction, the compiler MUST evaluate and emit all pending `defer` expressions in reverse order of registration.

In LLVM IR: emits `ret <type> <value>` or `ret void`. If the expression type differs from the return type, the compiler MUST emit an implicit cast equivalent to the `castType` form.

If `ensures` contract clauses are present on the enclosing function, the compiler MUST emit checks of those clauses in debug builds before the `ret`.

### Defer Statement (`DeferStmt`)

```antlr
'defer' expression
```

Registers `expression` for deferred evaluation. The expression is NOT evaluated when the `defer` statement is executed. Instead, it is evaluated just before the enclosing function returns, in reverse order relative to other `defer` statements.

Typical use is cleanup actions:

```hopper
defer deallocate buf
defer obj.close()
```

Multiple `defer` statements are evaluated LIFO (last-in, first-out). The deferred expression is an ordinary expression, typically a function call or method call. A conforming implementation MUST execute all deferred expressions in reverse registration order at every `return` point.

### Deallocate Statement (`DeallocateStmt`)

```antlr
'deallocate' expression
```

Frees heap memory previously obtained with `allocate`. The expression MUST evaluate to type `address`. This emits `call void @free(i8* <ptr>)`.

Calling `deallocate` on an address that was not obtained from `allocate`, or calling it more than once on the same address, is undefined behavior.

Example:
```hopper
deallocate buf
deallocate self.data
```

### Inline Assembly Statement (`AsmStmt`)

```antlr
AsmBlock
```

An `asm { ... }` block is captured as a single lexer token. The block body contains register assignments and instruction names, one per line, in a restricted inline assembly syntax.

Line forms within the block:

- `register = expression` — input: evaluates the expression and places the result in the named register before the assembly executes.
- `variable = register` — output: after the assembly executes, reads the register value into the named variable.
- `instruction` — a bare instruction mnemonic (e.g., `syscall`, `cpuid`).

The compiler emits LLVM inline assembly using Intel dialect (`inteldialect`) with the `sideeffect` flag. All integer inputs are widened to 64-bit before being passed to the assembly constraint. All SIMD registers use their native LLVM vector type.

Example:
```hopper
asm {
    rax = 1
    rdi = fd
    rsi = buf
    rdx = count
    syscall
    result = rax
}
```

A conforming implementation MUST NOT allow `asm {}` blocks to appear inside functions that call non-assembly functions in the same compilation unit, per design goal G8. *This restriction is currently enforced by convention rather than grammar.*

### Callback Declaration (`CallbackDeclTyped`)

```antlr
'callback' Identifier '=' Identifier '(' (type (',' type)*)? ')' type
```

Declares a callback (function pointer) variable with a fully specified signature. The first `Identifier` is the variable name. The second `Identifier` is the initial function to bind. The parenthesized type list gives the parameter types. The trailing `type` gives the return type.

Example:
```hopper
callback handler = onEvent(int, bool) int
```

This creates a variable `handler` of type `callback(int, bool) int`, initialized to the address of `onEvent`. The type signature MUST match the declared type of `onEvent`.

To call a callback variable: `handler(arg1, arg2)` in expression position.

### Constrain Clause

```antlr
constrainClause : 'constrain' '[' type ']' ;
```

An optional annotation on variable declarations that specifies a range constraint. In debug builds, the compiler MUST emit a runtime check that the initialized value is representable in the named type. In release builds, the check is omitted and the annotation serves as documentation.

Example:
```hopper
int x = computeOffset() constrain [byte]
```

This asserts at runtime (in debug) that `x` is in the range [0, 255].
