## Associativity

### General Rules

**All binary operators are left-associative.** When the same binary operator (or two operators of equal precedence) appears in sequence without parentheses, the leftmost operation is performed first.

**All unary operators are right-associative.** When multiple unary operators or cast prefixes are stacked, they bind from right to left (innermost-first).

**Assignment is a statement, not an expression.** There is no associativity to define for assignment because it cannot appear in expression position and therefore cannot be chained.

These rules are encoded directly in the grammar. Left-associativity for binary operators is expressed by the repetition form `rule ( op rule )*`, which builds a left-skewed parse tree. Right-associativity for unary operators is expressed by the recursive form `unary : op unary | primary`, which builds a right-skewed parse tree.

### Left-Associative Binary Operators

All operators in the following grammar rules are left-associative:

| Rule | Operators | Example | Groups As |
|------|-----------|---------|-----------|
| `logicalOr` | `\|\|` | `a \|\| b \|\| c` | `(a \|\| b) \|\| c` |
| `logicalAnd` | `&&` | `a && b && c` | `(a && b) && c` |
| `bitwiseOr` | `\|` | `a \| b \| c` | `(a \| b) \| c` |
| `bitwiseXor` | `^` | `a ^ b ^ c` | `(a ^ b) ^ c` |
| `bitwiseAnd` | `&` | `a & b & c` | `(a & b) & c` |
| `equality` | `==` `!=` | `a == b != c` | `(a == b) != c` |
| `relational` | `<` `<=` `>` `>=` | `a < b < c` | `(a < b) < c` |
| `shift` | `<<` `>>` | `a << b << c` | `(a << b) << c` |
| `additive` | `+` `-` | `a - b - c` | `(a - b) - c` |
| `multiplicative` | `*` `/` `%` | `a / b / c` | `(a / b) / c` |

**Important note on chained comparisons:** Because relational and equality operators are left-associative, `a < b < c` does not test whether `b` is between `a` and `c`. It tests whether the boolean result of `a < b` (0 or 1) is less than `c`. Programs MUST NOT write chained comparisons with the intent of range testing. Use `a < b && b < c`.

### Right-Associative Unary Operators

The unary rule is right-recursive:

```antlr
unary : ('!' | '-' | '~') unary | castType unary | primary ;
```

Stacked unary operators bind innermost-first:

| Example | Groups As | Notes |
|---------|-----------|-------|
| `--x` | `-(- x)` | No decrement operator exists |
| `~~x` | `~(~x)` | Double bitwise NOT = identity |
| `!!x` | `!(! x)` | Double logical NOT = identity |
| `int -x` | `int(- x)` | Cast applied after negation |
| `int byte x` | `int(byte x)` | Inner cast first, then outer |
| `-!x` | `-(! x)` | Negate the bool result (coerced to int) |

### No Operator Overloading for Precedence

A class that defines an operator overload (§2.3) does NOT change the precedence or associativity of that operator. The overloaded operator `+` on a class type still binds at the `additive` level and is left-associative. The class method is called at the point where the primitive instruction would otherwise be emitted; the grammar structure is unchanged.

### Assignment Is a Statement

Assignment (`Identifier '=' expression`) is a **statement form**, not an expression. A conforming implementation MUST NOT permit assignment to appear in expression position. Specifically:

- `a = b = c` is not valid Hopper — there is no chained assignment.
- `if (a = b)` is not valid Hopper — assignment does not produce a value that can be used as a condition.
- `x + (a = b)` is not valid Hopper — assignment cannot appear inside an expression.

This design eliminates a common class of bugs in C-family languages where `=` and `==` are confused.

The only form of initialization that resembles an expression is the `VarDecl` statement `type name = expr`, which is a declaration statement, not an assignment expression.

### Evaluation Order

The evaluation order of subexpressions within a binary operation is **left-to-right** in the reference implementation. That is, in `f() + g()`, `f()` is evaluated before `g()`.

A conforming implementation MAY evaluate subexpressions in any order, as Hopper does not specify evaluation order for binary operands. Programs MUST NOT depend on evaluation order of operands for correctness.

Exception: the `defer` statement guarantees LIFO evaluation order for deferred expressions at function return, and this ordering IS normative.
