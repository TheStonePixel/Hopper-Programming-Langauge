## Expressions

### Expression Grammar

An expression is a syntactic form that produces a value. The expression grammar is a strict hierarchy of precedence levels. Each level is a separate grammar rule that delegates to the next-higher-precedence rule for its operands.

```antlr
expression     : logicalOr ;
logicalOr      : logicalAnd ( '||' logicalAnd )* ;
logicalAnd     : bitwiseOr  ( '&&' bitwiseOr  )* ;
bitwiseOr      : bitwiseXor ( '|'  bitwiseXor )* ;
bitwiseXor     : bitwiseAnd ( '^'  bitwiseAnd )* ;
bitwiseAnd     : equality   ( '&'  equality   )* ;
equality       : relational ( ('==' | '!=') relational )* ;
relational     : shift ( ('<' | '<=' | '>' | '>=') shift )* ;
shift          : additive ( ('<<' | '>>') additive )* ;
additive       : multiplicative ( ('+' | '-') multiplicative )* ;
multiplicative : unary ( ('*' | '/' | '%') unary )* ;
unary          : ('!' | '-' | '~') unary | castType unary | primary ;
```

The complete precedence ordering is specified in §2.6 (Precedence). Associativity rules are specified in §2.7 (Associativity).

### Logical OR (`||`)

```antlr
logicalOr : logicalAnd ( '||' logicalAnd )* ;
```

The `||` operator evaluates to `true` if either operand is truthy. Both operands MUST be coercible to `bool`. The result type is `bool`.

**Important:** Hopper's `||` does NOT perform short-circuit evaluation in this version of the specification. Both operands are evaluated before the logical OR is applied. A conforming implementation MAY implement short-circuit evaluation as an optimization, but programs MUST NOT depend on short-circuit behavior for correctness.

*Future: short-circuit semantics are under consideration.*

### Logical AND (`&&`)

```antlr
logicalAnd : bitwiseOr ( '&&' bitwiseOr )* ;
```

The `&&` operator evaluates to `true` if both operands are truthy. Both operands MUST be coercible to `bool`. The result type is `bool`.

The same caveat about short-circuit evaluation applies as for `||`.

### Bitwise OR (`|`)

```antlr
bitwiseOr : bitwiseXor ( '|' bitwiseXor )* ;
```

Bitwise OR of two integer values. Both operands MUST have the same integer type (`int`, `byte`, `char`, `unsigned int`, `unsigned byte`). The result type is the type of the left operand.

In LLVM IR, this emits `or <type> <lhs>, <rhs>`.

### Bitwise XOR (`^`)

```antlr
bitwiseXor : bitwiseAnd ( '^' bitwiseAnd )* ;
```

Bitwise exclusive OR of two integer values. Same type requirements as bitwise OR. Result type is the type of the left operand.

In LLVM IR, this emits `xor <type> <lhs>, <rhs>`.

### Bitwise AND (`&`)

```antlr
bitwiseAnd : equality ( '&' equality )* ;
```

Bitwise AND of two integer values. Same type requirements as bitwise OR. Result type is the type of the left operand.

In LLVM IR, this emits `and <type> <lhs>, <rhs>`.

### Equality (`==`, `!=`)

```antlr
equality : relational ( ('==' | '!=') relational )* ;
```

Equality comparison. Both operands MUST have the same type, or one MUST be coercible to the type of the other. The result type is `bool`.

For integer types, LLVM IR emits `icmp eq` or `icmp ne`. For `float`, emits `fcmp oeq` or `fcmp one`. The `fcmp one` predicate returns `false` if either operand is NaN; this is ordered comparison behavior.

### Relational (`<`, `<=`, `>`, `>=`)

```antlr
relational : shift ( ('<' | '<=' | '>' | '>=') shift )* ;
```

Ordered comparison. Result type is `bool`.

For signed integer types (`int`, `byte`, `char`), LLVM IR emits `icmp slt`, `icmp sle`, `icmp sgt`, `icmp sge`. For unsigned integer types (`unsigned int`, `unsigned byte`), emits `icmp ult`, `icmp ule`, `icmp ugt`, `icmp uge`. For `float`, emits `fcmp olt`, `fcmp ole`, `fcmp ogt`, `fcmp oge`.

**Note on grammar ambiguity:** The `<` and `>` tokens also appear in template function calls and template type instantiations. The grammar resolves this by the position of the `<` relative to an identifier: `Identifier '<' type '>'` in primary position is a template function call; `'<'` in relational position is a comparison. The parser MUST handle this disambiguation correctly.

### Shift (`<<`, `>>`)

```antlr
shift : additive ( ('<<' | '>>') additive )* ;
```

Bitwise shift. The left operand MUST be an integer type. The right operand MUST be an integer type. Result type is the type of the left operand.

Left shift emits `shl`. Right shift emits `lshr` (logical shift right, zero-filling) for all integer types in this version of the specification. Arithmetic right shift (`ashr`) is not exposed through the `>>` operator.

### Additive (`+`, `-`)

```antlr
additive : multiplicative ( ('+' | '-') multiplicative )* ;
```

Addition and subtraction. For numeric types, emits `add`/`sub` (integers) or `fadd`/`fsub` (floats).

**Address arithmetic.** When the left operand is `address` or a typed `address:T`, addition and subtraction with an `int` right operand are permitted and produce address arithmetic:

- `address + int` → `getelementptr i8, i8* <ptr>, i64 <offset>`, result type `address`
- `address - int` → negates the offset, then `getelementptr i8, i8* <ptr>, i64 <negoffset>`, result type `address`
- `address:T + int` → scales the offset by `sizeof(T)` via a typed GEP, result type `address:T`
- `address:T - int` → same with negated offset, result type `address:T`

This means `ptr + 1` on a plain `address` advances by 1 byte, while `ptr + 1` on an `address:int` advances by 8 bytes (the size of `int`). A conforming implementation MUST implement this distinction.

A program MUST NOT add two addresses together. Address-address subtraction is not defined in this version of the specification.

### Multiplicative (`*`, `/`, `%`)

```antlr
multiplicative : unary ( ('*' | '/' | '%') unary )* ;
```

Multiplication, division, and remainder. Both operands MUST have the same numeric type.

- `*` → `mul` (integers), `fmul` (floats)
- `/` → `sdiv` (signed integers), `udiv` (unsigned integers), `fdiv` (floats)
- `%` → `srem` (signed integers), `urem` (unsigned integers), `frem` (floats)

Division by zero is undefined behavior. A conforming implementation is not required to detect it at compile time or runtime.

### Unary Operators

```antlr
unary : ('!' | '-' | '~') unary | castType unary | primary ;
```

Unary operators are right-recursive: `--x` parses as `-(- x)`, not as a decrement. Hopper has no decrement operator.

**Logical NOT (`!`):** Operand MUST be `bool`. Emits `xor i1 <val>, 1`. Result type is `bool`.

**Numeric negation (`-`):** Operand MUST be a numeric type. For integers, emits `sub <type> 0, <val>`. For floats, emits `fneg double <val>`. Result type is the operand type.

**Bitwise NOT (`~`):** Operand MUST be an integer type. Emits `xor <type> <val>, -1`. Result type is the operand type.

**Primitive cast (`castType unary`):** Applies a compile-time primitive type conversion to the operand. See §2.5 (Operators) for the full cast table. The `castType` form is an internal grammar production; programs SHOULD use the `cast<T>(expr)` template function form instead, which is defined in the `cast` module.

```antlr
castType
    : 'int'
    | 'byte'
    | 'char'
    | 'float'
    | 'bool'
    | 'bit'
    | 'address'
    | 'unsigned' 'int'
    | 'unsigned' 'byte'
    ;
```

The `castType` prefix is evaluated as a conversion from the operand's type to the named type. The conversion rules are identical to those used by `cast<T>(expr)`. See §4 (Types) for the complete conversion table.

**Not yet implemented.** Although `bit` appears in the `castType` grammar rule, `emitCast()` in `codegenExpr.js` has no handler for `bit` as a cast target or source type. Using `bit` as a `castType` prefix (e.g., `bit someExpr`) will throw a compile-time error (`Cannot cast from '...' to 'bit'`). Cast to and from `bit` is only meaningful inside bitfield read/write paths, which use dedicated shift-and-mask codegen rather than the generic `emitCast` function.

### Primary Expressions

```antlr
primary
    : IntegerLiteral
    | HexLiteral
    | CharLiteral
    | UnicodeLiteral
    | FloatLiteral
    | StringLiteral
    | 'true'
    | 'false'
    | 'null'
    | Identifier '[' expression ']' '::' 'address'
    | Identifier '::' 'address'
    | Identifier '::' 'value'
    | Identifier '::' 'size'
    | Identifier '<' type '>' '(' argList? ')'
    | Identifier '(' argList? ')'
    | Identifier '.' fieldName '.' fieldName '(' argList? ')'
    | Identifier '.' fieldName '(' argList? ')'
    | 'String' '(' argList? ')'
    | Identifier '.' fieldName '[' expression ']'
    | Identifier '[' expression ']'
    | Identifier '.' fieldName
    | Identifier
    | 'value'
    | 'address'
    | 'size'
    | '(' expression ')'
    ;
```

The primary rule is matched top-to-bottom. The parser attempts alternatives in order and selects the first that matches. Where multiple alternatives share a common prefix (e.g., `Identifier '(' ...` vs. `Identifier '.' ...`), the parser uses lookahead to distinguish them.

#### Literals

Integer literals (`IntegerLiteral`) produce type `int`. Hex literals (`HexLiteral`) produce type `int`. Character literals (`CharLiteral`) produce type `int` (the numeric value of the character). Unicode literals (`UnicodeLiteral`) produce type `int`. Float literals (`FloatLiteral`) produce type `float`. String literals (`StringLiteral`) produce type `string`.

The `true` and `false` keywords produce type `bool`. The `null` keyword produces type `address` with value zero.

#### Template Function Call

```antlr
Identifier '<' type '>' '(' argList? ')'
```

This is a call to a `template function` with an explicit type argument. The canonical use is the `cast` module:

```hopper
int n = cast<int>(3.7)
```

The compiler resolves the call by looking up all `template function` specializations registered under the given name, finding the one whose type parameter matches `type` and whose parameter types match the argument types, and emitting a call to the monomorphized function. If no matching specialization exists, the compiler MUST produce an error.

#### Function Call

```antlr
Identifier '(' argList? ')'
```

A call to a named free function or a constructor. If the identifier names a class, this is a constructor call that allocates storage and invokes `ClassName_constructor`. If the identifier names a known `callback` variable, this is an indirect call through the stored function pointer. Otherwise, the identifier MUST name a previously declared free function.

The argument count MUST match the declared parameter count, unless the function is declared `extern` with a variadic `...` parameter list, in which case additional arguments are permitted.

#### Method Call

```antlr
Identifier '.' fieldName '(' argList? ')'
```

A call to a method on a class or struct instance. The identifier MUST name a variable whose type is a class or struct. The `fieldName` MUST be a method declared in that class or struct. The argument count MUST match the method's declared parameter count.

Method calls are compiled as calls to the mangled name `TypeName_methodName`, with the receiver object passed as the first (implicit) argument. Programs MUST use explicit receiver syntax — `obj.method()` — for all method calls. There is no implicit `self` dispatch.

#### Chained Method Call

```antlr
Identifier '.' fieldName '.' fieldName '(' argList? ')'
```

A method call on a field of an object. `obj.field.method(args)` first obtains a pointer to `obj.field`, then calls `FieldType_method` with that pointer as the implicit self argument.

This form is limited to exactly two levels of field access before the method. Deeper chains are not supported in this version of the grammar.

#### Field Index Access

```antlr
Identifier '.' fieldName '[' expression ']'
```

Subscript access on a field. `obj.field[i]` obtains a pointer to `obj.field` and calls the `[]` operator defined on the field's type.

#### Array Access

```antlr
Identifier '[' expression ']'
```

Accesses element `i` of array variable `name`. If the variable has type `array:T:N`, this emits `getelementptr` to compute the element address and `load` to read the value. If the variable has type `string[]` (argv), this loads a pointer from the `i8**`. If the variable's type is a class with a `[]` operator defined, the call is dispatched to that operator.

#### Address Operators

```antlr
Identifier '::' 'address'
Identifier '[' expression ']' '::' 'address'
Identifier '::' 'value'
Identifier '::' 'size'
```

- `name::address` — produces the address of variable `name` as type `address`. If `name` is a function, produces the function pointer cast to `address`. If `name` is an array, produces the address of element 0.
- `name[i]::address` — produces the address of element `i` of array `name`.
- `name::value` — dereferences `name`, which MUST have type `address` or `address:T`. Loads the value at the pointed-to address.
- `name::size` — produces the byte size of the variable `name` or, if `name` is a type name, the byte size of that type. Result type is `int`.

The `::` operator binds tighter than any binary operator and is parsed as part of the primary rule.

#### Grouped Expression

```antlr
'(' expression ')'
```

Parenthesized expressions are evaluated normally. Parentheses may be used to override precedence by grouping subexpressions. They have no other semantic effect.

### Argument Lists

```antlr
argList : expression (',' expression)* ;
```

An argument list is one or more expressions separated by commas. The empty argument list (zero arguments) is represented by the absence of `argList` in the grammar (`argList?`).

### Class Operator Dispatch

When a binary expression's left operand is a class instance and the class defines an operator overload for the operator in use, the compiler MUST dispatch to the operator method instead of generating a primitive instruction. The operator method is called as `TypeName_op_<safe>`, where `<safe>` is the operator symbol encoded as a safe identifier string.

This dispatch takes priority over primitive code generation. If the left operand's class defines `operator +`, then `a + b` calls the operator method; if it does not, `a + b` generates primitive integer or float addition.

Operator dispatch is only supported on binary operators listed in the `operatorSymbol` rule. See §2.3 (Declarations) for the class operator declaration form.
