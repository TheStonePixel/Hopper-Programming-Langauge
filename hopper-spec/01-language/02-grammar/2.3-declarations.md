## Declarations

### Overview

Top-level declarations define the entities available in a module: functions, classes, interfaces, templates, enums, imports, and hardware directives. All declarations appear at the top level of a source file; no declaration form is valid inside a block.

```antlr
topLevelDecl
    : functionDecl
    | templateFunctionDecl
    | structDecl
    | classDecl
    | templateDecl
    | enumDecl
    | importDecl
    | aliasDecl
    | entryDecl
    | bindDecl
    | mmioDecl
    | bitfieldDecl
    | interfaceDecl
    ;
```

### Function Declaration (`functionDecl`)

```antlr
functionDecl
    : 'extern' 'function' Identifier '(' externParamList? ')' type   # ExternFuncDecl
    | 'extern' 'function' Identifier '(' externParamList? ')'        # ExternProcDecl
    | 'function' Identifier '(' paramList? ')' type (NEWLINE+ contractClause)* block   # FuncDecl
    | 'function' Identifier '(' paramList? ')' (NEWLINE+ contractClause)* block        # ProcDecl
    ;
```

A **function** has a return type. A **procedure** has no return type (implicitly `void`). The `extern` variants declare C-compatible functions with no body; the compiler MUST emit these as `declare` in LLVM IR with no definition.

```antlr
paramList     : param (',' param)* ;
externParamList : param (',' param)* (',' '...')? | '...' ;
param         : type paramName ;
paramName     : Identifier | 'value' | 'address' | 'size' ;
```

Variadic C functions MUST be declared with `extern function` and the `...` tail. Hopper functions MUST NOT be variadic.

**Contract clauses** appear between the parameter list (or return type) and the opening brace, one per line:

```antlr
contractClause
    : 'requires' expression   # RequiresClause
    | 'ensures'  expression   # EnsuresClause
    ;
```

`requires` clauses specify preconditions checked at function entry (in debug builds). `ensures` clauses specify postconditions checked at every `return` point (in debug builds). Both are omitted in release builds. A conforming implementation MUST evaluate contract expressions after all arguments are bound but before any function body statement executes (`requires`), and immediately before `ret` is emitted (`ensures`).

Example:
```hopper
function clamp(int x, int lo, int hi) int
requires lo <= hi
ensures result >= lo
{
    if (x < lo) {
        return lo
    }
    if (x > hi) {
        return hi
    }
    return x
}
```

### Template Function Declaration (`templateFunctionDecl`)

```antlr
templateFunctionDecl
    : 'template' 'function' Identifier '<' type '>' '(' paramList? ')' type (NEWLINE+ contractClause)* block   # TemplateFuncDecl
    | 'template' 'function' Identifier '<' type '>' '(' paramList? ')' (NEWLINE+ contractClause)* block        # TemplateProcDecl
    ;
```

A template function is parameterized by a single compile-time type argument. Multiple specializations may exist for the same name, each with a different type argument. Calls are resolved by matching the type argument and parameter types to the available specializations.

The canonical use is the `cast` module, which provides specializations for all supported type conversions:

```hopper
template function cast<int>(float src) int {
    return int src
}

template function cast<float>(int src) float {
    return float src
}
```

Each specialization is compiled to a distinct monomorphized function. The mangled name is implementation-defined but MUST be unique per (name, typeArg, paramTypes) triple.

A conforming implementation MUST produce a compile-time error if a call to a template function finds no matching specialization.

### Struct Declaration (`structDecl`)

```antlr
structDecl
    : 'struct' Identifier '{' NEWLINE* (structMember (NEWLINE+ structMember)* NEWLINE*)? '}'
    ;

structMember
    : type fieldName   # StructField
    | 'pad' IntegerLiteral   # StructPad
    ;
```

A struct is a flat memory layout: a sequence of typed fields with no methods and no constructor. The compiler lays out fields in declaration order with no padding between them unless `pad` directives are present.

`pad N` inserts `N` bytes of unnamed padding at that position in the layout. This is used to match hardware register layouts or protocol headers.

Structs MUST NOT have constructors, destructors, or methods. They are pure data aggregates. Field access uses `obj.field` syntax.

Example:
```hopper
struct Point {
    int x
    int y
    pad 0
}
```

> **Correction.** An earlier version of this example showed `byte dst[6]` and similar array-in-struct syntax. The `structMember` grammar rule is `type fieldName` with no `[N]` array size — array-sized fields are NOT valid struct member syntax. Fields with bracketed sizes belong to `bitfieldDecl` (the `BitfieldArrayField` rule), not to `structDecl`.

### Bitfield Declaration (`bitfieldDecl`)

```antlr
bitfieldDecl
    : 'bitfield' Identifier '{' NEWLINE* (bitfieldMember (NEWLINE+ bitfieldMember)* NEWLINE*)? '}'
    ;

bitfieldMember
    : type fieldName '[' IntegerLiteral ']'   # BitfieldArrayField
    | type fieldName                           # BitfieldField
    | 'pad' IntegerLiteral                     # BitfieldPad
    ;
```

A bitfield is a bit-level hardware register layout. Fields are packed sequentially from the least significant bit (LSB). Each field specifies the number of bits it occupies via `[N]` (for array fields) or implicitly 1 bit (for single-bit `bit` fields).

The container type for all bitfield operations is determined from the total bit count. Access to bitfield fields emits shift-and-mask operations on the container integer (for reads) and read-modify-write sequences (for writes).

`pad N` inserts `N` padding bits.

Example:
```hopper
bitfield StatusReg {
    bit ready
    bit error
    pad 2
    byte mode[4]
}
```

Bitfields are used with `@` declarations for MMIO register access. See `mmioDecl` below and §9.2.

### Class Declaration (`classDecl`)

```antlr
classDecl
    : 'class' className implementsList? '{' NEWLINE* (classMember (NEWLINE+ classMember)* NEWLINE*)? '}'
    ;

className : Identifier | 'String' ;

implementsList : 'satisfies' Identifier (',' Identifier)* ;
```

A class combines data fields with methods. Classes may optionally implement one or more interfaces. When `satisfies` is present, the compiler MUST verify at compile time that all method signatures declared in each named contract are present in the class body with matching signatures.

```antlr
classMember
    : type fieldName                                              # ClassField
    | 'function' fieldName '(' paramList? ')' type block         # ClassMethod
    | 'function' fieldName '(' paramList? ')' block              # ClassProcMethod
    | 'operator' operatorSymbol '(' Identifier (',' param)* ')' type block   # ClassOperator
    | 'constructor' '(' paramList? ')' block                     # ClassConstructor
    | 'destructor' '(' ')' block                                 # ClassDestructor
    ;
```

**Fields** are declared as `type fieldName`. Fields have no default value; the constructor MUST assign every field before it returns.

**Methods** are compiled as free functions with the mangled name `ClassName_methodName`. The implicit first parameter is a pointer to the class instance, accessible inside the method body as `self`.

**Constructors** are called automatically when a class is instantiated with `ClassName(args)`. The constructor receives a pointer to the freshly allocated (or stack-allocated) instance as `self`. Constructors have no return value.

**Destructors** are not called automatically. The programmer MUST call them explicitly: `obj.destructor()`. This is deliberate — in systems code, object lifetime is explicit.

**Operator overloads** define behavior for binary and subscript operators. The `operatorSymbol` rule enumerates the overloadable operators:

```antlr
operatorSymbol
    : '+' | '-' | '*' | '/' | '%'
    | '==' | '!=' | '<' | '<=' | '>' | '>='
    | '&' | '|' | '^' | '<<' | '>>'
    | '[' ']' '='
    | '[' ']'
    ;
```

`[]` is the subscript read operator; `[]=` is the subscript write operator. Their mangled names are `TypeName_op_idx` and `TypeName_op_setidx` respectively.

**Field access restriction.** Direct access to a class field from code outside the class MUST produce a compile-time error. Template instances are exempt from this restriction. Only accessor and mutator methods should be used to read or write class fields from external code.

Example:
```hopper
class Buffer {
    address data
    int size
    int capacity

    constructor(int cap) {
        self.data = allocate cap
        self.size = 0
        self.capacity = cap
    }

    destructor() {
        deallocate self.data
    }

    function length() int {
        return self.size
    }
}
```

### Template Declaration (`templateDecl`)

```antlr
templateDecl
    : 'template' templateName '<' templateParam (',' templateParam)* '>' '{' NEWLINE* (classMember (NEWLINE+ classMember)* NEWLINE*)? '}'
    ;

templateName  : Identifier | 'String' ;

templateParam
    : Identifier        # FreeParam
    | 'int'             # FixedParam
    | 'byte'            # FixedParam
    | 'char'            # FixedParam
    | 'float'           # FixedParam
    | 'bool'            # FixedParam
    | 'string'          # FixedParam
    | 'address'         # FixedParam
    | 'unsigned' 'int'  # FixedParam
    | 'unsigned' 'byte' # FixedParam
    ;
```

A template is a parameterized class definition. It has the same member forms as a class. Template parameters are either **free type variables** (identifiers like `T`, `K`, `V`) or **fixed primitive types** (concrete keywords like `int`, `byte`).

Free-parameter templates are instantiated with explicit type arguments at use sites: `List<int>`, `Map<String<byte>, int>`. Fixed-parameter templates are monomorphized at declaration time; no `<>` is required at use sites — the monomorphized name becomes a standalone type.

A conforming implementation MUST monomorphize each distinct instantiation to a distinct set of type definitions and method function definitions.

### Enum Declaration (`enumDecl`)

```antlr
enumDecl
    : 'enum' Identifier '{' NEWLINE* (enumVariant (NEWLINE+ enumVariant)* NEWLINE*)? '}'
    ;

enumVariant
    : Identifier ('=' ('-'? (IntegerLiteral | HexLiteral) | StringLiteral))?
    ;
```

An enum declares a set of named constants. Variants are integer-valued by default. If the first variant has no explicit value, it takes value 0. Subsequent variants without explicit values increment from the preceding variant's value.

String-valued variants are also supported. A variant with `= "string"` has a string value rather than an integer value.

Enum variants are accessed as `EnumName.VariantName`. The type of the access expression is `int` for integer variants and `string` for string variants.

Enums are the preferred replacement for top-level `const` declarations.

Example:
```hopper
enum FileMode {
    RDONLY = 0
    WRONLY = 1
    RDWR   = 2
    CREAT  = 64
}

enum Tag {
    NULL_TAG    = "null"
    BOOL_TAG    = "bool"
    INT_TAG     = "int"
}
```

Enums may also appear as members of interfaces:

```antlr
interfaceMember
    : 'function' fieldName '(' paramList? ')' type   # InterfaceFunc
    | 'function' fieldName '(' paramList? ')'        # InterfaceProc
    | enumDecl                                        # InterfaceEnum
    ;
```

### Interface Declaration (`interfaceDecl`)

```antlr
interfaceDecl
    : 'contract' Identifier '{' NEWLINE* (interfaceMember (NEWLINE+ interfaceMember)* NEWLINE*)? '}'
    ;
```

An contract is a compile-time contract: a named set of function signatures. A class that declares `satisfies InterfaceName` MUST provide every function and procedure listed in the contract with identical parameter types and return type.

Interfaces contain only function/procedure signatures and enum declarations. They MUST NOT contain field declarations, constructor declarations, or destructor declarations.

Interface conformance is checked at compile time. A conforming implementation MUST produce an error if a class claims to implement an contract but is missing any required method or has a method with a mismatched signature.

Example:
```hopper
contract IO {
    function read(int fd, address buf, int count) int
    function write(int fd, address buf, int count) int
    function pread64(int fd, address buf, int count, int offset) int
}
```

### Import Declaration (`importDecl`)

```antlr
importDecl
    : 'import' Identifier (',' Identifier)* 'from' Identifier   # ImportFrom
    | 'import' Identifier                                         # ImportModule
    ;
```

**Interface import** (`ImportFrom`): imports one or more named interfaces from a module. The contract names MUST appear in the consuming project's `hopper.json` targets section as bindings. The binding maps the contract name to a concrete contract file and implementation file. Free functions from the implementation file come into scope and may be called directly.

```hopper
import IO from linux
import FileSystem from linux
import SIMD from x86_64
```

**Module import** (`ImportModule`): imports a module by name using file-path resolution. This form is used only by legacy non-conforming modules. New code MUST use the `ImportFrom` form.

```hopper
import io
```

A conforming implementation MUST resolve contract imports through the project's `hopper.json` targets binding, not by file-path heuristics.

### Alias Declaration (`aliasDecl`)

```antlr
aliasDecl : 'alias' Identifier '=' type ;
```

Defines a type alias. The new name becomes interchangeable with the original type throughout the module.

Example:
```hopper
alias Size = int
alias Buffer = address
```

### Entry Declaration (`entryDecl`)

```antlr
entryDecl
    : 'entry' Identifier '(' paramList ')' block   # EntryBlockParams
    | 'entry' Identifier block                      # EntryBlock
    | 'entry' Identifier '=' expression             # EntryAddr
    ;
```

Declares the program entry point. A program MUST have exactly one `entry` declaration. Libraries MUST NOT have an `entry` declaration.

The `EntryBlockParams` form accepts arguments. For a standard `main`-style entry point:

```hopper
entry main(int argc, string[] argv) {
    string path = argv[1]
    ...
}
```

The `EntryBlock` form takes no arguments:

```hopper
entry main {
    ...
}
```

The `EntryAddr` form binds the entry point to the address of an existing function, useful in bare-metal contexts:

```hopper
entry reset = startup::address
```

The entry identifier is compiled to the `main` symbol (or the bare-metal reset vector) depending on the target. The compiler MUST emit this symbol at the address specified by the linker script or ABI for the target.

### Bind Declaration (`bindDecl`)

```antlr
bindDecl : 'bind' HexLiteral '=' Identifier '::' 'address' ;
```

A hardware linker directive. Places the function pointer of `Identifier` at the absolute address `HexLiteral` in the output image. Used for bare-metal interrupt vector tables.

Example:
```hopper
bind 0x00000004 = reset::address
bind 0x00000008 = nmi::address
```

A conforming implementation MUST emit a linker directive (e.g., a linker script section) that places the function's address at the specified location. The address MUST be a hex literal.

### MMIO Placement Declaration (`mmioDecl`)

```antlr
mmioDecl : type Identifier '@' HexLiteral ;
```

Places a named variable at a fixed hardware address. The `@` sigil binds the identifier to the physical address. Every read emits a `volatile load`; every write emits a `volatile store`. No stack allocation is used.

Example:
```hopper
int     uart_dr @ 0x09000000
int     uart_fr @ 0x09000018
```

If `type` names a `bitfield` type, field-level reads emit a volatile load + shift + mask; field-level writes emit a volatile load + mask + OR + volatile store (read-modify-write).

```hopper
bitfield PL011FR {
    bit cts
    bit dsr
    bit dcd
    bit busy
    bit rxfe    // bit 4: receive FIFO empty
    bit txff    // bit 5: transmit FIFO full
    bit rxff
    bit txfe
    pad 24
}

PL011FR uart_fr @ 0x09000018

// Reads uart_fr, extracts bit 5, compares to 1 — all volatile
while (uart_fr.txff == 1) {}
```

A conforming implementation MUST mark all loads and stores to MMIO variables as `volatile` in the LLVM IR. See §9.1 for full semantics and §9.2 for bitfield register access.
