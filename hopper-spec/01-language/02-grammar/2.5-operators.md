## Operators

### Overview

Hopper operators are defined by grammar production rules. Each operator belongs to a specific precedence level in the expression hierarchy. The precedence levels from lowest to highest are listed in §2.6. Associativity rules are in §2.7.

Operators in Hopper are NOT overloadable by default. Class operator overloads are a separate mechanism defined in class declarations (§2.3). The grammar-defined operators below apply to primitive types.

---

### Arithmetic Operators

| Operator | Grammar Production | LLVM Instruction | Operand Types | Result Type |
|----------|--------------------|------------------|---------------|-------------|
| `+` | `additive` | `add` / `fadd` / `getelementptr` | numeric, or `address + int` | same as left |
| `-` | `additive` | `sub` / `fsub` / `getelementptr` | numeric, or `address - int` | same as left |
| `*` | `multiplicative` | `mul` / `fmul` | numeric | same as left |
| `/` | `multiplicative` | `sdiv` / `udiv` / `fdiv` | numeric | same as left |
| `%` | `multiplicative` | `srem` / `urem` / `frem` | numeric | same as left |

For integer types, signed instructions (`sdiv`, `srem`) are used for `int`, `byte`, and `char`. Unsigned instructions (`udiv`, `urem`) are used for `unsigned int` and `unsigned byte`.

For `address + int` and `address - int`, the compiler emits `getelementptr i8, i8* <ptr>, i64 <offset>` (byte-granularity). For `address:T + int`, the compiler emits a typed GEP that scales by `sizeof(T)`.

Division by zero is undefined behavior.

---

### Bitwise Operators

| Operator | Grammar Production | LLVM Instruction | Operand Types | Result Type |
|----------|--------------------|------------------|---------------|-------------|
| `&` | `bitwiseAnd` | `and` | integer | same as left |
| `\|` | `bitwiseOr` | `or` | integer | same as left |
| `^` | `bitwiseXor` | `xor` | integer | same as left |
| `<<` | `shift` | `shl` | integer | same as left |
| `>>` | `shift` | `lshr` | integer | same as left |

All bitwise operators require integer operands. Float operands MUST NOT be used with bitwise operators; doing so is a compile-time error.

Right shift (`>>`) is a **logical right shift** (`lshr`) for all types: zero bits are shifted in from the left. There is no arithmetic right shift available through an operator in this version of the specification. To perform arithmetic right shift, use inline assembly.

Both operands MUST have the same type. No implicit promotion is performed.

---

### Comparison Operators

| Operator | Grammar Production | Result Type | Notes |
|----------|--------------------|-------------|-------|
| `==` | `equality` | `bool` | `icmp eq` (int), `fcmp oeq` (float) |
| `!=` | `equality` | `bool` | `icmp ne` (int), `fcmp one` (float) |
| `<` | `relational` | `bool` | `icmp slt`/`ult` (int), `fcmp olt` (float) |
| `<=` | `relational` | `bool` | `icmp sle`/`ule` (int), `fcmp ole` (float) |
| `>` | `relational` | `bool` | `icmp sgt`/`ugt` (int), `fcmp ogt` (float) |
| `>=` | `relational` | `bool` | `icmp sge`/`uge` (int), `fcmp oge` (float) |

Float comparisons use ordered predicates (`oeq`, `one`, `olt`, etc.): comparisons involving NaN return `false`.

For signed integer types (`int`, `byte`, `char`), signed predicates are used. For unsigned integer types (`unsigned int`, `unsigned byte`), unsigned predicates are used.

---

### Logical Operators

| Operator | Grammar Production | LLVM Instruction | Operand Types | Result Type |
|----------|--------------------|------------------|---------------|-------------|
| `&&` | `logicalAnd` | `and i1` | `bool` (coerced) | `bool` |
| `\|\|` | `logicalOr` | `or i1` | `bool` (coerced) | `bool` |

Non-bool operands are coerced to `bool` by emitting `icmp ne <type> <val>, 0` before the logical operation. This coercion is implicit and applies to any integer or address type.

Logical operators do NOT short-circuit in the current implementation. Both operands are evaluated unconditionally.

---

### Unary Operators

| Operator | Grammar Production | LLVM Instruction | Operand Types | Result Type |
|----------|--------------------|------------------|---------------|-------------|
| `!` | `unary` | `xor i1 <val>, 1` | `bool` | `bool` |
| `-` | `unary` | `sub <t> 0, <val>` / `fneg` | numeric | same as operand |
| `~` | `unary` | `xor <t> <val>, -1` | integer | same as operand |

Unary minus on floats emits `fneg double`. Unary minus on integers emits `sub <type> 0, <val>`.

Logical NOT (`!`) requires a `bool` operand. Bitwise NOT (`~`) requires an integer operand.

---

### Cast Operator

```antlr
castType unary
```

The `castType` prefix applies an explicit primitive type conversion to its operand. This is an internal grammar production; programs SHOULD use the `cast<T>(expr)` template function from the `cast` module instead.

The supported conversions and their LLVM instructions:

| From \ To | `int` | `byte` / `char` | `float` | `bool` | `address` | `string` |
|-----------|-------|-----------------|---------|--------|-----------|---------|
| `int` | — | `trunc i64 to i8` | `sitofp i64 to double` | `icmp ne i64 0` | `inttoptr i64 to i8*` | — |
| `byte`/`char` | `sext i8 to i64` | — | `sitofp i8 to double` | `icmp ne i8 0` | — | — |
| `unsigned int` | — (same bits) | `trunc i64 to i8` | `sitofp i64 to double` | `icmp ne i64 0` | `inttoptr i64 to i8*` | — |
| `unsigned byte` | `zext i8 to i64` | — | `sitofp i8 to double` | `icmp ne i8 0` | — | — |
| `float` | `fptosi double to i64` | `fptosi double to i8` | — | `fcmp une double 0.0` | — | — |
| `bool` | `zext i1 to i64` | `zext i1 to i8` | `uitofp i1 to double` | — | — | — |
| `address` | `ptrtoint i8* to i64` | — | — | — | — | `bitcast i8* to i8*` |
| `string` | — | — | — | — | `bitcast i8* to i8*` | — |

For `byte` → `int` with a signed source, `sext` is used. With an unsigned source (`unsigned byte`), `zext` is used.

The `address` ↔ `string` cast is a bitcast; both are `i8*` in LLVM IR, so no instruction is emitted.

### Address Operators

These are primary-position operators bound to the `::` token sequence:

| Syntax | Operation | Result Type |
|--------|-----------|-------------|
| `name::address` | Address of variable or function | `address` |
| `name[i]::address` | Address of array element | `address` |
| `name::value` | Dereference (load through pointer) | type pointed to |
| `name::size` | Byte size of variable or type | `int` |

**`::address`** emits a bitcast of the variable's LLVM alloca pointer to `i8*`. For function names, emits a bitcast of the function pointer to `i8*`. For array variables, emits a GEP to element 0, then bitcast to `i8*`.

**`::value`** emits a load through the pointer: load `i8*` from the alloca, bitcast to `T*`, then `load T`.

**`::size`** is a compile-time constant: the byte size of the variable's declared type, computed by the compiler from the type system. No runtime instruction is emitted.

---

### Array Subscript Operator

```antlr
Identifier '[' expression ']'
```

Applied to array variables: emits `getelementptr [N x T], [N x T]* <ptr>, i32 0, i64 <i>` followed by `load T`. Applied to class instances with a `[]` operator: calls the operator method.

---

### Assignment Operator

```antlr
Identifier '=' expression
```

Assignment is a **statement**, not an expression. It has no value and MUST NOT be used in expression position. Chained assignment (`a = b = c`) is not valid syntax.

A conforming implementation MUST reject assignment in expression position at the grammar level. The statement rule for `Assign` does not appear inside the expression hierarchy.
