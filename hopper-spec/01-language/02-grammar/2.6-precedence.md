## Precedence

### Precedence Table

The following table lists all Hopper operators from **lowest to highest** precedence. Operators on the same row have equal precedence and share the associativity noted. A conforming implementation MUST parse expressions according to this table.

| Level | Operators | Associativity | Grammar Rule |
|-------|-----------|---------------|--------------|
| 1 (lowest) | `\|\|` | Left | `logicalOr` |
| 2 | `&&` | Left | `logicalAnd` |
| 3 | `\|` | Left | `bitwiseOr` |
| 4 | `^` | Left | `bitwiseXor` |
| 5 | `&` | Left | `bitwiseAnd` |
| 6 | `==` `!=` | Left | `equality` |
| 7 | `<` `<=` `>` `>=` | Left | `relational` |
| 8 | `<<` `>>` | Left | `shift` |
| 9 | `+` `-` | Left | `additive` |
| 10 | `*` `/` `%` | Left | `multiplicative` |
| 11 | `!` `-` (unary) `~` `castType` | Right | `unary` |
| 12 (highest) | literals, calls, field access, subscript, `::`, `()` | n/a | `primary` |

### Rationale for Ordering

The precedence ordering follows established practice for C-family languages, with the following notes:

**Bitwise below logical:** In Hopper, `|`, `^`, and `&` sit between `&&`/`||` and the equality operators. This matches C precedence, where `a | b == c` groups as `a | (b == c)` due to `==` being higher than `|`. Programs SHOULD use explicit parentheses when mixing bitwise and comparison operators.

**Shift below additive:** `a + b << c` groups as `(a + b) << c` because `<<` and `>>` have lower precedence than `+` and `-`. This is the same as C. Programs SHOULD parenthesize shift expressions when combined with addition.

**Equality below relational:** `a < b == c < d` groups as `(a < b) == (c < d)`, which tests whether two comparisons have the same truth value. This is the C ordering and is preserved.

**Cast as unary:** The `castType` prefix cast has the same precedence as unary `-`, `!`, and `~`. The expression `int -x` is parsed as `int (-x)` — cast of the negated value. To cast first and then negate, write `-(int x)`.

### Precedence and Template Angle Brackets

The `<` and `>` tokens appear both as comparison operators (in `relational`) and as delimiters in template type arguments and template function calls (in `primary`). The grammar resolves this by context:

- `Identifier '<' type '>' '(' argList? ')'` in `primary` position is a template function call.
- `'<'` in `relational` position is a comparison.

A conforming implementation MUST use the full grammar to disambiguate; a heuristic such as "look for a type keyword after `<`" is not sufficient. The ANTLR4 LL(*) adaptive lookahead resolves this unambiguously.

### Worked Examples

The following examples show how the precedence rules apply:

```
a || b && c
  = a || (b && c)          — && binds tighter than ||

a | b & c
  = a | (b & c)            — & binds tighter than |

a == b != c
  = (a == b) != c          — equality is left-associative

a < b == c > d
  = (a < b) == (c > d)     — relational binds tighter than equality

a + b << 2
  = (a + b) << 2           — shift binds looser than additive

a * b + c * d
  = (a * b) + (c * d)      — multiplicative binds tighter than additive

!a && b
  = (!a) && b              — unary binds tighter than &&

int -x
  = int(-x)                — castType applies to the unary expression

a + b * -c
  = a + (b * (-c))         — unary minus binds tightest among non-primary
```

### Precedence and Parentheses

Parentheses `( expression )` override all precedence rules. A conforming implementation MUST evaluate parenthesized sub-expressions before using their results in the surrounding expression. Programs SHOULD use parentheses whenever the intended grouping is not obvious from the table.
