## Patterns

### No Pattern Matching

Hopper does not have pattern matching. There is no `match`, `switch`, `when`, or `case` construct in the grammar. A conforming implementation MUST NOT accept any of these keywords as syntactic forms.

The rationale is stated in §0.4 (Anti-Goals): Hopper prioritizes explicit, auditable control flow. Pattern matching introduces implicit binding, exhaustiveness checking machinery, and runtime dispatch patterns that increase language complexity without a commensurate benefit in systems code.

Conditional logic that would be expressed with pattern matching in other languages MUST be expressed in Hopper using `if`/`else if`/`else` chains or indexed dispatch through function pointers.

Example of idiomatic Hopper conditional dispatch:
```hopper
if (kind == JsonKind.NULL_VAL) {
    handleNull()
} else if (kind == JsonKind.BOOL) {
    handleBool(value)
} else if (kind == JsonKind.INT) {
    handleInt(value)
} else {
    handleOther(kind, value)
}
```

### Limited Destructuring

Hopper provides two forms of limited structural access that resemble destructuring in other languages, but which are simple field access and array indexing operations with no binding semantics.

#### Field Access

A field of a struct or class is accessed via dot notation:

```hopper
int x = point.x
int y = point.y
```

There is no syntax to bind multiple fields simultaneously. Each field access is a separate statement. The value must be assigned to a separately declared variable.

#### Array Indexing

Array elements are accessed by index:

```hopper
int first = values[0]
int second = values[1]
```

There is no slice syntax, spread syntax, or rest-element syntax. Each element is accessed individually.

#### Enum Variant Access

Enum variants are accessed by name:

```hopper
int mode = FileMode.RDONLY
```

This is a compile-time integer substitution. There is no destructuring of enum values because Hopper enums carry no payload — they are named integers.

### Future Pattern Matching

*Not yet specified.*

Pattern matching over algebraic types or tagged unions is not yet designed. Adding it would require:

1. A sum type (tagged union) declaration form not yet in the grammar.
2. A `match` statement or expression form.
3. Exhaustiveness checking in the type system.
4. Binding semantics for matched fields.

None of these are part of the current specification. Any future pattern matching design MUST be compatible with the newline-significant grammar and MUST NOT require a runtime type-descriptor table (per design goal G1, zero mandatory runtime).
