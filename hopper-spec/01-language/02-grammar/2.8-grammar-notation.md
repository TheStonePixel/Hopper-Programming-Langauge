## Grammar Notation

### Overview

This section describes the ANTLR4 grammar notation used throughout Chapter 2. Grammar excerpts in this specification are reproduced verbatim from `seed/compiler/grammar/Hopper.g4`. A conforming implementation is not required to use ANTLR4, but MUST accept the same language that the ANTLR4 grammar defines.

### Rule Definition

A grammar rule is defined as:

```
ruleName
    : alternative1
    | alternative2
    | alternativeN
    ;
```

The rule name appears on the left. Alternatives are separated by `|`. The rule definition ends with `;`. A rule defines a **parser rule** if its name begins with a lowercase letter, or a **lexer rule** if its name begins with an uppercase letter.

### Quoted Strings: Literal Tokens

A quoted string in double quotes or single quotes is a **literal terminal**. It matches exactly those characters in the input:

```antlr
'if'         — matches the four characters i, f
'=='         — matches the two characters =, =
'+'          — matches a single plus sign
```

Literals are case-sensitive. The literal `'int'` does NOT match `Int` or `INT`.

### Uppercase Names: Lexer Rules

An uppercase name refers to a **lexer token type**. The token is produced by the lexer rule of that name. In grammar excerpts, these appear as token references:

```antlr
IntegerLiteral   — a sequence of decimal digits [0-9]+
HexLiteral       — 0x followed by hex digits
Identifier       — [a-zA-Z_][a-zA-Z0-9_]*
NEWLINE          — \r?\n
AsmBlock         — asm { ... } captured as a single token
```

Lexer rules MUST be matched before parser rules. The lexer is responsible for producing a token stream; the parser consumes that stream.

### Lowercase Names: Parser Rules

A lowercase name refers to a **parser rule**. It is defined elsewhere in the grammar file:

```antlr
expression       — defined as the entry point of the expression hierarchy
statement        — defined as the union of all statement forms
block            — defined as a brace-delimited sequence of statements
```

### `?`: Optional Element

A `?` suffix makes the preceding element optional — it may appear zero or one time:

```antlr
expression?      — zero or one expression
argList?         — zero or one argList
elseClause?      — zero or one elseClause
constrainClause? — zero or one constrainClause
```

An optional element that is absent contributes nothing to the parse tree.

### `*`: Zero or More

A `*` suffix means the preceding element may appear zero or more times. ANTLR4 matches as many repetitions as possible (greedy):

```antlr
NEWLINE*         — zero or more newlines
(topLevelDecl NEWLINE*)*  — zero or more top-level declarations, each followed by zero or more newlines
```

### `+`: One or More

A `+` suffix means the preceding element MUST appear at least once and may repeat:

```antlr
NEWLINE+         — one or more newlines (required separator)
(statement (NEWLINE+ statement)*)   — one or more statements, newline-separated
```

The `NEWLINE+` between statements is the mechanism that enforces the one-statement-per-line rule.

### `|`: Alternation

The `|` operator separates alternatives within a rule. The parser tries each alternative left-to-right and selects the first that successfully matches. ANTLR4's LL(*) algorithm may use unbounded lookahead to distinguish alternatives that share a common prefix:

```antlr
statement
    : 'if' '(' expression ')' block elseClause?
    | 'while' '(' expression ')' block
    | ...
    ;
```

### `( ... )`: Grouping

Parentheses group elements within an alternative. The group may then have a `?`, `*`, or `+` quantifier applied to it:

```antlr
( ',' param )*          — zero or more comma-param pairs
( NEWLINE+ contractClause )*   — zero or more newline+clause pairs
(type (',' type)*)?     — optional type list with comma separators
```

### `#`: Rule Label (Alternative Label)

The `#` notation assigns a name to an alternative within a rule. This name becomes the AST node kind in the generated parser. It is used to dispatch on specific alternatives in the AST builder:

```antlr
statement
    : type Identifier '=' expression   # VarDecl
    | type Identifier                  # VarDeclNoInit
    | Identifier '=' expression        # Assign
    ;
```

Labels do not affect the grammar accepted by the parser. They are annotations for the AST-walking code.

### `->`: Lexer Action

Lexer rules may have an action directive following `->`:

```antlr
WS          : [ \t]+ -> skip ;
LINE_COMMENT : '//' ~[\r\n]* -> skip ;
```

`skip` instructs the lexer to discard the matched token entirely; it is never passed to the parser. This is how whitespace and comments are removed from the token stream.

Without `skip`, the token is passed to the parser. `NEWLINE` has no `->` directive and therefore IS passed to the parser, making it available as a statement separator.

### `~`: Character Complement

In lexer rules, `~[chars]` matches any character that is NOT in the specified set:

```antlr
~['\r\n\\]   — any character except single-quote, CR, LF, or backslash
~[\r\n]*     — any number of characters that are not CR or LF
~[}]*        — any number of characters that are not a closing brace
```

This is used in string and character literal rules, the comment rule, and the `AsmBlock` rule.

### `[chars]`: Character Class

A character class `[...]` in a lexer rule matches any single character in the named set. Ranges are written with `-`:

```antlr
[0-9]            — any decimal digit
[a-zA-Z_]        — any letter or underscore
[0-9a-fA-F]      — any hexadecimal digit
[ \t]            — space or tab
```

### Lexer Rule Precedence

When multiple lexer rules could match the same input, ANTLR4 selects the rule defined **earliest** in the grammar file. This gives `Identifier` lower priority than keyword literals — the literal `'if'` is always matched as the keyword token, not as an `Identifier`, because keyword literals are defined earlier in the grammar than the `Identifier` rule. A conforming implementation MUST apply the same precedence.

### Complete Lexer Rules

For reference, the full set of Hopper lexer rules in priority order:

```antlr
IntegerLiteral  : [0-9]+ ;
HexLiteral      : '0x' [0-9a-fA-F]+ ;
CharLiteral     : '\'' (~['\r\n\\] | '\\' .) '\'' ;
UnicodeLiteral  : 'U+' [0-9a-fA-F]+ ;
FloatLiteral    : [0-9]+ '.' [0-9]+ ;
StringLiteral   : '"' (~["\r\n\\] | '\\' .)* '"' ;
Identifier      : [a-zA-Z_][a-zA-Z0-9_]* ;
NEWLINE         : '\r'? '\n' ;
LINE_COMMENT    : '//' ~[\r\n]* -> skip ;
WS              : [ \t]+ -> skip ;
AsmBlock        : 'asm' [ \t]* '{' ~[}]* '}' ;
```

Note that `AsmBlock` is listed last but its pattern is highly specific (begins with `asm` followed by `{`), so it does not conflict with `Identifier` in practice. A conforming implementation MUST ensure that the sequence `asm {` is recognized as the beginning of an `AsmBlock` token and not as an `Identifier` followed by `{`.
