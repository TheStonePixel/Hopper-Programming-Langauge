## 4.8 Conversions

Hopper does not perform implicit type conversions. When an expression of type A appears where type B is expected and A ≠ B, the program is ill-formed. All type conversions MUST be explicit. The sole exception to this rule is the `address`/`string` representational overlap, which permits direct assignment between these two types.

---

### 4.8.1 No Implicit Coercion

A conforming implementation SHALL NOT silently convert a value from one type to another. The following examples are all type errors:

```hopper
int n = 3
float f = n        // ERROR: int is not implicitly convertible to float

byte b = 42        // ERROR: integer literal 42 has type int; byte expected
byte b = byte 42   // CORRECT: explicit castType unary

bool flag = 1      // ERROR: int is not implicitly convertible to bool
bool flag = 1 != 0 // CORRECT: explicit comparison produces bool

int x = 3
unsigned int u = x // ERROR: int and unsigned int are distinct types
```

The rule is absolute at the language level. There are no "safe" implicit conversions, no integer promotion rules, and no "usual arithmetic conversions" as exist in C.

---

### 4.8.2 The address/string Exception

`address` and `string` both compile to `i8*` in LLVM IR. Because a conversion between them is a no-op at the instruction level, Hopper permits direct assignment between `address` and `string` values without a cast.

```hopper
string path = "/etc/hosts"
address p   = path          // permitted: string → address, both are i8*
string s    = p             // permitted: address → string, both are i8*
```

This exception is explicitly limited to the assignment context between these two types. It does not extend to:
- Passing an `address` where a `string` parameter is declared (implementation-defined; a conforming implementation MAY or MAY NOT permit this).
- Passing a `string` where an `address` parameter is declared (same qualification).
- Any arithmetic or comparison between `address` and `string` values.

The reason this exception exists is practical: in systems programming, the boundary between a raw memory pointer and a null-terminated string pointer is frequently crossed. Requiring a cast at every such crossing would add syntactic noise without adding safety, because both types are `i8*` with no runtime distinction.

---

### 4.8.3 Explicit Conversion via the Cast Module

All type conversions other than the `address`/`string` exception MUST be explicit. The primary mechanism for explicit conversion between primitive types is the `cast<T>(x)` function from the `cast` module.

```hopper
import Cast from cast

int   n     = cast<int>(3.9)      // float → int, truncates toward zero
float f     = cast<float>(7)      // int → float
address p   = cast<address>(42)   // int → address
byte  b     = cast<byte>(300)     // int → byte, truncates to low 8 bits (300 mod 256 = 44)
bool  flag  = cast<bool>(1)       // int → bool (0 → false, nonzero → true)
string s    = cast<string>(p)     // address → string
```

`cast<T>(x)` is not a compiler intrinsic or special syntax — it is a call to a template function defined in `hopper/modules/cast/src/Cast.hop`. The cast module is the authoritative list of all legal primitive type conversions in Hopper.

A conforming program that needs a type conversion MUST either:
1. Use `cast<T>(x)` from the cast module (preferred for all primitive conversions).
2. Use the `castType unary` grammar production (see §4.8.4) for inline narrowing.

---

### 4.8.4 The castType Unary Expression

The grammar provides a second explicit conversion syntax: a primitive type keyword used as a unary prefix operator. This is the `castType unary` grammar production:

```hopper
byte b = byte 65
int n  = int  b
float f = float 7
bool ok = bool 1
```

The `castType unary` form is semantically equivalent to the corresponding `cast<T>(x)` call. It is available for any expression, not just in declaration initializers:

```hopper
int n0 = int (p0::value)      // dereference and widen to int
int n1 = int (b0) & 0xFF      // widen byte to int for bitwise arithmetic
```

The `castType unary` form works with the following type keywords:
`int`, `byte`, `char`, `float`, `bool`, `bit`, `address`, `unsigned int`, `unsigned byte`.

It does not work with `string`, `string[]`, `void`, or user-defined type names. For `string`/`address` conversion, use `cast<string>` or rely on the assignment exception.

---

### 4.8.5 Complete Table of Defined Primitive Conversions

The following table lists all explicit primitive type conversions defined in the Hopper cast module. A conversion marked — is not defined; performing such a conversion requires an intermediate step.

| From \ To       | `int`      | `byte`     | `float`    | `bool`     | `address`  | `string`   |
|-----------------|------------|------------|------------|------------|------------|------------|
| `int`           | —          | `trunc`    | `sitofp`   | `icmp ne`  | `inttoptr` | —          |
| `byte`          | `sext`     | —          | `sitofp`   | `icmp ne`  | —          | —          |
| `char`          | `sext`     | (identity) | `sitofp`   | `icmp ne`  | —          | —          |
| `float`         | `fptosi`   | `fptosi`   | —          | `fcmp une` | —          | —          |
| `bool`          | `zext`     | `zext`     | `uitofp`   | —          | —          | —          |
| `address`       | `ptrtoint` | —          | —          | —          | —          | `bitcast`  |
| `string`        | —          | —          | —          | —          | `bitcast`  | —          |
| `unsigned byte` | `zext`     | (identity) | `sitofp`   | `icmp ne`  | —          | —          |

The LLVM instruction listed is the instruction emitted by the compiler when the corresponding `cast<T>(src)` or `castType src` form is used. The instruction names are as they appear in LLVM IR documentation.

Additional conversions defined in the cast module:
- `cast<unsigned int>(byte src)` — zero-extend byte to i64 (`zext`)
- `cast<unsigned int>(address src)` — pointer to unsigned integer (`ptrtoint`)
- `cast<unsigned byte>(int src)` — truncate int to 8 bits (`trunc`)
- `cast<char>(int src)` — truncate int to 8 bits (`trunc`)

Cells marked — are not defined in the cast module and cannot be expressed directly. To convert `float` to `address`, the programmer MUST convert `float` to `int` first, then `int` to `address`:

```hopper
float f   = 3.14
int   n   = cast<int>(f)       // float → int
address p = cast<address>(n)   // int → address
```

---

### 4.8.6 Behavior of Defined Conversions

**Integer truncation (`trunc`):**
When converting `int` to `byte` or `unsigned byte`, the high bits are discarded and only the low 8 bits are retained. The result is the value modulo 256 (for unsigned) or the low 8-bit two's complement value (for signed). This is defined behavior — no overflow, no fault.

```hopper
byte b = cast<byte>(300)    // 300 & 0xFF = 44; b = 44
byte c = cast<byte>(-1)     // −1 in 64-bit = 0xFFFFFFFFFFFFFFFF; low 8 bits = 0xFF = −1 as byte
```

**Sign extension (`sext`):**
When converting `byte` to `int`, the sign bit of the byte is extended to fill all 64 bits. A `byte` value of 127 becomes `int` 127; a `byte` value of −128 becomes `int` −128; a `byte` value of 255 (interpreted as signed: −1) becomes `int` −1.

**Zero extension (`zext`):**
When converting `bool` to `int` or `unsigned byte` to `int`, zeros fill the high bits. `true` becomes 1; `false` becomes 0.

**Float-to-int truncation (`fptosi`):**
Truncates toward zero. `cast<int>(3.9)` → 3. `cast<int>(-3.9)` → -3. If the float value is outside the range of `int` (e.g., infinity, NaN, or a value larger than 2^63−1), the result is implementation-defined (LLVM `fptosi` with out-of-range input produces a poison value).

**Float comparison for bool (`fcmp une`):**
`cast<bool>(float)` uses unordered not-equal to zero: a float is `false` if it is exactly 0.0, and `true` for all other values including NaN. This means NaN is truthy.

**Pointer-integer conversion (`inttoptr`, `ptrtoint`):**
These conversions allow treating integer values as memory addresses and vice versa. The result is platform-defined in terms of what memory the address refers to. These are intentionally unsafe operations used in systems programming (e.g., mapping hardware registers at known physical addresses).

**Bitcast for address/string:**
`cast<string>(address)` and `cast<address>(string)` are zero-cost: both types are `i8*` in LLVM IR, and the conversion is a `bitcast` — no instruction is emitted.

---

### 4.8.7 No User-Defined Conversions

Hopper does not provide a mechanism for user-defined implicit or explicit type conversions. There is no conversion constructor, no conversion operator, and no way for a class to define how it converts to or from another type. If two user-defined types need to be convertible, the programmer MUST write an explicit conversion function:

```hopper
function packetToFrame(Packet p) Frame {
    Frame f
    f.buf = p.data
    f.len = p.length
    return f
}
```

This function is then called explicitly at every conversion site. There is no way to make it implicit.
