## 4.4 Type Inference

Hopper does not have type inference. Every variable declaration MUST include an explicit type annotation. The compiler does not derive the type of a variable from its initializer expression, from how the variable is used later in the function, or from any other contextual information.

This is a deliberate design choice. In systems and safety-critical code, the type of every named entity must be immediately apparent from the declaration site, without requiring the programmer or reviewer to trace data flow through expressions or function calls.

---

### 4.4.1 Explicit Declaration Required

Every local variable declaration takes one of the following forms:

```hopper
type identifier = expression
type identifier
type identifier[N] = [elem0, elem1, ..., elemN]
type identifier[N]
```

The `type` token is mandatory. There is no `auto`, `var`, `let`, `val`, `infer`, or any other keyword that instructs the compiler to deduce the type. A conforming implementation SHALL reject any declaration that lacks an explicit type.

```hopper
int count = 0           // well-formed
float ratio = 0.5       // well-formed
bool found = false      // well-formed
address buf = allocate 128  // well-formed

count = 0               // ERROR: assignment to undeclared variable, not a declaration
x = getValue()          // ERROR: bare assignment with no prior declaration
```

---

### 4.4.2 Default Types of Literals

Although Hopper does not infer variable types, literals do have canonical types. When a literal appears in an expression or as an initializer, the compiler assigns it the following types:

| Literal form              | Type     | Example             |
|---------------------------|----------|---------------------|
| Integer decimal literal   | `int`    | `42`, `-1`, `0`     |
| Hexadecimal literal       | `int`    | `0xFF`, `0x1000`    |
| Floating-point literal    | `float`  | `3.14`, `0.5`       |
| String literal            | `string` | `"hello"`           |
| Character literal         | `char`   | `'A'`, `'\n'`       |
| Unicode literal           | `int`    | `U+0041`            |
| `true` / `false`          | `bool`   | `true`, `false`     |
| `null`                    | `address`| `null`              |

These default types are the types of the literal expressions themselves. When a literal is used as an initializer for a variable of a different type, a type mismatch error is issued unless the declared type and the literal's type are assignment-compatible (§4.8).

```hopper
int n   = 42            // well-formed: int literal → int variable
byte b  = 42            // ERROR: int literal 42 has type int; byte variable requires byte
byte b  = byte 42       // well-formed: castType unary converts the int literal to byte
```

The programmer MUST use an explicit cast when the declared type differs from the literal's default type.

---

### 4.4.3 No Flow-Sensitive Typing

Hopper does not perform flow-sensitive type narrowing. A variable's type is fixed at its declaration and does not change based on control flow, conditional checks, or assignments in branches. There is no union type, optional type, or nullable type narrowing.

```hopper
address p = getValue()
if (p != null) {
    // p is still type address — no narrowing to a non-null subtype
    byte b = p::value   // programmer is responsible for verifying safety
}
```

---

### 4.4.4 Function Return Types

Function return types MUST be explicitly declared. The compiler does not infer the return type from the function body.

```hopper
function add(int a, int b) int {    // return type int is explicit
    return a + b
}

function add(int a, int b) {        // procedure — return type is implicitly absent (void)
    // ...
}
```

A function body's `return` expressions must all have a type consistent with the declared return type. The compiler checks this at each `return` statement; it does not infer the return type by collecting and unifying the types of all `return` expressions.

---

### 4.4.5 Parameter Types

Function parameter types MUST be explicitly declared. There is no mechanism by which a parameter's type is inferred from call sites.

```hopper
function clamp(int value, int lo, int hi) int {
    if (value < lo) { return lo }
    if (value > hi) { return hi }
    return value
}
```

Every parameter in every function, method, constructor, and `entry` point MUST carry an explicit type annotation.

---

### 4.4.6 Template Parameters

Template functions use a type parameter (`<targetType>`) that is resolved at instantiation time, not inferred. The programmer specifies the target type explicitly at every call site:

```hopper
int n = cast<int>(3.9)      // targetType = int, explicitly provided
float f = cast<float>(7)    // targetType = float, explicitly provided
```

The argument type (the type of `3.9` or `7`) is used by the compiler to select the correct specialization from among those defined in the template function's body. This is compile-time dispatch, not type inference. The target type is always explicitly written.

---

### 4.4.7 Rationale

The absence of type inference is intentional. In code that interfaces directly with hardware, manages memory manually, and crosses address-space or privilege boundaries, the type of every value is a load-bearing safety property. Making types implicit — even when the compiler can determine them unambiguously — introduces a category of bugs that is invisible to code reviewers and difficult to audit.

Hopper's design preference is that every variable declaration is self-describing. A reviewer reading a function body MUST be able to determine the type of every variable without reading the rest of the function, without consulting a type inference algorithm, and without running the code.
