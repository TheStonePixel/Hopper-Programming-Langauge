## 4.7 Constraints

Hopper provides two forms of compile-time and runtime constraint specification: range constraints on variable declarations (`constrain`), and contract clauses on function declarations (`requires`, `ensures`). Both forms express assertions about values and relationships that the programmer guarantees hold at specific program points.

---

### 4.7.1 Range Constraints

A range constraint is an optional clause on a variable declaration that asserts the declared variable's value falls within the value range of a specified type. The syntax is:

```hopper
type identifier = expression constrain [targetType]
```

or, without initialization:

```hopper
type identifier constrain [targetType]
```

**Example:**

```hopper
int x = 10 constrain [byte]
```

This declaration asserts that `x`, which has type `int`, holds a value in the range [−128, 127] — the range of `byte`. The programmer is asserting that the value will always fit in a `byte`, even though the storage type is `int`.

**Motivation:**

Range constraints allow programmers to use wider types (e.g., `int`) for arithmetic while explicitly documenting and asserting that the actual values remain within a narrower range. This is common in systems programming: a byte count stored as an `int` for arithmetic convenience, but guaranteed to never exceed 255.

**Semantics:**

The semantics of `constrain [T]` depend on the compilation mode:

**Standard mode:** The constraint is a runtime assertion. The compiler emits a bounds check at the point of the declaration's initialization. If the initializer produces a value outside the range of `T`, the program faults (implementation-defined fault behavior — typically abort or trap). If the variable has no initializer, the constraint is checked at the first assignment.

**Strict mode:** The constraint is a compile-time assertion. The compiler attempts to verify at compile time that the initializer expression's value is provably within the range of `T`. If the compiler cannot verify this statically, the program is ill-formed and the compiler SHALL emit a diagnostic. Strict mode is activated by the `--strict` compiler flag.

**Note:** The `strict` keyword in Hopper source code is unrelated to constrain strict mode — it is used exclusively for MMIO register declarations (`strict int uart_dr = 0x40021000`). Do not confuse the two uses of the word "strict".

**Rules:**

- The target type in `constrain [T]` MUST be a primitive type that has a defined integer range. The compiler's `constrainRange()` function only recognises `byte`, `unsigned byte`, `bool`, `unsigned int`, and `int` as constraint targets with meaningful ranges. User-defined types, interface types, and composite types are not valid constraint targets.

  **Not yet implemented:** `float` and `bit` are not valid constrain targets in the current compiler — `constrainRange()` returns `null` for both, causing the constraint check to be silently skipped (a no-op). A conforming implementation MUST NOT specify `constrain [float]` or `constrain [bit]`; the spec table below reflects the actually implemented targets only.
- The declared variable's type MUST be a supertype of `T` in terms of range. Constraining a `byte` to `[int]` is valid syntax but tautological (byte's range is a strict subset of int's range). A conforming implementation MAY warn on tautological constraints.
- A `constrain` clause on a variable without an initializer causes the range check to be deferred to the first assignment to that variable.
- `constrain` is not a cast. The variable retains its declared type (`int` in the example above). The constraint only asserts a value-range property; it does not change the storage representation.
- Multiple `constrain` clauses on a single declaration are not permitted. Only one constraint may be specified per declaration.

**Range definitions for constrain targets (implemented):**

| Target type    | Valid range                 |
|----------------|-----------------------------|
| `bool`         | 0 to 1                      |
| `byte`         | −128 to 127                 |
| `unsigned byte`| 0 to 255                    |
| `int`          | −2^63 to 2^63−1 (no-op: full range, check is skipped) |
| `unsigned int` | 0 to 2^64−1                 |

**Not yet implemented:** `bit` (grammar accepts it; runtime check is a no-op because `constrainRange()` returns `null`) and `float` (same — no range check is emitted).

### 4.7.2 Contract Clauses

Contract clauses express logical preconditions and postconditions on function declarations. They are placed between the function signature and the opening brace of the function body.

**Syntax:**

```hopper
function name(params) returnType
requires preconditionExpression
ensures postconditionExpression
{
    body
}
```

**`requires` — precondition:**

A `requires` clause asserts a condition that MUST hold when the function is called. It is the caller's responsibility to ensure the precondition holds before calling the function.

```hopper
function divide(int a, int b) int
requires b != 0
{
    return a / b
}
```

```hopper
function charAt(address str, int i) byte
requires i >= 0
{
    address p = str + i
    return p::value
}
```

**`ensures` — postcondition:**

An `ensures` clause asserts a condition that will hold when the function returns. It is the callee's responsibility to ensure the postcondition holds at every return point.

```hopper
function max(int a, int b) int
ensures result >= a
ensures result >= b
{
    if (a >= b) { return a }
    return b
}
```

The identifier `result` in an `ensures` clause refers to the function's return value. A conforming implementation SHALL bind `result` to the return value of the function for the purpose of evaluating `ensures` expressions.

**Multiple clauses:**

A function MAY have multiple `requires` clauses and multiple `ensures` clauses. All `requires` clauses are conjoined (the function requires all of them to hold). All `ensures` clauses are conjoined (the function guarantees all of them at return).

```hopper
function clamp(int value, int lo, int hi) int
requires lo <= hi
ensures result >= lo
ensures result <= hi
{
    if (value < lo) { return lo }
    if (value > hi) { return hi }
    return value
}
```

**Semantics:**

In standard mode, contract clauses are runtime assertions:
- `requires` expressions are evaluated at function entry. If a `requires` expression evaluates to `false`, the behavior is implementation-defined (typically an abort or trap).
- `ensures` expressions are evaluated at each `return` point. If an `ensures` expression evaluates to `false` at a return point, the behavior is implementation-defined.

In strict mode (`--strict` compiler flag), the compiler runs a compile-time constant-folding pass (`strictAnalyze`) before IR generation. The current implementation checks:

- **`requires` at call sites:** if all arguments to a call are compile-time constants, the compiler evaluates the `requires` expressions and emits a hard compile error if any precondition is provably violated.
- **`constrain` on variable declarations:** if the initializer is a compile-time constant and its value is outside the constrain range, the compiler emits a hard compile error.

**Not yet implemented:** Compile-time verification of `ensures` clauses. `ensures` expressions are NOT checked in strict mode — they still only generate runtime assertions. A future conforming strict-mode implementation SHALL attempt to prove that all `ensures` clauses are satisfied at every return point in the function body.

**Rules:**

- Contract clauses MUST appear between the function signature and the opening `{` of the body.
- Contract clause expressions MUST be boolean expressions (type `bool`).
- Contract clause expressions MUST NOT have side effects. A conforming implementation SHOULD warn if a contract expression contains a function call or any expression that may modify program state.
- `requires` and `ensures` are only valid on function and procedure declarations. They are not valid on constructor or destructor declarations.
- The `result` identifier in `ensures` expressions is only bound if the function has a return type. For procedures (no return type), `result` is not in scope in `ensures` expressions.

### 4.7.3 Loop Invariants

A related constraint form is the loop invariant, expressed with the `invariant` keyword on `while` loops:

```hopper
while (i < n)
invariant i >= 0
{
    // ...
    i = i + 1
}
```

A loop invariant is an assertion that holds at the beginning of every iteration of the loop, including before the first iteration and after the last.

In standard mode, the invariant expression is evaluated at each loop-head evaluation (before the condition check). In strict mode, the invariant is a specification aid for static verification tools.

Loop invariants are particularly important for pointer arithmetic loops, where the invariant documents the valid range of a pointer or index used for memory access.

---

### 4.7.4 Relationship to Type System

Constraints are not a type-level mechanism. A variable declared `int x = 10 constrain [byte]` has type `int`. The constraint is an assertion about value range, not a type narrowing. The type system does not track which variables are constrained or what their constraints are. Constraints are assertions, not types.

If a programmer needs a value that is guaranteed to be in byte range for passing to a function that accepts `byte`, they MUST still use an explicit cast. A constrained `int` is not assignable to a `byte` variable without a cast.
