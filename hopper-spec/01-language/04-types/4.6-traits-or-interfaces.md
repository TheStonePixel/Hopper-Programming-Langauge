## 4.6 Interfaces

An interface is a named set of function signatures that a class may commit to implementing. Interfaces define behavioral specifications. A class that declares conformance to an interface MUST provide implementations for every function signature the interface declares. This conformance is verified at compile time.

Interfaces are Hopper's primary mechanism for separating specification from implementation, and for enabling hardware portability: the OS-level interface modules (`linux/contracts/`) declare what capabilities a program needs; the hardware implementation modules (`x86_64/src/`) supply the concrete implementations.

---

### 4.6.1 Interface Declarations

An interface is declared at the top level with the `interface` keyword:

```hopper
interface IO {
    function read(int fd, address buf, int count) int
    function write(int fd, address buf, int count) int
    function pread64(int fd, address buf, int count, int offset) int
    function pwrite64(int fd, address buf, int count, int offset) int
}
```

**Rules for interface declarations:**

- An interface body MAY contain function signatures and enum declarations. No other members are permitted.
- A function signature declares the function name, parameter types (with names), and return type. It MUST NOT include a body.
- Procedure signatures (functions that return no value) are permitted: `function name(params)` with no return type.
- A conforming implementation SHALL reject interface bodies that contain field declarations, default implementations, constructor signatures, destructor signatures, or operator declarations.
- Enum declarations inside an interface body (§4.2.5) are permitted. An enum declared inside an interface is in scope for any file that imports the interface.
- Constants (`const`) are NOT permitted inside interfaces. The grammar does not support `constDecl` inside `interfaceDecl`. Named integer sets inside an interface MUST be expressed as enums.

**Example with enum:**

```hopper
interface FileSystem {
    enum OpenFlag {
        O_RDONLY  = 0
        O_WRONLY  = 1
        O_RDWR    = 2
        O_CREAT   = 64
        O_TRUNC   = 512
        O_APPEND  = 1024
        O_NONBLOCK = 2048
    }

    function open(string path, int flags, int mode) int
    function close(int fd) int
    function stat(string path, address buf) int
    function unlink(string path) int
}
```

---

### 4.6.2 Interface Conformance

A class declares conformance to one or more interfaces using the `satisfies` keyword in the class declaration:

```hopper
class LinuxIO satisfies IO {
    function read(int fd, address buf, int count) int {
        return read(fd, buf, count)
    }
    function write(int fd, address buf, int count) int {
        return write(fd, buf, count)
    }
    function pread64(int fd, address buf, int count, int offset) int {
        return pread64(fd, buf, count, offset)
    }
    function pwrite64(int fd, address buf, int count, int offset) int {
        return pwrite64(fd, buf, count, offset)
    }
}
```

**Conformance rules:**

- A class that declares `satisfies I` MUST provide a method implementation for every function signature declared in `I`.
- Each implementing method's name, parameter count, parameter types (in order), and return type MUST match the corresponding interface signature exactly. A difference in any of these is a conformance failure.
- A conforming implementation SHALL emit a compile-time error if a class declares `satisfies I` but does not provide a conforming implementation for every function in `I`.
- A class MAY implement multiple interfaces: `class C satisfies I1, I2, I3 { ... }`. All interfaces in the list MUST be satisfied.
- If two interfaces in the `satisfies` list declare functions with the same name and signature, a single method implementation satisfies both simultaneously.
- If two interfaces in the `satisfies` list declare functions with the same name but different signatures, the class MUST provide separate implementations. If this is impossible (the same name cannot have two implementations in one class body), the program is ill-formed.
- A class MAY provide additional methods beyond those required by the interfaces it satisfies.

---

### 4.6.3 Interfaces Are Not Value Types

An interface name does not denote a value type. Variables MUST NOT be declared with an interface type. There are no interface-typed expressions, no interface-typed parameters, and no interface-typed return values.

```hopper
IO handle       // ERROR: IO is not a value type
```

This is a fundamental difference from interface types in languages like Java or Go. Hopper interfaces are purely a compile-time conformance check. When a class satisfies an interface, the compiler verifies that the required methods exist — but no vtable, interface table, or runtime type information is generated.

Interface conformance does not enable runtime polymorphism. Two classes that implement the same interface are not substitutable for each other through an interface-typed variable, because no such variable can exist.

---

### 4.6.4 Interfaces and the Module System

Interfaces are the boundary between hardware-independent and hardware-specific code in Hopper's module architecture.

An OS-level interface (e.g., `linux/contracts/IO.hop`) declares what a program requires from the operating system. The hardware implementation (e.g., `x86_64/src/LinuxSyscalls.hop`) provides the concrete implementations as free functions. The build manifest (`hopper.json` targets section) connects them.

When the build system resolves an interface binding:

1. The interface file is compiled and its function signatures are brought into scope.
2. The implementation file is compiled and its free functions (which satisfy the interface) are brought into scope.
3. Program code calls the free functions by name. The method call syntax (`obj.method()`) is only used when an explicit class instance exists.

This pattern means that the free functions from the implementation file come directly into scope in the program — programs do not need to instantiate a class to call `read()`, `write()`, or `open()`. The interface provides the specification; the implementation provides the functions.

---

### 4.6.5 Conformance Is Name-Based

Interface conformance is checked by name. If a class has a method with the same name and signature as an interface function, it satisfies that requirement — regardless of whether the class was written with knowledge of the interface.

However, conformance is not implicit. A class only participates in conformance checking for an interface if it explicitly declares `satisfies InterfaceName`. A class that happens to have a method `read(int, address, int) int` does not satisfy the `IO` interface unless it explicitly declares `satisfies IO`.

This keeps conformance relationships explicit and auditable in the source code.

---

### 4.6.6 Interface File Structure

By convention, each interface is declared in its own file under an `interfaces/` directory within the module that owns it:

```
linux/
  interfaces/
    IO.hop
    FileSystem.hop
    System.hop
    Process.hop
    Socket.hop
    Network.hop
    Memory.hop
```

Each file contains exactly one `interface` declaration. A conforming build tool SHALL resolve interface files relative to the consuming project's module directory, as specified in the `hopper.json` targets section.

### 4.6.7 What Interfaces May and May Not Contain

| Member kind                        | Permitted in interface? |
|------------------------------------|-------------------------|
| Function signature (with return)   | YES                    |
| Procedure signature (no return)    | YES                    |
| Enum declaration                   | YES                    |
| Constant (`const`)                 | NO — use enum          |
| Field declaration                  | NO                     |
| Default implementation             | NO                     |
| Constructor signature              | NO                     |
| Destructor signature               | NO                     |
| Operator signature                 | NO                     |
| Nested interface declaration       | NO                      |
| Nested class/struct/template decl  | NO                     |
