## 4.10 Type Safety

Hopper is a statically typed language. All type errors are compile-time errors; a well-formed Hopper program has no type errors at runtime in the sense that no operation is applied to a value of the wrong type at the language level. However, Hopper intentionally exposes several mechanisms that bypass or escape the type system. These mechanisms are necessary for systems programming and are explicitly permitted by the language specification.

---

### 4.10.1 Static Type Safety Guarantees

A conforming Hopper compiler SHALL guarantee the following properties for all well-formed programs:

- Every variable has a declared type that is fixed for its entire lifetime.
- Every expression has a type that is determined at compile time.
- Every assignment places a value of the correct type (or a permitted compatible type per §4.8) into the destination variable.
- Every function call passes arguments of the correct types and receives a return value of the declared return type.
- Every operator application uses operands of types in the operator's defined domain.

These guarantees hold without exception for well-formed programs that do not use the explicitly unsafe mechanisms described in §4.10.2–§4.10.5.

---

### 4.10.2 address Arithmetic Bypasses the Type System

The `address` type is an untyped pointer. It carries no information about what type of data it points to, how much data exists at that address, or whether the pointer is valid. All pointer arithmetic is unchecked.

```hopper
address buf = allocate 16
address mid = buf + 8       // advance pointer by 8 bytes
byte b      = mid::value    // read one byte — type system cannot verify this is safe
```

The compiler does not know whether `mid` points to valid memory, what type of data is stored there, or whether reading one byte is the correct amount to read. The type system's role ends at ensuring the dereference expression has type `byte` — it cannot ensure the pointer is valid.

**Implications:**

- Pointer arithmetic that goes out of bounds is undefined behavior (§4.10.6).
- Reading from an uninitialized or deallocated region is undefined behavior.
- There is no bounds checking on `address` arithmetic or dereference.

This is intentional. Systems programming requires direct memory access. The type system provides discipline at the named-type level; raw memory access operates below that level.

---

### 4.10.3 Inline Assembly Bypasses Type Checking

Inline assembly blocks (`asm { ... }`) contain raw x86-64 register assignments and instruction mnemonics. The compiler does not type-check the contents of an `asm` block beyond parsing its structure.

```hopper
function write(int fd, address buf, int count) int {
    int result
    asm {
        rax = 1
        rdi = fd
        rsi = buf
        rdx = count
        syscall
        result = rax
    }
    return result
}
```

Inside an `asm` block, the programmer is directly manipulating CPU registers. The compiler assigns values to registers and reads results from registers, but it does not verify that the register contents have types consistent with Hopper's type system. A value placed in `rax` is a 64-bit integer at the hardware level; whether it represents an `int`, an `address`, a `bool`, or something else is the programmer's responsibility.

**Implications:**

- The output of an `asm` block (values read from registers into Hopper variables) is trusted by the compiler as having the declared type of the destination variable. No runtime check is performed.
- Inline assembly may perform any operation the hardware supports, including those that are undefined behavior at the Hopper level (e.g., division by zero, writing to an unmapped page).
- By convention, `asm` blocks MUST NOT appear in program source files. They are confined to hardware-layer modules (`x86_64/src/`). This is an architectural convention enforced by code review, not by the language.

---

### 4.10.4 Casting from int to address Bypasses Pointer Safety

The conversion `cast<address>(int)` treats an arbitrary integer value as a memory address. This is a deliberate escape hatch for hardware programming: device drivers need to read and write specific physical addresses; boot code needs to access fixed memory-mapped register banks.

```hopper
address uartDr = cast<address>(0x40021000)
byte b = uartDr::value     // read from hardware register
```

The compiler has no way to know whether the integer value is a valid, accessible memory address at the time of program execution. The programmer is asserting that it is. If it is not, dereferencing the resulting pointer is undefined behavior.

**Implications:**

- Programs that use `cast<address>(int)` are accessing hardware at specific physical addresses. This is only meaningful in contexts where the program controls the address space (kernel mode, bare-metal).
- In user-mode programs, using arbitrary integer values as addresses will almost always cause a segmentation fault or similar fault. This is a programming error, not language undefined behavior in the usual sense — the hardware will deliver a fault signal.

---

### 4.10.5 The extern Declaration

`extern` function declarations allow Hopper programs to call functions whose implementations are written in C or another language that follows the C calling convention. The compiler trusts the declared signature without verification.

```hopper
extern function memcpy(address dst, address src, int n) address
extern function printf(string fmt, ...) int
```

The compiler emits a call matching the declared signature. If the actual C function has a different signature, the result is undefined behavior. There is no mechanism to verify that the declared signature matches the actual external function at compile time or link time.

**Implications:**

- Incorrect `extern` declarations produce undefined behavior. The programmer is responsible for matching the signature exactly.
- Variadic `extern` functions (those with `...`) have additional risk: Hopper passes arguments of the types present at the call site, but the called function may interpret them differently.
- Per the CLAUDE.md design rules, `extern` declarations to libc are acceptable temporarily for diagnostics only. Programs MUST NOT ship with `extern` declarations that delegate their core work to libc.

---

### 4.10.6 Undefined Behavior

The following conditions constitute undefined behavior in Hopper. A conforming implementation MAY assume these conditions do not occur and optimize accordingly. Programs that trigger undefined behavior are not conforming.

**Null address dereference:**
Dereferencing `null` (i.e., reading or writing `null::value`) is undefined behavior. The compiler does not check for null before generating load/store instructions.

```hopper
address p = null
byte b = p::value    // undefined behavior
```

**Reading uninitialized memory:**
Reading from a location that has been allocated (`allocate`) but never written is undefined behavior. The allocator does not zero-initialize memory. The value produced by reading uninitialized memory is unpredictable.

```hopper
address buf = allocate 64
byte b = buf::value    // undefined behavior: buf is not initialized
```

**Out-of-bounds array access:**
Accessing `arr[i]` where `i` is negative or `i >= N` (the array size) is undefined behavior. Hopper does not perform bounds checking on array accesses. The bounds check, if needed, is the programmer's responsibility.

```hopper
int arr[4] = [1, 2, 3, 4]
int x = arr[10]    // undefined behavior: index out of bounds
```

**Signed integer overflow:**
Arithmetic on `int` values that produces a result outside [−2^63, 2^63−1] is undefined behavior per LLVM semantics. The LLVM `nsw` (no signed wrap) flag is assumed on signed integer arithmetic, which allows the optimizer to assume overflow does not occur.

```hopper
int max = 9223372036854775807    // 2^63 - 1
int n   = max + 1               // undefined behavior: signed overflow
```

Unsigned integer overflow (`unsigned int`) is defined behavior: the result wraps modulo 2^64.

**Use after deallocate:**
Reading or writing memory through an `address` that has been passed to `deallocate` is undefined behavior.

```hopper
address buf = allocate 64
deallocate buf
byte b = buf::value    // undefined behavior: use after free
```

**Calling a null callback:**
Calling a `callback` variable that is null (has not been initialized or has been set to null) is undefined behavior.

**Double deallocate:**
Calling `deallocate` twice on the same address is undefined behavior.

---

### 4.10.7 Implementation-Defined Behavior vs. Undefined Behavior

**Implementation-defined behavior** is behavior that is not specified by this document but MUST be documented by a conforming implementation. Programs that depend on implementation-defined behavior are not portable, but they are not invalid.

Examples of implementation-defined behavior:
- The fault behavior when a constraint assertion fails (see §4.7).
- The fault behavior when a contract clause is violated at runtime.
- The result of `cast<int>(float)` when the float is NaN or infinity (LLVM poison; the implementation MAY define a specific result).

**Undefined behavior** is behavior that this specification makes no guarantees about. A conforming implementation MAY do anything when undefined behavior occurs — including emitting incorrect code, crashing, silently producing wrong results, or triggering security vulnerabilities. The canonical definition of undefined behavior in Hopper follows the LLVM IR semantics for the generated code.

---

### 4.10.8 Type Safety at the System Boundary

Hopper's type system provides full static safety for all code that remains within the named-type layer: class methods, free functions, template functions, enum accesses, and interface dispatch. The type system provides no safety guarantees for code that crosses into the hardware layer via `address`, `asm`, `extern`, or `cast<address>`.

This boundary is explicit and intentional. Systems programming requires operating below the type system. Hopper's design makes the unsafe operations named and visible — there are no hidden coercions, no implicit pointer casts, and no automatic type erasure. Every point where the type system's guarantees end is marked by explicit syntax: a dereference (`::value`), an assembly block (`asm { ... }`), an extern declaration, or a cast to `address`.

Programs SHOULD minimize the code that operates at or below the address layer, and SHOULD encapsulate all direct memory and hardware access in dedicated modules (`x86_64/src/`, hardware drivers) rather than scattering it through application logic.
