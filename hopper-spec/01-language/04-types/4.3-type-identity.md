## 4.3 Type Identity

Hopper uses a nominal type system. Two types are identical if and only if they have the same name. Structural equivalence — sharing the same field layout, the same method signatures, or the same LLVM representation — does not establish type identity.

---

### 4.3.1 Nominal Equality

The fundamental rule of type identity in Hopper:

> Two types are the same type if and only if they have the same declared name.

This rule applies without exception to all user-defined types: structs, classes, template instantiations, and enums. It applies regardless of whether the types' definitions are structurally identical.

Given:

```hopper
struct Point {
    int x
    int y
}

struct Size {
    int x
    int y
}
```

`Point` and `Size` are distinct types. A variable of type `Point` MUST NOT be assigned a value of type `Size`, even though both types have identical field layouts. A function that accepts `Point` MUST NOT be called with a `Size` argument.

A conforming implementation SHALL reject programs that treat nominally distinct types as interchangeable.

---

### 4.3.2 Primitive Type Identity

Primitive type names are distinct names in the Hopper type system. Each keyword names a distinct type:

- `int` and `unsigned int` are distinct types (same LLVM representation, different signedness).
- `byte` and `unsigned byte` are distinct types (same LLVM representation, different signedness).
- `bool` and `bit` are distinct types (same LLVM representation `i1`, different intended semantics).
- `float` is its own type with no unsigned or signed variant.
- `address` and `string` are the same underlying LLVM type (`i8*`) but are distinct named types in the Hopper source type system — with one explicit exception (§4.3.3).
- `string` and `string[]` are distinct types.

Arithmetic and comparison operators are defined for specific type pairs. Using a value of the wrong primitive type where another primitive type is required is a type error even when both types have the same LLVM representation.

---

### 4.3.3 The byte/char Identity

`char` is an alias for `byte`. These two names refer to the same type. A conforming implementation SHALL treat `char` and `byte` as identical for all purposes: assignment, parameter passing, return values, and operator application.

The alias exists for programmer readability: `char` signals intent to represent a character value; `byte` signals intent to represent a raw octet. At the type-checking level, no distinction is made.

```hopper
byte  b = 65
char  c = b         // permitted: char and byte are the same type
byte  b2 = c        // permitted: same type
```

This is the only aliasing relationship in the primitive type system.

---

### 4.3.4 The address/string Representation Overlap

`address` and `string` are distinct named types in Hopper's source-level type system. They are not aliases and they are not the same type. However, because both compile to `i8*` in LLVM IR, Hopper permits direct assignment between them without a cast.

This exception is representational, not nominal: the compiler permits the assignment because it would be a no-op at the LLVM level, not because the types are considered identical.

```hopper
string s   = "hello"
address p  = s          // permitted: both are i8*
string s2  = p          // permitted: both are i8*
```

Despite this assignment permission, `address` and `string` remain distinct types in all other respects. A function declared to accept `string` does not accept `address` as a parameter, and vice versa. A conforming implementation MAY relax this restriction for parameter passing as well, given the identical LLVM representation, but is not required to do so.

---

### 4.3.5 Template Instantiation Identity

Each distinct combination of type arguments to a template produces a distinct type. Two instantiations of the same template with different type arguments are not the same type, even if the instantiated layouts happen to be structurally identical.

```hopper
List<int>  ints  = List<int>(64)
List<byte> bytes = List<byte>(64)

// The following is a type error:
ints = bytes    // ERROR: List<int> and List<byte> are distinct types
```

The name of a template instantiation is the template name combined with its type arguments. Two instantiations with the same type arguments in the same order are the same type.

---

### 4.3.6 Signedness Is Part of Type Identity

For integer types, signedness is part of the type name and therefore part of type identity.

- `int` and `unsigned int` are different types.
- `byte` and `unsigned byte` are different types.

Mixing signed and unsigned types in arithmetic or assignment without an explicit cast is a type error.

```hopper
int n          = 100
unsigned int u = 200
int result     = n + u      // ERROR: type mismatch, int + unsigned int
```

---

### 4.3.7 Interface Types Are Not Value Types

Interface names do not denote value types. A variable MUST NOT be declared with an interface type. Interfaces define behavioral specifications; they have no storage representation.

```hopper
interface IO {
    function read(int fd, address buf, int count) int
    function write(int fd, address buf, int count) int
}

IO handle    // ERROR: IO is not a value type; cannot declare a variable of interface type
```

Interface identity matters for conformance checking: a class either does or does not implement a specific named interface. If a class declares `satisfies IO`, the compiler checks conformance against the interface named `IO`. It does not accept any structurally equivalent interface under a different name.

---

### 4.3.8 Type Names Are Scoped to the File

User-defined type names (structs, classes, templates, bitfields, enums) are visible within the file where they are declared and in any file that imports the declaring file. Two user-defined types with the same name declared in two different files are distinct types unless they are the same declaration (i.e., the same file imported via the module system).

A conforming implementation SHALL NOT silently unify types with the same name that originate from different source files. If a name collision would be ambiguous, the implementation SHALL emit a diagnostic.
