## 4.1 Primitive Types

Primitive types are the scalar value types built into the Hopper language. Each primitive type has a fixed, platform-independent LLVM representation. A conforming implementation SHALL compile each primitive type to exactly the LLVM type specified in this section.

---

### 4.1.1 int

**LLVM type:** `i64`
**Signedness:** signed
**Width:** 64 bits
**Range:** −2^63 to 2^63−1 (−9,223,372,036,854,775,808 to 9,223,372,036,854,775,807)

`int` is the default integer type in Hopper. Integer literals appearing in source code are typed as `int` unless the declaration context specifies otherwise. Arithmetic on `int` values uses signed LLVM instructions: `add`, `sub`, `mul`, `sdiv`, `srem`, `ashr`.

Integer overflow on signed `int` is undefined behavior per LLVM semantics. A conforming implementation MAY assume that signed integer overflow does not occur and optimize accordingly. Programs that depend on signed overflow wrapping are not conforming.

```hopper
int count = 0
int offset = -1
int limit  = 2147483647
```

---

### 4.1.2 byte

**LLVM type:** `i8`
**Signedness:** signed
**Width:** 8 bits
**Range:** −128 to 127

`byte` is used for raw memory access, small signed integer arithmetic, and character values where the signed interpretation matters. Reading a memory location via `address::value` yields a `byte`. Writing to a memory location via `address::value = expr` requires the right-hand side to be of type `byte`.

```hopper
address p = buf + i
byte b = p::value          // read one byte from memory
byte next = b + 1          // arithmetic on byte values
```

When a `byte` value is used in integer arithmetic alongside an `int`, the programmer MUST explicitly widen it using `cast<int>` or the bare `int` cast-unary expression. No implicit widening occurs.

---

### 4.1.3 char

**LLVM type:** `i8`
**Signedness:** signed
**Width:** 8 bits
**Range:** −128 to 127

`char` is an alias for `byte` at the representation level. Both types compile to `i8`. The distinction is semantic: `char` is used when the programmer intends to represent a single ASCII character; `byte` is used when the programmer intends to represent a raw octet or a small signed integer.

A conforming implementation SHALL treat `char` and `byte` as the same type for the purpose of type checking. A value of type `char` MAY be assigned to a variable of type `byte` and vice versa without a cast.

Character literals (e.g., `'A'`) have type `char`.

```hopper
char c = 'A'
byte b = c          // permitted: char and byte are the same type
int n = int (c)     // explicit widening required for int arithmetic
```

---

### 4.1.4 bool

**LLVM type:** `i1`
**Values:** `true` (1) and `false` (0)
**Width:** 1 bit

`bool` is the type of boolean values. The only valid `bool` literals are `true` and `false`. Comparison operators (`==`, `!=`, `<`, `<=`, `>`, `>=`) and logical operators (`&&`, `||`, `!`) produce `bool` results.

`bool` is not an integer type in Hopper. A `bool` value MUST NOT be used where an `int` is expected without an explicit conversion. An integer value MUST NOT be used where a `bool` is expected without an explicit conversion.

```hopper
bool found = false
bool ready = count > 0
if (found) { ... }          // condition is bool
if (found == true) { ... }  // also valid but redundant
```

The conditional expression in `if`, `while`, and `for` MUST have type `bool`. A conforming implementation SHALL reject programs where the condition has a non-bool type.

---

### 4.1.5 float

**LLVM type:** `double`
**Standard:** IEEE 754 binary64
**Width:** 64 bits
**Range:** approximately ±1.8 × 10^308
**Precision:** approximately 15–17 significant decimal digits

`float` in Hopper is a 64-bit double-precision floating-point value. There is no single-precision float type. The name `float` maps to LLVM `double`.

Floating-point literals (e.g., `3.14`, `0.5`, `1.0`) have type `float`. Arithmetic on `float` uses IEEE 754 instructions: `fadd`, `fsub`, `fmul`, `fdiv`, `frem`. Comparisons use `fcmp` with ordered predicates by default.

NaN behavior follows IEEE 754: any comparison involving NaN (except `!=`) returns `false`. `cast<bool>(float)` uses `fcmp une` (unordered not-equal to zero), so NaN is truthy.

```hopper
float pi    = 3.14159265358979
float ratio = 1.0 / 3.0
```

---

### 4.1.6 address

**LLVM type:** `i8*`
**Interpretation:** raw, untyped pointer to memory
**Width:** platform word size (64 bits on x86-64 and AArch64)

`address` is an untyped memory pointer. It carries no information about the type or size of the data it points to. All pointer arithmetic is done in byte units: `buf + n` advances the pointer by `n` bytes.

Dereferencing an `address` produces a `byte`:

```hopper
address p = buf + i
byte b = p::value          // read one byte at address p
p::value = b               // write one byte to address p
```

`address` arithmetic bypasses the type system. The compiler does not track what type of data an `address` points to. Dereferencing a null or dangerously invalid address is undefined behavior (§4.10).

`address` and `string` are the same LLVM type (`i8*`). Assignment between them is permitted without a cast (§4.8).

The special expression `identifier::address` produces the address of a variable or function as an `address` value. The expression `identifier::value` dereferences an `address` variable and produces a `byte`.

---

### 4.1.7 string

**LLVM type:** `i8*`
**Encoding:** null-terminated UTF-8 byte sequence

`string` is a pointer to the first byte of a null-terminated sequence of UTF-8 bytes. The pointer itself is the same LLVM type as `address` (`i8*`). The compiler treats `string` as semantically distinct from `address`: a `string` is intended to point to human-readable text, while an `address` is a raw pointer.

String literals have type `string`:

```hopper
string greeting = "hello, world"
string path     = "/etc/hosts"
```

Hopper has no built-in string length operation, string concatenation, or string comparison. These operations are provided by library modules. The raw null-terminated pointer is the only built-in string concept.

Because `string` and `address` share the same LLVM representation, assignment between them is permitted without an explicit cast (§4.8). A `string` MAY be assigned to an `address` variable, and an `address` MAY be assigned to a `string` variable.

---

### 4.1.8 string[]

**LLVM type:** `i8**`
**Interpretation:** array of string pointers

`string[]` is the type of the `argv` parameter in an entry point:

```hopper
entry main(int argc, string[] argv) {
    string path = argv[1]
}
```

`string[]` is not a general array type. It cannot be declared as a local variable type. Its only valid use is as the type of the second parameter in an `entry` declaration. A conforming implementation SHALL reject `string[]` in any other position.

The expression `argv[i]` has type `string`. Indexing is not bounds-checked.

---

### 4.1.9 unsigned int

**LLVM type:** `i64`
**Signedness:** unsigned
**Width:** 64 bits
**Range:** 0 to 2^64−1 (0 to 18,446,744,073,709,551,615)

`unsigned int` is the unsigned 64-bit integer type. It shares the same LLVM type as `int` (`i64`) but the compiler emits unsigned arithmetic instructions: `udiv`, `urem`, `lshr`, and unsigned comparison predicates (`ule`, `uge`, `ult`, `ugt`).

Unsigned integer overflow wraps modulo 2^64. This is defined behavior for `unsigned int`. Programs MAY rely on unsigned wrapping.

```hopper
unsigned int size = 0
unsigned int mask = 0xFFFFFFFF
```

Arithmetic mixing `int` and `unsigned int` is not permitted without an explicit cast. The programmer MUST convert between them explicitly.

---

### 4.1.10 unsigned byte

**LLVM type:** `i8`
**Signedness:** unsigned
**Width:** 8 bits
**Range:** 0 to 255

`unsigned byte` is the unsigned 8-bit integer type. It shares the same LLVM type as `byte` (`i8`) but unsigned arithmetic instructions apply. Unsigned byte overflow wraps modulo 256.

`unsigned byte` is used when raw byte values must be treated as non-negative quantities — for example, when decoding binary protocols or comparing byte values against unsigned thresholds.

```hopper
unsigned byte octet = 200
```

---

### 4.1.11 bit

**LLVM type:** `i1`
**Width:** 1 bit
**Values:** 0 and 1

`bit` is a 1-bit integer type. It is distinct from `bool` in intended semantics: `bool` is a logical predicate, while `bit` is a hardware-level binary value used in bitfield declarations and bit-manipulation contexts.

`bit` is valid as a field type inside a `bitfield` declaration (§4.2.4). It is also valid as a local variable type when a programmer needs to work with single-bit values from bitfield extractions.

```hopper
bitfield Flags {
    bit carry
    bit zero
    bit overflow
    bit sign
    bit[4] reserved
}
```

---

### 4.1.12 void

`void` denotes the absence of a value. It is not a storage type. `void` is valid only as the return type of a function declaration. It is not valid as a variable type, a parameter type, a field type, or in any expression position.

The preferred way to express a void-returning callable in Hopper is the procedure form (no return type annotation). The `void` keyword is accepted by the grammar as a type token to support certain interface declarations, but programs SHOULD use the procedure form for all normal function declarations that produce no value.

```hopper
function cleanup(address buf) {     // procedure form — preferred
    deallocate buf
}
```
