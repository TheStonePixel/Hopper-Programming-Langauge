## 4.5 Generics

Hopper provides compile-time parameterization through two mechanisms: template classes and template functions. Both are fully monomorphized at compile time — each unique combination of type arguments produces a distinct, independently compiled artifact. There is no runtime polymorphism, no type erasure, and no shared generic code at the LLVM IR level.

---

### 4.5.1 Template Classes

A template class is a class declaration parameterized by one or more type variables. Template classes are declared with the `template` keyword.

**Syntax:**

```hopper
template Name<T> {
    // class body — fields and methods
}

template Name<T, U> {
    // multi-parameter template
}
```

**Free type parameters:**

A free type parameter is a single identifier (conventionally `T`, `K`, `V`, `E`, etc.). It is a placeholder that is substituted with a concrete type at the instantiation site.

```hopper
template Stack<T> {
    address data
    int     top
    int     capacity

    constructor(int cap) {
        self.data     = allocate cap
        self.top      = 0
        self.capacity = cap
    }

    destructor() {
        deallocate self.data
    }

    function push(T value) {
        // store value at top
        self.top = self.top + 1
    }

    function pop() T {
        self.top = self.top - 1
        // return value at top
    }

    function isEmpty() bool {
        return self.top == 0
    }
}
```

Instantiation:

```hopper
Stack<int>  intStack  = Stack<int>(64)
Stack<byte> byteStack = Stack<byte>(128)
```

Each instantiation (`Stack<int>`, `Stack<byte>`) is a distinct type with a distinct LLVM struct type and distinct method implementations.

**Fixed type parameters:**

A fixed type parameter is a concrete primitive type keyword used directly as the template parameter. Fixed-parameter templates are fully monomorphized at declaration time. The instantiated type does not require `<>` at use sites — the name alone refers to the monomorphic type.

```hopper
template IntList<int> {
    address data
    int length

    constructor(int cap) {
        self.data   = allocate cap
        self.length = 0
    }

    function append(int value) {
        // ...
    }
}
```

Use site:

```hopper
IntList list = IntList(64)     // no <int> required — fixed-param template
```

Fixed-parameter templates are useful when a type needs to specialize for a particular primitive type without being a general generic.

---

### 4.5.2 Template Instantiation Rules

A conforming implementation SHALL enforce the following rules for template class instantiation:

- All type parameters MUST be provided at the instantiation site for free-parameter templates. Partial instantiation is not supported.
- Each unique combination of type arguments produces a distinct type, even if the resulting field layout is structurally identical to another instantiation.
- Template instantiation is eager: the full class body is compiled for each unique instantiation. There is no lazy or deferred instantiation.
- Circular instantiation (a template that directly or indirectly instantiates itself with the same type arguments) is not permitted and SHALL produce a compile-time error.
- Template classes MUST be declared at the top level. Template class declarations inside function bodies, method bodies, or other class bodies are not permitted.

---

### 4.5.3 Template Functions

A template function is a function parameterized by a target type. Template functions are declared with the `template function` keyword sequence. Each unique `(targetType, argumentTypes)` combination is a distinct function specialization compiled to a distinct function in LLVM IR under a mangled name.

**Syntax:**

```hopper
template function name<targetType>(params) returnType {
    body
}

template function name<targetType>(params) {
    body   // procedure form
}
```

The `<targetType>` is a type parameter that appears in the function signature or body. It is substituted with a concrete type at the call site.

**Example — the cast module:**

```hopper
template function cast<int>(float src) int {
    return int src
}

template function cast<int>(byte src) int {
    return int src
}

template function cast<float>(int src) float {
    return float src
}

template function cast<bool>(int src) bool {
    return bool src
}
```

Each declaration is a separate specialization. The compiler selects the correct specialization based on the explicit type argument and the inferred argument type at the call site.

**Call site:**

```hopper
int n    = cast<int>(3.9)       // selects cast<int>(float src)
int m    = cast<int>(b)         // selects cast<int>(byte src)
float f  = cast<float>(7)       // selects cast<float>(int src)
bool ok  = cast<bool>(count)    // selects cast<bool>(int src)
```

---

### 4.5.4 Template Function Mangling

Each specialization of a template function is compiled to a distinct function in LLVM IR under a mangled name that encodes the function name, the target type, and the argument types.

The mangled name convention is:

```
__tmpl_<functionName>__<targetType>__<argType1>__<argType2>...
```

Examples:

| Call expression           | Mangled name                            |
|---------------------------|-----------------------------------------|
| `cast<int>(float)`        | `__tmpl_cast__int__float`               |
| `cast<int>(byte)`         | `__tmpl_cast__int__byte`                |
| `cast<int>(address)`      | `__tmpl_cast__int__address`             |
| `cast<float>(int)`        | `__tmpl_cast__float__int`               |
| `cast<bool>(float)`       | `__tmpl_cast__bool__float`              |
| `cast<address>(int)`      | `__tmpl_cast__address__int`             |

The call `cast<int>(3.9)` at the call site compiles to a direct call to `__tmpl_cast__int__float`. After inlining, no call overhead remains — the emitted IR is a single LLVM conversion instruction.

---

### 4.5.5 Template Function Specialization Selection

At a template function call site, the compiler selects a specialization as follows:

1. The target type is read from the explicit `<type>` argument at the call site.
2. The types of the actual arguments are determined from the call expression.
3. The compiler searches all declared specializations of the function for a declaration whose `<targetType>` matches the call's target type and whose parameter types match the actual argument types exactly.
4. If exactly one matching specialization is found, that specialization is called.
5. If no matching specialization is found, the program is ill-formed and the compiler SHALL emit a diagnostic.
6. If more than one matching specialization is found (ambiguous), the program is ill-formed and the compiler SHALL emit a diagnostic.

Type matching is exact: no implicit coercion is applied during specialization selection. If the argument is of type `byte` and no specialization has a `byte` parameter, the call is ill-formed even if there exists a specialization with an `int` parameter.

---

### 4.5.6 Template Classes versus Template Functions

| Property                         | Template class          | Template function              |
|----------------------------------|-------------------------|--------------------------------|
| Keyword                          | `template Name<T>`      | `template function name<T>`    |
| Type parameter position          | After class name        | Between function name and `(`  |
| Produces                         | A type                  | A callable                     |
| Instantiation site               | Variable declaration    | Call expression                |
| Target type specified explicitly | Yes (`Name<int>`)       | Yes (`cast<int>(x)`)           |
| Each unique args combination     | Distinct struct type    | Distinct function               |
| Mangling                         | `%Name_T`               | `__tmpl_name__T__ArgType`      |

---

### 4.5.7 Constraints on Template Bodies

Template bodies are compiled independently for each instantiation. The compiler MUST verify that the body is type-correct under the substituted types. If a template body uses an operation on the type parameter that is not defined for a specific instantiation, the program is ill-formed for that instantiation.

Template functions MUST NOT be called with type arguments for which no specialization has been declared. There is no implicit instantiation of unspecialized template bodies — every specialization that may be called MUST be explicitly declared.

---

### 4.5.8 No Higher-Kinded Types

Hopper templates do not support higher-kinded types. A template parameter is always a concrete type (e.g., `int`, `byte`, `MyClass`) — it cannot itself be a template (e.g., `List<T>` is not valid as a template parameter). Template parameters that are themselves generic types are not supported.
