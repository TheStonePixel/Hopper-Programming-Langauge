## 4.2 Composite Types

Composite types are types that aggregate other types into a named structure. Hopper provides six composite type forms: `struct`, `class`, `template`, `bitfield`, `enum`, and `callback`. Each form has distinct semantics and distinct restrictions on where and how it may be used.

---

### 4.2.1 struct

A `struct` defines a named, fixed-size aggregate of typed fields with no associated behavior. Structs are memory-layout types: their only purpose is to describe how data is arranged in memory.

**Declaration syntax:**

```hopper
struct Packet {
    int  length
    int  checksum
    address data
    pad 4
}
```

**Rules:**

- A `struct` declaration MUST contain only field declarations (`type fieldName`) and padding directives (`pad N`).
- A `struct` MUST NOT contain methods, constructors, destructors, or operators.
- A `struct` MUST NOT implement an interface.
- Fields are laid out in declaration order. No padding is inserted automatically by the compiler unless explicitly specified with `pad N`.
- `pad N` inserts `N` bytes of unnamed padding at that position in the layout. This is used to match hardware register maps or C ABI layouts exactly.
- A `struct` with no fields has implementation-defined size.

**Field access:**

```hopper
Packet pkt
pkt.length   = 100
pkt.checksum = 0xDEAD
pkt.data     = buf
```

**LLVM representation:**

A struct named `S` with fields of types `T1, T2, ..., Tn` compiles to an LLVM named struct type: `%S = type { <T1_llvm>, <T2_llvm>, ..., <Tn_llvm> }`. Padding bytes are represented as `i8` array fields. The LLVM struct is not packed by default; however, because Hopper inserts explicit padding rather than relying on the compiler, the layout is fully deterministic.

---

### 4.2.2 class

A `class` defines a named aggregate of typed fields combined with associated methods. Classes are the primary abstraction mechanism in Hopper for encapsulating state and behavior together.

**Declaration syntax:**

```hopper
class Buffer {
    address data
    int     length
    int     capacity

    constructor(int cap) {
        self.data     = allocate cap
        self.capacity = cap
        self.length   = 0
    }

    destructor() {
        deallocate self.data
    }

    function append(byte b) {
        address p = self.data + self.length
        p::value  = b
        self.length = self.length + 1
    }

    function get(int i) byte {
        address p = self.data + i
        return p::value
    }
}
```

**Rules:**

- A class body MAY contain field declarations, a `constructor`, a `destructor`, instance methods, and operator overloads.
- A class MUST have at most one `constructor` and at most one `destructor`.
- The constructor is called when an instance is created with the class-name-as-function-call syntax: `Buffer buf = Buffer(1024)`.
- The destructor MUST be called explicitly by the programmer: `obj.destructor()`. Hopper has no automatic destructor invocation — no garbage collection, no RAII. The `deallocate` statement only frees raw memory (it calls `free()`); it does NOT invoke a class destructor. The programmer is responsible for calling the destructor before releasing the last reference to an object.
- Instance methods receive `self` as an implicit first parameter. Within a method body, field access MUST use the `self.field` form. Bare `field` is not in scope.
- Class fields are not directly accessible from outside the class. Accessing a class instance's field from outside its method bodies is a compile-time error: "Access failure: cannot access field of class directly. Use an accessor method." Programmers MUST expose fields through explicit accessor methods. (Structs do not have this restriction — struct fields are freely accessible.)
- Methods are compiled as free functions under a mangled name: `ClassName_methodName(self, arg1, arg2, ...)`.
- Operator overloads use the `operator` keyword with a recognized symbol. Overloaded operators are called using the operator syntax at the call site.

**Method calls:**

```hopper
Buffer buf = Buffer(1024)
buf.append(65)
byte b = buf.get(0)
```

All method calls require an explicit receiver. `append(65)` without a receiver is a free function call, not a method call on the current instance. There is no `this.method()` shorthand inside method bodies — `self.method()` is always required.

**Interface conformance:**

A class MAY declare that it satisfies one or more interfaces:

```hopper
class LinuxIO satisfies IO {
    function read(int fd, address buf, int count) int {
        return read(fd, buf, count)
    }
    function write(int fd, address buf, int count) int {
        return write(fd, buf, count)
    }
}
```

See §4.6 for interface conformance rules.

**LLVM representation:**

A class `C` with fields `T1, T2, ..., Tn` compiles to `%C = type { <T1_llvm>, ..., <Tn_llvm> }`. There is no vtable, no type tag, and no runtime type information. Method calls resolve statically at compile time to the mangled free function name.

### 4.2.3 template (parameterized class)

A `template` declaration defines a class parameterized by one or more type variables. Templates are instantiated at compile time by substituting concrete types for the type parameters.

**Free type parameter — generic class:**

```hopper
template List<T> {
    address data
    int     length
    int     capacity

    constructor(int cap) {
        self.data     = allocate cap
        self.length   = 0
        self.capacity = cap
    }

    function push(T value) {
        // ...
    }

    function get(int i) T {
        // ...
    }
}
```

A free type parameter (a single identifier like `T`, `K`, `V`) is a placeholder substituted at the instantiation site. Each unique combination of type arguments produces a distinct, independently compiled class.

**Fixed type parameter — monomorphic specialization:**

```hopper
template List<int> {
    // ... specialized implementation for int
}

template List<byte> {
    // ... specialized implementation for byte
}
```

When the template parameter is a concrete primitive type keyword (`int`, `byte`, `char`, `float`, `bool`, `string`, `address`, `unsigned int`, `unsigned byte`), the template is fully monomorphized immediately at the declaration site. The instantiated type name does not require `<>` at use sites.

**Instantiation syntax:**

```hopper
List<int> nums = List<int>(64)
List<byte> bytes = List<byte>(256)
```

**Rules:**

- Template parameters MUST be either a single free identifier (free type variable) or a concrete primitive type keyword (fixed specialization).
- Multiple template parameters are permitted: `template Map<K, V> { ... }`.
- Template classes MUST NOT be used without providing all type arguments (no partial application).
- Two instantiations with different type arguments are distinct types and are not assignment-compatible with each other.
- Templates do not support default type arguments.

**LLVM representation:**

Each unique instantiation `Name<T1, T2>` compiles to a distinct LLVM named struct `%Name_T1_T2` with the type parameters substituted. There is no shared code between instantiations.

### 4.2.4 bitfield

A `bitfield` declaration defines a bit-packed data structure where each field occupies a specified number of bits. Bits are packed sequentially from the least significant bit (LSB) of the containing word.

**Declaration syntax:**

```hopper
bitfield EFlags {
    bit     carry
    bit     reserved1
    bit     parity
    bit     reserved2
    bit     auxcarry
    bit     reserved3
    bit     zero
    bit     sign
    bit     trap
    bit     interrupt
    bit     direction
    bit     overflow
    bit[4]  iopl
    bit     nt
    bit[17] reserved4
}
```

**Rules:**

- Field types in a bitfield MUST be `bit`, `int`, `byte`, or `unsigned byte`. The field type specifies the signedness and arithmetic interpretation; the `[N]` suffix specifies the number of bits consumed in the packed layout.
- The `bit fieldName` form (without `[N]`) declares a single-bit field, equivalent to `bit fieldName[1]`.
- The `bit[N] fieldName` form declares an N-bit field. The value is extracted and sign- or zero-extended to the declared type.
- `pad N` inserts N bits of unnamed padding at that position.
- Fields are packed from LSB to MSB within each word.
- The total size of the bitfield is the minimum number of bytes required to hold all declared bits, rounded up to the next byte boundary.
- A conforming implementation SHALL lay out bitfield fields in exactly the declared order from LSB to MSB.

**LLVM representation:**

The compiler generates load/store operations with bitwise masks and shifts to extract and insert individual fields. The containing word is typed as the smallest integer type that can hold all bits (`i8`, `i16`, `i32`, or `i64`).

### 4.2.5 enum

An `enum` declaration defines a named set of integer or string constants. Enum values are referenced as `EnumName.VariantName`.

**Declaration syntax:**

```hopper
enum Color {
    Red   = 0
    Green
    Blue
}

enum Key {
    NONE  = -1
    CHAR  = 0
    ENTER
    BACKSPACE
    ARROW_UP
    ARROW_DOWN
    ARROW_LEFT
    ARROW_RIGHT
}

enum HttpMethod {
    GET    = "GET"
    POST   = "POST"
    PUT    = "PUT"
    DELETE = "DELETE"
}
```

**Rules:**

- Enum variants MAY have explicit integer values (decimal or hexadecimal), explicit negative integer values, or explicit string values.
- If a variant has no explicit value, its value is the predecessor's value plus one. The first variant without an explicit value has value 0.
- All variants in an enum MUST be of the same kind: either all integer-valued or all string-valued. Mixing integer and string variants in a single enum is not permitted.
- Integer-valued enum variants have type `int`. String-valued enum variants have type `string`.
- Enum names are accessed via the `EnumName.VariantName` syntax. Bare variant names are not in scope.
- Enums MUST be declared at top level or inside an interface body. Enums MUST NOT be declared inside function bodies, class bodies, or struct bodies.
- An enum is not a distinct type — `Color.Red` is an `int` value, not a `Color` value. There is no mechanism to restrict a variable to only holding values from a specific enum.

**LLVM representation:**

Enum variants are compile-time integer constants. References to `EnumName.VariantName` are replaced by the literal integer value at compile time. No enum type or symbol table is emitted into LLVM IR.

### 4.2.6 callback (function pointer type)

`callback` is the function pointer type in Hopper. It is declared as part of a variable or parameter declaration, not as a standalone named type.

**Syntax:**

```hopper
callback identifier = functionName(paramType1, paramType2) returnType
```

The `callback` declaration binds `identifier` to a function pointer that accepts the specified parameter types and returns the specified return type.

**Example:**

```hopper
callback compare = compareInts(int, int) bool
callback handler = onEvent(int, address) int
```

**Rules:**

- The `callback` declaration MUST appear inside a function or method body. It is a statement, not a top-level declaration.
- The function named on the right-hand side MUST exist and its signature MUST match the declared parameter types and return type exactly.
- A `callback` variable may be passed as a parameter whose type is declared as `callback(T1, T2, ...) R`.
- Calling a callback variable uses the normal function call syntax: `compare(a, b)`.
- Callbacks MUST NOT be null at the point of call. Calling a null callback is undefined behavior.
- Function pointer types are expressed in parameter lists as `callback(paramTypes...) returnType paramName`:

```hopper
function sort(address arr, int len, callback(int, int) bool lessThan) {
    // ...
}
```

**LLVM representation:**

A `callback` compiles to an LLVM function pointer type. The exact LLVM type is the function signature derived from the declared parameter types and return type. Calls through a callback variable compile to an indirect call instruction (`call` through a pointer).
