## 5.4 Stack Memory

### 5.4.1 Overview

Local variables in Hopper are allocated on the **call stack**. Stack allocation is implicit: declaring a local variable causes the compiler to reserve space for it in the current function frame. There is no explicit stack-allocation syntax beyond variable declaration.

The stack grows and shrinks automatically as functions are called and return. The programmer has no direct control over stack layout, stack size, or the order in which variables are placed within a frame.

---

### 5.4.2 Stack Allocation Mechanism

The Hopper compiler lowers local variable declarations to LLVM `alloca` instructions. Each `alloca` reserves a contiguous region of bytes in the current function's stack frame, aligned to the variable's natural alignment (see §5.9).

The compiler MAY hoist `alloca` instructions to the entry block of the function, regardless of where in the source the variable is declared. Programs MUST NOT rely on `alloca` being executed at the textual point of declaration.

```hopper
function sum(int a, int b) int {
    int result = a + b    // result is an alloca'd i64 on the stack
    return result
}
```

---

### 5.4.3 No Explicit Stack Allocation Syntax

Hopper provides **no syntax** for explicitly allocating stack memory beyond variable declaration. There is no `alloca` keyword, no stack-frame control directive, and no mechanism for dynamically sized stack allocation. All stack-allocated storage has a size that is known at compile time.

Use `allocate` for dynamically sized storage (see §5.5).

---

### 5.4.4 Stack-Allocated Arrays

An array with a compile-time constant size MAY be declared as a local variable. Such an array is allocated entirely on the stack.

```hopper
function process() {
    byte buf[256]          // 256 bytes on the stack, zero overhead
    int  indices[64]       // 512 bytes on the stack (64 × 8 bytes each)
    // use buf and indices ...
}
```

The element type and size MUST be compile-time constants. Variable-length arrays (VLAs) are **not supported**. A conforming implementation SHALL reject any array declaration whose size is not a compile-time integer constant.

The array occupies contiguous bytes in the stack frame. Element `i` starts at offset `i * element_size` from the base address of the array.

---

### 5.4.5 Taking the Address of a Stack Variable

The address of a stack-allocated variable is obtained with the `::address` qualifier:

```hopper
function example() int {
    int x = 10
    address p = x::address    // p points to x on the stack
    // ... use p ...
    return x
}
// After return: p is dangling. x no longer exists.
```

An `address` obtained from a stack variable is valid only within the scope where that variable is live. Allowing such an address to escape the enclosing scope — by returning it, storing it in a heap allocation, or passing it to a long-lived data structure — creates a dangling pointer. The compiler SHALL NOT diagnose this. The programmer MUST ensure stack addresses do not escape their variable's scope.

---

### 5.4.6 Stack Frame Size

The stack frame size for each function is determined by the compiler from the set of local variable declarations and compiler temporaries. The programmer CANNOT directly specify or query the frame size from within the program.

A conforming implementation MAY optimize the stack frame (e.g., by reusing stack slots for variables with non-overlapping live ranges), provided the observable behavior of the program is preserved.

---

### 5.4.7 Stack Overflow

Hopper makes **no guarantee** about behavior when the call stack is exhausted. Stack overflow is **undefined behavior**. Causes include:

- Unbounded recursion.
- Functions with very large local arrays.
- Deeply nested function calls on platforms with limited stack size.

A conforming implementation MAY install a guard page or signal handler to detect stack overflow, but it is NOT REQUIRED to do so. Programs that depend on stack overflow detection are non-portable.

Programs with deep recursion SHOULD be restructured to use heap allocation or explicit stacks when the recursion depth cannot be statically bounded.

---

### 5.4.8 Interaction with Inline Assembly

Inline `asm {}` blocks may read from and write to stack-allocated variables by loading their addresses. The implementation SHALL NOT move a stack variable to a register in a way that makes `var::address` incorrect within an `asm {}` block that uses that address. Implementations MUST ensure that stack variable addresses remain stable across inline assembly.

```hopper
function syscallResult() int {
    int result
    asm {
        rax = 39       // getpid
        syscall
        result = rax
    }
    return result
}
```

In the above example, `result` is stack-allocated; the `asm {}` block writes to it via the `result = rax` assignment. The implementation MUST ensure this write reaches the stack slot that `result` occupies.
