## 5.6 Memory Regions

### 5.6.1 Overview

A Hopper program's address space is divided into four distinct memory regions. Each region has well-defined access semantics, lifetime rules, and management responsibilities. The regions are:

1. **Stack** — automatic storage for local variables and function frames.
2. **Heap** — dynamic storage managed via `allocate` and `deallocate`.
3. **Code segment** — read-only storage for compiled functions and string literals.
4. **MMIO** — memory-mapped hardware registers accessed via `bind` declarations.

Memory-mapped files are **not supported** in the current version of Hopper. Programs requiring file-backed memory mappings MUST use the `mmap` syscall directly via the `Memory` interface (see the `linux` module) and manage the resulting `address` values manually.

---

### 5.6.2 Stack Region

The stack is a contiguous region of memory managed automatically by the CPU and compiler. Each function call pushes a **frame** onto the stack; each return pops it.

**Properties:**

- Stack frames are created on function entry and destroyed on function return.
- The stack grows downward on x86-64 (toward lower addresses).
- Local variable declarations cause the compiler to emit `alloca` instructions that carve space from the current frame.
- Stack-allocated arrays (`type name[N]`) occupy contiguous bytes in the frame.
- The stack pointer is a CPU register (`rsp` on x86-64); the programmer MUST NOT modify it directly except within `asm {}` blocks that restore it before exit.

**Lifetime:** Stack variables are live from declaration to the end of their enclosing block. Addresses into the stack are invalid after the enclosing function returns or after the enclosing block exits.

**Access:** Read/write, with native alignment guaranteed by the compiler.

**Size:** Determined by the OS at thread/process creation. Exceeding the stack causes undefined behavior (see §5.4.7).

---

### 5.6.3 Heap Region

The heap is a region of dynamically managed memory provided by the OS and organized by the runtime allocator (`malloc`/`free`).

**Properties:**

- Heap memory is acquired via `allocate N` and released via `deallocate ptr`.
- Heap allocations are independent of any function frame; they persist across function calls.
- The allocator returns 8-byte-aligned pointers.
- Heap contents are uninitialized at allocation time.
- The heap may be fragmented; successive `allocate` calls do not necessarily return contiguous addresses.

**Lifetime:** From `allocate` to `deallocate`. Allocations that are never freed persist until process exit.

**Access:** Read/write. No bounds checking is performed by the implementation.

**Size:** Limited by available virtual and physical memory, plus swap. The implementation SHALL NOT impose an artificial upper bound on heap usage beyond OS constraints.

---

### 5.6.4 Code Segment

The code segment (also called the text segment) contains:

- Compiled machine code for all functions and entry points.
- String literals (e.g., `"hello"`) embedded at compile time.
- Read-only constant data emitted by the compiler.

**Properties:**

- The code segment is mapped read-only and execute-only by the OS loader. Writing to the code segment is **undefined behavior** and will typically cause a hardware protection fault.
- String literals have type `string` (which is `i8*`) and point into the code segment. Passing a string literal pointer to `deallocate` is **undefined behavior**.
- Functions are represented as `address` values that point into the code segment. Calling a non-function address is **undefined behavior**.

**Lifetime:** The entire program lifetime. Code segment addresses are always valid while the program is running.

**Access:** Read and execute only. Writes produce undefined behavior.

```hopper
string msg = "fatal error\n"   // msg points into the read-only code segment
// deallocate msg              -- undefined behavior; MUST NOT do this
```

---

### 5.6.5 MMIO Region

Memory-mapped I/O (MMIO) registers are hardware resources accessed at fixed physical addresses. MMIO is used in embedded and low-level firmware contexts where hardware peripherals expose control and data registers as memory locations.

**Declaration:** MMIO registers MUST be declared using the `bind` keyword, which is a Hopper language keyword reserved for hardware address mapping. MMIO declarations associate a name with a specific physical address.

**Properties:**

- MMIO accesses are **volatile**: every read MAY produce a different result, and every write MAY have an observable side effect beyond storing the value.
- A conforming implementation SHALL NOT cache, reorder, or elide reads and writes to MMIO addresses.
- MMIO addresses MUST NOT be passed to `deallocate`.
- MMIO addresses MUST NOT be used for general data storage; their behavior is hardware-defined.

**Access:** Volatile read/write. The exact access width (byte, word, doubleword, quadword) MUST match the hardware register's specification.

**Lifetime:** Hardware-defined. MMIO addresses are valid as long as the hardware peripheral is powered and mapped.

> **Note:** The `bind` keyword is reserved in the Hopper grammar for hardware address declarations. The socket syscall `bind(2)` is exposed as `socketBind` in the `linux` and `x86_64` modules to avoid the keyword conflict.

---

### 5.6.6 No Memory-Mapped File Support

Hopper's language-level memory model does **not** include built-in memory-mapped file semantics. The concepts of file-backed pages, demand paging, copy-on-write mappings, and `msync` are not part of the language.

Programs that require file-backed memory MUST:

1. Import the `Memory` interface from the `linux` module.
2. Call `mmap` directly to obtain an `address` pointing to the mapped region.
3. Manage the resulting address as an ordinary heap-like `address`, with the understanding that it is backed by a file rather than anonymous memory.
4. Call `munmap` explicitly to release the mapping.

```hopper
import Memory from linux

// Map 4096 bytes of a file:
int PROT_READ  = 1
int PROT_WRITE = 2
int MAP_SHARED = 1
address region = mmap(cast<address>(0), 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0)

// ... use region ...

munmap(region, 4096)
```

The `region` address behaves like any other `address` from the Hopper memory model perspective. Lifetime and validity rules are the same as for heap allocations, with the additional constraint that the underlying file descriptor `fd` MUST remain open for the duration of the mapping.

---

### 5.6.7 Region Summary

| Region | Managed by | Lifetime | Access | `deallocate` allowed? |
|--------|-----------|---------|-------|----------------------|
| Stack | Compiler / CPU | Block scope | R/W | No |
| Heap | Programmer | `allocate` to `deallocate` | R/W | Yes (exactly once) |
| Code segment | Linker / OS | Program lifetime | R/X | No |
| MMIO | Hardware | Hardware-defined | Volatile R/W | No |
