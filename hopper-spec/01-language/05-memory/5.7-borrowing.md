## 5.7 Borrowing

### 5.7.1 No Borrowing System

Hopper has **no borrowing system**. There are no shared references, no exclusive (mutable) references, no borrow checker, no lifetime-parameterized reference types, and no enforcement of aliasing rules at compile time.

An `address` value in Hopper is a raw machine pointer. The language imposes no restrictions on how many variables hold copies of the same address, whether those copies are read-only or read-write, or how long they are kept alive. All aliasing and validity responsibility belongs to the programmer.

---

### 5.7.2 Taking the Address of a Variable

The address of any named, stack-allocated variable is obtained with the `::address` qualifier:

```hopper
int x = 100
address p = x::address    // p holds the address of x
```

This is an explicit operation. There is no implicit address-of operator. The programmer must use `::address` to obtain a pointer; the compiler will not take addresses implicitly.

The resulting `address` is valid for as long as the underlying variable `x` is in scope. The compiler SHALL NOT warn when an address of `x` is assigned to a variable with a broader scope.

---

### 5.7.3 Aliasing

Multiple `address` values MAY refer to the same memory location simultaneously. Hopper imposes no restrictions on aliasing. Programs MAY hold any number of copies of an address and read or write through any of them at any time.

```hopper
int value  = 0
address p1 = value::address
address p2 = value::address   // p1 and p2 alias the same location

// Both of the following are legal:
p1::value = 1
p2::value = 2
```

Because there is no borrow checker, the compiler cannot reason about aliasing. This means:

- The compiler MAY assume that two `address` values of different provenance do not alias, for optimization purposes, consistent with the LLVM aliasing model used by the backend.
- Programs that violate the backend's aliasing assumptions (e.g., by casting unrelated values to `address` and writing through both) may encounter unexpected optimization behavior.
- When in doubt, use explicit volatile loads/stores or `asm {}` blocks to prevent reordering.

---

### 5.7.4 No Shared/Exclusive Reference Semantics

Hopper does not distinguish between "shared" references (readable by many) and "exclusive" references (writable by one at a time). An `address` value may be read through and written through by any code that holds a copy of it, with no enforcement.

This means concurrent access to the same address from multiple threads or interrupt contexts is not automatically safe. Race conditions are the programmer's responsibility. Hopper provides no memory ordering primitives in the core language; concurrent access MUST be serialized using platform-level mechanisms available through the OS interface (e.g., futex, spinlock) or hardware memory barriers in `asm {}` blocks.

---

### 5.7.5 No Compiler Enforcement of Pointer Rules

The compiler SHALL NOT enforce any of the following properties:

- That a pointer is not null before it is dereferenced.
- That a pointer is not dangling before it is dereferenced.
- That a pointer stays within the bounds of its originating allocation.
- That two concurrent accesses through aliases are properly synchronized.
- That an address taken from a local variable does not outlive that variable.

All of these properties are the programmer's responsibility. Violations are undefined behavior.

---

### 5.7.6 Conventions for Safe Address Passing

Although Hopper provides no enforcement mechanism, the following conventions SHOULD be followed to reduce the likelihood of invalid pointer use:

**Convention 1: Caller-scoped addresses.** When passing the address of a local variable to a function, the callee SHOULD NOT store that address beyond the duration of the call.

```hopper
function readValue(address ptr) int {
    return ptr::value          // reads through ptr; does not store it
}

function caller() {
    int x = 42
    int v = readValue(x::address)   // safe: readValue does not outlive x
}
```

**Convention 2: Heap addresses as long-lived references.** When long-lived references are needed (e.g., storing in a data structure), use heap-allocated memory rather than addresses of stack variables.

```hopper
address node = allocate 8    // heap allocation outlives any particular function call
// store node in a list, return it from a function, etc.
```

**Convention 3: Document ownership at API boundaries.** Functions that receive an `address` and are expected to call `deallocate` on it SHOULD document this explicitly. Functions that receive an `address` for read-only access SHOULD document that they do not free it.

---

### 5.7.7 Relationship to Future Language Evolution

The absence of a borrow system is a deliberate design choice, not an oversight. Hopper's design philosophy prioritizes explicit, visible resource management over compiler-enforced ownership. Should future versions of Hopper introduce ownership annotations, they will be additive — existing programs without annotations will continue to compile.
