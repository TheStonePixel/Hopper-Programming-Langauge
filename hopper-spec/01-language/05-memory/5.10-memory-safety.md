## 5.10 Memory Safety

### 5.10.1 Safety Posture

Hopper does **not** guarantee memory safety. A conforming Hopper implementation MUST NOT insert automatic bounds checks, null pointer guards, use-after-free detectors, or any other runtime memory-safety mechanism unless the programmer has explicitly requested them through the contract system (see §5.10.5).

This is a deliberate design choice. Hopper targets systems programming contexts — compilers, operating systems, embedded firmware, high-performance servers — where the overhead of automatic safety checks is unacceptable and where the programmer is expected to maintain correctness through discipline and explicit contracts.

---

### 5.10.2 Undefined Behavior Categories

The following memory operations constitute **undefined behavior** in Hopper. When a program invokes undefined behavior, the implementation may produce any result: correct output, incorrect output, a crash, or silent data corruption. The implementation is under no obligation to detect, diagnose, or handle undefined behavior.

#### Null pointer dereference

Reading or writing through an `address` whose value is zero (the null address):

```hopper
address p = cast<address>(0)
int x = p::value    // undefined behavior
```

#### Out-of-bounds access

Accessing memory outside the bounds of any allocation or stack variable:

```hopper
address buf = allocate 8
int x = (buf + 16)::value    // undefined behavior: 16 bytes past a 8-byte allocation
```

#### Use-after-free

Reading or writing through an `address` after the underlying allocation has been freed:

```hopper
address p = allocate 16
deallocate p
int x = p::value    // undefined behavior
```

#### Double-free

Calling `deallocate` on a pointer that has already been freed:

```hopper
address p = allocate 16
deallocate p
deallocate p    // undefined behavior
```

#### Dangling stack pointer

Using an `address` derived from a stack variable after that variable's scope has ended:

```hopper
address saved

function setPtr() {
    int x = 99
    saved = x::address    // saved will be dangling after setPtr returns
}

function usePtr() {
    setPtr()
    int v = saved::value    // undefined behavior: x no longer exists
}
```

#### Uninitialized read

Reading from a heap allocation before writing any value to it, and treating the result as meaningful:

```hopper
address buf = allocate 8
int x = buf::value    // undefined behavior: buf is uninitialized
```

#### Stack overflow

Exceeding the available stack space through excessive recursion or oversized stack frames (see §5.4.7).

---

### 5.10.3 No Built-in Safety Mechanisms

A conforming Hopper implementation provides **none** of the following safety mechanisms automatically:

- **Bounds checking:** No runtime check that array or pointer indices are within allocation bounds.
- **Null checking:** No automatic guard before pointer dereferences.
- **Temporal safety:** No use-after-free or double-free detection.
- **Leak detection:** No tracking of unfreed allocations.
- **Stack canaries:** No automatic stack corruption detection (though the underlying platform or linker may provide them independently).
- **Address space layout randomization:** Not a language concern; managed at the OS/linker level.
- **Safe integer arithmetic:** Integer overflow is not a memory-safety issue in Hopper; it is a separate category of undefined behavior.

Providing any of these mechanisms is the programmer's explicit responsibility.

---

### 5.10.4 Programmer Responsibility

In exchange for the absence of automatic safety mechanisms, the programmer accepts the following obligations:

1. Every `allocate` MUST be matched by exactly one `deallocate` at a point in the program where the allocation is still live and its value is no longer needed.
2. Every pointer arithmetic expression MUST be checked by the programmer to remain within the bounds of the originating allocation.
3. No `address` value derived from a stack variable MUST be allowed to outlive the variable's scope.
4. No `address` value MUST be used after the underlying allocation has been freed.
5. `deallocate` MUST be called at most once per allocation.
6. Heap memory MUST be initialized before it is read.
7. Null pointers MUST be checked before dereferencing.

These obligations SHOULD be documented at API boundaries using `requires`/`ensures` contracts wherever they are not mechanically obvious.

---

### 5.10.5 Contract-Based Safety

In safety-critical contexts, programs SHOULD use Hopper's contract system to document and enforce memory invariants. Contracts provide a lightweight, opt-in mechanism for expressing and checking preconditions and postconditions without imposing overhead on programs that do not require them.

**Preconditions (`requires`)** express what must be true before a function is called:

```hopper
function copyBytes(address dst, address src, int count) {
    requires cast<int>(dst) != 0
    requires cast<int>(src) != 0
    requires count > 0
    // ... implementation ...
}
```

**Postconditions (`ensures`)** express what will be true after a function returns:

```hopper
function makeBuffer(int size) address {
    ensures cast<int>(result) != 0
    address buf = allocate size
    return buf
}
```

**Constrain clauses** express loop invariants and other mid-body invariants:

```hopper
function scanBytes(address buf, int len, byte target) int {
    int i = 0
    while (i < len) {
        constrain i >= 0
        constrain i < len
        // ...
        i = i + 1
    }
    return -1
}
```

Contracts are checked at runtime when enabled by the build configuration. A conforming implementation MAY optimize out contract checks in release builds. Whether contracts are checked in release builds is a build-system decision, not a language guarantee.

---

### 5.10.6 External Safety Tools

Programs SHOULD be tested with external memory safety tools during development. Hopper's use of LLVM as a backend enables compatibility with:

- **AddressSanitizer (ASan):** Detects out-of-bounds accesses, use-after-free, and double-free at runtime with approximately 2x overhead. Enable via compiler flags.
- **MemorySanitizer (MSan):** Detects reads from uninitialized memory.
- **Valgrind Memcheck:** Detects a broad range of memory errors with higher overhead.

Using these tools during testing is strongly RECOMMENDED for any program that manipulates raw `address` values. They do not substitute for correct program logic, but they significantly increase the probability of catching errors before deployment.

---

### 5.10.7 Safety and Performance

The absence of automatic memory safety is not an incidental property — it is the mechanism by which Hopper achieves predictable, measurable performance. Every safety check has a cost: a branch, a memory access, or a cache miss. By making safety checks explicit and opt-in, Hopper ensures that programs pay only for the checks they need.

Programs in which safety is critical SHOULD add explicit checks at module boundaries (e.g., at the entry to public API functions) while omitting them in inner loops and hot paths where the invariants are already established by the calling context.

This layered approach — enforced at external interfaces, trusted internally — is consistent with Hopper's overall design philosophy: visibility and control at every level of the system, with no hidden costs.
