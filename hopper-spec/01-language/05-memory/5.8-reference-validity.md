## 5.8 Reference Validity

### 5.8.1 Definition

An `address` value is **valid** if all of the following conditions hold:

1. The address refers to a heap allocation that has been acquired via `allocate` and not yet freed via `deallocate`.
2. **OR** the address refers to a stack-allocated variable that is currently in scope (i.e., the variable has been declared and the enclosing block has not yet exited).
3. **OR** the address refers to a location within the read-only code segment (e.g., a string literal).
4. **OR** the address refers to a declared MMIO register and the hardware peripheral is operational.
5. The address, when taken as a base-plus-offset into an allocation, does not exceed the bounds of that allocation.
6. The address is not zero (null), unless null is explicitly treated as a sentinel rather than being dereferenced.

An `address` value that does not satisfy all applicable conditions is **invalid**. Any read or write operation through an invalid address is **undefined behavior**.

---

### 5.8.2 Validity Conditions in Detail

#### Condition: Heap allocation is live

A heap allocation is live from the moment `allocate` returns to the moment `deallocate` is called with the same base address. An address derived from a freed allocation is invalid regardless of whether the memory has been reused.

```hopper
address p = allocate 64
deallocate p
// p is now invalid. The following is undefined behavior:
// byte b = p::value
```

#### Condition: Stack variable is in scope

An address taken from a stack variable is valid only within the block where that variable is declared. The compiler does not track this at runtime; the programmer is solely responsible.

```hopper
address escape

function bad() {
    int x = 5
    escape = x::address    // escape will be invalid after bad() returns
}

function caller() {
    bad()
    // escape is now invalid. Reading escape::value is undefined behavior.
}
```

#### Condition: Pointer arithmetic within bounds

Address arithmetic that computes a pointer within an allocation is valid. Arithmetic that goes past the end of the allocation is **undefined behavior**, even if the resulting address happens to fall within another allocation.

```hopper
address buf = allocate 16
address p   = buf + 8      // valid: within the 16-byte allocation
address q   = buf + 16     // undefined behavior: one byte past the end
address r   = buf + 24     // undefined behavior: clearly out of bounds
```

The "one past the end" position (address `buf + 16` for a 16-byte allocation) is **undefined behavior** in Hopper. Unlike C, there is no guarantee that computing a one-past-end pointer is safe.

#### Condition: Non-null

An `address` value of zero is the conventional null pointer. Null is not a valid address for reading or writing. Dereferencing a null `address` is undefined behavior.

Programs MAY use zero as a sentinel value (a "no value" indicator), but MUST check for zero before dereferencing:

```hopper
function findNode(address list, int target) address {
    address cur = list
    while (cast<int>(cur) != 0) {
        // ... traverse ...
    }
    return cast<address>(0)   // conventional "not found"
}
```

---

### 5.8.3 Use of Invalid Addresses

Using an invalid address — whether by reading, writing, or passing it to `deallocate` — is **undefined behavior**. A conforming implementation:

- SHALL NOT be required to detect invalid address use at compile time.
- SHALL NOT be required to detect invalid address use at runtime.
- MAY produce any observable result when an invalid address is used, including:
  - Reading an arbitrary value.
  - Writing to memory belonging to another allocation.
  - Triggering a hardware fault (segfault / access violation).
  - Silently corrupting program state.
  - Executing arbitrary code (if a function pointer is overwritten).

There is no catchable exception for invalid address use. Programs that need to handle potential invalid addresses MUST validate them before use.

---

### 5.8.4 Validity Is Not Tracked at Runtime

The implementation SHALL NOT maintain a validity table, allocation map, or shadow memory that tracks the validity of every `address` value at runtime. Validity checking is entirely the programmer's concern.

If runtime validity tracking is required (e.g., for debugging), the programmer MUST implement it explicitly — for example, by maintaining a table of known-live allocations.

---

### 5.8.5 Interactions with Type Casting

Casting an integer to `address` via `cast<address>` produces an address with the numeric value of the integer. The implementation cannot determine whether this address is valid. The programmer is responsible for ensuring that a cast-from-integer address points to a live, accessible memory region before using it.

```hopper
int  physical = 0x40020000   // a hardware register address
address mmio  = cast<address>(physical)
// mmio may be valid (if the hardware is mapped) or invalid (if not)
// The compiler performs no check.
```

---

### 5.8.6 Validity and the `::value` Qualifier

Reading a value from an address using `::value` requires the address to be valid and correctly aligned for the type being read. Reading from a misaligned address may produce undefined behavior on targets that do not support unaligned access (though x86-64 permits unaligned loads at a performance cost). The implementation's behavior for misaligned access is platform-defined.

### 5.8.7 Summary Table

| Scenario | Valid? |
|---------|-------|
| Pointer to live heap allocation, within bounds | Yes |
| Pointer to live stack variable, within its scope | Yes |
| Pointer to string literal | Yes |
| Pointer to declared MMIO register | Yes (hardware-defined) |
| Pointer to freed heap allocation | No — undefined behavior |
| Pointer to stack variable after block exit | No — undefined behavior |
| Pointer past the end of an allocation | No — undefined behavior |
| Null pointer (address 0) dereferenced | No — undefined behavior |
| Pointer cast from arbitrary integer | Programmer's responsibility |
