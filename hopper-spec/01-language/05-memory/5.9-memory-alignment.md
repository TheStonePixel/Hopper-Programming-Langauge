## 5.9 Memory Alignment

### 5.9.1 Overview

Memory alignment governs how the compiler and runtime system place values in memory. Aligned accesses are required for correctness on some hardware architectures and improve performance on all architectures. The Hopper compiler delegates alignment decisions to LLVM, which selects natural alignments for all types based on the target ABI.

Programs MUST NOT assume a specific byte offset for struct fields or stack variables beyond the alignment guarantees defined in this section. The compiler MAY insert padding between fields to satisfy alignment constraints.

---

### 5.9.2 Primitive Type Alignments

The following alignment guarantees apply on all currently supported targets (64-bit x86-64):

| Type | LLVM IR type | Size (bytes) | Alignment (bytes) |
|------|-------------|-------------|------------------|
| `int` | `i64` | 8 | 8 |
| `unsigned int` | `i64` | 8 | 8 |
| `float` | `double` | 8 | 8 |
| `byte` | `i8` | 1 | 1 |
| `unsigned byte` | `i8` | 1 | 1 |
| `char` | `i8` | 1 | 1 |
| `bool` | `i1` | 1 | 1 |
| `bit` | `i1` | 1 | 1 |
| `address` | `i8*` | 8 | 8 |
| `string` | `i8*` | 8 | 8 |

These alignments are fixed for the x86-64 ABI. A conforming implementation targeting a different architecture MUST document the alignment for each type on that architecture.

---

### 5.9.3 Struct (Class) Alignment

A class in Hopper is laid out as a C-compatible struct by LLVM. The following rules apply:

1. **Field order:** Fields are laid out in the order they appear in the class definition.
2. **Field alignment:** Each field is placed at the next offset that is a multiple of that field's alignment.
3. **Padding:** The compiler MAY insert implicit padding bytes between fields to satisfy field alignment requirements. The programmer MUST NOT rely on fields being contiguous in memory.
4. **Struct alignment:** The overall struct is aligned to the largest alignment of any of its fields.
5. **Trailing padding:** The compiler MAY append trailing padding to make the struct size a multiple of its alignment, so that arrays of structs are correctly aligned.

```hopper
class Record {
    byte  tag        // offset 0, size 1
    // 7 bytes implicit padding
    int   value      // offset 8, size 8
    byte  flags      // offset 16, size 1
    // 7 bytes trailing padding
    // total size: 24 bytes, alignment: 8 bytes
}
```

The programmer MUST NOT assume any particular struct layout when interpreting raw memory. If precise layout control is required (e.g., for network protocols or hardware registers), bitfield types SHOULD be used (see §5.9.5).

---

### 5.9.4 Pointer and Function Pointer Alignment

All pointer types (`address`, `string`, `callback`) are pointer-sized and pointer-aligned:

- **Size:** 8 bytes on 64-bit targets.
- **Alignment:** 8 bytes.

This matches the system pointer size on x86-64. A conforming implementation targeting a 32-bit architecture MUST use 4-byte pointer size and alignment.

---

### 5.9.5 Bitfield Alignment

Bitfields allow the programmer to specify the exact bit-level layout of a region of memory. Bitfields in Hopper are packed into their declared container integer type without any implicit padding between consecutive bitfield fields.

```hopper
// A 16-bit hardware control register:
class ControlReg {
    bit  enable       // bit 0
    bit  direction    // bit 1
    bit  overflow     // bit 2
    // ... etc.
}
```

When all fields of a class are bitfields, the struct's total size and alignment are determined by the container type (typically `byte` or `int`). The programmer specifies the layout; the compiler does not insert padding between consecutive bitfield fields within the same container.

The precise semantics of bitfield packing are defined by the LLVM target data layout for the selected architecture. A conforming implementation MUST document any deviation from natural packing order on its target.

---

### 5.9.6 Array Element Alignment

Elements of an array (stack-allocated `type name[N]`) are placed at consecutive offsets, each a multiple of the element type's alignment. For a type with alignment `A` and size `S` (where `S` is a multiple of `A`), element `i` is at offset `i * S` from the array base.

For primitive types, `S` equals the alignment `A`, so there is no padding between elements.

---

### 5.9.7 Heap Allocation Alignment

The `allocate` instruction returns a pointer with **at least 8-byte alignment**, consistent with the `malloc` guarantee on 64-bit systems. This is sufficient for any primitive type defined in §5.9.2.

Programs that require **stricter alignment** (e.g., 16-byte alignment for SSE2 operations) MUST arrange this manually by:

1. Allocating additional bytes to allow for alignment adjustment.
2. Rounding the returned pointer up to the required alignment boundary.
3. Tracking the original (unadjusted) pointer for deallocation.

```hopper
// Allocate 64 bytes with 16-byte alignment:
int   size     = 64
int   align    = 16
address raw    = allocate size + align
int   rawInt   = cast<int>(raw)
int   aligned  = (rawInt + align - 1) & -(align)
address ptr    = cast<address>(aligned)
// Use ptr for 16-byte-aligned access.
// MUST deallocate raw, not ptr.
deallocate raw
```

The implementation is NOT REQUIRED to provide aligned allocation primitives. Any alignment beyond 8 bytes is the programmer's responsibility.

---

### 5.9.8 Misaligned Access

On x86-64, misaligned loads and stores are permitted by the hardware and produce correct results, though with a potential performance penalty. A conforming implementation targeting x86-64 MAY generate misaligned load/store instructions without error.

On architectures that do not permit misaligned access (e.g., some embedded ARM targets), a misaligned access causes a hardware alignment fault. Hopper does not insert alignment fixups for misaligned accesses. Programs that must run on strict-alignment architectures MUST ensure all accesses are naturally aligned.

A conforming implementation MAY emit a warning when it can statically detect a misaligned access, but it is NOT REQUIRED to do so.

---

### 5.9.9 The `::size` Qualifier

The byte size of any named variable or type is obtained with the `::size` qualifier:

```hopper
int   n  = x::size      // size of variable x in bytes
int   s  = int::size    // size of the int type: 8
int   p  = address::size  // size of the address type: 8
```

This value is the total number of bytes the variable occupies in memory, including any trailing padding added for alignment. It is always a compile-time constant for primitive types and classes with no dynamically sized fields.
