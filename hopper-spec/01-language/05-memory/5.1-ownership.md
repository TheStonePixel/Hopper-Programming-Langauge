## 5.1 Ownership

### 5.1.1 No Ownership System

Hopper has **no ownership system**. There is no borrow checker, no ownership type annotation, no move semantics enforced by the compiler, and no reference counting. The compiler does NOT track which variable holds the "owner" of a heap allocation.

The programmer is solely responsible for ensuring that every heap allocation is freed **exactly once** at the correct point in the program's execution.

---

### 5.1.2 Ownership as Convention

In the absence of an enforced ownership model, Hopper adopts the following normative convention:

> **The entity that allocates memory owns it and is responsible for its deallocation.**

"Entity" means either a function or a class:

- A **function** that calls `allocate` SHOULD either free the result before returning, pass it to a data structure that takes ownership, or document the transfer of ownership to the caller.
- A **class** that calls `allocate` in its `constructor` SHOULD call `deallocate` on that same allocation in its `destructor`.

This convention is not enforced by the compiler. It is a design contract that MUST be maintained by the programmer to avoid memory leaks, double-frees, and use-after-free errors.

---

### 5.1.3 Constructor/Destructor Pairing

The canonical ownership pattern in Hopper is to pair `allocate` in the constructor with `deallocate` in the destructor. This makes the full resource lifecycle visible within a single class definition.

```hopper
class Buffer {
    address data
    int length

    constructor(int size) {
        self.length = size
        self.data   = allocate size
    }

    destructor() {
        deallocate self.data
    }

    function write(int offset, byte value) {
        address slot = self.data + offset
        slot::value  = value
    }
}
```

A class following this pattern is said to **own** its allocation. The allocation is acquired at construction time and released at destruction time. A reader of the class definition can determine the complete resource lifecycle without examining call sites.

A conforming program SHOULD use this pattern whenever a class holds a heap allocation as a field.

---

### 5.1.4 Ownership Transfer

Ownership may be transferred explicitly, but this MUST be documented at the call site and in any affected interface. There is no language syntax for expressing an ownership transfer. It is expressed through convention and documentation.

A common pattern is returning an `address` from a function with the expectation that the caller is responsible for freeing it:

```hopper
function makeString(int length) address {
    address s = allocate length + 1
    // ... populate s ...
    return s
    // Caller MUST call deallocate on the returned address.
}
```

A function that transfers ownership to the caller SHOULD document this in a `// Caller owns result` comment or equivalent.

---

### 5.1.5 Non-Owning References

A value of type `address` that does NOT own the underlying allocation — for example, a pointer into the middle of an allocation, or a pointer passed to a function for reading — is called a **non-owning reference**. A function holding a non-owning reference MUST NOT call `deallocate` on it.

There is no syntactic distinction between an owning `address` and a non-owning `address`. The distinction is semantic and MUST be maintained by the programmer.

---

### 5.1.6 Errors: Double-Free and Use-After-Free

**Double-free** occurs when `deallocate` is called on a pointer whose underlying allocation has already been freed. Double-free is **undefined behavior**. The implementation is not required to detect or report it.

**Use-after-free** occurs when any read or write is performed through an `address` after the underlying allocation has been freed. Use-after-free is **undefined behavior**.

Neither error is detected at compile time. Neither error is guaranteed to be detected at runtime. Programs that invoke these errors have no guaranteed behavior.

---

### 5.1.7 Null Pointers

Hopper has a `null` literal (grammar: `primary : 'null'`) that produces an `address` value of zero. It is the conventional representation of the absence of a valid allocation:

```hopper
address p = null               // null address (address zero)
```

Passing a null pointer to `deallocate` is **undefined behavior**. Programs SHOULD check for null before deallocating a pointer whose validity is uncertain:

```hopper
if (p != null) {
    deallocate p
}
```

Dereferencing a null pointer is **undefined behavior**.
