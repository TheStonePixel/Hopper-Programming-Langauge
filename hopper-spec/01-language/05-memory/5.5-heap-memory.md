## 5.5 Heap Memory

### 5.5.1 Overview

Heap memory is the primary mechanism for dynamic, programmer-controlled storage in Hopper. Heap allocations have lifetimes that are independent of any function frame: they persist from the point of `allocate` until the point of `deallocate`, regardless of how many function calls or scope boundaries intervene.

Heap memory MUST be managed explicitly. There is no garbage collector, no reference counting, and no automatic reclamation.

---

### 5.5.2 The `allocate` Expression

**Syntax:**
```
allocate <size-expr>
```

`allocate N` allocates exactly `N` bytes of heap memory and returns an `address` pointing to the first byte of the allocated region.

- `N` MUST be a positive integer expression. The type of `N` is `int`.
- `N` MUST evaluate to a value greater than zero. Allocating zero bytes is **undefined behavior**.
- The returned `address` is **8-byte aligned** (guaranteed by the underlying allocator). Programs requiring stricter alignment MUST arrange it manually.
- The contents of the allocated region are **uninitialized**. Reading from uninitialized memory and treating the result as meaningful is **undefined behavior**.
- The result type of `allocate N` is `address`.

```hopper
address buf   = allocate 4096         // 4096 uninitialized bytes
address node  = allocate 24           // space for a 3-field struct (3 × 8 bytes)
address block = allocate bufSize      // dynamic size — bufSize is an int variable
```

The size expression is evaluated exactly once at the point of the `allocate` expression.

---

### 5.5.3 Equivalence to `malloc`

On all currently supported targets, `allocate N` is lowered to a call to `malloc(N)` from the C standard library. The behavior of `allocate` is therefore identical to `malloc`:

- Returns a pointer to at least `N` usable bytes.
- Returns a pointer with alignment suitable for any fundamental type (at least 8-byte aligned on 64-bit targets).
- Returns a null-like value (`address 0`) if the allocation fails.

A conforming implementation MUST handle allocation failure. A program SHOULD check the return value of `allocate` when operating in environments where allocation may fail:

```hopper
address buf = allocate 1048576        // 1 MiB
if (cast<int>(buf) == 0) {
    // allocation failed; handle the error
}
```

Failing to check the result and then using a null address is **undefined behavior** (null pointer dereference).

---

### 5.5.4 The `deallocate` Statement

**Syntax:**
```
deallocate <address-expr>
```

`deallocate ptr` releases the heap allocation that begins at `ptr`. After this statement executes, the memory at `ptr` is no longer owned by the program. Any access to `ptr` or any derived address after deallocation is **undefined behavior**.

Rules for `deallocate`:

1. The `ptr` argument MUST be an `address` value that was previously returned by `allocate` and has not yet been freed.
2. The `ptr` argument MUST point to the **beginning** of the allocation. Passing a pointer into the middle of an allocation is **undefined behavior**.
3. Passing a null pointer (address value `0`) to `deallocate` is **undefined behavior**. (This differs from C's `free(NULL)`, which is defined. Hopper does not guarantee the same.)
4. Calling `deallocate` twice on the same pointer is **double-free**, which is **undefined behavior**.
5. The size passed to the original `allocate` need not be repeated; the allocator tracks the size internally.

```hopper
address buf = allocate 256
// ... use buf ...
deallocate buf
// buf is now dangling. Do not use it.
```

---

### 5.5.5 Equivalence to `free`

On all currently supported targets, `deallocate ptr` is lowered to a call to `free(ptr)`. The behavior is identical to `free` with the exception that passing `NULL` (address `0`) is not guaranteed safe (see §5.5.4 rule 3).

---

### 5.5.6 Using Heap Memory

Heap memory is accessed through the `::value` qualifier applied to the `address`. The address itself is a number; interpreting the bytes at that address as a particular type is the programmer's explicit responsibility.

```hopper
address buf = allocate 8
// Write an int (8 bytes) to the allocation:
int val = 42
address src = val::address
// (manual copy; or use a module that provides memcpy semantics)

// Read it back:
// int readback = (buf::value) as int -- requires cast module
```

Because `address` is untyped, all type-specific interpretation is explicit. There is no automatic deref operator.

---

### 5.5.7 Heap Arrays

A heap-allocated array of `N` elements of type `T` (where each element is `S` bytes) occupies `N * S` contiguous bytes starting at the returned `address`:

```hopper
int elemSize = 8                      // int is 8 bytes (i64)
int count    = 100
address arr  = allocate count * elemSize

// Element i starts at: arr + (i * elemSize)
address elem3 = arr + (3 * elemSize)
```

The programmer MUST ensure index calculations remain within the allocated range. Out-of-bounds access is **undefined behavior**.

---

### 5.5.8 Nested Allocations

A heap allocation may contain `address` fields that point to further heap allocations. Each allocation MUST be freed independently, starting from the deepest level outward:

```hopper
// Allocate a node with a payload
address node    = allocate 16         // 2 fields: address + int
address payload = allocate 64
// Store payload address in node field (manual, platform-specific offset)

// To free:
deallocate payload   // inner allocation first
deallocate node      // outer allocation second
```

Freeing the outer allocation without first freeing inner allocations causes a memory leak. A conforming implementation SHALL NOT detect or prevent this automatically.
