## 5.2 Lifetimes

### 5.2.1 No Lifetime Annotations

Hopper has **no lifetime annotation system**. There are no lifetime parameters, no borrow lifetimes, no lifetime elision rules, and no lifetime inference. The language does not track, name, or reason about lifetimes at compile time.

The lifetime of a value is determined by straightforward rules described in this section. Violations of these rules — such as using a dangling pointer — produce undefined behavior, not a compile-time error.

---

### 5.2.2 Local Variable Lifetimes

A local variable is live from the point of its declaration to the end of the innermost enclosing block in which it is declared. When control flow reaches the closing `}` of that block, the variable ceases to exist and any `address` values pointing to its storage become **dangling pointers**.

```hopper
function example() {
    {
        int x = 42
        address p = x::address   // p is valid here
    }
    // x no longer exists. p is dangling. Using p is undefined behavior.
}
```

The compiler SHALL NOT warn about dangling pointers derived from local variables. The programmer is responsible for ensuring that no `address` value derived from a local variable escapes its lifetime.

---

### 5.2.3 Parameter Lifetimes

Function parameters are live from the start of the function body to the end of the function body. Parameters behave identically to local variables for lifetime purposes.

An `address` parameter pointing into a caller's allocation is valid only for the duration of the call. If the callee stores the address in a data structure that outlives the call, the address becomes dangling when the caller's allocation is freed.

---

### 5.2.4 Heap Allocation Lifetimes

A heap allocation is live from the point of the `allocate` expression that created it to the point of the `deallocate` statement that frees it.

```hopper
address buf = allocate 1024    // allocation begins here
// buf is valid throughout this region
deallocate buf                 // allocation ends here
// buf is dangling. Any use of buf after this point is undefined behavior.
```

The implementation SHALL NOT extend or shorten the lifetime of a heap allocation beyond the programmer-specified `allocate`/`deallocate` pair.

If `deallocate` is never called, the allocation lives for the entire remaining execution of the program — its memory is reclaimed by the OS when the process exits. This is a memory leak; it is not undefined behavior in and of itself, but it violates the ownership convention in §5.1.2.

---

### 5.2.5 Class Instance Lifetimes

A class instance allocated on the stack (via `ClassName x = ClassName(...)`) is live from the point of declaration to the end of the enclosing block. Its destructor MUST be called explicitly if the class holds resources:

```hopper
function process() {
    Buffer buf = Buffer(4096)
    // ... use buf ...
    buf.destructor()   // must be called explicitly; no automatic RAII
}
```

> **Note:** Hopper does not guarantee automatic destructor invocation at scope exit. The programmer MUST call `destructor()` explicitly when the instance is no longer needed. This is a deliberate design choice: implicit side effects at scope boundaries violate Hopper's visibility principle.

---

### 5.2.6 String Literal Lifetimes

String literals (e.g., `"hello"`) are stored in the read-only code segment. Their lifetime is the entire program lifetime. An `address` or `string` value pointing to a string literal is always valid and MUST NOT be passed to `deallocate`.

```hopper
string greeting = "hello"      // points into read-only segment
// greeting is valid for the entire program
// deallocate greeting          -- undefined behavior; MUST NOT do this
```

---

### 5.2.7 Stack-Allocated Arrays

Arrays declared with a compile-time size (`type name[N]`) are stack-allocated. Their lifetime is the same as any other local variable: from declaration to end of enclosing block.

```hopper
function example() {
    byte buf[256]              // 256 bytes on the stack
    // buf is valid here
    address p = buf::address   // valid here
}
// buf is gone. p is dangling.
```

The compiler SHALL NOT warn about this. The programmer MUST NOT allow stack array addresses to escape their scope.

---

### 5.2.8 Dangling Pointers

A **dangling pointer** is an `address` value that once referred to a valid allocation or stack location, but that allocation or location has since ended. The Hopper implementation:

- SHALL NOT detect dangling pointers at compile time.
- SHALL NOT detect dangling pointers at runtime (unless the programmer has added explicit checks).
- SHALL treat any read or write through a dangling pointer as undefined behavior.

Avoiding dangling pointers is entirely the programmer's responsibility.
