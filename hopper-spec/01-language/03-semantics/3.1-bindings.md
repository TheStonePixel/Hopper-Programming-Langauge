## 3.1 Bindings

A *binding* is an association between a name and a storage location that holds a typed value. Every local variable declaration in Hopper creates exactly one binding. Bindings are introduced by variable declaration statements and by function parameters.

### 3.1.1 Variable Declaration

The general form of a local variable declaration is:

```hopper
type name = expression
```

or, without an initializer:

```hopper
type name
```

A conforming implementation MUST allocate storage for `name` sufficient to hold a value of the declared type. The storage MUST be distinct from any other named variable in the same scope. The name `name` MUST NOT already be bound in the same scope (see section 3.2 for scope rules and section 3.2.2 for shadowing).

When an initializer expression is present, the expression MUST be evaluated first, and then the resulting value MUST be stored into the allocated storage. The name becomes usable immediately after the declaration statement — that is, the name is NOT in scope within its own initializer expression.

When no initializer is present, the storage is allocated but its contents are unspecified. Reading such a variable before assigning to it is *undefined behavior* (see section 3.3).

### 3.1.2 Stack Allocation

Local variable bindings are allocated on the call stack. The lifetime of the storage begins when control reaches the declaration and ends when control leaves the enclosing block. The compiler MUST NOT reuse the same storage slot for two variables whose lifetimes overlap, but MAY reuse storage for non-overlapping variables as an optimization.

In the reference implementation, each `VarDecl` emits an `alloca` instruction in the LLVM IR function prologue. All allocas for a function are collected at the top of the entry basic block regardless of where the declaration appears in source. This means the storage is technically live for the entire function, but the Hopper semantics treat the variable as live only from its declaration point.

### 3.1.3 The Stack Slot Is Stable

The name introduced by a binding always refers to the same stack slot for the entire duration of the binding's lifetime. This property has two consequences:

1. **Assignment modifies the slot in place.** The statement `name = newValue` stores `newValue` into the storage already allocated for `name`. It does not create a new binding or move the storage.

2. **Taking the address of a variable is stable.** The expression `name::address` yields the address of the stack slot. That address is valid and constant for the entire lifetime of the binding. Programs MUST NOT use the address of a local variable after the enclosing block exits, as the storage is no longer live (doing so is undefined behavior).

### 3.1.4 Type Annotation Is Mandatory

Every variable declaration MUST include an explicit type annotation. There is no type inference on variable declarations. The declared type determines:

- The size of the allocated storage.
- The LLVM IR type used for loads and stores.
- The interpretation of the initializer expression (the compiler MAY emit a type mismatch error if the initializer type is incompatible with the declared type).

### 3.1.5 Constrain Clause

A variable declaration MAY include a `constrain` clause:

```hopper
int x = 300 constrain [byte]
```

The `constrain` clause specifies that the value of `x` MUST fall within the range of the constraining type (`byte` in this example). If contracts are not disabled (i.e., not compiled with `--release`), the compiler MUST emit a runtime range check immediately after the initializer store. If the runtime value is outside the constrained range, behavior is implementation-defined; the reference implementation traps.

In `--release` mode, all `constrain` checks are omitted. Programs MUST NOT rely on `constrain` for correctness of release builds.

### 3.1.6 Array Declarations

Arrays are declared with a fixed compile-time size:

```hopper
int nums[8]
int nums[4] = [1, 2, 3, 4]
```

An array declaration creates a contiguous block of storage on the stack containing exactly `N` elements of the element type. Element `i` is accessed with `nums[i]`. Array bounds checking is NOT performed at runtime unless a future version of the language mandates it; out-of-bounds access is undefined behavior.

The name of an array variable refers to the entire array. There is no implicit conversion to a pointer-to-first-element in Hopper; use `name::address` to obtain the address of the first element.

### 3.1.7 Function Parameter Bindings

Function parameters behave identically to locally declared variables: each parameter is bound to a stack slot at function entry, and the caller's value is copied into that slot. Mutations to a parameter inside the function body do NOT affect the caller's variable. For class types, the parameter holds a pointer to the caller's instance; mutations through `self` fields DO affect the caller's instance (this is the pointer-semantics rule from section 3).
