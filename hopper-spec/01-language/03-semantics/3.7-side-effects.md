## 3.7 Side Effects

A *side effect* is any observable change to program state or the external environment that occurs as a consequence of evaluating an expression or executing a statement. Hopper has no pure/effect system and no mechanism for annotating or restricting side effects. This section defines what counts as a side effect and what constraints a conforming implementation MUST respect when handling them.

### 3.7.1 Classification of Side Effects

The following operations are side effects in Hopper:

1. **Assignment to a variable.** Storing a new value into a local variable, a function parameter, or a field of a class instance changes observable program state.

2. **Function calls.** Any call to a user-defined function, a template function, or an extern-declared function is a side effect, because the called function may itself perform any of the operations listed here. The compiler MUST NOT eliminate a function call on the basis that its return value is unused, unless it can prove through whole-program analysis that the function is pure (has no observable effects). The reference implementation does not perform such analysis; all calls are preserved.

3. **Memory allocation and deallocation.** `allocate` invokes the system allocator and changes the heap state. `deallocate` returns memory to the allocator. Both are side effects that MUST NOT be reordered with respect to other memory operations that use the same pointer.

4. **Inline assembly.** An `asm { ... }` block is an opaque side-effecting operation. The compiler MUST treat the entire block as a black box: no inputs may be assumed unchanged after it, no outputs may be assumed before it executes, and no other operations may be reordered across it. In the reference implementation, `asm` blocks are emitted as LLVM inline assembly with the `~{memory}` clobber, which prevents the LLVM optimizer from reordering memory operations across the block.

5. **MMIO register access (`strict`).** Reads and writes to `strict`-declared names are memory-mapped I/O operations. They are volatile: every read MUST produce a fresh value from the hardware register, and every write MUST be written to the hardware address. These accesses MUST NOT be cached, combined, or reordered by the compiler (see section 3.5.6).

6. **System calls.** System calls are performed through inline assembly in the `x86_64` implementation layer. They are side effects because they interact with the operating system kernel, which maintains state outside the program's address space. All constraints on inline assembly (item 4) apply to system call sequences.

7. **Writing to an `address` or `string` target.** A store through a pointer modifies memory that may be shared with other parts of the program or with external code. Such stores are side effects.

### 3.7.2 Preservation of Side Effects

A conforming implementation MUST NOT eliminate, deduplicate, or reorder side-effecting operations in ways that change the observable behavior of the program.

*Observable behavior* consists of:
- The sequence of writes to memory-mapped hardware registers (`strict`).
- The sequence of inline assembly executions.
- The values passed to and returned from system calls.
- The sequence of allocations and deallocations.
- The final exit status of the process.

The compiler MAY reorder or eliminate operations that are demonstrably free of side effects (e.g., pure arithmetic on values with no aliases). The compiler MUST preserve the relative order of any two side-effecting operations that appear in program order.

### 3.7.3 No Pure/Effect System

Hopper does not have a `pure` keyword, an effect type system, or any annotation mechanism for declaring that a function has no side effects. The compiler MUST conservatively assume that any user-defined function call may have side effects. The compiler MUST NOT hoist, sink, or eliminate a call based on an assumption of purity.

This is a deliberate design decision. The language is intended for systems programming where side effects are the primary means of interaction with hardware and the operating system. A pure/effect system would add complexity without providing proportionate benefit in this domain.

### 3.7.4 Defer

The `defer expr` statement registers a side-effecting expression (typically a function call) to be executed when the enclosing function returns, in last-in-first-out order. Deferred expressions are a sequenced side effect: they execute after all other statements in the function body have been evaluated, but before the function's return value is delivered to the caller.

```hopper
function withCleanup(int fd) {
    defer close(fd)    // 'close(fd)' executes when the function returns
    // ... do work ...
}
```

A conforming implementation MUST guarantee that all deferred expressions registered within a function execute exactly once when that function returns, regardless of which `return` statement is reached. A deferred expression MUST NOT execute if the function never returns (e.g., due to an infinite loop or process exit).

The order in which multiple `defer` statements execute MUST be the reverse of their registration order (last-registered, first-executed), matching the typical use pattern for paired open/close or lock/unlock operations.

### 3.7.5 Evaluation of Deferred Expressions

The arguments to a deferred call are evaluated at the point of the `defer` statement, not at the point of execution. The saved argument values are used when the deferred call runs at function exit.

```hopper
function example() {
    int fd = open("file.txt", 0)
    defer close(fd)   // 'fd' is captured here with its current value
    fd = 42           // this does NOT affect what is passed to close()
}
```

### 3.7.6 Interaction with the Compiler Optimizer

In `--release` mode, the compiler MAY apply additional optimizations that eliminate contract checks (`requires`, `ensures`, `invariant`, `constrain`). These checks are not side effects in the observable sense defined above — they are diagnostic assertions. Their elimination in release mode does not violate the side-effect preservation rule.

The compiler MUST NOT eliminate or reorder any of the operations listed in section 3.7.1 regardless of optimization level.
