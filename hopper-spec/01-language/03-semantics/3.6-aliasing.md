## 3.6 Aliasing

An *alias* occurs when two or more names refer to the same region of memory. In Hopper, aliasing is possible because `address` and `string` values are raw untyped pointers. The language provides no alias analysis, no alias annotations, and no enforcement of alias freedom. Programs that create aliases are responsible for the correctness of their memory accesses.

### 3.6.1 Address and String Are Untyped Pointers

Both `address` and `string` values are represented as LLVM `i8*` — an 8-bit pointer with no type information beyond the fact that it points to bytes. The compiler does not track what type of data a pointer refers to, how many bytes are valid after the pointer, or whether two pointer values alias one another.

Assigning an `address` value to two different variables creates two names for the same memory:

```hopper
function example(address buf) {
    address p = buf    // p and buf alias the same memory
    address q = buf    // q, p, and buf all alias the same memory

    // Writing through q is visible through p and buf.
    // The compiler does not know or enforce this.
}
```

### 3.6.2 Pointer Arithmetic Creates Aliases

Hopper does not have a dedicated pointer arithmetic operator, but programs can perform arithmetic on an `address` value by converting it to `int`, performing arithmetic, and converting back:

```hopper
function elementAddress(address base, int index) address {
    int baseInt = cast<int>(base)
    int elemInt = baseInt + (index * 8)
    return cast<address>(elemInt)
}
```

The result of such arithmetic is an alias into the interior of the same allocation as `base`. The compiler has no knowledge that the result is related to `base`. Programs performing interior pointer arithmetic are responsible for ensuring:

- The resulting address lies within the bounds of a live allocation.
- The resulting address is correctly aligned for the type that will be loaded or stored through it.
- No concurrent write to overlapping memory occurs without synchronization.

### 3.6.3 No Restrict or NoAlias Annotations

Hopper provides no equivalent of C's `restrict` keyword or LLVM's `noalias` attribute. The compiler MUST assume that any two `address` values may alias unless it can prove otherwise through local analysis. This is a conservative assumption that is always correct but may prevent some alias-based optimizations.

A conforming implementation MUST NOT emit `noalias` IR attributes for `address` parameters unless it can prove through whole-program analysis that no aliasing occurs.

### 3.6.4 Class Instance Aliasing

Class instances live at a specific memory address (either a stack alloca or a heap allocation from `allocate`). Passing a class variable to a function communicates its address; the function receives a pointer to the instance's storage. When two function calls both hold a pointer to the same class instance, mutations through either pointer are immediately visible through the other.

The language provides no mechanism to detect or prevent such aliasing. Programs that pass the same instance to concurrent operations MUST synchronize those accesses externally.

### 3.6.5 String Literals

String literal values (e.g., `"hello"`) are stored in the program's read-only data section. The compiler MAY merge identical string literals into a single storage location (*string interning*). Programs MUST NOT write through a `string` value that points to a string literal; doing so is undefined behavior. A program may take the address of a string literal (`literal::address`) to obtain an `address` value pointing to the first byte of the literal's storage; it MUST NOT write to that address.

### 3.6.6 Compiler Assumptions

Because Hopper declares no alias restrictions, the compiler MUST be conservative when optimizing code that contains `address` or `string` values. Specifically:

- A load from an `address` immediately following a store to a possibly-aliasing `address` MUST read the stored value, not a cached copy.
- A function call that takes an `address` argument MAY modify any memory accessible through that pointer; the compiler MUST NOT assume that other `address` values are unchanged across such a call.

These conservative assumptions match LLVM's default pointer aliasing rules and do not require special annotation in the reference implementation.
