## 3.3 Name Resolution

Name resolution is the process of determining, for each identifier that appears in source code, which binding it refers to. Hopper uses *lexical scoping with an outward search order*: the compiler begins in the innermost enclosing scope and works outward until the name is found or all scopes are exhausted.

### 3.3.1 Resolution Order

Given an unqualified identifier `x` at a use site, the compiler MUST search for a binding in the following order:

1. **Current block scope.** Variables declared within the current `{...}` block, from the top of the block up to (but not including) the use site. (No hoisting — see section 3.2.2.)
2. **Enclosing block scopes.** Each successive enclosing block, from innermost to outermost, until the function body boundary is reached.
3. **Function parameters.** The parameters of the enclosing function or entry point.
4. **Module scope.** Functions, classes, enums, `extern` declarations, and `const` declarations defined at the top level of the current source file.
5. **Imported names.** Free functions and interface definitions brought into scope by `import` declarations. The rules for import resolution are specified in section 8.

If the name is not found after exhausting all scopes, the compiler MUST emit a compile-time error.

### 3.3.2 Qualified Names

Some names are accessed through a qualifier that restricts the search:

- **Enum variants** are always qualified: `EnumName.VariantName`. The compiler MUST look up `EnumName` in module scope, then look up `VariantName` within the enum's variant set.
- **Field access** uses the form `obj.fieldName`. The compiler MUST resolve `obj` first (as a regular identifier), then look up `fieldName` within the type of `obj`.
- **Method calls** use the form `obj.method(args)`. The compiler MUST resolve `obj` first, then look up `method` within the class definition of `obj`'s type.

Qualified access does NOT search outer scopes for `fieldName` or `method` — the search is confined to the type of the left-hand side.

### 3.3.3 Explicit Self — No Implicit Field Access

Inside a class method or constructor, fields of the current instance are NOT automatically in scope as bare names. Every field access MUST use an explicit `self` receiver.

```hopper
class Counter {
    int value

    constructor(int start) {
        self.value = start     // correct — explicit self
        // value = start       // compile error — 'value' is not in scope
    }

    function increment() {
        self.value = self.value + 1   // correct
    }

    function get() int {
        return self.value      // correct
        // return value        // compile error
    }
}
```

A conforming implementation MUST reject bare field name references inside method bodies. The `self` keyword is a reserved identifier in scope for the entire body of every class method, constructor, and destructor; it refers to the current instance.

### 3.3.4 Unqualified Function Calls in Method Bodies

When an unqualified function call `f(args)` appears inside a class method body, the compiler MUST resolve `f` to a *free function* — a function defined at module scope or imported from another module. The call MUST NOT be interpreted as a recursive call to the enclosing method.

This rule follows from the method-call requirement in section 3.3.3: method calls always require an explicit receiver (`self.f(args)` or `obj.f(args)`). A bare call `f(args)` cannot be a method call, therefore it can only be a free function call.

This distinction is critical for class implementations of interfaces. A class method that satisfies a syscall wrapper calls the free function from the imported implementation file, not itself:

```hopper
import IO from linux

class LinuxIO satisfies IO {
    function write(int fd, address buf, int count) int {
        // 'write(...)' here resolves to the free function imported from
        // the x86_64 LinuxSyscalls implementation, NOT to this method.
        return write(fd, buf, count)
    }
}
```

A conforming implementation MUST resolve unqualified calls to free functions before considering any method of the enclosing class.

### 3.3.5 Template Function Resolution

Template functions (e.g., `cast<int>(x)`) are resolved by matching both the function name and the type parameter. The compiler MUST find a template function declaration that matches the given name and type argument. If no matching specialization exists, the compiler MUST emit a compile-time error.

### 3.3.6 Overload Resolution

When multiple free functions share the same name but differ in parameter types (overloading), the compiler MUST select the candidate whose parameter types most closely match the argument types at the call site. The exact overload resolution rules are implementation-defined for the current version; a conforming implementation MUST document its strategy. Ambiguous calls where no single candidate is uniquely best MUST result in a compile-time error.

### 3.3.7 Interface Binding Resolution

When a program declares `import IO from linux`, the compiler resolves `IO` through the `targets` section of the closest enclosing `hopper.json`. The binding specifies an interface file and an implementation file. Both files are compiled; the free functions from the implementation file come into module scope and are available for resolution under step 5 of section 3.3.1. The interface name itself (`IO`) is used only at the type level and in `satisfies` clauses; it is not a callable entity.

### 3.3.8 `extern` Declarations

An `extern function` declaration introduces a name into module scope that refers to a function defined outside the Hopper translation unit (typically a C library function). Extern names are resolved in step 4 of section 3.3.1 alongside module-scope functions. The compiler MUST NOT emit a definition for an extern-declared function; it MUST emit an external linkage reference that the linker resolves.

### 3.3.9 `const` and `enum` Resolution

Top-level `const` names and `enum` type names are in module scope and resolved in step 4. `const` values are substituted inline at their use sites; they do not occupy addressable storage. `enum` variant values are substituted similarly — `Color.Red` is replaced by its integer value during code generation.
