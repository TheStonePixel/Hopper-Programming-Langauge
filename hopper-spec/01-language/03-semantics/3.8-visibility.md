## 3.8 Visibility

*Visibility* governs which names declared in one module may be referenced from another module. Hopper's current visibility model is intentionally minimal: there is no module-level access control keyword, and all module-scope declarations are accessible to any module that imports the declaring module.

### 3.8.1 Current Model — All Declarations Are Public

Every function, class, enum, struct, bitfield, `extern` declaration, and (deprecated) `const` declaration at module scope is visible to importing modules. A conforming implementation MUST NOT prevent a module from referencing any module-scope name it can reach through an import.

This means that implementation details exposed at module scope are technically accessible to all consumers. Module authors SHOULD use interface types (see section 3.8.2) to express the intended public API and discourage direct use of implementation details, but the language currently provides no enforcement mechanism.

### 3.8.2 Interface Conformance as Encapsulation

The primary encapsulation mechanism in Hopper is the `interface` and `implements` system. A module author defines an interface that expresses the contract:

```hopper
interface IO {
    function read(int fd, address buf, int count) int
    function write(int fd, address buf, int count) int
}
```

A class that implements the interface declares:

```hopper
class LinuxIO implements IO {
    function read(int fd, address buf, int count) int { ... }
    function write(int fd, address buf, int count) int { ... }
}
```

Consumers of the module are encouraged to depend only on the interface — calling the free functions declared there — rather than on internal implementation details of the class. The compiler enforces conformance (a class that declares `implements IO` MUST provide all methods listed in `IO`) but does not prevent callers from reaching into implementation internals.

This is a *by-convention* encapsulation model. Its strength depends on module authors documenting their intended API clearly and consumers respecting it.

### 3.8.3 Import Scope Is Not Re-Exported

When module A imports module B, the names from B that come into scope in A are NOT automatically re-exported to any module C that imports A. A module C that needs a name from B MUST import B directly. This prevents inadvertent re-export of transitive dependencies and keeps dependency graphs explicit.

Specifically, free functions imported into A via a `targets` binding (e.g., the syscall free functions from `x86_64/src/LinuxSyscalls.hop`) are available within A's source files but are not part of A's public API. A module C importing A cannot call those free functions unless C also declares the appropriate `targets` binding that resolves to the same implementation file.

### 3.8.4 Planned Private-by-Default Visibility

A future version of Hopper SHOULD introduce explicit visibility control. The anticipated design is *private-by-default* at module scope: every declaration is private unless explicitly annotated as public. Interfaces would then be the primary (and required) mechanism for exposing a module's API to consumers.

When this feature is introduced, it MUST be specified in this section with full normative detail. Until then, the current public-by-default model governs.

### 3.8.5 Class-Internal Visibility

There is currently no distinction between public and private fields or methods within a class. Any code that has a reference to a class instance MAY access any field or call any method by name. This is a known limitation. Programs SHOULD treat fields beginning with a conventional prefix (e.g., an underscore) as private by convention, but the compiler does not enforce this.

The interface system provides one partial remedy: if a class is only ever passed around as an interface type, callers cannot directly access the implementation class's fields or non-interface methods without casting to the concrete type. Since Hopper has no safe downcast mechanism, this provides a weak encapsulation boundary.

### 3.8.6 Entry Points

The `entry` declaration names the program entry point. It is not a function in the module-scope sense and is not callable from other modules. A conforming implementation MUST ensure that the entry point is only invoked by the runtime as the initial point of execution.

### 3.8.7 `extern` Declarations

An `extern function` declaration makes a name from outside the Hopper module system available inside the module. It does NOT affect the visibility of Hopper-defined names. An `extern` name is visible within the module where the declaration appears, subject to the same rules as any module-scope function name. It is NOT automatically visible to importing modules; importers that need the same extern MUST re-declare it.
