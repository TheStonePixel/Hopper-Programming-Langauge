## 3.5 Mutability

In Hopper, **all variables are mutable by default**. There is no `const` qualifier on local variables and no `final`, `let`, or `readonly` modifier. Any named binding may have its value changed through an assignment statement at any point during its lifetime.

### 3.5.1 Local Variable Mutability

A local variable declared with `type name = expr` may be assigned a new value at any point after its declaration, as many times as needed:

```hopper
function countdown(int start) {
    int n = start
    while (n > 0) {
        n = n - 1   // reassignment is always permitted
    }
}
```

There is no mechanism in the current language to declare a local variable as immutable. A conforming implementation MUST permit assignment to any local variable in scope.

### 3.5.2 Class Field Mutability

Class fields are mutable through the `self` receiver inside any method of the class. There is no read-only or private-field distinction at the language level in the current version. A field assigned in the constructor MAY be reassigned in any subsequent method call.

```hopper
class Buffer {
    address data
    int size

    constructor(int n) {
        self.data = allocate n
        self.size = n
    }

    function resize(int newSize) {
        deallocate self.data          // free old storage
        self.data = allocate newSize  // reassign the field
        self.size = newSize
    }
}
```

### 3.5.3 No `const` Qualifier on Local Variables

The grammar includes a `const` variant of the variable declaration statement (`const type name = expr`), but its semantics are implementation-defined in the current version. A conforming implementation MAY choose to enforce immutability for `const`-declared locals, or it MAY treat them identically to mutable declarations. Programs SHOULD NOT rely on `const` local declarations for program correctness in a portable way.

This is distinct from top-level `const` declarations, which are being deprecated (see section 3.5.4).

### 3.5.4 Top-Level `const` — Deprecated

Top-level `const` declarations of the form:

```hopper
const STDIN  = 0
const STDOUT = 1
const NAME   = "hello"
```

are currently supported by the grammar and compiler but are **being removed** from the language. A conforming implementation MUST continue to compile top-level `const` declarations for backward compatibility until the removal is complete, but programs SHOULD migrate away from them.

The replacement for top-level integer constants is `enum`:

```hopper
enum FileDescriptor { STDIN = 0  STDOUT  STDERR }
```

For named string constants there is currently no direct replacement; programs should use `string` local variables initialized to the desired literal.

A conforming implementation SHOULD emit a deprecation warning for any top-level `const` declaration.

### 3.5.5 No Immutability Mechanism

Hopper currently provides no mechanism to express or enforce immutability beyond the deprecated `const` local qualifier. There is no equivalent of Rust's `let` bindings, C's `const` pointers, or C++'s `const` methods. Programs that require immutability guarantees MUST enforce them through design discipline, not through language features.

This is a deliberate design decision for the current version of the language. A future version MAY introduce a first-class immutability mechanism. When such a mechanism is added, it will be specified in a new subsection of this section.

### 3.5.6 MMIO Registers (strict)

The `strict` declaration introduces a named alias for a memory-mapped hardware register:

```hopper
strict int uart_dr = 0x40021000
```

A `strict` name behaves like a mutable variable whose storage is at the specified absolute address. Reads and writes to a `strict` name are treated as volatile memory operations: the compiler MUST NOT cache the value in a register across multiple reads, and MUST NOT eliminate or reorder any write. These constraints are stronger than the general rules in section 3.7.

`strict` names are not writable through ordinary assignment in the current grammar; they are read and written through the `name::value` dereference syntax. A conforming implementation MUST emit volatile load/store instructions (or their equivalent) for all accesses to `strict` names.
