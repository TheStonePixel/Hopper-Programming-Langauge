## 3.5 Mutability

In Hopper, **all variables are mutable by default**. A named binding introduced by `type name = expr` may be reassigned at any time during its lifetime. There is no `var` or `let` distinction.

### 3.5.1 Local Variable Mutability

A local variable declared with `type name = expr` may be assigned a new value at any point after its declaration:

```hopper
function countdown(int start) {
    int n = start
    while (n > 0) {
        n = n - 1
    }
}
```

### 3.5.2 Class Field Mutability

Class fields are mutable through the `self` receiver inside any method. There is no read-only field or private-field distinction at the language level. A field assigned in the constructor may be reassigned in any subsequent method.

### 3.5.3 The `const` Immutability Modifier

A local variable declaration may be prefixed with `const` to declare it immutable:

```
'const' type Identifier '=' expression
```

A `const`-declared variable is initialized once and MUST NOT be reassigned. A conforming implementation MUST produce a compile-time error if any assignment statement targets a `const`-declared name.

```hopper
const int limit = 100
const string label = "max"
limit = 200           // compile-time error: TypeError — 'limit' is declared const
```

`const` is a statement-level modifier. It may appear on any local variable declaration inside a function, method, constructor, or entry point body. It is NOT a top-level declaration form.

The compiler stores `const` variables on the stack identically to mutable variables. `const` is an assertion for the programmer and compiler, not a directive to inline the value.

### 3.5.4 No Top-Level `const`

There is no top-level constant declaration in Hopper. The form:

```
const NAME = value   // NOT valid
```

is not in the grammar. Programs that need named integer constants MUST use `enum`:

```hopper
enum FileDescriptor {
    STDIN  = 0
    STDOUT = 1
    STDERR = 2
}
```

For named string constants there is currently no module-level form. Use a local variable or a function that returns the value.

### 3.5.5 Class Field Mutability and Encapsulation

Fields declared in a class are accessible and mutable through any method that holds a `self` reference to that instance. There is no `private`, `protected`, or `readonly` field modifier in the current version.

### 3.5.6 MMIO Registers (`strict`)

The `strict` declaration introduces a named alias for a memory-mapped hardware register:

```hopper
strict int uart_dr = 0x40021000
```

A `strict` name behaves like a mutable variable whose storage is at the specified absolute address. Reads and writes are treated as volatile memory operations: the compiler MUST NOT cache the value in a register across multiple reads, and MUST NOT eliminate or reorder any write.

`strict` names are accessed through the `name::value` dereference syntax.
