## 3.4 Evaluation Order

This section defines the order in which subexpressions are evaluated. Evaluation order is significant whenever subexpressions have side effects (assignments, function calls, memory operations, inline assembly). A conforming implementation MUST evaluate expressions in the order defined here; it MUST NOT reorder or eliminate evaluations in ways that change observable behavior as defined by section 3.7.

### 3.4.1 Left-to-Right Evaluation

Within any expression, subexpressions are evaluated **left-to-right**. For a binary operator expression `A op B`, the implementation MUST evaluate `A` completely before evaluating `B`. For a unary operator expression `op A`, `A` is evaluated first.

```hopper
function sideEffect(int n) int {
    // suppose this has a side effect
    return n
}

function example() {
    int r = sideEffect(1) + sideEffect(2)
    // sideEffect(1) is evaluated first, then sideEffect(2), then the addition.
}
```

### 3.4.2 Function Argument Evaluation

When a function is called with multiple arguments, the arguments MUST be evaluated **left-to-right** before the call is made. All arguments are fully evaluated (including all side effects) before control transfers to the called function.

```hopper
function f(int a, int b, int c) int {
    return a + b + c
}

function g(int x) int {
    // assume this has a side effect
    return x * 2
}

function example() {
    int r = f(g(1), g(2), g(3))
    // g(1) evaluated first, then g(2), then g(3), then f is called.
}
```

A conforming implementation MUST NOT evaluate arguments in any other order.

### 3.4.3 Method Call Evaluation

For a method call `obj.method(arg1, arg2)`, the receiver expression `obj` MUST be evaluated first. Then the arguments MUST be evaluated left-to-right. Then the method is called.

For a chained call `obj.field.method(arg1)`, evaluation proceeds left to right through the chain: `obj` first, then `.field` access, then `arg1`, then the call.

### 3.4.4 Short-Circuit Evaluation

The logical operators `&&` (AND) and `||` (OR) use *short-circuit* evaluation. The right-hand operand MUST NOT be evaluated if the result is determined by the left-hand operand alone.

- For `A && B`: `A` is evaluated first. If `A` is `false`, the result is `false` and `B` MUST NOT be evaluated.
- For `A || B`: `A` is evaluated first. If `A` is `true`, the result is `true` and `B` MUST NOT be evaluated.

```hopper
function expensive() bool {
    // this function has side effects
    return true
}

function safeCheck(address ptr) bool {
    // 'expensive()' is only called if ptr is not null
    return (ptr != null) && expensive()
}
```

Short-circuit behavior is observable: if `B` has side effects and `A` determines the result, those side effects MUST NOT occur. A conforming implementation MUST NOT speculatively evaluate `B`.

### 3.4.5 Assignment Evaluation

In an assignment statement `name = expr`, the expression `expr` MUST be fully evaluated before the result is stored into `name`. The old value of `name` is overwritten atomically with respect to the current thread (not with respect to other threads — see section 7).

In a field assignment `obj.field = expr`, the receiver `obj` is evaluated first, then `expr` is evaluated, then the store is performed.

### 3.4.6 Determinism

Evaluation order in Hopper is fully deterministic. For any given set of inputs, executing a program produces the same sequence of operations every time. There is no unspecified evaluation order for any construct in the language. A conforming implementation MUST NOT introduce non-determinism through reordering of observable side effects.

### 3.4.7 Operator Precedence and Grouping

Operator precedence (defined in section 2.6) and explicit parenthesization affect which subexpressions are evaluated as operands to which operators, but they do NOT change the left-to-right evaluation order within those subexpressions. Parentheses group operands; they do not reorder evaluation.

```hopper
function example(int a, int b, int c) int {
    // The grouping (a + b) * c means: evaluate 'a + b' as the left operand of '*'.
    // Evaluation order: 'a', then 'b', then '+', then 'c', then '*'.
    return (a + b) * c
}
```

### 3.4.8 Inline Assembly Evaluation Order

An `asm { ... }` block is treated as an atomic unit with respect to evaluation order. When the control flow reaches the `asm` statement, all prior expressions in the current statement have been evaluated and stored. The inline assembly body executes in document order (top to bottom within the `asm` block). After the `asm` block completes, subsequent statements proceed.

### 3.4.9 Allocate Expression

In `type name = allocate expr`, the size expression `expr` MUST be evaluated to determine the allocation size before `allocate` is invoked. The resulting pointer is then stored into `name`.
