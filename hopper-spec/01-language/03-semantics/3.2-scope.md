## 3.2 Scope

*Scope* defines the region of source text in which a name is visible and can be used. Hopper uses *lexical scoping* — the scope of every name is determined entirely by its textual position in the source file, without regard to the order in which code executes.

### 3.2.1 Block Scope

A *block* is a sequence of statements enclosed by `{` and `}`. Every block introduces a new scope. The scope of a name declared inside a block begins immediately after the declaration statement and ends at the closing `}` of that block.

```hopper
function example(int x) int {
    int result = x        // 'result' comes into scope here
    if (x > 0) {
        int inner = x + 1 // 'inner' comes into scope here
        result = inner    // OK: 'inner' is in scope
    }                     // 'inner' goes out of scope here
    return result         // 'inner' is NOT in scope here — compile error if used
}
```

A conforming implementation MUST reject any use of a name before its declaration within the same block. A conforming implementation MUST reject any use of a name outside the block in which it was declared.

### 3.2.2 No Hoisting

Hopper does NOT hoist declarations. A variable declared on line N is not available on any line before N in the source text, even if the control flow could reach the use before the declaration at runtime. A conforming implementation MUST report a compile-time error for any such use.

### 3.2.3 No Shadowing Within the Same Scope

A name MUST NOT be redeclared within the same block scope. If a block already contains a binding for name `x`, introducing a second declaration of `x` in the same block is a compile-time error.

Shadowing across nested scopes — declaring `x` in an inner block when `x` is already declared in an outer block — is permitted by the grammar but SHOULD be diagnosed with a warning by a conforming implementation. When shadowing occurs, the inner declaration takes precedence within the inner block; the outer binding is inaccessible under that name for the duration of the inner scope.

```hopper
function shadow(int x) int {
    int y = x          // outer 'y'
    if (x > 0) {
        int y = x * 2  // inner 'y' shadows outer — implementation SHOULD warn
        return y       // refers to inner 'y'
    }
    return y           // refers to outer 'y'
}
```

### 3.2.4 Function Parameter Scope

Function parameters are in scope for the entire function body — from the opening `{` of the body block to the closing `}`. Parameters are treated as if they were declared at the top of the function body, before any statement in the body. A parameter name MUST NOT be redeclared as a local variable in the top-level function body scope.

```hopper
function add(int a, int b) int {
    // 'a' and 'b' are in scope throughout
    int sum = a + b
    return sum
}
```

### 3.2.5 Control-Flow Block Scope

The body block of an `if`, `else`, `while`, or `for` statement is a distinct nested scope. Variables declared inside these bodies are not visible outside.

```hopper
function check(int n) {
    if (n > 0) {
        int msg = 1   // scoped to this if-body
    }
    // 'msg' is NOT in scope here
}
```

The `for` loop initializer introduces a variable that is scoped to the for statement — it is visible in the loop condition, the loop update, and the loop body, but not after the closing `}` of the for body.

```hopper
function count() {
    for (int i = 0; i < 10; i = i + 1) {
        // 'i' is in scope here
    }
    // 'i' is NOT in scope here
}
```

### 3.2.6 Top-Level (Module) Scope

Names declared at the top level of a source file — functions, classes, enums, interfaces, `extern` declarations, and `const` declarations — occupy the *module scope*. Module scope is outside all block scopes. Names in module scope are visible throughout the entire file in which they appear and, when imported, in any importing file (see section 3.3 and section 8).

There is no ordering constraint for module-scope declarations: a function declared later in the file MAY be called by a function declared earlier. A conforming implementation MUST process all module-scope declarations before resolving any references within them.

### 3.2.7 Enum Scope

An `enum` declaration introduces a new scope for its variants. Variants are accessed with the qualified form `EnumName.VariantName`; they are NOT introduced into the enclosing scope as bare names.

```hopper
enum Color { Red = 0  Green  Blue }

function paint() {
    int c = Color.Red   // correct
    // int d = Red      // compile error — 'Red' is not in scope
}
```

### 3.2.8 Class Member Scope

Class fields and methods are not visible as bare names outside the class body. Inside a class method, field access MUST use an explicit `self` receiver (see section 3.3.3). There is no implicit injection of field names into the method body scope.
