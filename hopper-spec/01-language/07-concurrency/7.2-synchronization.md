## 7.2 Synchronization

### 7.2.1 No Built-in Synchronization Primitives

Hopper provides no built-in synchronization primitives. The language has no
mutex type, no semaphore type, no condition variable type, no lock statement,
no critical section keyword, and no `synchronized` annotation. No Hopper
statement is a synchronization operation.

Programs that require synchronization between threads MUST implement it
explicitly using OS system calls or inline assembly.

### 7.2.2 Futex-Based Synchronization

On Linux, the `futex` syscall (nr 202) is the kernel-level primitive on which
all POSIX thread synchronization objects are built. `futex` provides:

- **FUTEX_WAIT** (op=0): sleep the calling thread if `*uaddr == val`.
- **FUTEX_WAKE** (op=1): wake up to `val` threads sleeping on `uaddr`.
- **FUTEX_WAIT_PRIVATE** (op=128): same as FUTEX_WAIT but restricted to
  threads within the same process (faster; uses no global hash table).
- **FUTEX_WAKE_PRIVATE** (op=129): same as FUTEX_WAKE but process-private.

The `x86_64/src/thread.hop` module exposes `futex`, `futex_wait`, and
`futex_wake` as free functions. Programs MUST link these functions by declaring
an appropriate target binding in `hopper.json`.

#### Mutex Pattern

A minimal non-recursive mutex can be implemented using a single `int` word
stored in an `address`-typed location shared between threads:

- State 0 = unlocked.
- State 1 = locked, no waiters.
- State 2 = locked, one or more waiters sleeping in the kernel.

Lock: attempt `lock cmpxchg` to transition 0→1; if the word is not 0, call
`futex_wait(addr, 2)` after setting state to 2 via `lock xchg`.

Unlock: use `lock xchg` to set the word to 0; if the previous value was 2,
call `futex_wake(addr, 1)` to wake one waiter.

The `lock cmpxchg` and `lock xchg` instructions MUST be issued as inline
assembly (§7.3.2). The futex word MUST be naturally aligned (`int` is 8-byte
aligned in Hopper's type system; a 4-byte futex word MUST be accessed via
`address`-typed pointer arithmetic with verified alignment).

### 7.2.3 Semaphores

Two families of semaphore API are available:

**Futex-based (preferred for new code):** Implement a counting semaphore using
a futex word containing the current count. Decrement is attempted with
`lock cmpxchg`; if count is already zero, call `futex_wait`. Increment uses
`lock xadd` or `lock cmpxchg`; if there were waiters, call `futex_wake`.

**SysV semaphores (for cross-process use):** `semget`, `semop`, and `semctl`
(syscalls nr 64, 65, 66) are exposed in `x86_64/src/sem.hop`. SysV semaphores
are kernel-resident objects that survive process exit until explicitly destroyed
with `semctl(IPC_RMID)`. They are appropriate when synchronization is required
across unrelated processes that do not share memory.

### 7.2.4 Condition Variables

Hopper provides no condition variable abstraction. A condition variable MUST be
implemented using futex primitives following the standard futex condvar protocol:

1. Hold the mutex.
2. Copy the current sequence number (a futex word used as a generation counter).
3. Release the mutex.
4. Call `futex_wait(cond_addr, seq)` — the kernel will return immediately if
   the sequence number has already changed.
5. On return, re-acquire the mutex and check the actual condition.

Signal: increment the sequence number atomically; call `futex_wake(cond_addr, 1)`.
Broadcast: increment the sequence number atomically; call `futex_wake(cond_addr, INT_MAX)`.

### 7.2.5 Preventing Data Races Is the Programmer's Responsibility

The language does not enforce mutual exclusion at the type level. No Hopper
construct prevents two threads from accessing the same memory location
concurrently without synchronization. The programmer MUST:

1. Identify all memory locations that are accessed from more than one thread.
2. Choose an appropriate synchronization strategy for each such location.
3. Ensure that the implementation is correct — the compiler will not diagnose
   synchronization errors.

A program that accesses shared mutable state without adequate synchronization
has a data race and exhibits undefined behavior (§7.7.2).

### 7.2.6 Signal Safety

OS signal handlers run asynchronously and may interrupt any instruction,
including instructions that are part of a synchronization sequence. Functions
that are not async-signal-safe MUST NOT be called from a signal handler. In
particular:

- `futex` is async-signal-safe (it is a raw syscall).
- Heap allocators based on `allocate`/`deallocate` are NOT guaranteed to be
  async-signal-safe unless the implementation explicitly supports this.
- Any function that acquires a mutex that the interrupted thread may already
  hold MUST NOT be called from a signal handler (deadlock).

Programs MUST carefully limit the operations performed inside signal handlers.
