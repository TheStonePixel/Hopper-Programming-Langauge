## 7.4 Memory Ordering

### 7.4.1 Single-Threaded Memory Model

For single-threaded programs, Hopper's memory model is the LLVM memory model:
loads and stores are sequentially consistent with respect to the single thread
of execution. The compiler and the LLVM backend MAY reorder memory operations
that have no observable effect within the single thread, but MUST NOT eliminate
or reorder operations that affect observable program state as defined by the
sequential semantics of the language.

In single-threaded execution there are no data races (there is only one thread)
and therefore no undefined behavior arising from concurrency.

### 7.4.2 Multi-Threaded Memory Model

For multi-threaded programs, the memory model is NOT sequentially consistent.

Hopper does not specify a formal multi-threaded memory model. The effective
memory model is determined by:

1. **The hardware**: On x86-64, the Total Store Order (TSO) memory model
   applies at the microarchitecture level. TSO guarantees that all processors
   observe stores from a single processor in program order, but permits stores
   from different processors to be observed in different orders by different
   processors. In particular, a load MAY observe a stale (pre-store) value from
   another processor's store buffer.

2. **The compiler**: LLVM may reorder, merge, or eliminate loads and stores
   when it can prove (from single-threaded reasoning) that this does not change
   the result. Because LLVM does not know which variables are shared across
   threads unless told explicitly (via LLVM atomic intrinsics, which Hopper does
   not expose directly), it MAY transform shared-variable accesses in ways that
   break the programmer's intended ordering.

The combination of compiler reordering and hardware store buffering means that
**sequential consistency between threads is NOT guaranteed**. Programs MUST NOT
assume that a write by thread A to location X is immediately visible to thread
B, even if thread B executes after thread A in wall-clock time.

### 7.4.3 Happens-Before and Synchronization Edges

A *happens-before* relationship between two operations means that the effect of
the first operation is guaranteed to be visible to the second. In Hopper
programs, happens-before edges are established only by:

1. **`lock`-prefixed instructions** (§7.3.2): a `lock`-prefixed instruction is
   a full memory barrier on x86-64. All memory operations that appear before
   it in program order are complete before it executes; all memory operations
   that appear after it execute after it completes. This establishes a
   happens-before edge at that point in the executing thread.

2. **`mfence` / `sfence` / `lfence` instructions** (§7.3.4): explicit fence
   instructions in inline assembly establish ordering constraints as defined
   by the Intel/AMD architecture manuals.

3. **System calls with ordering semantics**: `futex_wait` and `futex_wake`
   include implicit memory barriers. A `futex_wake` that observes a change to
   the futex word and wakes a waiter ensures that all stores by the waking
   thread that precede the wake are visible to the woken thread after it returns
   from `futex_wait`. This is the mechanism by which futex-based mutexes provide
   their ordering guarantees.

4. **`fork`**: all stores by the parent before `fork` are visible to the child.
   After fork, the parent and child share no memory unless shared memory is
   explicitly established.

No other Hopper construct establishes a happens-before edge. In particular:

- Ordinary assignments do NOT establish happens-before across threads.
- Function calls do NOT establish happens-before across threads.
- The order in which threads are scheduled does NOT establish happens-before.

### 7.4.4 Visibility Without Synchronization

Without a happens-before edge established by a synchronization operation, a
write by thread A to location X is NOT guaranteed to ever become visible to
thread B. The compiler MAY cache the value of X in a register across any
number of loads. Thread B MAY observe any value that was previously valid for
X, not necessarily the most recently written one.

This applies even on x86-64, where the hardware itself provides TSO: the
compiler's register caching may prevent the hardware memory model from being
observed at all.

**Programs MUST ensure that all accesses to shared mutable state are covered
by appropriate synchronization (§7.2). Accesses not covered by synchronization
are data races with undefined behavior (§7.7.2).**

### 7.4.5 Volatile Accesses

Hopper has no `volatile` keyword. There is no mechanism in the language to
suppress compiler optimizations on a per-access basis. For memory-mapped I/O or
any access that must not be optimized away, programs MUST use inline assembly
to issue the access directly, bypassing the LLVM optimizer.

### 7.4.6 Relationship to C11

Programs that interoperate with C11 code via `extern` declarations (e.g.,
calling a pthreads mutex from Hopper) MUST ensure that the C11 synchronization
object's requirements are met. In particular:

- C11 `_Atomic` objects have defined behavior under concurrent access. Hopper
  has no equivalent type; programs passing Hopper variables to C atomic
  operations MUST manually ensure alignment and sizing requirements.
- C11 mutex lock/unlock operations establish happens-before. Hopper code that
  uses a C mutex via `extern` declarations benefits from those guarantees
  without any additional barrier instructions, because the mutex implementation
  itself includes the required barriers.
