## 7.7 Concurrency Safety

### 7.7.1 No Compile-Time Concurrency Safety Guarantees

Hopper provides no compile-time concurrency safety guarantees. The type system
does not track thread ownership, thread affinity, or whether a value may be
sent across a thread boundary. The compiler does not perform data-race
detection, lock-order analysis, or deadlock detection.

A conforming Hopper implementation is NOT REQUIRED to issue any diagnostic for
code that, when executed, would produce a data race. The absence of a compiler
error or warning does NOT indicate that a program is free of concurrency bugs.

Thread safety is entirely the programmer's responsibility.

### 7.7.2 Data Races Are Undefined Behavior

A **data race** occurs when:

1. Two or more threads access the same memory location concurrently (without
   one completing before the other begins), AND
2. At least one of the accesses is a write, AND
3. The accesses are not ordered by a synchronization operation that establishes
   a happens-before relationship between them (§7.4.3).

A program containing a data race has **undefined behavior**. This specification
does not define the outcome of executing such a program. A conforming
implementation MAY:

- Produce incorrect results silently.
- Crash.
- Expose stale, arbitrary, or partially-written data.
- Behave differently on each execution.
- Behave differently across compiler optimizations, CPU generations, or kernel
  versions.

The undefined behavior is not limited to the racy access itself. A data race
MAY corrupt the entire program's state, because the LLVM optimizer transforms
the program under the assumption that undefined behavior does not occur; if that
assumption is violated, optimizations may produce globally incorrect code.

Undefined behavior from a data race is distinct from a race condition in the
application logic (an error in how the program uses synchronization primitives,
such as checking a condition without holding a lock). Both are bugs; the former
is undefined at the language specification level, the latter is merely incorrect.

### 7.7.3 Deadlock

Hopper does not define behavior for deadlocked programs. A program that
deadlocks — for example, by acquiring mutexes in inconsistent order across
threads, or by waiting on a futex that no thread will ever wake — will block
indefinitely. The specification does not require any diagnostic, timeout, or
automatic recovery.

Programs MUST establish and enforce a consistent global lock-ordering discipline
to prevent deadlock. The compiler provides no assistance.

### 7.7.4 Signal Races

Concurrent signal delivery is subject to the same undefined behavior rules as
thread races. If a signal handler accesses memory that the interrupted thread is
also accessing, and the access in the signal handler is not protected by async-
signal-safe synchronization, the result is undefined behavior.

Programs MUST ensure that signal handlers access only:

- `volatile sig_atomic_t`-equivalent locations (a single machine word, read or
  written atomically on the target architecture).
- Values set before the signal mask was configured.
- Results of async-signal-safe syscalls.

Because Hopper has no `volatile` qualifier (§7.4.5), a single-word flag shared
between a signal handler and the main program MUST be protected by inline
assembly barriers or written via `lock xchg`.

### 7.7.5 No Runtime Race Detector

A conforming Hopper implementation is NOT REQUIRED to provide a runtime data-
race detector (analogous to ThreadSanitizer). A future version of the
specification MAY define an optional instrumentation mode for race detection.
Until such a mode is specified, programmers MUST use external tools (e.g.,
Valgrind helgrind, or re-implement the program in a language with a built-in
race detector) to validate concurrent programs.

### 7.7.6 Future Direction

*This section is informative.*

A future version of this specification MAY introduce one or more of the
following mechanisms to enable compile-time concurrency checking:

- **Ownership annotations**: a type modifier (e.g., `shared<T>`) that marks
  values intended for cross-thread sharing and requires the programmer to
  annotate sharing explicitly. The compiler could then reject unguarded accesses
  to `shared`-annotated locations.

- **Effect annotations**: a function-level annotation (e.g., `@thread_safe`)
  that causes the compiler to verify (under a simplified model) that the
  function's callee set consists only of other thread-safe functions.

- **Send/Sync interfaces**: interfaces (analogous to Rust's `Send` and
  `Sync` traits) that, when declared on a class, assert that instances MAY be
  safely moved across thread boundaries or accessed from multiple threads.
  Violating these would be a compile-time error.

None of these mechanisms are present in the current version of the language.
Programs MUST NOT assume any future mechanism will be backward-compatible with
current code. The introduction of compile-time concurrency checking, if and when
it occurs, MAY require source-level changes to programs that currently rely on
unguarded cross-thread access.

Until the specification is updated, the normative rule is: **Hopper programs
with data races have undefined behavior, and the language provides no tool to
prevent them at compile time.**
