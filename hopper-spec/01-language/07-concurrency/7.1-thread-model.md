## 7.1 Thread Model

### 7.1.1 Default Execution Model

A Hopper program MUST begin execution in a single thread of control. That
thread is created by the operating system when the process is started, before
the program's `entry` function is called.

The language provides no syntax for creating, naming, joining, or destroying
threads. There is no thread type, no `spawn` keyword, no `async` keyword, and
no `await` keyword. Parallelism is not expressible in the Hopper type system.

### 7.1.2 Creating Threads via OS System Calls

A program MAY create additional threads by invoking OS system calls through the
`linux` module. On Linux/x86-64, the canonical mechanism is the `clone` syscall
with the `CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD`
flag combination, which creates a new kernel thread sharing the caller's address
space.

The `CloneFlag` enum in `linux/interfaces/Process.hop` exposes these flags:

```
CLONE_VM      = 256
CLONE_FS      = 512
CLONE_FILES   = 1024
CLONE_SIGHAND = 2048
CLONE_THREAD  = 65536
```

A new thread starts executing at the function address passed to `clone` (or to
the wrapper used by the threading library). The new thread receives its own
stack, whose address and size MUST be provided by the caller. Allocating a
thread stack is the programmer's responsibility; the `Memory` contract's
`mmap` function with `MAP_PRIVATE | MAP_ANON` is the recommended mechanism.

Process-level forking is also available via `fork()` and `vfork()` in the
`Process` contract. `fork` creates a new process with a copy of the parent's
address space; the two processes share no memory after the fork unless shared
memory is explicitly established (§7.5).

### 7.1.3 Stack

Each thread has its own stack. The main thread's stack is allocated by the
operating system. Stacks for additional threads MUST be allocated by the
program before the `clone` call and MUST NOT overlap each other or any other
mapped region.

The language makes no provision for stack overflow detection. Stack overflows
produce undefined behavior (typically a segmentation fault from writing below
the stack guard page, if a guard page is installed).

### 7.1.4 No Thread-Local Storage Syntax

Hopper provides no `thread_local` storage class or equivalent syntax. Programs
that require per-thread state MUST implement it explicitly, for example by:

- Passing a context pointer as a function argument to the thread entry function.
- Allocating per-thread structures on the heap and storing the address in a
  caller-allocated struct passed to the thread at creation time.
- Using the Linux `set_tid_address` / `arch_prctl` mechanism for a raw
  FS-segment TLS slot (x86-64 only, requires inline assembly).

The `gettid()` function (syscall nr 186, exposed in `x86_64/src/thread.hop`)
returns the calling thread's kernel thread ID and MAY be used to index into
per-thread arrays.

### 7.1.5 No Built-in Scheduler

Hopper does not include a cooperative or work-stealing scheduler. All scheduling
is performed by the Linux kernel. Programs MUST NOT assume any particular thread
scheduling policy or ordering unless they establish that ordering through
explicit synchronization (§7.2).

`sched_yield()` (syscall nr 24, exposed in `x86_64/src/thread.hop`) is
available as a hint to the kernel that the calling thread is willing to
relinquish the CPU. It provides no synchronization guarantee.

### 7.1.6 Thread Identity

Each thread has a thread ID (`tid`) assigned by the kernel. The main thread's
tid equals the process ID (`pid`). Additional threads created by `clone` receive
distinct tids within the same thread group (`tgid`).

- `getpid()` returns the thread group ID (same for all threads in a process).
- `gettid()` returns the calling thread's individual kernel thread ID.
- `tgkill(tgid, tid, sig)` sends a signal to a specific thread.

### 7.1.7 Thread Termination

A thread terminates when:

1. Its entry function returns.
2. It calls `exit()` (which terminates the entire process and all threads).
3. It calls `exit_group()` (which terminates all threads in the thread group).
4. It receives an unhandled signal that terminates the process.

There is no `pthread_exit`-equivalent in the language. A threading library
built on top of the `clone` syscall MUST manage thread termination using
`set_tid_address` and `futex_wait` on the thread's tid address, which the
kernel zeros automatically when the thread exits.
