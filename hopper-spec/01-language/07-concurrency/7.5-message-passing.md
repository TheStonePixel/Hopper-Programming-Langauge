## 7.5 Message Passing

### 7.5.1 No Built-in Message Passing

Hopper provides no channel type, no actor abstraction, no mailbox, and no
built-in queue. The language does not define any message-passing primitive.
Programs that require message passing between threads or processes MUST
implement it using OS IPC mechanisms accessed through the `linux` module.

### 7.5.2 Pipes

Pipes are the simplest IPC mechanism. A pipe is a unidirectional byte stream
with a kernel-managed buffer. The `FileSystem` interface in `linux` exposes
`pipe(address pipefd)`, which creates a pipe and writes the read end and write
end file descriptors into `pipefd[0]` and `pipefd[1]`.

Properties of pipes:

- Writes up to `PIPE_BUF` bytes (at least 4096 bytes on Linux) are atomic:
  no partial write from another concurrent writer will be interleaved.
- Reads and writes larger than `PIPE_BUF` are NOT atomic.
- A write to a full pipe blocks the writing thread until the reader consumes
  data (unless `O_NONBLOCK` is set).
- A read from an empty pipe blocks the reading thread until data is available
  or all write ends are closed.
- Closing all write ends causes a read to return 0 (EOF).
- Writing to a pipe with no read ends remaining delivers `SIGPIPE` and
  returns `EPIPE`.

Pipes are suitable for producer-consumer communication between a parent and
child process after `fork`, or between two threads within a process. Because
the kernel serializes pipe reads and writes, no additional synchronization is
required for the data transfer itself; however, the application protocol
(framing, sequencing) is the programmer's responsibility.

### 7.5.3 Sockets

The `Socket` and `Network` interfaces in `linux` provide stream (`SOCK_STREAM`)
and datagram (`SOCK_DGRAM`) sockets for both inter-process communication (via
`AF_UNIX` domain sockets) and network communication (via `AF_INET` / `AF_INET6`).

`AF_UNIX` sockets offer:

- **Stream sockets**: reliable, ordered, connection-oriented byte streams.
  Similar to pipes but bidirectional and accessible to unrelated processes.
- **Datagram sockets**: unordered, unconnected, message-boundary-preserving.
  Datagrams up to `SO_SNDBUF` bytes are delivered atomically.
- **Seqpacket sockets** (`SOCK_SEQPACKET`): ordered, connection-oriented,
  with preserved message boundaries.

Socket operations are accessed via:

- `socket(domain, type, protocol)` — create a socket.
- `socketBind(fd, addr, addrlen)` — bind to an address (note: `bind` is a
  reserved keyword in Hopper; the function is named `socketBind`).
- `listen(fd, backlog)` — mark socket as passive.
- `accept(fd, addr, addrlen)` — accept a connection.
- `connect(fd, addr, addrlen)` — initiate a connection.
- `send(fd, buf, len, flags)` and `recv(fd, buf, len, flags)` — transfer data.

### 7.5.4 Shared Memory

Shared memory allows threads or processes to communicate by reading and writing
a common memory region without copying data through the kernel.

Two mechanisms are available:

**`mmap` with `MAP_SHARED | MAP_ANON`** (preferred for threads and
parent-child processes):

```
address region = mmap(0, size, PROT_READ | PROT_WRITE,
                      MAP_SHARED | MAP_ANON, -1, 0)
```

After `fork`, both parent and child see the same physical pages. Changes are
visible immediately across the shared mapping, subject to memory ordering
constraints (§7.4).

**`memfd_create` + `mmap`** (for unrelated processes):

1. Parent creates an anonymous file: `int fd = memfd_create("shm", 0)`.
2. Parent maps the file: `address region = mmap(..., fd, 0)`.
3. Parent sends `fd` to the child via a `sendmsg` with `SCM_RIGHTS` ancillary
   data over a `AF_UNIX` socket.
4. Child maps the same file descriptor.

Both processes now share the same physical memory.

**SysV shared memory** (`shmget`, `shmat`, `shmdt`, `shmctl`) is also
available via `x86_64/src/shm.hop` for compatibility. For new code, `mmap`
with `MAP_SHARED` is preferred.

**Critical:** shared memory is shared mutable state (§7.6). Programs MUST
protect concurrent accesses to a shared memory region with appropriate
synchronization (§7.2). The shared memory mechanism itself provides no
synchronization.

### 7.5.5 SysV Message Queues

The `x86_64/src/msg.hop` module exposes `msgget`, `msgsnd`, `msgrcv`, and
`msgctl`. SysV message queues are kernel-resident; each message has a type
field that can be used for selective receive. `msgsnd` and `msgrcv` are
individually atomic with respect to each other (a message is either sent or
not; it is never partially delivered).

SysV message queues are suitable for cross-process communication between
unrelated processes. For new code within a single process or between
parent/child processes, pipes or sockets are preferred.

### 7.5.6 Ordering Guarantees

The kernel provides the following ordering guarantees for IPC mechanisms:

- **Pipes**: writes are visible to the reader in the order they were written.
  A read that returns N bytes observes the first N bytes written since the
  last read. The kernel pipe buffer is a synchronization point: a successful
  `read` happens-after the `write` that produced those bytes.

- **`AF_UNIX SOCK_STREAM` sockets**: messages are delivered in order; the
  stream is a synchronization point equivalent to a pipe.

- **`AF_UNIX SOCK_DGRAM` sockets**: individual datagrams are atomic, but
  ordering between different datagrams is not guaranteed.

- **Shared memory**: NO ordering guarantees beyond the hardware memory model.
  Programs MUST use explicit synchronization (§7.2, §7.4).

- **SysV message queues**: the kernel serializes `msgsnd` and `msgrcv`.
  A successful `msgrcv` happens-after the `msgsnd` that produced the message.
