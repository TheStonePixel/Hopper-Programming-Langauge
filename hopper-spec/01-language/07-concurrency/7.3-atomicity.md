## 7.3 Atomicity

### 7.3.1 No Atomic Type

Hopper has no `atomic` type modifier, no `_Atomic` qualifier, and no
`atomic<T>` generic. The language provides no mechanism for declaring a
variable as atomically accessed, and the compiler does not generate atomic
load or store instructions for ordinary variable accesses.

### 7.3.2 Atomic Operations via Inline Assembly

Programs that require atomic read-modify-write operations MUST implement them
using inline assembly. On x86-64, the relevant instructions are:

| Operation | Instruction | Effect |
|-----------|-------------|--------|
| Atomic exchange | `lock xchg reg, [mem]` | Atomically replace `[mem]` with `reg`; old value returned in `reg` |
| Compare-and-swap | `lock cmpxchg [mem], reg` | If `[mem] == rax`, write `reg` to `[mem]`; else load `[mem]` to `rax` |
| Atomic add | `lock xadd [mem], reg` | Atomically add `reg` to `[mem]`; old value returned in `reg` |
| Atomic increment | `lock inc [mem]` | Atomically increment `[mem]` by 1 |
| Atomic decrement | `lock dec [mem]` | Atomically decrement `[mem]` by 1 |
| Atomic AND | `lock and [mem], reg` | Atomically AND `[mem]` with `reg` |
| Atomic OR | `lock or [mem], reg` | Atomically OR `[mem]` with `reg` |

The `lock` prefix on x86-64 ensures that the operation is indivisible with
respect to other CPUs and is a full memory barrier (§7.4.3).

#### Example: Atomic Compare-and-Swap (CAS)

```hopper
// Returns 1 if the swap succeeded (old value was expected), 0 otherwise.
// addr must point to a naturally-aligned 8-byte location.
function cas(address addr, int expected, int desired) int {
    int result
    asm {
        rax = expected
        rdx = desired
        rdi = addr
        lock cmpxchg [rdi], rdx
        // ZF is set if the exchange happened; sete writes 1 or 0 to al.
        sete al
        result = rax
    }
    return result
}
```

#### Example: Atomic Exchange

```hopper
// Atomically writes newVal to *addr and returns the old value.
function atomicExchange(address addr, int newVal) int {
    int old
    asm {
        rax = newVal
        rdi = addr
        lock xchg [rdi], rax
        old = rax
    }
    return old
}
```

These patterns MUST be placed in a hardware-specific module (e.g.,
`x86_64/src/`) and called through an interface, following the same
architecture separation used for SIMD and syscalls. Application source files
MUST NOT contain `lock` prefix instructions directly.

### 7.3.3 Atomicity of Ordinary Load and Store

Ordinary Hopper assignments (`x = y`, `ptr[i] = v`) compile to LLVM `load`
and `store` instructions with no atomicity guarantee beyond what the
underlying hardware provides for naturally-aligned accesses.

On x86-64:

- Aligned loads and stores of 1, 2, 4, and 8 bytes are architecturally
  guaranteed to be single-bus-transaction operations. Concurrent aligned reads
  and writes from different threads will not produce a "torn" value (a mix of
  old and new bytes).
- However, this hardware guarantee does NOT constitute an atomic operation in
  the C11 or LLVM sense. The compiler MAY reorder, eliminate, or transform
  loads and stores that it observes to be non-atomic, breaking programs that
  rely on this hardware behavior without memory ordering constraints.
- Unaligned accesses have NO atomicity guarantee, even on x86-64.

**Programs MUST NOT rely on the architectural torn-read guarantee as a
substitute for proper synchronization.** Use inline assembly atomics (§7.3.2)
or memory barriers (§7.4) wherever thread-safety is required.

### 7.3.4 No Atomic Fence Statement

Hopper has no `fence` keyword or memory barrier statement. Memory barriers MUST
be issued via inline assembly. On x86-64:

- `mfence` — full store-fence/load-fence (serializes all memory operations).
- `sfence` — store-fence (prevents store reordering with subsequent stores).
- `lfence` — load-fence (prevents load reordering with subsequent loads).

A `lock`-prefixed instruction also acts as a full memory barrier and is
typically faster than a standalone `mfence` when the atomic operation and the
barrier are needed together.

### 7.3.5 OS-Provided Atomic Syscalls

The `futex` syscall (§7.2.2) provides an atomic compare-and-sleep operation
that is guaranteed correct even across fork and exec boundaries within the same
physical memory region. For synchronization that does not require a
read-modify-write on the same word, `futex` is the preferred mechanism because
it avoids busy-waiting.

### 7.3.6 Alignment Requirements for Atomic Accesses

Any location used as the target of a `lock`-prefixed instruction MUST be
naturally aligned:

- A 4-byte futex word MUST be 4-byte aligned.
- An 8-byte counter MUST be 8-byte aligned.

Misaligned atomic accesses are a general-protection fault on x86-64 when
crossing a cache-line boundary, and produce undefined behavior on other
architectures. Programs MUST verify alignment before performing atomic
operations on dynamically-allocated or externally-provided memory.
