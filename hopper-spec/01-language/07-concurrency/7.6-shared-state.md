## 7.6 Shared State

### 7.6.1 Sharing Memory Between Threads

Hopper's type system does not distinguish between memory that is accessed by a
single thread and memory that is shared across multiple threads. A value of
type `address` is an untyped pointer; the language does not record whether the
pointed-to memory is private to the current thread or shared with others.

A program MAY share memory between threads by passing an `address` value from
the creating thread to a newly created thread (for example, as the stack or
context pointer argument to `clone`). Once the address is in the possession of
both threads, both threads may read and write the pointed-to memory. No
language construct prevents this.

### 7.6.2 No Compile-Time Race Detection

The compiler performs no alias analysis across thread boundaries, no
happens-before tracking, and no escape analysis for thread safety. There is no
`Send` trait, no `Sync` marker, and no ownership transfer required for
inter-thread sharing. The compiler WILL NOT diagnose a data race, even if one
is statically detectable.

Detecting data races is entirely the programmer's responsibility.

### 7.6.3 Requirements for Correct Shared-State Access

A program MUST satisfy ALL of the following conditions for every memory
location L that is accessed from more than one thread:

1. **Mutual exclusion for writes**: if any thread writes to L, no other thread
   MAY read or write L concurrently. Concurrent access is prevented by a
   synchronization operation (§7.2) that establishes a happens-before edge
   such that one access is ordered before the other.

2. **Read-only sharing**: if all threads only read L (no thread writes to L
   after it becomes shared), no synchronization is required for the reads
   themselves. However, a synchronization operation MUST establish happens-
   before between the last write to L (which set it up for sharing) and the
   first read by any thread other than the writer.

3. **Atomic read-modify-write**: if the update to L is a read-modify-write
   that MUST be seen as indivisible by other threads, it MUST be performed with
   a `lock`-prefixed instruction (§7.3.2), not an ordinary load followed by a
   store.

4. **Memory ordering**: after any update to L, the writing thread MUST issue a
   memory barrier (or use a `lock`-prefixed instruction, which implies a
   barrier) before signaling other threads that the update is complete (§7.4).

Failure to satisfy any of these conditions is a data race (§7.7.2).

### 7.6.4 Shared State Patterns

#### Pattern 1: Mutex-Protected Data

A struct containing a futex word and the protected data MUST be allocated in
memory accessible to all participating threads. Threads MUST acquire the mutex
before accessing the protected fields and release it afterward. No thread MAY
access the protected fields without holding the mutex.

#### Pattern 2: Producer-Consumer with Ring Buffer

A ring buffer in shared memory with a head index (written only by the producer)
and a tail index (written only by the consumer) can avoid mutex contention if:

- The producer uses `lock xadd` or `sfence` after writing a slot before
  updating the head.
- The consumer uses `lfence` before reading a slot after observing an updated
  tail.
- Only one producer and one consumer access the buffer (single-producer,
  single-consumer invariant). Multiple producers or consumers require a mutex
  or CAS loop.

#### Pattern 3: Once Initialization

To initialize a shared data structure exactly once, regardless of how many
threads race to initialize it:

1. Use a futex word as a state machine: 0=uninitialized, 1=initializing, 2=done.
2. Thread attempting to initialize: CAS 0→1. If CAS succeeds, initialize, then
   atomically set to 2 and `futex_wake` all waiters.
3. Thread observing state=1: `futex_wait(state_addr, 1)` until state becomes 2.
4. Thread observing state=2: proceed directly; the successful CAS + `futex_wake`
   establish happens-before with the `futex_wait` return.

#### Pattern 4: Reference Counting

An `int` reference count in shared memory, incremented with `lock xadd` and
decremented with `lock xadd`. When the decrement returns 1 (old value was 1,
new value is 0), the decrementing thread owns the object and MUST free it.
A `mfence` or `lock dec` is required before the free to ensure that no other
thread's writes to the object are reordered to after the free.

### 7.6.5 Global Variables and Shared State

The Hopper language forbids mutable global variables (§ clause 5 of Critical
Rules). This restriction eliminates an entire class of accidental shared state:
because there are no module-level variables, there are no variables that are
automatically shared between threads by virtue of being global.

All shared state MUST be explicitly allocated and its address explicitly
communicated to each participating thread. This makes the sharing relationship
visible in the program's source code, even though the compiler does not
enforce any access discipline.

### 7.6.6 Stack Variables Are Not Shared by Default

Each thread has its own stack (§7.1.3). Local variables declared inside a
function are allocated on the calling thread's stack and are NOT shared with
other threads unless the program explicitly passes a pointer to them across a
thread boundary.

Programs MUST NOT pass a pointer to a stack variable to another thread and then
allow the stack frame to be destroyed (by returning from the function) while
the other thread still holds the pointer. Such a pointer becomes dangling, and
any access through it is undefined behavior. This is a special case of the
general lifetime rules in Chapter 5 and is not diagnosed by the compiler.
