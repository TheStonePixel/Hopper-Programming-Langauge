## 1.3 Line Termination

### 1.3.1 The NEWLINE Token

Line terminators are **lexically significant** in Hopper. Unlike whitespace, they are not discarded; they are delivered to the parser as `NEWLINE` tokens. The ANTLR4 lexer rule is:

```antlr
NEWLINE : '\r'? '\n' ;
```

This rule matches both:

- **LF** (`\n`, U+000A): a single LINE FEED character.
- **CRLF** (`\r\n`, U+000D U+000A): a CARRIAGE RETURN immediately followed by a LINE FEED.

Both sequences produce exactly one `NEWLINE` token regardless of length in bytes. The `\r` is consumed as part of the `NEWLINE` token and is not separately visible to the parser.

A standalone `\r` (CR not followed by LF) does not match this rule. As specified in section 1.1.5, a standalone CR MUST cause a compilation error.

### 1.3.2 Role in Statement Separation

`NEWLINE` tokens are the primary mechanism for separating statements within a block. Hopper does not use a statement terminator character such as `;`. Within a block body, statements are separated by one or more `NEWLINE` tokens:

```antlr
block
    : NEWLINE* '{'
      NEWLINE*
      ( statement (NEWLINE+ statement)*
        NEWLINE* )?
      '}'
    ;
```

Each `statement` alternative in the parser grammar produces a complete statement on a single logical line. A statement MUST NOT be split across multiple lines unless it is expressed as a single unbroken token sequence without an intervening `NEWLINE` token.

### 1.3.3 Multiple Consecutive NEWLINEs

Multiple consecutive `NEWLINE` tokens — produced by blank lines in the source — are treated as a single separator wherever the grammar uses `NEWLINE+` or `NEWLINE*`. This means any number of blank lines between two statements has the same effect as a single newline: they separate the statements but do not produce additional parse nodes. Blank lines are a visual formatting choice and carry no semantic weight.

For example, the following two blocks are semantically identical:

```hopper
{
    int x = 1
    int y = 2
}
```

```hopper
{
    int x = 1


    int y = 2
}
```

### 1.3.4 NEWLINE Before an Opening Brace

An opening brace `{` that begins a block MAY appear on the same line as the statement or declaration that introduces the block, or it MAY appear on the following line. In both cases the `NEWLINE` between the header and the brace is consumed by the leading `NEWLINE*` of the `block` rule and does NOT act as a statement separator. The following two forms are therefore equivalent:

```hopper
function add(int a, int b) int {
    return a + b
}
```

```hopper
function add(int a, int b) int
{
    return a + b
}
```

The same rule applies to `if`, `while`, `for`, `class`, `interface`, `struct`, `enum`, and `entry` bodies: the opening brace may appear on any line, and any intervening newlines before the `{` are consumed without effect.

### 1.3.5 NEWLINE at Top Level

At the top level of a source file (the `program` rule), `NEWLINE*` appears before the first declaration, between declarations, and after the last declaration:

```antlr
program
    : NEWLINE* (topLevelDecl NEWLINE*)* EOF
    ;
```

Leading blank lines, blank lines between top-level declarations, and trailing blank lines are all accepted and ignored. A top-level declaration MUST be complete on a single logical line or within a single block structure; a raw `NEWLINE` token that appears inside a top-level declaration where the grammar does not expect it MUST cause a parse error.

### 1.3.6 NEWLINE in Enum and Struct Bodies

Within `enumDecl` and `structDecl` bodies, members are separated by `NEWLINE+` in the same way as statements in a block. Blank lines between members are accepted. The grammar explicitly requires at least one `NEWLINE` between successive members; placing two members on the same line MUST cause a parse error.

### 1.3.7 NEWLINE and Contract Clauses

`requires` and `ensures` contract clauses are separated from the function signature and from each other by `NEWLINE+`:

```antlr
functionDecl
    : 'function' Identifier '(' paramList? ')' type (NEWLINE+ contractClause)* block
    ;
```

Each contract clause MUST appear on its own line. The opening brace of the function body, which follows the last contract clause (or the signature if there are none), is consumed by `block`'s leading `NEWLINE*` and does not require a separate line.

### 1.3.8 Line Count and Diagnostics

A conforming implementation MUST track the line number of each token for diagnostic purposes. Line numbering MUST start at 1. Each `NEWLINE` token increments the current line number by 1. Diagnostic messages (errors, warnings) MUST include the source file name, line number, and column number of the relevant token.

Column numbers MUST start at 1 and MUST be measured in Unicode code points from the start of the line, not in bytes.
