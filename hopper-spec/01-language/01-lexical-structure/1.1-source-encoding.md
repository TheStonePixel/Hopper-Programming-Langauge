## 1.1 Source Encoding

### 1.1.1 Required Encoding

Every Hopper source file MUST be encoded as UTF-8 as defined by RFC 3629. A conforming implementation MUST reject any source file whose byte content is not valid UTF-8 with a compilation error naming the file and the byte offset of the first invalid byte sequence.

Hopper does not support any other encoding. Implementation-specific facilities that accept source from non-UTF-8 sources (e.g., a language server receiving input from an editor that uses a different encoding) MUST transcode that input to UTF-8 before presenting it to the compiler and MUST report an error if the transcoding is lossy or ambiguous.

### 1.1.2 Byte Order Mark

A UTF-8 byte order mark (BOM), the byte sequence `0xEF 0xBB 0xBF` which encodes the code point U+FEFF, MUST NOT appear at the beginning of a Hopper source file. A conforming implementation MUST report a compilation error if a BOM is present. This prohibition exists because the BOM is not part of the source text in UTF-8 and its presence indicates either a tooling error or a file that was re-encoded from a UTF-16 or UTF-32 source.

A BOM appearing anywhere other than the first three bytes of the file is the code point U+FEFF encoded normally; it is not treated specially and is subject to the restrictions on valid source code characters (section 1.1.3).

### 1.1.3 Valid Code Points

The source file, once decoded from UTF-8, consists of a sequence of Unicode scalar values (code points in the range U+0000 to U+10FFFF, excluding the surrogate range U+D800 to U+DFFF). The following constraints apply:

- The maximum permitted code point is U+10FFFF. Code points above U+10FFFF cannot be encoded in valid UTF-8 and are therefore rejected by the UTF-8 validity check in section 1.1.1.
- Unicode surrogate code points (U+D800 through U+DFFF) are not valid Unicode scalar values. They cannot appear in valid UTF-8 and are rejected by the UTF-8 validity check.
- The null code point U+0000 MUST NOT appear anywhere in a source file. A conforming implementation MUST report a compilation error if U+0000 is encountered.

### 1.1.4 Code Points in Tokens

The lexer operates on the decoded code-point stream. Most tokens are restricted to ASCII characters:

- `IntegerLiteral`, `HexLiteral`, `FloatLiteral`, `CharLiteral`, `StringLiteral`, `UnicodeLiteral`, `Identifier`, `AsmBlock`, `LINE_COMMENT`, `WS`, and `NEWLINE` are all defined using character class notation that matches only ASCII code points (U+0000 through U+007F).
- Non-ASCII Unicode code points (U+0080 and above) MAY appear only inside `StringLiteral` content, where they are valid as literal Unicode characters within the quoted string body. They MUST NOT appear in `CharLiteral`, identifiers, keywords, numeric literals, whitespace, or comments.

> **Rationale.** Restricting identifiers to ASCII eliminates ambiguities from visually confusable Unicode characters and simplifies tooling. The restriction may be relaxed in a future version of this specification.

### 1.1.5 Line Endings

Hopper accepts two line-ending conventions:

- **LF only**: the single code point U+000A (LINE FEED, `\n`).
- **CRLF**: the two-code-point sequence U+000D U+000A (CARRIAGE RETURN followed by LINE FEED, `\r\n`).

A standalone CR (U+000D) not followed by LF is not a recognized line ending. A conforming implementation MUST report a compilation error if a standalone CR is encountered.

Both LF and CRLF produce a single `NEWLINE` token (see section 1.3). A single source file MAY contain a mixture of LF and CRLF line endings; each terminator is normalized independently.

### 1.1.6 File Termination

A source file MUST end with a line terminator. A conforming implementation SHOULD warn if a source file does not end with a newline. The final `NEWLINE` is included in the token stream and consumed by the top-level `NEWLINE*` clause of the `program` rule.

A source file MAY be empty (zero bytes). An empty source file is a valid Hopper source file that declares no top-level items.
