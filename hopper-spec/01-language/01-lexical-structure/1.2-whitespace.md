## 1.2 Whitespace

### 1.2.1 Definition

Whitespace in Hopper consists of the following ASCII characters:

| Character | Code Point | Name |
|-----------|-----------|------|
| ` `       | U+0020    | SPACE |
| `\t`      | U+0009    | CHARACTER TABULATION (horizontal tab) |

No other characters are whitespace in Hopper. In particular:

- Vertical tab (U+000B), form feed (U+000C), and other Unicode space characters (U+00A0 NO-BREAK SPACE, U+2003 EM SPACE, etc.) are NOT whitespace. Their appearance in source code outside of a string literal MUST cause a compilation error.
- Carriage return (U+000D) is not independently whitespace; it is only recognized as part of the CRLF line-ending sequence (see section 1.3).

The ANTLR4 lexer rule for whitespace is:

```antlr
WS : [ \t]+ -> skip ;
```

### 1.2.2 Significance

Whitespace is **lexically insignificant**. The `WS` rule carries the ANTLR4 `-> skip` directive, which means whitespace tokens are discarded immediately by the lexer and are never delivered to the parser. Whitespace tokens do not appear in the token stream and have no effect on parsing or semantics.

The following pairs of lines are lexically identical:

```hopper
int x = 42
int    x   =   42
```

```hopper
function add(int a, int b) int {
function add( int a , int b ) int {
```

### 1.2.3 Whitespace as Token Separator

Although whitespace is insignificant as a token in its own right, it serves as a separator between tokens that would otherwise be ambiguous. The maximal munch rule (section 1, overview) means that without whitespace, adjacent token-forming characters are consumed into a single token. For example:

- `intx` is a single `Identifier` token, not the keyword `int` followed by `Identifier` `x`. A space or other separator is required: `int x`.
- `0xdeadbeef` is a single `HexLiteral` token, not `0` followed by `x` followed by `deadbeef`.
- `1.5` is a single `FloatLiteral` token, not `1` followed by `.` followed by `5`.

Whitespace is therefore necessary wherever two adjacent characters would otherwise be consumed as a single token.

### 1.2.4 Indentation

Hopper is not an indentation-sensitive language. Indentation is purely a visual convention for human readers. A conforming implementation MUST NOT infer block structure, statement boundaries, or any other grammatical information from indentation.

The canonical indentation style for Hopper source code is four spaces per level. A conforming formatter SHOULD enforce this style. Tabs in indentation are accepted by the compiler but SHOULD be converted to spaces by a formatter.

### 1.2.5 Trailing Whitespace

Trailing whitespace on a line (spaces or tabs before the line terminator) is permitted and ignored by the lexer. A conforming formatter SHOULD strip trailing whitespace.

### 1.2.6 Whitespace Inside the AsmBlock Token

The `AsmBlock` lexer rule captures the content between `asm` and the closing `}` as a single opaque token. Inside an `AsmBlock`, horizontal whitespace retains its textual content because it is not re-scanned by the `WS` rule; the assembly body parser is responsible for interpreting spacing within that content. This is an exception to the general rule that whitespace is insignificant and applies only to the raw text content of the `AsmBlock` token.
