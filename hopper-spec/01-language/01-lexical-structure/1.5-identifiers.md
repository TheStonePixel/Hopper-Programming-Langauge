## 1.5 Identifiers

### 1.5.1 Lexical Form

An identifier is a name that refers to a declared entity: a variable, function, class, interface, enum, type alias, module, or template. The ANTLR4 lexer rule for identifiers is:

```antlr
Identifier : [a-zA-Z_][a-zA-Z0-9_]* ;
```

An identifier MUST begin with an ASCII letter (`a`–`z`, `A`–`Z`) or an ASCII underscore (`_`). Subsequent characters MUST each be an ASCII letter, an ASCII decimal digit (`0`–`9`), or an ASCII underscore. An identifier MUST contain at least one character.

Non-ASCII Unicode characters are not permitted in identifiers. A conforming implementation MUST report a compilation error if a non-ASCII character appears in a position where an identifier is expected.

### 1.5.2 Case Sensitivity

Hopper identifiers are **case-sensitive**. The identifiers `count`, `Count`, and `COUNT` are three distinct names. A conforming implementation MUST treat them as distinct in all contexts: name lookup, type checking, and code generation.

### 1.5.3 Length

There is no specified maximum length for an identifier. A conforming implementation MUST accept identifiers of any length composed of valid characters. A conforming implementation MAY impose a practical upper limit on identifier length provided that limit is at least 1023 characters; if such a limit is imposed, the implementation MUST document it.

### 1.5.4 Keywords Cannot Be Used as Identifiers

A string that matches a keyword (see section 1.6) is always lexed as that keyword token, never as an `Identifier`. Because ANTLR4 keyword matching takes precedence over the `Identifier` rule for exact keyword strings, programs MUST NOT use keywords as identifier names.

For example, the following are all ill-formed:

```hopper
int int = 5       // 'int' is a keyword, not a variable name
int return = 0    // 'return' is a keyword
int class = 1     // 'class' is a keyword
```

A conforming implementation MUST report a compilation error when a keyword appears in a position where an identifier is expected.

### 1.5.5 Contextual Keywords

Three identifiers are **contextual keywords**: `value`, `address`, and `size`. They appear as keywords in the `::` context (scope-resolution operator expressions) and as special names in the `fieldName` and `paramName` grammar rules:

```antlr
fieldName
    : Identifier
    | 'value'
    | 'address'
    | 'size'
    ;
```

Despite having special meaning in these positions, `value`, `address`, and `size` remain valid `Identifier` tokens in all other positions — they are NOT reserved and may be used as field names, parameter names, and local variable names. For example:

```hopper
class Node {
    int value         // 'value' as a field name — valid
    address ptr       // 'address' as a field name — valid
}

function process(int value, int size) int {   // contextual keywords as param names — valid
    return value + size
}
```

However, `address` also appears as a **type keyword** in the grammar (the raw-pointer type). When `address` is used as a type annotation, it is the keyword form, not an identifier. Context determines which interpretation applies:

- In a type position (before a variable name in a declaration, after `(` in a parameter list, as a return type), `address` is the pointer type keyword.
- In a field-access or parameter-name position (after a type keyword), `address` is a contextual identifier.

A conforming implementation MUST resolve this ambiguity using the grammar production being parsed, not by post-hoc renaming.

### 1.5.6 Naming Conventions

The following naming conventions are used throughout the Hopper standard library and are RECOMMENDED for user programs. They are not enforced by the compiler.

| Entity | Convention | Example |
|--------|------------|---------|
| Types (class, struct, interface, template) | `PascalCase` | `LinuxIO`, `FileSystem`, `String` |
| Functions and methods | `camelCase` | `readLine`, `scanByte16` |
| Variables and parameters | `camelCase` | `bufSize`, `lineCount` |
| Enum variants | `UPPER_SNAKE_CASE` | `STDIN`, `BACKSPACE`, `COLOR_RED` |
| Constants (top-level `const`) | `UPPER_SNAKE_CASE` | `MAX_RETRIES` |
| Module-level free functions | `camelCase` | `open`, `read`, `write` |

### 1.5.7 Leading Underscore Convention

Identifiers beginning with a single underscore (e.g., `_buf`, `_impl`) are by convention reserved for implementation-internal use or generated code. User programs SHOULD NOT define identifiers beginning with underscore in exported interfaces, but the compiler does not enforce this restriction. Identifiers beginning with two underscores (e.g., `__builtin`) are reserved for future use by this specification; a conforming implementation SHOULD warn when such identifiers appear in user source code.

### 1.5.8 The `self` Identifier

The identifier `self` is a keyword (see section 1.6). Within a class method or constructor body, `self` refers to the receiver object. Programs MUST NOT use `self` as the name of a variable, parameter, or field. All method calls on the receiver MUST use an explicit `self.` prefix; there is no implicit `self` dereference.

### 1.5.9 Identifiers in Template Instantiation

When a template is instantiated, the type argument is provided in angle brackets: `List<int>`, `Map<String, int>`. The type argument is a type, not an identifier, and follows the `type` grammar rule. The template name itself (`List`, `Map`) is an identifier. The special template name `String` is handled by the `templateName` production, which allows `String` — a keyword — to appear as a template class name.
