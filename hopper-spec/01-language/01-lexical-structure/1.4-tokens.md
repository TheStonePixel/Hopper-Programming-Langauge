## 1.4 Tokens

### 1.4.1 Token Categories

The Hopper lexer produces tokens in the following categories. All token types are exhaustive: every byte in a valid source file is consumed by exactly one token rule or by the skip rules for whitespace and comments.

| Category | Token Types |
|----------|-------------|
| Literals | `IntegerLiteral`, `HexLiteral`, `FloatLiteral`, `StringLiteral`, `CharLiteral`, `UnicodeLiteral` |
| Names | `Identifier` |
| Keywords | One token per keyword (see section 1.6) |
| Operators and punctuation | Matched as literal strings in parser rules |
| Structure | `NEWLINE`, `AsmBlock` |
| Discarded | `WS`, `LINE_COMMENT` |

### 1.4.2 Literal Tokens

#### IntegerLiteral

Matches one or more decimal digits.

```antlr
IntegerLiteral : [0-9]+ ;
```

An `IntegerLiteral` represents a non-negative integer value in base 10. The value MUST fit in a 64-bit signed integer (`int`, i.e., i64) at the point of use; if it does not, a conforming implementation MUST report a compilation error. Leading zeros are permitted syntactically (`007` is the integer 7) but SHOULD be rejected by a conforming linter or formatter as confusing.

Note that the unary minus operator `-` is not part of this token. The expression `-1` is parsed as the unary negation operator applied to `IntegerLiteral` `1`, not as a single negative integer literal token.

#### HexLiteral

Matches the prefix `0x` followed by one or more hexadecimal digits.

```antlr
HexLiteral : '0x' [0-9a-fA-F]+ ;
```

A `HexLiteral` represents a non-negative integer value in base 16. Both upper-case (`A`–`F`) and lower-case (`a`–`f`) hex digits are accepted. The value MUST fit in a 64-bit signed integer at the point of use. `HexLiteral` takes precedence over `IntegerLiteral` by virtue of maximal munch: `0x10` is one `HexLiteral` token (value 16), not `IntegerLiteral` `0` followed by identifier `x10`.

#### FloatLiteral

Matches one or more decimal digits, a literal dot, and one or more decimal digits after the dot.

```antlr
FloatLiteral : [0-9]+ '.' [0-9]+ ;
```

A `FloatLiteral` represents a double-precision IEEE 754 floating-point value. Both the integer part and the fractional part MUST contain at least one digit; `.5` and `1.` are NOT valid `FloatLiteral` tokens. Scientific notation (e.g., `1.5e10`) is not supported. The lexer's maximal munch rule means `1.5` is always one `FloatLiteral` token, not `IntegerLiteral` `1` plus `.` plus `IntegerLiteral` `5`.

#### StringLiteral

Matches a double-quoted sequence of characters, with support for escape sequences.

```antlr
StringLiteral : '"' (~["\r\n\\] | '\\' .)* '"' ;
```

A `StringLiteral` begins and ends with a double-quote character (`"`). The body may contain:

- Any Unicode code point other than `"`, `\r`, `\n`, or `\`.
- An escape sequence: a backslash `\` followed by any single character.

A `StringLiteral` MUST NOT span more than one line; an unescaped newline (LF or CR) inside a string literal MUST cause a lexical error. The recognized escape sequences and their meanings are specified in section 1.7.3.

The `StringLiteral` token includes the surrounding double-quote characters. The compiler strips the quotes and processes escape sequences during compilation.

#### CharLiteral

Matches a single-quoted character, with support for a single escape sequence.

```antlr
CharLiteral : '\'' (~['\r\n\\] | '\\' .) '\'' ;
```

A `CharLiteral` begins and ends with a single-quote character (`'`). The body contains exactly one character, which is either:

- Any single code point other than `'`, `\r`, `\n`, or `\`.
- An escape sequence: a backslash `\` followed by any single character.

A `CharLiteral` representing zero characters or more than one character (after escape processing) MUST cause a compilation error.

#### UnicodeLiteral

Matches the prefix `U+` followed by one or more hexadecimal digits.

```antlr
UnicodeLiteral : 'U+' [0-9a-fA-F]+ ;
```

A `UnicodeLiteral` denotes a Unicode code point by its scalar value. The value MUST be in the range U+0000 to U+10FFFF. Surrogate code points (U+D800 to U+DFFF) MUST cause a compilation error. A `UnicodeLiteral` is used to denote a specific code point value as an integer; it evaluates to the integer value of the code point at compile time.

### 1.4.3 The Identifier Token

```antlr
Identifier : [a-zA-Z_][a-zA-Z0-9_]* ;
```

An `Identifier` token is produced for any name-forming sequence that is not reclassified as a keyword. See section 1.5 for the full specification of identifiers, and section 1.6 for keyword reclassification.

### 1.4.4 Keyword Tokens

ANTLR4 does not define separate lexer rules for keywords. Instead, the literal strings used in parser rules (e.g., `'int'`, `'function'`, `'return'`) cause ANTLR4 to generate implicit lexer rules that are tested before the `Identifier` rule. A conforming implementation MUST treat keyword strings as keyword tokens, not as identifiers.

The full list of keywords and their token names is given in section 1.6.

### 1.4.5 Operator and Punctuation Tokens

Operators and punctuation characters (`+`, `-`, `*`, `/`, `%`, `=`, `==`, `!=`, `<`, `<=`, `>`, `>=`, `&&`, `||`, `!`, `~`, `&`, `|`, `^`, `<<`, `>>`, `(`, `)`, `{`, `}`, `[`, `]`, `,`, `.`, `;`, `::`, `...`) are matched as anonymous literal tokens within the parser rules. The lexer recognizes them via ANTLR4's implicit literal mechanism. Multi-character operators (`==`, `!=`, `<=`, `>=`, `&&`, `||`, `<<`, `>>`, `::`) are matched by maximal munch before their single-character components.

### 1.4.6 The NEWLINE Token

```antlr
NEWLINE : '\r'? '\n' ;
```

The `NEWLINE` token is the normalized line terminator. It is delivered to the parser and participates in statement separation. See section 1.3 for the full specification.

### 1.4.7 The AsmBlock Token

```antlr
AsmBlock : 'asm' [ \t]* '{' ~[}]* '}' ;
```

The `AsmBlock` token captures an entire inline assembly block as a single opaque unit. The token begins at the `a` of `asm` and ends at the first `}` encountered. The content between `{` and `}` is treated as raw text by the lexer and is not scanned for sub-tokens. The `AsmBlock` token takes precedence over the `Identifier` rule for the text `asm` because it appears later in the ANTLR4 rule list but is longer (maximal munch gives it priority). See section 1 for the assembly parsing model.

### 1.4.8 Discarded Tokens

Two token types are immediately discarded by the lexer and are never delivered to the parser:

- **`WS`** (`[ \t]+`): horizontal whitespace. See section 1.2.
- **`LINE_COMMENT`** (`'//' ~[\r\n]*`): a line comment. See section 1.8.

A conforming implementation MAY retain discarded tokens for tooling purposes (e.g., formatting, documentation extraction) but MUST NOT allow them to influence parsing or semantics.

### 1.4.9 Token Precedence Summary

When multiple rules could match at a given position, the following precedence rules determine which token is produced:

1. The rule that produces the longest match wins (maximal munch).
2. When two rules produce the same length match, the rule that appears earlier in the ANTLR4 grammar wins.
3. Keywords take precedence over `Identifier` for strings that are exact keyword matches.
4. `AsmBlock` takes precedence over `Identifier` for the text `asm` followed by optional whitespace and `{`, because the full `AsmBlock` pattern is longer.
5. `HexLiteral` takes precedence over `IntegerLiteral` for any input beginning with `0x`, because `HexLiteral` produces a longer match.
6. `FloatLiteral` takes precedence over `IntegerLiteral` for any input of the form `digits.digits`, because `FloatLiteral` produces a longer match.
