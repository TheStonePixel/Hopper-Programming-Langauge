## 1.7 Literals

### 1.7.1 Overview

A literal is a syntactic form that denotes a fixed, compile-time value directly in source code. Hopper supports the following literal forms:

| Literal | Token | Example |
|---------|-------|---------|
| Integer | `IntegerLiteral` | `42`, `0`, `1024` |
| Hexadecimal integer | `HexLiteral` | `0xFF`, `0x1A2B` |
| Floating-point | `FloatLiteral` | `3.14`, `0.5` |
| String | `StringLiteral` | `"hello"`, `"line\n"` |
| Character | `CharLiteral` | `'A'`, `'\n'` |
| Unicode code point | `UnicodeLiteral` | `U+0041`, `U+1F600` |
| Boolean | Keywords `true`/`false` | `true`, `false` |
| Null pointer | Keyword `null` | `null` |

Literals have no leading sign. A negative numeric literal is represented as the unary negation operator (`-`) applied to a non-negative literal.

### 1.7.2 Integer Literals

```antlr
IntegerLiteral : [0-9]+ ;
```

An `IntegerLiteral` is a sequence of one or more decimal digit characters (`0`–`9`). It denotes a non-negative integer value in base 10.

**Type.** An `IntegerLiteral` has type `int` (i64, 64-bit signed integer) unless context requires otherwise. The value MUST be in the range 0 to 9223372036854775807 (2⁶³ − 1) when used as an `int`. When an `IntegerLiteral` is used in a position that requires a `byte` (i8), `char` (i8), `bool` (i1), or `unsigned int` (u64), the value MUST be in the range appropriate to that type or a compilation error MUST be reported.

**Leading zeros.** Leading zeros are syntactically accepted (`007` denotes the integer 7) but do not indicate octal notation. Hopper has no octal literal syntax. A conforming linter SHOULD warn on integer literals with leading zeros.

**Examples:**

```hopper
int x = 0
int y = 255
int z = 1000000
int big = 9223372036854775807    // 2^63 - 1, the maximum int value
```

### 1.7.3 Hexadecimal Literals

```antlr
HexLiteral : '0x' [0-9a-fA-F]+ ;
```

A `HexLiteral` begins with the prefix `0x` (lowercase only; `0X` is NOT accepted) followed by one or more hexadecimal digit characters (`0`–`9`, `a`–`f`, `A`–`F`). It denotes a non-negative integer value in base 16.

**Type.** A `HexLiteral` has type `int` (i64) unless context requires otherwise. The value MUST be in the range 0 to 0x7FFFFFFFFFFFFFFF when used as an `int`.

**Use cases.** Hexadecimal literals are commonly used for hardware addresses, bitmasks, and other values where the bit pattern is significant:

```hopper
bind 0x00000004 = reset::address
strict int uart_dr = 0x40021000
int mask = 0xFF
int flags = 0x0001
```

### 1.7.4 Floating-Point Literals

```antlr
FloatLiteral : [0-9]+ '.' [0-9]+ ;
```

A `FloatLiteral` consists of an integer part (one or more decimal digits), a literal decimal point (`.`), and a fractional part (one or more decimal digits). Both parts are mandatory; neither `1.` nor `.5` is a valid `FloatLiteral`.

**Type.** A `FloatLiteral` has type `float` (IEEE 754 double-precision, 64-bit). The value is converted from decimal to binary floating-point using round-to-nearest-even. If the value is outside the representable range of double-precision (greater than approximately 1.8 × 10³⁰⁸), a conforming implementation MUST report a compilation error.

**Scientific notation.** Hopper does not support scientific notation (e.g., `1.5e10`). Programs requiring very large or very small floating-point values MUST use arithmetic expressions.

**Examples:**

```hopper
float pi = 3.14159
float small = 0.001
float neg = -2.718      // unary minus applied to FloatLiteral 2.718
```

### 1.7.5 String Literals

```antlr
StringLiteral : '"' (~["\r\n\\] | '\\' .)* '"' ;
```

A `StringLiteral` is a sequence of characters enclosed in double-quote characters (`"`). The body may contain any number of:

- **Literal characters**: any Unicode code point other than `"`, `\r`, `\n`, or `\`.
- **Escape sequences**: a backslash followed by exactly one character.

**Type.** A `StringLiteral` has type `string` (i8*, a null-terminated byte string). The compiler stores the string content — with escape sequences decoded — as a null-terminated sequence of bytes in the data segment of the output binary. The `string` value at runtime is a pointer to the first byte.

**Encoding.** The string content is stored as its raw byte representation in the source file (UTF-8). No transcoding is performed. Programs that embed multibyte UTF-8 sequences in string literals receive those bytes verbatim in the output.

**Escape sequences.** The following escape sequences are recognized:

| Sequence | Meaning | Byte value |
|----------|---------|------------|
| `\n` | LINE FEED (newline) | 0x0A |
| `\t` | CHARACTER TABULATION (horizontal tab) | 0x09 |
| `\r` | CARRIAGE RETURN | 0x0D |
| `\\` | BACKSLASH | 0x5C |
| `\"` | DOUBLE QUOTATION MARK | 0x22 |
| `\0` | NULL character | 0x00 |
| `\'` | APOSTROPHE | 0x27 |

A backslash followed by any other character is implementation-defined behavior. A conforming implementation SHOULD warn and SHOULD treat the sequence as the literal character that follows the backslash.

**Multiline strings.** A string literal MUST NOT span multiple source lines. To include a newline character in a string, use the `\n` escape sequence. To compose a multi-line string, concatenate expressions using the `+` operator on a `String` object or write the bytes explicitly.

**Examples:**

```hopper
string greeting = "Hello, world!\n"
string path = "/usr/local/bin"
string tab = "col1\tcol2"
string quote = "She said \"hello\""
string backslash = "C:\\Users\\name"
```

### 1.7.6 Character Literals

```antlr
CharLiteral : '\'' (~['\r\n\\] | '\\' .) '\'' ;
```

A `CharLiteral` is a single character enclosed in single-quote characters (`'`). The body contains exactly one character, which is either a literal character (any code point other than `'`, `\r`, `\n`, or `\`) or a single escape sequence (backslash followed by exactly one character).

**Type.** A `CharLiteral` has type `byte` (i8). Its value is the byte value of the encoded character. For ASCII characters, this is the ASCII code point value (0–127). For non-ASCII characters, the result is the first byte of the UTF-8 encoding. This behavior means that `CharLiteral` is only well-defined for ASCII characters; non-ASCII characters in a `CharLiteral` SHOULD be rejected by a conforming linter.

**Escape sequences.** The same escape sequences as for `StringLiteral` are recognized (see section 1.7.5).

**Zero-character and multi-character literals.** A `CharLiteral` body that, after escape processing, contains zero characters or more than one character MUST cause a compilation error.

**Examples:**

```hopper
byte letter = 'A'        // value: 65
byte newline = '\n'      // value: 10
byte nul = '\0'          // value: 0
byte slash = '\\'        // value: 92
```

### 1.7.7 Unicode Code Point Literals

```antlr
UnicodeLiteral : 'U+' [0-9a-fA-F]+ ;
```

A `UnicodeLiteral` denotes a Unicode scalar value using the conventional `U+XXXX` notation. The prefix is `U+` (capital U, literal plus sign) followed by one or more uppercase or lowercase hexadecimal digits.

**Type.** A `UnicodeLiteral` has type `int`. Its value is the numeric code-point value of the specified Unicode scalar. For example, `U+0041` has value 65 (LATIN CAPITAL LETTER A).

**Valid range.** The value MUST be a valid Unicode scalar value: 0 (U+0000) through 1,114,111 (U+10FFFF), excluding the surrogate range 55,296 (U+D800) through 57,343 (U+DFFF). A value outside this range or within the surrogate range MUST cause a compilation error.

**Number of hex digits.** Any number of hex digits is accepted (`U+41`, `U+0041`, `U+000041` are all the same value). Leading zeros are insignificant.

**Examples:**

```hopper
int latin_a = U+0041         // 65, LATIN CAPITAL LETTER A
int euro = U+20AC            // 8364, EURO SIGN
int snowman = U+2603         // 9731, SNOWMAN
int smiley = U+1F600         // 128512, GRINNING FACE emoji
```

### 1.7.8 Boolean Literals

The boolean literals `true` and `false` are keywords (see section 1.6) that denote the two values of type `bool` (i1).

- `true` has type `bool` and value 1 (i1 one).
- `false` has type `bool` and value 0 (i1 zero).

Boolean literals are the only valid boolean values; Hopper does not implicitly convert integers to `bool`. To convert an integer to `bool`, use `cast<bool>(x)`.

**Examples:**

```hopper
bool done = false
bool valid = true
if (done) { ... }
```

### 1.7.9 The Null Literal

The keyword `null` is the null pointer literal. It has type `address` and represents the null pointer (a pointer with integer value 0).

`null` MUST only be used in a context where a pointer type (`address`, `string`, or any class type passed as an address) is expected. Using `null` in a numeric arithmetic context is undefined behavior. A conforming implementation SHOULD warn when `null` is compared with `==` or `!=` against a non-pointer type.

**Examples:**

```hopper
address buf = null
if (buf == null) { ... }
```

### 1.7.10 Literals in Enum Variants

Enum variant values MAY be specified as integer literals, hexadecimal literals, or string literals:

```hopper
enum Color {
    Red   = 0
    Green = 1
    Blue  = 2
}

enum Escape {
    Tab    = 0x09
    Newline = 0x0A
}

enum Mode {
    Read  = "r"
    Write = "w"
}
```

Unvalued variants inherit an automatically incremented integer value (see the grammar specification for `enumVariant`). Mixing integer-valued and string-valued variants in the same enum MUST cause a compilation error.
