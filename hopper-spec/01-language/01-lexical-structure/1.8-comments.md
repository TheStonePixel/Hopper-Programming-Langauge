## 1.8 Comments

### 1.8.1 Line Comments

Hopper supports exactly one form of comment: the **line comment**. A line comment begins with the two-character sequence `//` and extends to, but does not include, the line terminator. The ANTLR4 lexer rule is:

```antlr
LINE_COMMENT : '//' ~[\r\n]* -> skip ;
```

The character class `~[\r\n]*` matches any sequence of characters that are neither `\r` (U+000D) nor `\n` (U+000A). This means the comment body is terminated at the first carriage return or line feed encountered after `//`.

The line comment token carries the `-> skip` directive and is **immediately discarded** by the lexer. Line comment tokens do not appear in the token stream delivered to the parser and have no effect on parsing, semantics, or code generation.

### 1.8.2 Comment Termination

A line comment does NOT consume the line terminator that ends it. The `\n` or `\r\n` that follows the comment body is recognized as a `NEWLINE` token by the subsequent application of the `NEWLINE` rule. This ensures that a comment on the same line as a statement does not suppress the `NEWLINE` that separates it from the next statement.

For example:

```hopper
int x = 42    // declare x
int y = x + 1 // y is one more than x
```

The tokenization of this fragment is:

```
Identifier('int') Identifier('x') '=' IntegerLiteral('42') NEWLINE
Identifier('int') Identifier('y') '=' Identifier('x') '+' IntegerLiteral('1') NEWLINE
```

The comment text is not present. The `NEWLINE` between the two declarations is produced from the actual newline character that follows the comment, which is not consumed by the `LINE_COMMENT` rule.

### 1.8.3 No Block Comments

Hopper does NOT support block comments (also known as multiline comments, C-style `/* ... */`, or nested `(* ... *)`). Any occurrence of `/*` in source code is not a comment delimiter; it is a `//` line comment that begins a line comment body, or it is the division operator `/` followed by the multiplication operator `*`. In the absence of a preceding `//`, the character `/` is an operator token and `*` is an operator token, and they do not form a comment.

Programs MUST NOT rely on `/* ... */` syntax. A conforming implementation MUST NOT accept `/* ... */` as a comment. Introducing block-comment syntax is reserved for a future version of this specification.

### 1.8.4 Comment Content

The content of a line comment (everything after `//` up to but not including the line terminator) is opaque to the compiler. Any sequence of non-newline characters MAY appear inside a comment, including:

- Other `//` sequences (which do not start nested comments).
- String delimiters `"` or `'`.
- Operator characters, keywords, and any other token-forming characters.
- Non-ASCII Unicode code points (subject to the UTF-8 encoding requirement of section 1.1).

The compiler MUST NOT interpret the content of a comment as source code.

### 1.8.5 Documentation Comments

Hopper does not currently define a documentation comment syntax. Some tools in the Hopper ecosystem use `///` (triple-slash) by convention to denote API documentation, but this is not specified by this version of the language. A `///` line is lexically identical to a `//` line comment and is discarded by the compiler in the same way.

A future version of this specification MAY define a normative documentation comment syntax. Programs SHOULD use `///` for documentation comments to maximize forward compatibility.

### 1.8.6 Comments and the AsmBlock Token

Comments are not recognized inside an `AsmBlock` token. Because the lexer matches the entire `asm { ... }` body as a single opaque token using the rule `AsmBlock : 'asm' [ \t]* '{' ~[}]* '}' ;`, the character sequence `//` inside an assembly block is part of the raw text and is not treated as a comment. The assembly body parser is responsible for its own comment handling, if any.

### 1.8.7 Comment-Only Lines and Blank Lines

A line that contains only a comment (after optional leading whitespace) results in no tokens in the stream other than the `NEWLINE` that terminates it. This `NEWLINE` is indistinguishable from a blank line's `NEWLINE`. Both are handled by the `NEWLINE*` and `NEWLINE+` clauses of the grammar and are treated as insignificant separators between declarations and statements.

A source file that consists entirely of comments is valid and produces no compiled output.
