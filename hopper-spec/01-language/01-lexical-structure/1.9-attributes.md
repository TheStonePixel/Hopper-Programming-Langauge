## 1.9 Attributes

*Not yet specified.*

This section is reserved for the specification of compile-time attributes — syntactic annotations that modify the behavior of a declaration without changing its runtime semantics. Examples from other systems languages include attributes for controlling alignment, linkage, inlining, deprecation, and platform-specific code generation.

Hopper does not currently define an attribute syntax. No attribute form is accepted by the grammar at this time. A conforming implementation MUST NOT accept any attribute-like syntax (e.g., `@name`, `#[name]`, `[[name]]`) as valid Hopper source.

When an attribute system is introduced, it will be specified in this section. Until then, programs MUST NOT use any such syntax, and any occurrence of it in source code MUST cause a compilation error.

> **Design note.** Several behaviors that attributes typically control in other languages are handled in Hopper through alternative mechanisms:
>
> - **Alignment**: the `strict` declaration provides memory-mapped register access with implied alignment requirements.
> - **Inlining**: the compiler decides inlining; no source annotation is currently available.
> - **Linkage**: `extern function` declarations control external linkage for C ABI functions.
> - **Hardware binding**: the `bind` declaration places a function pointer at a hardware address.
> - **Deprecation**: not yet supported.
> - **Platform guards**: handled entirely at build time through `hopper.json` target bindings; no source-level conditional compilation exists.
