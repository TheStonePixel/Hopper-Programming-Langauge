## 8.7 Linkage

This section specifies how Hopper symbols are emitted in LLVM IR, what linkage attributes they receive, how names are mangled for overloaded and template functions, and how `extern`, `bind`, and `strict` declarations interact with the linker.

### 8.7.1 Regular Functions

Every non-extern, non-overloaded function is emitted as an LLVM `define` directive with the function's plain Hopper name as the LLVM symbol:

```llvm
define i64 @write(i8* %p0, i64 %p1, i8* %p2) {
entry:
    ...
}
```

LLVM linkage is not explicitly specified by the compiler for regular functions; LLVM defaults to `external` linkage for `define` directives. All Hopper functions are therefore externally visible in the LLVM IR. The linker may perform dead-code elimination at link time; the compiler does not mark functions as private or internal by default.

**Inline functions:** When the `binding.inline` flag is set in `hopper.json` (indicating the implementation file is a thin translation layer), all non-extern functions in the implementation AST are marked with the LLVM attribute `alwaysinline`:

```llvm
define i64 @write(i8* %p0, i64 %p1, i8* %p2) alwaysinline {
```

### 8.7.2 Extern Functions

`extern function` declarations are emitted as LLVM `declare` directives, which tell LLVM that the function exists in a separately compiled object (typically a C library or the OS):

```hopper
extern function memset(address dst, int c, int n) address
```

emits:

```llvm
declare i8* @memset(i8*, i64, i64)
```

Variadic extern functions (declared with `...`) emit a variadic LLVM `declare`:

```hopper
extern function printf(string fmt, ...) int
```

emits:

```llvm
declare i64 @printf(i8*, ...)
```

If a concrete (non-extern) function definition exists for the same name in the same compilation unit, the compiler MUST suppress the `declare` — the `define` acts as its own forward declaration in LLVM IR. Duplicate `declare` directives for the same name are also suppressed.

### 8.7.3 Entry Point

The `entry` declaration compiles to an LLVM function named after the identifier used in the `entry` keyword. The return type is always `i64`.

```hopper
entry main {
    ...
}
```

emits:

```llvm
define i64 @main() {
entry:
    ...
    ret i64 0
}
```

A parameterized entry with `int argc, string[] argv` emits:

```llvm
define i64 @main(i32 %argc, i8** %argv) {
```

Note: the `argc` parameter is received as `i32` (matching the C ABI) and is sign-extended to `i64` internally.

When a non-`main` identifier is used (e.g. `entry startup { ... }`), the compiler emits the entry function as `@startup` and also emits a thin `@main` stub that calls `@startup` and returns its result. This ensures the C runtime can find the program entry point regardless of the entry identifier used.

If the `entry` block does not contain an explicit `return` statement, the compiler emits `ret i64 0` implicitly.

### 8.7.4 Function Overload Mangling

When multiple top-level functions share the same name (overloaded functions), each overload is emitted under a mangled LLVM symbol name. The mangling scheme is:

```
<name>__<type1>_<type2>_..._<typeN>
```

where each type is the normalized Hopper type of the corresponding parameter, with non-alphanumeric characters replaced by underscores. If there are no parameters, the mangled name is `<name>__`.

Examples:

| Hopper declaration | Mangled LLVM name |
|---|---|
| `function read(int fd, address buf, int count) int` | `read__int_address_int` |
| `function open(string path, int flags, int mode) int` | `open__string_int_int` |
| `function close(int fd) int` | `close__int` |
| `function exit()` | `exit__` |

The mangled name for each overload is registered in `functionReturnTypes` so the compiler can find the return type when generating call sites.

A call site to an overloaded function resolves the correct mangled name by matching the argument types against the registered overload group. If no overload matches the argument types, the compiler MUST report an error.

Same-signature duplicates (two imported files both defining a function with identical name and parameter types) are silently deduplicated: only the first occurrence is emitted.

### 8.7.5 Template Function Mangling

Template function specializations are emitted under a mangled name that encodes the type parameter and all parameter types:

```
__tmpl_<name>__<typeParam>__<paramType1>__<paramType2>...
```

Examples:

| Hopper declaration | Mangled LLVM name |
|---|---|
| `template function cast<int>(address src) int` | `__tmpl_cast__int__address` |
| `template function cast<float>(int src) float` | `__tmpl_cast__float__int` |
| `template function cast<address>(int src) address` | `__tmpl_cast__address__int` |

Template function mangling is designed to produce globally unique LLVM symbols for each specialization. The mangled names are registered in `templateFuncRegistry` for call-site resolution.

### 8.7.6 Class Method Mangling

Class methods are emitted as top-level LLVM functions with names of the form:

```
ClassName_methodName
```

The first parameter is always the implicit `self` pointer (`%class.ClassName*`). For example:

```hopper
class Buffer {
    function write(int fd, address buf, int count) int {
        ...
    }
}
```

emits:

```llvm
define i64 @Buffer_write(%class.Buffer* %p0, i64 %p1, i8* %p2, i64 %p3) {
```

Constructor and destructor methods use:

```
ClassName_constructor
ClassName_destructor
```

Class operators use:

```
ClassName_op_<safe-name>
```

See §8.6.3 for the operator safe-name table.

### 8.7.7 String Constants

String literals used anywhere in the compilation unit are collected into a deduplication pool during code generation. Each unique string value is emitted once as a private unnamed LLVM global constant:

```llvm
@.str.0 = private unnamed_addr constant [6 x i8] c"hello\00"
@.str.1 = private unnamed_addr constant [12 x i8] c"world\0a\00"
```

String constants are `private` — they are not externally visible symbols. The `unnamed_addr` attribute permits the linker to merge identical string constants across object files (when LTO is applied).

### 8.7.8 Bind Declarations (Bare-Metal)

`bind` declarations place function pointers at specific hardware addresses for use in interrupt vector tables:

```hopper
bind 0x00000004 = reset::address
bind 0x00000008 = nmi::address
```

These are emitted as LLVM global variables at the specified address using the `@llvm.used` mechanism and section attributes. The compiler emits a pointer-to-function global variable for each `bind` entry. The variable is placed in a linker section that the linker script maps to the hardware address.

### 8.7.9 Strict Declarations (MMIO Registers)

`strict` declarations name memory-mapped hardware registers:

```hopper
strict int uart_dr = 0x40021000
```

These are not emitted as LLVM global variables. Instead, they are registered in `mmioBindings` with their hardware address and LLVM type. All reads and writes to the named identifier within function bodies are compiled directly to LLVM volatile load/store instructions targeting the literal hardware address, cast to the appropriate pointer type:

```llvm
; Read from uart_dr:
%1 = inttoptr i64 1073807360 to i64*
%2 = load volatile i64, i64* %1

; Write to uart_dr:
%3 = inttoptr i64 1073807360 to i64*
store volatile i64 %value, i64* %3
```

The `volatile` attribute is mandatory for MMIO registers to prevent the optimizer from eliminating or reordering the access.

### 8.7.10 Heap Runtime Declarations

For all targets except `armv6-bare`, the compiler emits forward declarations for the C heap allocator:

```llvm
declare i8* @malloc(i64)
declare void @free(i8*)
```

These correspond to `allocate N` and `deallocate ptr` expressions in Hopper source. On the `armv6-bare` target, these declarations are suppressed and `allocate`/`deallocate` MUST NOT be used in bare-metal programs (no heap is available without a custom runtime).

When a build is compiled in debug mode (not `--release`), the compiler also emits:

```llvm
declare void @abort()
```

for use by contract-checking code (`requires`/`ensures` clause violations call `abort()`).
