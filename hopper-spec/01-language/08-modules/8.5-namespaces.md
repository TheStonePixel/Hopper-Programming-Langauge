## 8.5 Namespaces

Hopper has no namespace system. All top-level names — functions, classes, structs, interfaces, enums, aliases, templates — exist in a single flat global scope after all imports have been merged. There is no `namespace`, `module`, `package`, or `using` keyword.

### 8.5.1 Flat Global Scope

When the compiler merges all imported module ASTs, every top-level name from every imported file is placed into the same global namespace. Names do not carry any module qualifier after import. A function `open` from `LinuxSyscalls.hop` is simply `open` in the merged scope; it is not `linux.open` or `x86_64.open`.

This means:

- There is no qualified-name syntax such as `pkg.func()` or `Module::Type`.
- Names from imported modules are used directly without any prefix.
- Two imported modules that both define a top-level name with the same identifier produce a name collision.

### 8.5.2 Name Collision

If two imported modules define top-level names with the same identifier, the behavior is implementation-defined. The current implementation does not detect all name collisions at the AST level; instead, collisions between non-overloaded functions of the same name and parameter-type signature are detected during LLVM IR emission when duplicate `define` directives would be generated.

A conforming implementation SHOULD detect name collisions between non-function top-level declarations (classes, structs, interfaces, enums) and MUST report a compile error. For functions, see §8.6.2 (overloading).

Programs SHOULD avoid name collisions by using descriptive, unique, globally distinct names for all top-level declarations. The standard library module naming convention uses PascalCase for class and interface names (`LinuxIO`, `X86SIMD`) to minimize collision risk with application-defined names.

### 8.5.3 Enum Variant Namespacing

Enum variants are accessed through the enum type name using dot notation:

```hopper
enum Color { Red = 0   Green = 1   Blue = 2 }
enum Direction { North = 0   South = 1   East = 2   West = 3 }

// Variants are accessed as Color.Red, Direction.North — not as plain Red or North
Color c = Color.Green
Direction d = Direction.East
```

This provides a limited form of namespacing for named integer constants. The variant names `Red`, `Green`, `Blue` do NOT exist as plain identifiers in the global scope; they MUST be accessed as `Color.Red`, `Color.Green`, `Color.Blue`. This is the primary mechanism in Hopper for avoiding collisions among named constants.

The compiler resolves `Color.Green` during code generation by looking up the enum type `Color` in `enumTypes`, then looking up the variant `Green` within that enum. The result is an integer literal inline at the call site.

### 8.5.4 Method Namespacing via Type Name

Method calls require an explicit receiver expression:

```hopper
obj.method(args)
```

This provides implicit namespacing: `buf.write(fd, data, len)` is unambiguous even if multiple classes define a method named `write`, because the receiver type determines which method is called. The compiler resolves `obj.method(args)` to the LLVM IR call `ClassName_method(obj, args)` where `ClassName` is the static type of `obj`.

There is no ambiguity between a class method `write` and a free function `write` because method calls always require an explicit receiver (the dot notation) while free function calls use plain call syntax (`write(fd, buf, len)`).

### 8.5.5 Interface Type Names

Interface names occupy the global namespace like any other type name. The binding mechanism in `hopper.json` creates a type alias from the interface name to the concrete implementation class name. For example, if `IO` is an interface and the implementation class is `X86LinuxIO`, the compiler registers `alias IO = X86LinuxIO`. Both names refer to the same underlying type. Programs MAY use either name, but SHOULD use the interface name (`IO`) for portability.

### 8.5.6 No `using` or `open` Import

There is no syntax to selectively bring specific names from a module into scope while excluding others. An import always brings all top-level declarations from the resolved files into scope. Selective import is not supported.

### 8.5.7 Naming Recommendations

To minimize name collisions in a flat global namespace, library authors SHOULD follow these conventions:

- Use PascalCase for class, interface, struct, and enum type names.
- Use camelCase for free function names.
- Choose names that include the semantic domain, e.g. `socketBind` instead of `bind` (which is a grammar keyword in Hopper), `epollCreate` instead of `create`.
- Prefix internal helper names with an underscore.
- Prefer full words over abbreviations for globally-scoped names.
