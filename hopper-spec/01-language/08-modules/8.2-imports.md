## 8.2 Imports

An import declaration brings the names defined in another module into the current compilation unit. Hopper has two import forms: the *interface import* (preferred) and the *module import* (legacy).

### 8.2.1 Import Syntax

The import grammar (as matched by the compiler's preprocessor step before ANTLR4 parsing) is:

```
import Identifier (',' Identifier)* from Identifier   // interface import or named module import
import Identifier                                       // bare module import (legacy)
```

Examples:

```hopper
import IO from linux                   // interface import — single interface
import IO, FileSystem from linux       // interface import — two interfaces from same package
import SIMD from x86_64               // interface import — hardware capability
import io from arch                   // legacy module import (lowercase)
import tty from linux                 // legacy module import (lowercase)
```

Import declarations MUST appear on their own lines. They MAY be placed anywhere among the top-level declarations of a module file. Multiple import declarations MAY appear in a single file. The order of import declarations does not affect which names are visible, because the compiler resolves all imports before codegen begins.

### 8.2.2 Interface Import

An interface import has the form:

```hopper
import InterfaceName from packageName
```

where `InterfaceName` starts with an uppercase letter (by convention) and `packageName` is the name of a package directory.

**Resolution algorithm:**

1. The compiler treats the name before `from` as a binding key.
2. It walks up the directory tree from the importing source file's location, looking for a `hopper.json` file.
3. In the first `hopper.json` found, it reads the `targets.<target>.<InterfaceName>` object. If no such binding exists, the compiler falls back to legacy module resolution (§8.2.3).
4. The binding object MUST contain three fields:
   - `from` — the package name. MUST match the package name in the import statement. A mismatch is a compile error.
   - `interface` — path to the interface `.hop` file (relative to `hopper.json`).
   - `implementation` — path to the implementation `.hop` file (relative to `hopper.json`).
5. Both files are resolved to absolute paths and checked for existence. A missing file is a compile error.
6. The interface file is parsed and its AST is merged into the current compilation unit (unless already visited).
7. The implementation file is parsed and its AST is merged into the current compilation unit (unless already visited).

The compiler maintains a *visited set* of absolute file paths across the entire recursive import resolution. Any file that is already in the visited set MUST be skipped. This prevents duplicate definitions when multiple modules import the same package.

**Effect on scope:**

After an interface import, all of the following from both the interface file and the implementation file are in scope in the importing module:

- Free functions (both `function` and `extern function` declarations)
- Class declarations
- Interface declarations
- Struct declarations
- Enum declarations
- Type aliases
- Template declarations and template functions

The `entry` declaration from an imported file is never inherited (see §8.1.3).

**Multiple interfaces from one package:**

```hopper
import IO, FileSystem from linux
```

This is shorthand for two separate interface imports from the same package. Both `IO` and `FileSystem` MUST have binding entries in `hopper.json` under the same target. The compiler resolves each name independently: if `IO` and `FileSystem` both bind to the same implementation file (e.g. `LinuxSyscalls.hop`), that file MUST be loaded only once (enforced by the visited set).

**Type alias injection:**

When an interface import's implementation file contains a class whose name differs from the binding key, the compiler automatically registers a type alias: `bindingKey → concretClassName`. This allows programs to declare variables of the interface type name while actually holding an instance of the concrete class. For example, if `IO` binds to an implementation class named `X86LinuxIO`, the compiler registers `alias IO = X86LinuxIO`, so `IO io = IO()` compiles correctly.

**Example `hopper.json` binding:**

```json
{
  "targets": {
    "host": {
      "IO": {
        "from":           "linux",
        "interface":      "modules/linux/interfaces/IO.hop",
        "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
      },
      "FileSystem": {
        "from":           "linux",
        "interface":      "modules/linux/interfaces/FileSystem.hop",
        "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
      },
      "SIMD": {
        "from":           "x86_64",
        "interface":      "modules/x86_64/interfaces/SIMD.hop",
        "implementation": "modules/x86_64/src/SIMD.hop"
      }
    }
  }
}
```

### 8.2.3 Module Import (Legacy)

A legacy module import has the form:

```hopper
import symbolName from moduleName
import symbolName1, symbolName2 from moduleName
```

where both `symbolName` and `moduleName` are typically lowercase by convention.

**Resolution algorithm:**

1. No binding lookup is performed in `hopper.json`.
2. The compiler walks up the directory tree from the importing source file looking for a `modules/<moduleName>/` directory.
3. If found, it looks for `modules/<moduleName>/src/<symbolName>.hop`. If `src/` does not exist, it looks directly in `modules/<moduleName>/<symbolName>.hop`.
4. For a bare `import moduleName` (no `from`), the compiler loads all `.hop` files in `modules/<moduleName>/src/` (or `modules/<moduleName>/` if no `src/` exists).
5. Each resolved file is checked against the visited set and parsed if not yet loaded.

The compiler MUST report an error of the form:

```
Module '<moduleName>' not found
  Run: hopper install <moduleName>
```

if no `modules/<moduleName>/` directory is found anywhere in the directory tree up to the filesystem root.

**Legacy module import is deprecated for application code.** It SHOULD only be used within `hopper/nonconform/` legacy code that has not yet been migrated to the interface import system. New programs MUST use the interface import form.

### 8.2.4 Import Order and Merging

Imports are collected in order of appearance and resolved sequentially. When a file is resolved:

- Its functions are prepended to the current AST's function list (placed before the importing file's own functions).
- Its classes, structs, enums, interfaces, aliases, templates, and template functions are prepended to the respective lists.
- Its `bind` and `strict` declarations are prepended to their lists.

Prepending ensures that imported definitions are in scope before the importing file's definitions are processed, which supports the case where a local class calls a free function from an import.

### 8.2.5 Error Conditions

A conforming implementation MUST report a compile error when:

- An interface import's binding in `hopper.json` specifies a `from` field that does not match the package name in the import statement.
- The `interface` file path in a binding does not exist.
- The `implementation` file path in a binding does not exist.
- A legacy module import names a module directory that does not exist.
- A legacy module import names a symbol file that does not exist within the module directory.
- An import is syntactically malformed (e.g., missing `from`, bare `import` of a non-existent identifier).

A conforming implementation SHOULD include the name of the importing source file in error messages about import resolution failures.
