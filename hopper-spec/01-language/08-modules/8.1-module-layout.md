## 8.1 Module Layout

A module file is a plain UTF-8 text file with the `.hop` extension. It contains zero or more top-level declarations in any order. There is no required header, no package declaration, and no module-name declaration at the top of the file. The file name need not match any class name, interface name, or other identifier within the file.

### 8.1.1 Top-Level Declaration Kinds

The following declaration kinds MAY appear at the top level of a module file, in any order:

| Declaration Kind | Keyword(s) | Description |
|---|---|---|
| Import declaration | `import` | Brings names from another module into scope (see §8.2) |
| Function declaration | `function` | A named free function with a return type |
| Procedure declaration | `function` (no return type) | A named free function that returns void |
| Extern function declaration | `extern function` | A forward declaration of a C ABI function |
| Template function declaration | `template function` | A type-parameterized free function (compile-time dispatch) |
| Class declaration | `class` | A named type with fields, methods, constructor, destructor |
| Struct declaration | `struct` | A layout-only named type with no methods |
| Bitfield declaration | `bitfield` | A bit-level packed layout type |
| Interface declaration | `interface` | A named set of method signatures |
| Enum declaration | `enum` | A named set of integer or string constants |
| Template declaration | `template` | A parameterized class instantiated at use sites |
| Alias declaration | `alias` | A type alias: `alias Name = ExistingType` |
| Entry declaration | `entry` | The program entry point (at most one per compilation unit) |
| Bind declaration | `bind` | Place a function pointer at a hardware address (bare-metal) |
| Strict declaration | `strict` | Name a memory-mapped hardware register (bare-metal) |

The ANTLR4 grammar rule for a top-level declaration is:

```antlr
topLevelDecl
    : functionDecl
    | templateFunctionDecl
    | structDecl
    | classDecl
    | templateDecl
    | enumDecl
    | importDecl
    | aliasDecl
    | entryDecl
    | bindDecl
    | strictDecl
    | bitfieldDecl
    | interfaceDecl
    ;
```

A conforming implementation MUST accept any ordering of these declarations within a file. Forward references are permitted: a function MAY call another function or reference a class that is declared later in the same file or in any imported module, because the compiler resolves all names after merging all module ASTs.

### 8.1.2 Import Declarations

`import` declarations MAY appear anywhere among the top-level declarations of a module file. The compiler strips `import` lines before the ANTLR4 parser sees the file; the grammar itself does not parse import syntax at this phase. Imports are collected into a list and resolved after parsing the rest of the file.

Because import declarations are stripped before parsing, they MUST appear on lines by themselves with no other tokens. The full syntax is:

```
import Identifier (',' Identifier)* from Identifier
import Identifier
```

The first form is the standard interface or named-module import. The second form is a bare-module import (legacy).

### 8.1.3 Entry Declaration

A compilation unit MUST contain at most one `entry` declaration. An `entry` declaration MUST appear in the entry-point source file, not in any imported module. If an `entry` declaration is encountered in an imported file, it is silently ignored: the compiler never inherits an entry point from an import.

Three syntactic forms of `entry` exist:

```hopper
// Parameterless entry block
entry main {
    ...
}

// Entry block with parameters (argc/argv)
entry main(int argc, string[] argv) {
    ...
}

// Entry at a hardware address (bare-metal)
entry reset = startup::address
```

### 8.1.4 Const Declarations

The grammar permits a `const` statement form inside function/method bodies:

```hopper
const int LIMIT = 100
```

Top-level `const` declarations (outside any function or class) are NOT part of the current grammar. Compile-time named constants at module scope SHOULD be expressed as `enum` declarations. This design is intentional: named integer groups belong in `enum`, and the grammar will not add top-level `const` in future versions.

### 8.1.5 No Namespace or Package Header

A Hopper module file contains no package declaration, module declaration, or namespace header. There is no syntax of the form `module foo.bar;` or `package io;` at the top of a file. The file name is not normatively significant: a file named `Foo.hop` is not required to contain a class named `Foo`. The build tool (`hopper.json`) is the authoritative source of package identity and dependency relationships.

### 8.1.6 Top-Level Mutable Variables

Top-level mutable variable declarations (`int x = 5` at module scope) DO NOT exist in Hopper. All mutable state MUST be declared inside a class or passed as function parameters. A conforming implementation MUST reject any top-level variable declaration that is not a `const`, `alias`, `enum`, `strict`, or `bind` declaration.
