## 8.6 Symbol Resolution

Symbol resolution in Hopper occurs after the compiler has merged all imported module ASTs into a single compilation unit. All name lookups happen in this merged scope; there is no per-module scope chain or import-ordering sensitivity during code generation.

### 8.6.1 Resolution Phases

Symbol resolution proceeds in two phases:

**Phase 1 â€” Registration (before codegen):**

The compiler scans the merged AST and registers all top-level declarations into the following tables before emitting any code:

- `classTypes` â€” maps class names to their field and method lists.
- `structTypes` â€” maps struct names to their field lists.
- `bitfieldTypes` â€” maps bitfield names to their field lists.
- `enumTypes` â€” maps enum names to their variant maps.
- `interfaceDefs` â€” maps interface names to their method signature lists.
- `typeAliases` â€” maps alias names to their target type names.
- `templateDefs` â€” maps template names to their template AST nodes (for free-variable templates).
- `functionReturnTypes` â€” maps function names (or mangled names) to their return types and parameter lists.
- `templateFuncRegistry` â€” maps template function names to their specialization list.
- `overloadGroups` â€” maps function names to their overload sets (when multiple functions share a name).

Registration order: interfaces, aliases, templates (monomorphized), bitfields, enums, structs, classes, functions.

**Phase 2 â€” Code generation:**

During LLVM IR emission, every name reference is looked up in the registered tables. Lookups happen by normalized type name (after alias expansion) and by exact identifier string.

### 8.6.2 Free Function Resolution

Free function calls (`foo(a, b)`) are resolved as follows:

1. If the function name is in `overloadGroups`, the compiler selects the correct overload by matching the argument types against each overload's parameter types using `overloadTypeMatch`. The string type matches the address type for overload resolution purposes.
2. If no overload group exists for the name, the compiler looks up the name directly in `functionReturnTypes`.
3. If the name is found in `templateFuncRegistry`, the call is a template function call and is resolved by matching the type parameter and argument types (see Â§8.6.5).
4. If the name is not found in any table, the compiler MUST report an undefined-name error.

### 8.6.3 Method Call Resolution

Method calls (`obj.method(args)`) are resolved to LLVM function calls using the naming convention:

```
ClassName_methodName(self, args...)
```

where `ClassName` is the static type of `obj` (after alias expansion). For example:

```hopper
X86SIMD simd = X86SIMD()
int mask = simd.scanByte16(ptr, 10)
```

resolves to the LLVM call `@X86SIMD_scanByte16(%class.X86SIMD* %simd, i64 10)`.

The compiler looks up `ClassName_methodName` in `functionReturnTypes` to determine the return type and parameter types of the method. If no such entry exists, the compiler MUST report an error of the form `Unknown method 'methodName' on type 'ClassName'`.

Operators defined on a class (e.g. `operator +`, `operator []`) are resolved to:

```
ClassName_op_<safe-name>(self, arg)
```

where `<safe-name>` is the operator symbol mapped through the operator name table:

| Operator | Safe name |
|---|---|
| `+` | `plus` |
| `-` | `minus` |
| `*` | `mul` |
| `/` | `div` |
| `%` | `mod` |
| `==` | `eq` |
| `!=` | `ne` |
| `<` | `lt` |
| `<=` | `le` |
| `>` | `gt` |
| `>=` | `ge` |
| `&` | `band` |
| `\|` | `bor` |
| `^` | `bxor` |
| `<<` | `shl` |
| `>>` | `shr` |
| `[]` | `idx` |
| `[]=` | `setidx` |

### 8.6.4 Type Resolution and Alias Expansion

Type names are resolved through alias expansion. Whenever the compiler needs the LLVM type for a Hopper type string `T`:

1. If `T` is in `typeAliases`, it is replaced by `typeAliases.get(T)` recursively until a non-alias type is reached.
2. If the resulting type is in `classTypes`, the LLVM type is `%class.ClassName`.
3. If the resulting type is in `structTypes`, the LLVM type is `%struct.StructName`.
4. If the resulting type is a primitive keyword (`int`, `byte`, `char`, `bool`, `float`, `string`, `address`, `unsigned int`, `unsigned byte`), it maps to its LLVM equivalent:
   - `int` â†’ `i64` (32-bit on `armv6-bare` target)
   - `byte`, `char` â†’ `i8`
   - `bool` â†’ `i1`
   - `float` â†’ `double`
   - `string`, `address` â†’ `i8*`
   - `unsigned int` â†’ `i64` (or `i32` on `armv6-bare`)
   - `unsigned byte` â†’ `i8`
   - `string[]` â†’ `i8**`
5. Callback types (`callback(T1, T2) R`) â†’ `i8*` (opaque function pointer).
6. Template instantiation types (`List<int>`) â†’ `%class.List_int_` (mangled name).

### 8.6.5 Template Function Resolution

Template function calls (`cast<int>(x)`) are resolved by:

1. Looking up the function name (`cast`) in `templateFuncRegistry`.
2. Matching the type argument (`int`) against each registered specialization's `typeParam` field (after normalization).
3. Matching the argument types against each specialization's `paramTypes` list.
4. The first matching specialization is selected. If no specialization matches, the compiler MUST report an error.

Template functions are emitted under mangled names of the form:

```
__tmpl_<name>__<typeParam>__<paramType1>__<paramType2>...
```

For example, `cast<int>(address src)` compiles to the mangled name `__tmpl_cast__int__address`.

### 8.6.6 Enum Variant Resolution

Enum variant access (`EnumName.VariantName`) is resolved at compile time:

1. The compiler looks up `EnumName` in `enumTypes`.
2. It looks up `VariantName` in the variants map of that enum.
3. The result is an integer or string literal inlined at the use site.

Enum variants are never emitted as LLVM global variables. They are always substituted as immediate integer constants during code generation.

### 8.6.7 Interface Conformance Checking

When the compiler encounters a class declaration with an `satisfies` clause:

```hopper
class LinuxIO satisfies IO {
    ...
}
```

It verifies that:

1. The interface `IO` exists in `interfaceDefs`.
2. The class defines a method for every method declared in the interface.
3. Each method's parameter count matches the interface signature.
4. Each method's return type matches the interface signature (both `null` means void/procedure).

A violation of any of these requirements MUST be a compile error. The error message MUST name the class, the interface, and the specific method that is missing or mismatched.

### 8.6.8 Constructor and Destructor Resolution

A class constructor call `ClassName(args)` resolves to `ClassName_constructor(self, args)` where `self` is a stack-allocated `%class.ClassName` value. The compiler allocates space for the class on the stack, passes a pointer to the constructor, and binds the result to the declared variable.

A `deallocate obj` statement resolves to a call to `ClassName_destructor(obj)` if one is defined, followed by a `free` of any heap-allocated memory the destructor releases.

### 8.6.9 Chained Method Calls

Chained method calls (`obj.field.method(args)`) are resolved in two steps:

1. `obj.field` is resolved as a field access: the compiler emits a GEP into `obj`'s struct at the index of `field`, yielding a pointer to the field's type.
2. `.method(args)` is resolved as a method call on the field's type: `FieldType_method(fieldPtr, args)`.

The field type MUST be a class type for the chained call to be valid. If the field type is a primitive, the compiler MUST report an error.
