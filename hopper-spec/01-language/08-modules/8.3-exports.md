## 8.3 Exports

Hopper has no explicit export syntax. Every top-level declaration in a module is implicitly exported. There is no `export` keyword and no mechanism to selectively hide a top-level name from importing modules.

### 8.3.1 Implicit Export of All Top-Level Names

All of the following top-level declarations in a module file are visible to any module that imports it:

- Free function declarations (`function`, `extern function`)
- Template function declarations (`template function`)
- Class declarations (`class`)
- Struct declarations (`struct`)
- Bitfield declarations (`bitfield`)
- Interface declarations (`contract`)
- Enum declarations (`enum`)
- Template declarations (`template`)
- Alias declarations (`alias`)

This means that importing a module makes all of its names available in the importing file's scope. The compiler does not enforce any boundary between "public" and "private" declarations at the language level.

### 8.3.2 Intended Public API via Interface Files

The intended mechanism for declaring a library's public API is the *contract file*. A library package declares one or more interfaces in its `interfaces/` directory. The contract file lists only the function signatures that callers SHOULD use. The implementation file contains the concrete class and any helper functions that are internal to the module.

For example, the `linux` package's `IO` contract:

```hopper
// modules/linux/interfaces/IO.hop

enum StdFd {
    STDIN  = 0
    STDOUT = 1
    STDERR = 2
}

contract IO {
    function read(int fd, address buf, int count) int
    function write(int fd, address buf, int count) int
    function pread64(int fd, address buf, int count, int offset) int
    function pwrite64(int fd, address buf, int count, int offset) int
    function readv(int fd, address iov, int iovcnt) int
    function writev(int fd, address iov, int iovcnt) int
    function sendfile(int out_fd, int in_fd, address offset, int count) int
    function splice(int fd_in, address off_in, int fd_out, address off_out, int len, int flags) int
    function tee(int fd_in, int fd_out, int len, int flags) int
    function ioctl(int fd, int request, address arg) int
}
```

Callers import `IO from linux` and call only the functions declared in the `IO` contract. The implementation file (`LinuxSyscalls.hop`) may contain additional helper functions; these are technically visible but callers SHOULD NOT call them directly.

### 8.3.3 The `exports` Field in `hopper.json`

A library's `hopper.json` MAY contain an `exports` field:

```json
{
  "exports": {
    "IO": {
      "contract":      "interfaces/IO.hop",
      "implementation": "src/LinuxIO.hop"
    }
  }
}
```

This field is documentation and tooling metadata. The current build system does not enforce or consume the `exports` field during compilation. Consuming projects still declare full binding paths in their own `hopper.json` `targets` section.

### 8.3.4 Enum Declarations Inside Interfaces

An `contract` declaration MAY contain `enum` declarations as members. These enums are exported as part of the contract and are brought into scope when the contract file is imported:

```hopper
// In an contract file:
contract IO {
    // An enum nested inside an contract declaration
    enum StdFd { STDIN = 0   STDOUT = 1   STDERR = 2 }
    function write(int fd, address buf, int count) int
}
```

The enum variants are accessed as `StdFd.STDOUT` in calling code. The compiler registers contract-level enums in the global `enumTypes` map when it processes the contract declaration, making them available throughout the compilation unit.

### 8.3.5 No Re-Export

There is no mechanism to re-export a name imported from another module. If module `A` imports `function foo` from module `B`, a third module `C` that imports `A` does NOT automatically get `foo` in scope. Module `C` MUST import `B` (or the package that provides `B`) directly if it needs `foo`.

This is a consequence of the flat import model: each import explicitly names the package and contract to load. There is no transitive re-export.
