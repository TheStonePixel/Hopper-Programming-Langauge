## 8.4 Visibility

Hopper has no access modifiers. There are no `public`, `private`, `protected`, or `internal` keywords. All names at the top level of a module file are visible to any module that imports it, and all fields and methods of a class are visible to all callers. The compiler does not enforce any visibility boundary.

### 8.4.1 No Access Control Keywords

The following constructs DO NOT exist in Hopper:

- `public`, `private`, `protected`, `internal` modifiers on functions, classes, fields, or methods.
- Module-private declarations visible only within a package.
- Friend declarations.
- Sealed or final classes.

A field declared in a class is always accessible from outside the class:

```hopper
class Buffer {
    int capacity       // always visible — no private keyword
    address data

    constructor(int cap) {
        self.capacity = cap
        self.data = allocate cap
    }
}

// Calling code may access capacity directly — no enforcement prevents it:
int cap = buf.capacity
```

### 8.4.2 Encapsulation by Convention

Encapsulation in Hopper is enforced by *convention*, not by the compiler. The intended mechanism is:

1. A library declares one or more `contract` files in its `interfaces/` directory listing only the function signatures that callers SHOULD use.
2. Implementation-internal functions and helpers live in `src/` files.
3. Callers import the contract and call only the functions declared there.

Programs SHOULD only call functions declared in an contract file. Calling implementation-internal functions directly is technically possible but constitutes a violation of the library's API contract and MAY break without notice when the implementation changes.

### 8.4.3 Class Field Access

All class fields are directly readable and writable by any code that holds a reference to an instance. The class itself cannot prevent external reads or writes to its fields. Library authors SHOULD document which fields are part of the stable API and which are internal state.

### 8.4.4 Method Access

All class methods are callable by any code that holds a reference to an instance. There is no private method syntax. Internal helper methods SHOULD be named with a leading underscore by convention to communicate intent:

```hopper
class Parser {
    // Internal helper — not part of the public API by convention
    function _advance(int n) int {
        ...
    }

    // Public API method
    function parse(address src, int len) int {
        ...
    }
}
```

A conforming implementation MUST NOT treat a leading underscore as any kind of access restriction; it is a naming convention only.

### 8.4.5 Extern Function Declarations

`extern function` declarations are module-visible. They declare that a function with C ABI linkage exists and will be resolved at link time. They are treated identically to regular function declarations in terms of visibility: any code in the compilation unit may call an `extern function`.

### 8.4.6 Implementation Rationale

The absence of access control reflects Hopper's design focus on systems programming for trusted environments where the programmer is responsible for correct module layering. The contract-file mechanism provides a clear, enforceable-by-review API boundary without compiler overhead for access checking. Future versions of the language MAY introduce access modifiers; any such addition would be a backwards-incompatible grammar change.
