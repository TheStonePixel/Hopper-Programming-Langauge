## 9.4 Bare-Metal Targets

A bare-metal target is a build configuration that produces a standalone binary with no operating system. The program starts from a hardware reset vector, initializes the processor, and calls the Hopper entry point directly. Everything below the entry point — stack setup, BSS zeroing, MMIO configuration — is handled by a board-specific boot stub that the build system links automatically.

### Board Configuration

A project targets a specific board by setting the `"board"` field in `hopper.json`:

```json
{
  "name": "uart-demo",
  "type": "executable",
  "board": "qemu-virt",
  "entry": "src/main.hop"
}
```

The build system resolves the board name to a configuration that includes:

- **Compiler target** — the LLVM triple and CPU model (`aarch64-none-elf`, `cortex-a53`)
- **Load address** — where the binary is placed in memory (`0x40000000` for QEMU virt)
- **Boot stub** — assembly that runs before `entry main` (stack setup, BSS zeroing)
- **Flash method** — how the binary is delivered to the target

### Supported Boards

| Board | Architecture | Load address | Flash method |
|-------|-------------|--------------|--------------|
| `pi-zero` | ARM Cortex-A53 (AArch64) | `0x80000` | SD card (`kernel8.img`) |
| `pi-zero-2w` | ARM Cortex-A53 (AArch64) | `0x80000` | SD card (`kernel8.img`) |
| `qemu-virt` | ARM Cortex-A53 (AArch64) | `0x40000000` | QEMU emulator |
| `pico` | ARM Cortex-M0+ (Thumb-2) | `0x10000000` | UF2 (`firmware.uf2`) |

### Build and Deploy

```bash
hopper build   # cross-compile to bare-metal ELF (or raw binary)
hopper flash   # deploy: copy to SD card, mount UF2, or launch QEMU
```

For QEMU targets, `hopper flash` launches the emulator directly:

```
hopper: launching QEMU  [qemu-virt]
  qemu-system-aarch64 -M virt -cpu cortex-a53 -kernel build/uart-demo.elf -nographic
```

For physical boards, `hopper flash` copies the binary to the board's boot media.

### Boot Process

The boot stub runs before the Hopper entry point. It is board-specific assembly, invisible to the Hopper programmer, linked automatically by the build system. For AArch64 boards, the typical sequence is:

1. Park all CPU cores except core 0
2. Drop from EL2 to EL1 (if required by the board)
3. Set the initial stack pointer
4. Zero the BSS segment
5. Call `main` (the compiled `entry main` block)

After `main` returns, the boot stub enters an infinite loop. On real hardware, returning from the entry point is not meaningful — the program should spin or reset.

### QEMU Virt Machine

The QEMU `virt` machine is a generic AArch64 virtual platform with no board-specific peripheral mapping. It is the recommended target for development and testing because it requires no physical hardware.

Key peripherals on `qemu-virt`:

| Peripheral | Base address | Purpose |
|------------|-------------|---------|
| PL011 UART | `0x09000000` | Serial I/O — write a byte to DR to transmit |
| PL031 RTC  | `0x09010000` | Real-time clock |
| GIC        | `0x08000000` | Generic Interrupt Controller |

The PL011 UART is active from boot with no initialization required. Writing bytes to the data register at `0x09000000` produces immediate output on the QEMU terminal.

Exit QEMU with `Ctrl-A X`.

### No Runtime, No OS

A bare-metal Hopper program has no runtime library, no operating system, and no standard library. The only things present in the binary are:

- The boot stub (boot.s, linked by the build system)
- The compiled Hopper entry point and all functions it calls
- Any MMIO declarations (emitted as address constants in the code)

`allocate` and `deallocate` are not available without a heap allocator. On bare metal, memory management must be implemented explicitly or avoided. For register-level hardware programming, no dynamic allocation is needed — MMIO variables are address constants, and working state fits in local variables on the stack.
