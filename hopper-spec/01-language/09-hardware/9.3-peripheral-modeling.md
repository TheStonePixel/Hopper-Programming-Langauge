## 9.3 Peripheral Modeling

A hardware peripheral is a structured component: it has a fixed register layout, a defined set of operations, and a contract for how those operations interact. Hopper's type system maps directly onto this structure.

The core idea: **a bitfield describes the register, `@` places it at hardware, and functions that operate on those variables form the driver**. The result is a self-contained software component that mirrors the hardware component — same structure, same names, same contract.

### The Pattern

For any peripheral:

1. Declare a `bitfield` for each register that needs field-level access.
2. Place the registers at their hardware addresses with `@`.
3. Write functions that operate on those registers.
4. The functions become the public API of the driver.

No class required, no abstraction layer, no runtime overhead. The hardware description and the software interface are the same thing.

### Example — PL011 UART Driver

The ARM PL011 UART is the UART controller used on QEMU's `virt` machine (base address `0x09000000`) and on ARM development boards. Its register map includes:

| Offset | Register | Purpose |
|--------|----------|---------|
| 0x000  | DR       | Data register — TX byte (write), RX byte (read) |
| 0x018  | FR       | Flag register — FIFO status bits |

A complete, functional UART driver in Hopper:

```hopper
// Flag register — named bits from the PL011 datasheet
bitfield PL011FR {
    bit cts     // bit 0: clear to send
    bit dsr     // bit 1: data set ready
    bit dcd     // bit 2: data carrier detect
    bit busy    // bit 3: UART busy
    bit rxfe    // bit 4: receive FIFO empty
    bit txff    // bit 5: transmit FIFO full
    bit rxff    // bit 6: receive FIFO full
    bit txfe    // bit 7: transmit FIFO empty
    pad 24      // bits 8–31 reserved
}

// Place registers at their physical addresses
int     uart_dr @ 0x09000000   // data register
PL011FR uart_fr @ 0x09000018   // flag register

// Send one byte — poll TXFF so we never overrun the FIFO
function putc(int c) {
    while (uart_fr.txff == 1) {}
    uart_dr = c
}

// Receive one byte — poll RXFE so we only read when data has arrived
function getc() int {
    while (uart_fr.rxfe == 1) {}
    return uart_dr
}
```

This is the entire UART driver. The datasheet's register map is the type declaration. The datasheet's bit names are the field names. The polling behavior described in the datasheet's programming model is the function body.

### Comparing to C

The equivalent C requires explicit casting, volatile qualifiers on every declaration, and manual shift-mask expressions at every field access:

```c
#define UART_BASE 0x09000000UL
#define UART_DR   (*(volatile uint32_t *)(UART_BASE + 0x000))
#define UART_FR   (*(volatile uint32_t *)(UART_BASE + 0x018))
#define UART_FR_TXFF (1 << 5)
#define UART_FR_RXFE (1 << 4)

void putc(int c) {
    while (UART_FR & UART_FR_TXFF) {}
    UART_DR = c;
}
```

In Hopper, `uart_fr.txff` replaces `(UART_FR >> 5) & 1`. The volatile guarantee comes from the `@` declaration, not from a per-variable qualifier. The type system enforces that `txff` is a valid field of the flag register — there is no equivalent check in C.

### Modeling a Peripheral as a Class

When a peripheral has state beyond its registers — initialization state, buffer management, error tracking — it can be wrapped in a class. The MMIO declarations remain at the top level (they are global hardware facts), and the class provides the higher-level API:

```hopper
bitfield PL011FR {
    bit cts
    bit dsr
    bit dcd
    bit busy
    bit rxfe    // receive FIFO empty
    bit txff    // transmit FIFO full
    bit rxff
    bit txfe
    pad 24
}

int     uart_dr @ 0x09000000
PL011FR uart_fr @ 0x09000018

class UART {
    int base_address
    bool initialized

    constructor(int base) {
        self.base_address = base
        self.initialized  = false
    }

    function init() {
        // Configure baud rate, line control, etc.
        // (board-specific register writes)
        self.initialized = true
    }

    function putc(int c) {
        while (uart_fr.txff == 1) {}
        uart_dr = c
    }

    function getc() int {
        while (uart_fr.rxfe == 1) {}
        return uart_dr
    }

    function puts(int a, int b, int c, int d) {
        self.putc(a)
        self.putc(b)
        self.putc(c)
        self.putc(d)
    }
}
```

The class models the peripheral as a software component with its own lifecycle. The MMIO declarations are still module-level — they are a hardware truth, not an instance variable. Multiple `UART` instances could in principle address different UARTs by holding different base addresses, using those to compute the correct MMIO addresses.

### The Hardware Component Model

Hopper's approach to hardware modeling has three layers:

| Layer | Mechanism | Purpose |
|-------|-----------|---------|
| Register layout | `bitfield` | Describe bit-level structure of a hardware register |
| Register placement | `@` | Bind that type to a physical address with volatile semantics |
| Driver API | Functions / class | Expose hardware behavior as ordinary Hopper functions |

Each layer has a single responsibility. The bitfield knows the layout. The `@` declaration knows the address. The functions know the behavior. A programmer reading the driver can check each claim against the datasheet independently.

### Peripheral Modeling Rules

- MMIO declarations (`@`) are facts about the hardware, not instance state. They MUST appear at the top level and are shared across all code in the module.
- A `bitfield` type used with `@` carries volatile semantics for all field accesses. The same `bitfield` type used on the stack or heap does not.
- Functions operating on MMIO variables are stateless with respect to the hardware — the hardware state lives in the registers, not in the function. This matches the hardware reality.
- A class wrapping an MMIO peripheral SHOULD NOT redeclare register addresses as fields. The `@` declarations at module level serve that purpose. Class fields SHOULD hold derived or bookkeeping state.
- A peripheral driver SHOULD name its types and functions to match the peripheral's datasheet. This makes cross-referencing between code and hardware documentation natural.
