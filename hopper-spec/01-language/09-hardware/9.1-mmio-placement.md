## 9.1 MMIO Placement (`@`)

The `@` sigil binds a named variable to a fixed physical address. Every read or write of that variable becomes a volatile memory operation at that address — no pointer cast, no intermediate variable, no risk of the optimizer eliminating the access.

### Syntax

```antlr
mmioDecl : type Identifier '@' HexLiteral ;
```

`type` is any Hopper type. `Identifier` is the name used in source code. `HexLiteral` is the physical address of the register.

```hopper
int     uart_dr @ 0x09000000   // PL011 data register
int     uart_fr @ 0x09000018   // PL011 flag register
```

### Semantics

An MMIO declaration does not allocate stack or heap memory. The identifier is a compile-time alias for a specific address. Every access through that identifier is emitted as a `volatile` load or store directly to the declared address.

**Reads** compile to a volatile load:

```hopper
int status = uart_fr   // volatile load from 0x09000018
```

**Writes** compile to a volatile store:

```hopper
uart_dr = 65           // volatile store to 0x09000000
```

The `volatile` qualifier prevents the compiler from:
- Caching the value in a register across accesses
- Eliminating reads or writes it judges to have no effect
- Reordering accesses relative to each other

These guarantees are essential for hardware registers, where every read may have a side effect (clearing a status bit, advancing a FIFO) and every write may trigger a hardware action.

### LLVM Representation

An MMIO declaration for address `A` and type `T` compiles to a pointer constant in LLVM IR:

```llvm
; int uart_dr @ 0x09000000
; Read:
%val = load volatile i64, i64* inttoptr (i64 150994944 to i64*), align 4
; Write:
store volatile i64 65, i64* inttoptr (i64 150994944 to i64*), align 4
```

The address literal is emitted as `inttoptr` from the hex constant. No global variable, no symbol, no relocation — the address is a compile-time constant embedded in every load and store instruction.

### Rules

- The address MUST be a hexadecimal literal (`0x...`). Symbolic or computed addresses are not permitted in an MMIO declaration.
- MMIO declarations MUST appear at the top level of a source file. They are not valid inside function bodies, class bodies, or struct bodies.
- Any type valid in Hopper may be used with `@`. When the type is a `bitfield`, field-level access emits volatile read-modify-write sequences (see §9.2).
- An MMIO variable has no stack representation. Taking the address of an MMIO variable is a compile-time **MMIO safety violation** (see §9.1.1).
- Multiple MMIO declarations MAY share the same address. This is used when different views of the same register are useful (e.g., a scalar `int` for raw access and a `bitfield` for field-level access).

### 9.1.1 MMIO Safety

MMIO variables are hardware facts, not addressable memory locations. The compiler enforces this with a dedicated error category.

**Prohibited: taking the address of an MMIO variable**

```hopper
int uart_dr @ 0x09000000

entry main {
    address ptr = uart_dr::address   // ERROR
}
```

```
│ Error: MMIO safety violation
│        cannot take the address of MMIO variable 'uart_dr'
│ Hint: 'uart_dr' is a hardware register bound to 0x09000000
│   via @. MMIO variables have no stack representation —
│   read and write 'uart_dr' directly.
```

Obtaining a raw pointer to an MMIO variable would allow pointer arithmetic, bypassing the named, typed interface that `@` provides. The only operations permitted on an MMIO variable are direct reads and writes.

**Why this matters.** In C, a common pattern is:

```c
volatile uint32_t *reg = (volatile uint32_t *)0x09000000;
reg++;  // now pointing at 0x09000004 — silent, legal, dangerous
```

Hopper prevents this class of error entirely. An MMIO binding is a fixed name for a fixed address. There is no pointer to increment.

### Contrast With Other Approaches

| Approach | Hardware access |
|---|---|
| C `volatile uint32_t *` | Manual cast, easy to write wrong address |
| C `#define UART_DR *((volatile uint32_t *)0x09000000)` | Macro, no type safety |
| Hopper `int uart_dr @ 0x09000000` | Named, typed, volatile by construction |

The `@` declaration is not syntax sugar over a cast — it is a distinct declaration form whose volatile semantics are guaranteed by the compiler regardless of optimization level.
