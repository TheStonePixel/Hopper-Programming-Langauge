## 9.2 Register Types (`bitfield`)

Hardware registers are not flat integers. A 32-bit control register may contain a 1-bit enable flag at bit 13, a 2-bit mode field at bits 4–5, and 19 bits of reserved padding. Reading and writing individual fields by hand requires shifting and masking on every access — error-prone and opaque.

Hopper's `bitfield` declaration captures the register layout at the type level. The compiler generates the correct shifts and masks automatically. Field names come from the datasheet, not from comments in the code.

### Syntax

```antlr
bitfieldDecl
    : 'bitfield' Identifier '{' NEWLINE*
        (bitfieldMember (NEWLINE+ bitfieldMember)* NEWLINE*)?
      '}'
    ;

bitfieldMember
    : 'bit' Identifier '[' IntegerLiteral ']'   // multi-bit field
    | 'bit' Identifier                           // single-bit field
    | 'pad' IntegerLiteral                       // unnamed padding bits
    ;
```

Fields are packed from the **least significant bit (LSB)** in declaration order. `bit fieldName` occupies one bit. `bit fieldName[N]` occupies N bits. `pad N` inserts N unnamed bits of padding.

### Example — PL011 UART Flag Register

The ARM PL011 UART flag register (FR, offset 0x18) has this layout per the datasheet:

| Bits | Name | Description |
|------|------|-------------|
| 0    | CTS  | Clear to send |
| 1    | DSR  | Data set ready |
| 2    | DCD  | Data carrier detect |
| 3    | BUSY | UART busy |
| 4    | RXFE | Receive FIFO empty |
| 5    | TXFF | Transmit FIFO full |
| 6    | RXFF | Receive FIFO full |
| 7    | TXFE | Transmit FIFO empty |
| 8–31 | —    | Reserved |

In Hopper:

```hopper
bitfield PL011FR {
    bit cts     // bit 0: clear to send
    bit dsr     // bit 1: data set ready
    bit dcd     // bit 2: data carrier detect
    bit busy    // bit 3: UART busy
    bit rxfe    // bit 4: receive FIFO empty
    bit txff    // bit 5: transmit FIFO full
    bit rxff    // bit 6: receive FIFO full
    bit txfe    // bit 7: transmit FIFO empty
    pad 24      // bits 8–31 reserved
}
```

The type mirrors the datasheet exactly. Every bit has a name. Padding matches the reserved range.

### MMIO Placement of a Bitfield

Combine `bitfield` with `@` to place the typed register at its hardware address:

```hopper
PL011FR uart_fr @ 0x09000018
```

Field reads and writes now go through the bitfield's named fields:

```hopper
// Read TXFF before writing — never overrun the transmit FIFO
while (uart_fr.txff == 1) {}

// Read RXFE before reading — only read when data has arrived
while (uart_fr.rxfe == 1) {}
```

No manual bit shifting. The register name and field name appear directly in the code.

### Generated Code for Field Access

**Field read** — `uart_fr.txff`:

1. Volatile load of the full register from 0x09000018
2. Right-shift by the field's bit offset (5)
3. Mask with `1` (single-bit field)

```llvm
%reg = load volatile i32, i32* inttoptr (i64 150994968 to i32*), align 4
%shifted = lshr i32 %reg, 5
%field = and i32 %shifted, 1
```

**Field write** — `uart_cr.ue = 1`:

1. Volatile load of the full register (read-modify-write)
2. Clear the field's bits with an AND mask
3. OR the new value into the cleared bits
4. Volatile store back to the hardware address

```llvm
%old = load volatile i32, i32* inttoptr (i64 150994992 to i32*), align 4
%cleared = and i32 %old, 4294959103     ; mask clears bit 13
%shifted = shl i32 1, 13
%new = or i32 %cleared, %shifted
store volatile i32 %new, i32* inttoptr (i64 150994992 to i32*), align 4
```

The volatile qualifier appears on both the initial load and the final store. This guarantees the hardware sees the full read-modify-write even under optimization.

### Rules

- `bitfield` declarations MUST appear at the top level of a source file.
- Bits are packed from LSB to MSB in declaration order. The first declared field occupies the lowest-numbered bits.
- `pad N` inserts N bits of unnamed padding. Padding bits are preserved in write operations (cleared to 0) and ignored in read operations.
- The container type for the bitfield is the smallest integer type that holds all declared bits: `i8` (≤8 bits), `i16` (≤16 bits), `i32` (≤32 bits), or `i64` (≤64 bits).
- A bitfield used without `@` is a valid aggregate type like any struct — it can be allocated on the stack, passed to functions, and returned by value. Only bitfield variables declared with `@` emit volatile MMIO accesses.
- A conforming implementation MUST NOT reorder or merge volatile accesses to bitfield fields, even when multiple fields of the same register are accessed in sequence.
