## Return Semantics

### 6.3.1 The `return` Statement

The `return` statement exits the current function and transfers control to the call site. It takes one of two forms:

```hopper
return expression      // valued return — for functions with a declared return type
return                 // bare return — for procedures (no return type)
```

A `return` statement MUST appear inside a function or procedure body. It is a compile-time error to use `return` at the top level or inside an `entry` block.

### 6.3.2 Type Agreement

The expression in a valued `return` MUST have a type that is compatible with the function's declared return type. Type compatibility is strict: the expression type MUST match the declared return type exactly. Implicit coercion does not occur. If the types do not match, the program MUST NOT compile.

```hopper
function clamp(int x, int lo, int hi) int {
    if (x < lo) {
        return lo       // int — compatible
    }
    if (x > hi) {
        return hi       // int — compatible
    }
    return x            // int — compatible
}
```

A function declared to return `float` MUST return a `float` expression; returning an `int` literal is a type error. Use an explicit cast:

```hopper
function toFloat(int x) float {
    return cast<float>(x)
}
```

### 6.3.3 Bare `return` in Procedures

A procedure is a function with no declared return type. A procedure MAY contain a bare `return` statement to exit early:

```hopper
function log(int level, string msg) {
    if (level < MIN_LEVEL) {
        return            // early exit
    }
    write(2, msg, len)
}
```

A procedure that reaches its closing brace without executing a `return` exits normally at that point. A bare `return` MUST NOT carry an expression; the form `return expr` in a procedure is a compile-time error.

### 6.3.4 All Paths Must Return

A function with a declared return type MUST return on all code paths. Falling off the end of a non-void function — reaching the closing brace without executing a `return expr` — is undefined behavior (see §6.9). The compiler SHOULD detect this condition and emit a compile-time error or warning. A conforming implementation MUST NOT silently accept a non-void function with a code path that does not terminate in `return`.

The following code paths MUST each contain a `return` expression of the correct type:
- The normal (non-branching) path through the function body.
- Each branch of every `if`/`else` that is not guaranteed to `return`.
- Any path that exits a `while` or `for` loop.

```hopper
// CORRECT — all paths return
function abs(int x) int {
    if (x < 0) {
        return -x
    } else {
        return x
    }
}

// INCORRECT — falls off the end if the if-branch is not taken (compile error)
function bad(int x) int {
    if (x < 0) {
        return -x
    }
    // missing return — undefined behavior if x >= 0
}
```

### 6.3.5 Return in Entry Blocks

The `entry` block does not use `return`. An `entry` block is not a function; it is the program's start point. To exit the program, the program MUST call an appropriate exit syscall (e.g., `exit(0)`) or fall off the end of the `entry` block. The exit status in the latter case is implementation-defined.

### 6.3.6 Return Value Lifetime

The expression in a `return` statement is evaluated in the callee's frame. Primitive return values are copied into the return register(s) before the frame is torn down. Class return values are written into the caller-allocated slot before the frame is torn down.

A function MUST NOT return a pointer to a local variable. The local variable's storage is released when the function returns. Any pointer to a stack-allocated local that escapes through a `return` produces a dangling pointer, which is undefined behavior.

```hopper
// INCORRECT — returns pointer to stack storage (undefined behavior)
function dangling() address {
    int local = 42
    return local::address     // local is destroyed on return — dangling pointer
}
```

### 6.3.7 Multiple Return Points

Functions MAY have multiple `return` statements. All `return` statements in a non-void function MUST carry an expression of the correct type. Early returns are idiomatic for error handling:

```hopper
function openAndRead(string path, address buf, int count) int {
    int fd = open(path, 0)
    if (fd < 0) {
        return fd             // early return: negative error code
    }
    int n = read(fd, buf, count)
    close(fd)
    return n
}
```
