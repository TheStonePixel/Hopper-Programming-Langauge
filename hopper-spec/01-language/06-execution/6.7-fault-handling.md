## Fault Handling

### 6.7.1 Hardware Faults Are OS-Level Events

Hardware faults — memory access violations (segmentation faults), bus errors, divide-by-zero traps, illegal instruction traps, and alignment faults — are handled by the operating system or hardware, not by the Hopper language or runtime.

When a hardware fault occurs, the processor delivers a trap or interrupt to the kernel. On POSIX systems, the kernel converts the hardware event into a signal delivered to the process (`SIGSEGV`, `SIGBUS`, `SIGFPE`, `SIGILL`). The process's default signal disposition for these signals is termination with a core dump. Hopper provides no language-level interception of this sequence.

There is no Hopper equivalent of C++'s structured exception handling, Java's `NullPointerException`, or Rust's `panic!` with a backtrace. A null pointer dereference terminates the process.

### 6.7.2 Fault Handling on Hosted Targets

On hosted targets (Linux, other POSIX), programs MAY install signal handlers using the `Process` contract to intercept hardware-fault signals:

```hopper
import Process from linux

// install a SIGSEGV handler that logs a message before terminating
sigaction(11, handler::address, null)
```

A signal handler installed for `SIGSEGV`, `SIGBUS`, or `SIGFPE` MUST be written with extreme care. The program is in an undefined state when the signal is delivered. Only async-signal-safe operations (as defined by POSIX) are permitted inside a signal handler. The signal handler SHOULD terminate the process (e.g., by calling `_exit()`) rather than returning to the faulting instruction, unless the handler has corrected the fault condition at the memory level (e.g., a custom page fault handler using `mprotect`).

### 6.7.3 Fault Handling on Bare-Metal Targets

On bare-metal targets, there is no operating system to deliver signals. Hardware faults deliver control to the processor's fault vector (e.g., the ARM Cortex-M HardFault handler, the x86 IDT entries for `#PF`, `#GP`, `#DF`).

Hopper programs targeting bare metal MUST set up fault handlers explicitly using the `bind` declaration, which places a function pointer at a hardware-defined vector address:

```hopper
bind 0x00000008 = hardFault::address    // ARM Cortex-M HardFault vector
```

The function at the bound address MUST conform to the hardware's expectations for the fault vector (calling convention, register state, return mechanism). This is platform-specific and documented in the target's hardware reference manual, not in this specification.

A bare-metal program that does not install fault handlers will reset, hang, or exhibit undefined behavior when a fault occurs, depending on the processor's reset state for that vector.

### 6.7.4 Divide by Zero

Integer division by zero is undefined behavior (see §6.9). On x86-64, it delivers a `#DE` (divide exception) to the processor, which results in `SIGFPE` on Linux. On ARM, the result of integer division by zero is implementation-defined by the hardware (many ARM processors return zero rather than faulting).

Programs MUST NOT divide by integer zero. Use a precondition or an explicit check:

```hopper
function safeDivide(int a, int b) int
requires b != 0 {
    return a / b
}
```

Floating-point division by zero follows IEEE 754: dividing a non-zero float by zero produces positive or negative infinity (not a fault). Dividing zero by zero produces NaN. These are not faults; they are defined IEEE 754 results.

### 6.7.5 Stack Overflow

Stack overflow — exhausting the thread stack by deep recursion or large stack allocations — is not detected by the Hopper language. On Linux, the kernel may deliver `SIGSEGV` when the stack guard page is touched, but this is not guaranteed for all stack overflow patterns. On bare-metal targets, stack overflow typically corrupts adjacent memory silently.

Stack overflow is undefined behavior. Programs MUST bound their recursion depth or use iterative algorithms for deep computations (see §6.4).

### 6.7.6 No Structured Fault Handling

Hopper does not provide and will not provide a structured fault handling mechanism (no `try-fault`, no `__try/__except`, no segfault-catching mechanism at the language level). Programs that require fault resilience MUST implement it at the system level through signal handlers or hardware fault vectors, using the tools provided by the `Process` contract and the `bind` declaration.

This is consistent with the zero-runtime principle: structured fault handling requires a runtime unwinder and a persistent fault context — infrastructure that is not available on all targets.
