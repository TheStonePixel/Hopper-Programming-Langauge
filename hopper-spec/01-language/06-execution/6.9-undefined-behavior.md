## Undefined Behavior

### 6.9.1 Definition

Undefined behavior (UB) is a condition that this specification explicitly does not define. When undefined behavior occurs, a conforming Hopper implementation MAY do anything: produce incorrect output, terminate the program, corrupt memory, delete files, or behave in any other manner. The implementation is not required to diagnose, report, or produce a consistent result for undefined behavior.

A conforming implementation MAY assume that undefined behavior never occurs and use that assumption as the basis for optimizations. This means that a program that contains undefined behavior MAY produce results that appear correct in unoptimized builds and silently fail in optimized builds, because the optimizer has eliminated code it "knows" cannot execute.

Programs that trigger undefined behavior are not conforming Hopper programs.

### 6.9.2 Complete List of Undefined Behaviors

The following is the complete normative list of undefined behaviors in Hopper.

---

#### UB1 — Null Pointer Dereference

Reading or writing through a pointer that holds the value `null` (address zero) is undefined behavior.

```hopper
address ptr = null
byte b = ptr::value      // UB: dereferencing null
```

The compiler MAY assume that any pointer being dereferenced is non-null. Programs that dereference potentially-null pointers MUST check for null first.

---

#### UB2 — Out-of-Bounds Memory Access

Reading or writing a memory location outside the bounds of the allocated object or array is undefined behavior.

```hopper
address buf = allocate 16
int x = (buf + 20)::value   // UB: 20 > 16, out of bounds
```

```hopper
int arr[4] = [1, 2, 3, 4]
int x = arr[4]              // UB: valid indices are 0–3
```

Hopper performs no bounds checking. The programmer is responsible for ensuring that all memory accesses are within bounds.

---

#### UB3 — Use-After-Free

Reading or writing through a pointer to memory that has been deallocated with `deallocate` is undefined behavior.

```hopper
address buf = allocate 64
deallocate buf
int x = buf::value          // UB: buf has been freed
```

After `deallocate ptr`, the pointer `ptr` MUST NOT be read from, written to, or passed to another `deallocate` call.

---

#### UB4 — Double-Free

Calling `deallocate` on a pointer that has already been freed is undefined behavior.

```hopper
address buf = allocate 64
deallocate buf
deallocate buf              // UB: double-free
```

Calling `deallocate` on a pointer that was not returned by `allocate` (e.g., a pointer into the middle of an allocated block, a stack pointer, a null pointer) is undefined behavior.

---

#### UB5 — Reading Uninitialized Memory

Reading the value of a variable that has been declared but not assigned is undefined behavior.

```hopper
int x           // declared, not initialized
int y = x + 1  // UB: x is uninitialized
```

A variable declared with `type Identifier` (no `= expression`) has an unspecified initial value. The compiler MAY produce any value when such a variable is read. The compiler MAY assume the variable will be initialized before it is read and optimize accordingly.

---

#### UB6 — Signed Integer Overflow

Arithmetic overflow of a signed integer type (`int`, `byte`, `char`) is undefined behavior.

```hopper
int x = 9223372036854775807   // INT64_MAX
int y = x + 1                 // UB: signed overflow
```

The compiler MAY assume that signed integer arithmetic does not overflow. Programs that rely on two's-complement wraparound of signed integers are not conforming. Use `unsigned int` if wraparound semantics are required.

Unsigned integer overflow is defined: it wraps modulo 2^N where N is the bit width of the type.

---

#### UB7 — Invalid Shift Amount

Shifting an integer by a negative amount or by an amount greater than or equal to the bit width of the type is undefined behavior.

```hopper
int x = 1
int y = x << 64     // UB: shift >= 64 (bit width of int)
int z = x << -1     // UB: negative shift
```

Valid shift amounts for `int` (64-bit) are 0–63. Valid shift amounts for `byte` (8-bit) are 0–7.

---

#### UB8 — Falling Off a Non-Void Function

If execution reaches the closing brace of a function that has a declared return type without executing a `return expr` statement, the behavior is undefined.

```hopper
function broken(int x) int {
    if (x > 0) {
        return x
    }
    // UB if x <= 0: no return on this path
}
```

The compiler SHOULD reject this pattern at compile time. If the compiler does not detect the missing return, executing the function on the missing-return path is undefined behavior.

---

#### UB9 — Invalid Function Pointer Call

Calling a function through a `callback` pointer that does not point to a valid, compatible function is undefined behavior.

```hopper
callback(int) int fn = null
int x = fn(42)              // UB: null function pointer
```

Causes of invalid function pointer calls include:
- The pointer is null.
- The pointer was never assigned.
- The pointer points to a function whose actual signature differs from the `callback` type (type mismatch between caller and callee).
- The pointer points to freed or uninitialized memory.

---

#### UB10 — Writing to a String Literal

String literals (e.g., `"hello"`) are stored in read-only memory. Writing through a pointer to a string literal is undefined behavior.

```hopper
string msg = "hello"
msg::value = 'H'            // UB: modifying a string literal
```

On hosted targets with virtual memory, writing to a string literal typically causes a segmentation fault because the literal is in a read-only segment. On bare-metal targets, behavior is entirely implementation-defined.

To work with mutable byte buffers, allocate memory explicitly and copy the bytes in:

```hopper
address buf = allocate 6
// copy string contents into buf before writing
```

---

#### UB11 — Data Races

When a Hopper program uses OS threads and two threads access the same memory location concurrently, where at least one access is a write, and the accesses are not synchronized, the behavior is undefined.

```hopper
// Thread A:
counter = counter + 1      // UB if Thread B may also be reading or writing counter

// Thread B:
int x = counter            // UB if Thread A is concurrently writing counter
```

Data races corrupt the values of shared variables and MAY corrupt surrounding memory depending on the access size and alignment. The compiler MAY assume that data races do not occur and reorder or eliminate memory operations accordingly.

Synchronization MUST be performed using kernel-mediated mechanisms (`futex`, `pthread_mutex`, memory barriers via inline assembly) when shared state is accessed from multiple threads.

---

### 6.9.3 Undefined Behavior Is Not a Runtime Error

Hopper does not provide runtime checks for undefined behavior in release builds. Undefined behavior does not throw an exception, signal an error, or produce a defined trap in release mode. It silently corrupts program state, produces wrong results, or causes a crash — often far from the source of the problem.

In non-release builds, the compiler SHOULD insert checks for a subset of undefined behaviors (null pointer dereferences, out-of-bounds accesses where the bounds are statically known, violated contract clauses) and call `abort()` on violation. This is a debugging aid, not a specification guarantee.

### 6.9.4 Sanitizers

Implementations MAY provide instrumentation modes (analogous to AddressSanitizer, UBSanitizer) that detect undefined behavior at runtime with a defined error. When such instrumentation is active, the implementation SHOULD report the specific undefined behavior and terminate the process. The exact behavior of such instrumentation is implementation-defined.

### 6.9.5 Implementation Latitude

When undefined behavior occurs, the implementation is permitted to:
- Produce the "expected" result on one platform and silently fail on another.
- Produce a correct result in unoptimized builds and an incorrect result in optimized builds.
- Corrupt memory adjacent to the operation that triggered the UB.
- Terminate the process.
- Do nothing observable (the UB is eliminated by the optimizer).

No behavior produced by a program with undefined behavior constitutes a conformance failure of the implementation. The non-conforming program is the one that triggered the UB.
