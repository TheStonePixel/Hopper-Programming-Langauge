## Function Calls

### 6.2.1 Call Syntax

A free function is called by name followed by a parenthesized argument list:

```hopper
int n = read(fd, buf, count)
write(1, msg, len)
```

A method is called with an explicit receiver:

```hopper
int mask = simd.scanByte16(ptr, 10)
obj.reset()
```

There is no implicit receiver. Every method call MUST name the object on which the method is dispatched. Unqualified calls (without a `.`) always resolve to free functions in scope, never to methods of the current class.

Template functions are called with an explicit type argument:

```hopper
int x = cast<int>(3.9)
```

### 6.2.2 Argument Evaluation Order

Arguments in a call expression are evaluated left-to-right before the call is made. Given `f(a, b, c)`, the evaluation order is `a`, then `b`, then `c`, then the call to `f`. This order is normative. A conforming implementation MUST evaluate arguments in left-to-right order.

If any argument expression has a side effect (a function call, a write through a pointer), those side effects occur in left-to-right order. Programs MUST NOT rely on any other evaluation order.

### 6.2.3 Argument Passing — Primitive Types

Primitive types (`int`, `byte`, `char`, `bool`, `float`, `address`, `string`, `unsigned int`, `unsigned byte`) are passed by value. The callee receives a copy. Modifications to a parameter inside a callee have no effect on the variable at the call site.

On x86-64 with the System V AMD64 ABI:
- Integer and pointer arguments (`int`, `byte`, `bool`, `address`, `string`) are passed in registers `rdi`, `rsi`, `rdx`, `rcx`, `r8`, `r9` for the first six arguments, then on the stack.
- Floating-point arguments (`float`) are passed in `xmm0`–`xmm7` for the first eight, then on the stack.

This ABI behavior is implementation-defined for non-x86-64 targets.

### 6.2.4 Argument Passing — Class Types

Class instances are not passed by copy in the source language. When a function parameter has a class type, the compiler passes a pointer to the caller's instance. The caller MUST ensure the instance remains live for the duration of the call.

```hopper
function process(MyClass obj) int {
    return obj.value
}
```

In the generated LLVM IR, `obj` becomes a pointer parameter (`%MyClass*`). The source-level illusion of pass-by-value for class types is not enforced — the callee receives a pointer to the caller's object and MAY observe mutations made through that pointer. Programs SHOULD NOT mutate class parameters unless they explicitly intend to.

### 6.2.5 Return Values — Primitive Types

Primitive return values are returned by value. On x86-64 with the System V AMD64 ABI:
- Integer and pointer types are returned in `rax`.
- Floating-point types are returned in `xmm0`.

A function's return value is an expression; it may be used directly in a larger expression:

```hopper
int total = read(fd, buf, 256) + read(fd, buf2, 256)
```

Note: this expression has defined left-to-right evaluation order per §6.2.2.

### 6.2.6 Return Values — Class Types

When a function declares a class return type, the LLVM backend uses the standard aggregate return convention. The caller allocates a slot on its stack for the returned object, and the callee writes into that slot. The mechanics are implementation-defined and depend on the target ABI.

From the source language perspective, a function returning a class value produces a class instance that may be bound to a local variable:

```hopper
MyClass instance = makeInstance(42)
```

The returned class instance occupies a stack slot in the caller's frame. No heap allocation occurs implicitly.

### 6.2.7 Variadic Functions

Hopper does not support variadic user-defined functions. Only `extern` function declarations MAY include a trailing `...` to declare a C-compatible variadic function:

```hopper
extern function printf(string fmt, ...) int
```

Variadic `extern` functions are called with the standard C variadic ABI (promotion rules apply per the C standard: `float` is promoted to `double`, integer types narrower than `int` are promoted to `int`). The compiler MUST pass arguments to a variadic call using the appropriate ABI-level promotion.

The number and types of arguments passed after the fixed parameters are not checked by the Hopper compiler. Passing arguments of incorrect type or count to a variadic extern function is undefined behavior.

### 6.2.8 Function Pointer Calls

A function pointer (declared with type `callback(T...) R`) is called using the same syntax as a regular function:

```hopper
callback(int, int) int op = add
int result = op(3, 4)
```

The types of the arguments MUST match the declared callback signature. Calling a function pointer that does not point to a valid function is undefined behavior (see §6.9).

### 6.2.9 Recursion

Hopper supports recursive function calls. There is no language-level limit on recursion depth; the practical limit is determined by the platform's stack size. Deep recursion MAY cause stack overflow, which is undefined behavior. See §6.4 for a discussion of tail call optimization.

### 6.2.10 Overloading

Free functions with the same name but different parameter types ARE supported by the current compiler. The compiler (see `codegenLLVM.js`) detects same-named free functions, builds an `overloadGroups` map, and resolves calls by matching argument types at the call site. Each overloaded function is emitted under a mangled LLVM name; the call site selects the best match based on parameter type compatibility.

Template functions with the same name but different type parameters are also distinct and are dispatched via the template specialization mechanism (see §6.2.1).

Note: method names within a single class MUST be unique. Class methods do not participate in the free-function overload resolution mechanism.
