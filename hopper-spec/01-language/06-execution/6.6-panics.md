## Panics

### 6.6.1 No Panic Mechanism

Hopper has no `panic` keyword, no `panic!` macro, no runtime panic handler, and no stack unwinding on failure. There is no mechanism by which the language runtime raises a structured error that unwinds the stack.

This is consistent with the zero-runtime design: a panic handler requires a runtime thread context, stack unwinding tables, and a known execution environment — none of which are available on bare-metal targets.

### 6.6.2 Compiler-Generated `abort()` in Non-Release Builds

In non-release builds, the compiler SHOULD emit a call to `abort()` for the following conditions when they can be detected at runtime:

- A violated `invariant` clause in a `while` loop (see §6.1.3.1).
- A violated `requires` precondition or `ensures` postcondition on a function (see contract clauses in the grammar).
- A violated `constrain` clause on a variable declaration (see the `constrainClause` production in the grammar).

When such a check fails, the compiler-emitted code calls `abort()`. `abort()` sends `SIGABRT` to the process (on POSIX systems), which terminates the process immediately and MAY produce a core dump. There is no error message, no stack trace, and no cleanup of resources.

```hopper
function divide(int a, int b) int
requires b != 0 {
    return a / b
}
```

In a non-release build, calling `divide(10, 0)` violates the `requires b != 0` precondition. The compiler inserts a runtime check:

```
if b == 0: abort()
```

### 6.6.3 Release Mode

In release mode (compilation with optimization enabled), assertion checks MUST be omitted. The compiler MAY use the assertions as optimization assumptions — a violated `requires b != 0` clause in release mode means the compiler is permitted to assume `b != 0` and optimize the division accordingly. Violating an assertion in release mode is undefined behavior.

Programs MUST NOT rely on assertion checks executing in release builds. Assertions are documentation and debug aids, not substitute error handling.

### 6.6.4 No Stack Unwinding

When `abort()` is called, the process terminates immediately. No destructors run. No `defer`red calls execute. No buffers are flushed. The OS reclaims all resources (file descriptors, memory, etc.) at process exit.

This is intentional. Stack unwinding requires a separate runtime mechanism (e.g., libunwind, DWARF tables) that is incompatible with the zero-runtime goal. Programs that require cleanup on abnormal exit MUST set up signal handlers through the `Process` contract and handle `SIGABRT` explicitly.

### 6.6.5 `abort()` Is an External Function

`abort()` is not a built-in language feature. When the compiler emits an abort call, it generates a call to the C standard library `abort` function (or a syscall equivalent on bare-metal targets). Programs that use contract clauses implicitly depend on this symbol being available at link time in non-release builds.

Programs MAY declare and call `abort()` directly when they detect a non-recoverable condition:

```hopper
extern function abort()

function invariantViolated() {
    abort()
}
```

### 6.6.6 Assertions vs. Error Handling

The distinction between assertions and error handling is important:

- **Assertions** (`requires`, `ensures`, `invariant`, `constrain`) express conditions that MUST be true if the program is correct. A violation is a programming error, not a runtime condition to recover from.
- **Error handling** (return codes, null checks, §6.5) expresses conditions that are part of normal operation: files that do not exist, network connections that drop, allocations that fail. These are not programming errors; they are expected input conditions.

Programs MUST NOT use assertions as a substitute for error handling. Checking `requires fd >= 0` is correct for an assertion that the caller passed a valid file descriptor. It is NOT a substitute for checking the return value of `open()`.
