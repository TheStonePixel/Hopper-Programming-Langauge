## Errors

### 6.5.1 No Exception Mechanism

Hopper has no exception mechanism. There is no `throw`, no `try`, no `catch`, no `finally`, and no exception propagation. Functions cannot signal errors by raising an exception that unwinds the stack to a distant handler. This is a permanent design decision (see Anti-Goal AG4 in §0.4).

The rationale: exception handling requires a runtime unwinder, DWARF unwind tables embedded in every binary, and nondeterministic control flow that is not visible in source code. None of these are acceptable on zero-runtime targets.

### 6.5.2 Error Communication by Return Value

Errors are communicated through return values. A function that can fail SHOULD return an integer error code, a negative value, a null address, or a sentinel value that the caller can check.

**Integer error codes.** Functions that wrap system calls return the system call's result directly. On Linux, system calls return a negative value on failure (the negated `errno` code) and a non-negative value on success:

```hopper
int fd = open(path, flags)
if (fd < 0) {
    // fd is a negated errno: -2 = ENOENT, -13 = EACCES, etc.
    return fd
}
```

**Null address.** Allocation functions return `null` when allocation fails:

```hopper
address buf = allocate 4096
if (buf == null) {
    // out of memory — buf is null, do not dereference
    return -12      // ENOMEM
}
```

**Sentinel integer.** Functions that count items, parse data, or scan buffers may return -1 or another out-of-band value to indicate failure:

```hopper
function findByte(address buf, int len, int b) int {
    for (int i = 0; i < len; i = i + 1) {
        if (buf[i] == b) {
            return i
        }
    }
    return -1       // not found
}
```

### 6.5.3 Callers SHOULD Check Return Values

Programs SHOULD check the return value of every call that can fail. Ignoring an error return is not a compile-time error in the current specification, but it is a programming defect. A future version of this specification MAY introduce a `must_use` annotation or a checked return type to enforce error checking at compile time.

The following pattern is idiomatic for propagating errors upward:

```hopper
function readAll(int fd, address buf, int count) int {
    int total = 0
    while (total < count) {
        int n = read(fd, buf + total, count - total)
        if (n < 0) {
            return n        // propagate error code
        }
        if (n == 0) {
            return total    // EOF
        }
        total = total + n
    }
    return total
}
```

### 6.5.4 No Result or Option Type

There is no `Result<T, E>` type, no `Option<T>` type, and no monadic error-handling construct in the standard language. Error handling is explicit: integer codes, null pointers, and out-of-band sentinels. This keeps the type system simple and eliminates the need for generics-over-generics.

Modules in the standard library MAY define named error code sets using `enum`:

```hopper
enum Errno {
    OK      = 0
    EPERM   = 1
    ENOENT  = 2
    EINTR   = 4
    EIO     = 5
    ENOMEM  = 12
    EACCES  = 13
    EBADF   = 9
}
```

Using an enum makes error code comparisons self-documenting:

```hopper
if (fd == -Errno.ENOENT) {
    // file not found
}
```

### 6.5.5 Functions That Cannot Fail

A function that cannot fail under any circumstance SHOULD have a return type appropriate to its computation, not a wrapped type or error-unioned type. Simplicity of the common case is a design value.

```hopper
function max(int a, int b) int {
    if (a > b) {
        return a
    }
    return b
}
```

No error return is needed or expected for `max`. The caller receives the value directly.

### 6.5.6 Non-Recoverable Conditions

Some conditions are non-recoverable: the program has encountered a situation from which it cannot proceed (corrupted internal state, a violated invariant, a programming error). For these conditions, the appropriate action is to terminate the process. This is done by calling the `exit` or `abort` syscall, not by raising an exception.

See §6.6 for compiler-generated `abort()` in non-release builds. See §6.7 for hardware faults.
