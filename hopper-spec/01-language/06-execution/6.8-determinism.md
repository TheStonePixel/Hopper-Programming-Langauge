## Determinism

### 6.8.1 Single-Threaded Determinism

A single-threaded Hopper program is deterministic: given the same inputs, it produces the same outputs. This is a normative guarantee of the language model, not merely an aspiration.

"Same inputs" means the same sequence of values returned by all system calls the program issues. "Same outputs" means the same sequence of arguments passed to all side-effecting system calls and the same program exit status.

This guarantee holds at the language level. The operating system and hardware may introduce sources of nondeterminism (ASLR, scheduler preemption timing, hardware interrupts) that affect the program's inputs (e.g., the value returned by a time syscall), but these are inputs to the program, not nondeterminism in the program's computation of its outputs from those inputs.

### 6.8.2 No Built-In Sources of Nondeterminism

The Hopper language contains no built-in sources of nondeterminism:

- There is no `random()` built-in.
- There is no `date()` or `time()` built-in.
- There is no `thread_id()` or `process_id()` built-in.
- There is no hash function that seeds itself from process state.
- There are no global constructors with unspecified initialization order.

A program that produces different outputs on successive runs with identical inputs has introduced nondeterminism through its inputs (system calls or inline assembly). The language itself does not introduce nondeterminism.

### 6.8.3 Nondeterminism Through System Calls

Programs that call time-returning syscalls (`clock_gettime`, `gettimeofday`), entropy syscalls (`getrandom`, reading from `/dev/urandom`), or process-state syscalls (`getpid`, `getuid`) observe values that differ between runs. This nondeterminism is in the system call's return value — it is an input to the program's computation. The program's computation from those inputs is still deterministic per §6.8.1.

Programs SHOULD make their nondeterministic inputs explicit and clearly separated from deterministic computation. A program that uses nondeterministic inputs SHOULD expose those inputs as parameters or configuration rather than embedding them in the middle of deterministic algorithms.

### 6.8.4 Nondeterminism Through Inline Assembly

Inline `asm {}` blocks have access to the full machine state, including registers that contain values from the system or hardware environment. A program that reads a hardware timestamp counter (`rdtsc` on x86-64), polls a hardware random number generator, or accesses memory-mapped I/O through inline assembly introduces nondeterminism through that path.

The language does not restrict what inline assembly may access. The programmer is responsible for understanding and documenting any nondeterminism introduced through `asm {}` blocks.

### 6.8.5 Memory Layout Determinism

The layout of data structures in memory is deterministic: the compiler assigns fixed offsets to struct fields, class fields, and local variables. The layout does not change between runs of the same binary.

However, the absolute addresses at which data resides MAY vary between runs if the operating system applies Address Space Layout Randomization (ASLR). ASLR is an OS-level feature applied to the process's virtual address space at load time; it is not a language-level nondeterminism. A program that computes values based on absolute pointer values (e.g., using a pointer as a hash key) will produce different values on different runs when ASLR is active.

Programs SHOULD NOT depend on absolute pointer values being stable across runs. Pointer equality between two pointers within a single run is stable and defined. Pointer equality across runs is not.

### 6.8.6 Argument Evaluation Order and Determinism

Arguments are evaluated left-to-right (see §6.2.2). This evaluation order is deterministic. Programs MAY rely on left-to-right evaluation order within a call expression.

### 6.8.7 Multi-Threaded Programs

When a Hopper program uses OS threads (via `clone`, `pthread_create`, or equivalent), the program is no longer single-threaded and the determinism guarantee of §6.8.1 no longer applies. Threads share an address space, and the interleaving of their operations is determined by the OS scheduler. Data races on shared memory are undefined behavior (see §6.9).

Multi-threaded programs MAY achieve determinism through explicit synchronization (mutexes, condition variables, atomic operations via system calls), but the Hopper language provides no built-in synchronization primitives. All synchronization is done through the `linux` contract using syscalls.
