## Tail Calls

### 6.4.1 Definition

A tail call is a function call that is the last operation performed before a function returns — the call's result is returned directly without further computation. In many languages, tail calls can be optimized so that the callee reuses the caller's stack frame, enabling recursion that does not grow the stack.

### 6.4.2 No Tail Call Guarantee

**Tail call optimization is not a language guarantee in the current specification.**

Hopper does not provide a `tail` keyword, a `@tailcall` annotation, or any other mechanism for the programmer to demand tail call optimization. A conforming implementation is not required to apply tail call optimization to any call, even one that is structurally a tail call.

Recursive Hopper functions MUST be treated by the programmer as stack-consuming. Each level of recursion allocates a new stack frame. Programs that rely on tail call optimization to avoid stack overflow are not portable and are not conforming.

### 6.4.3 LLVM-Level TCO

The Hopper compiler emits LLVM IR. LLVM's optimizer MAY recognize tail calls and apply the `musttail` or `tail` annotation to the generated call instruction, depending on the optimization level and target. This optimization is an emergent property of the LLVM backend, not a guarantee of the Hopper language.

Specifically:
- At `-O0`, LLVM does not apply tail call optimization.
- At `-O2` or `-O3`, LLVM MAY transform a structurally recursive tail call into a loop or a reuse of the caller's frame, subject to ABI constraints.

Programs MUST NOT be written in a way that depends on this optimization to be correct. If a function is correct only when TCO is applied (e.g., it would overflow the stack without it), the program is not a conforming Hopper program.

### 6.4.4 Iteration as the Correct Alternative

Any algorithm that requires tail call optimization for correctness SHOULD be expressed as an iterative loop instead. Hopper's `while` and `for` constructs are the idiomatic replacement for tail-recursive functions.

**Tail-recursive (do not rely on TCO):**
```hopper
function sum(int n, int acc) int {
    if (n == 0) {
        return acc
    }
    return sum(n - 1, acc + n)   // tail call — TCO not guaranteed
}
```

**Iterative (correct regardless of optimization level):**
```hopper
function sum(int n) int {
    int acc = 0
    while (n > 0) {
        acc = acc + n
        n = n - 1
    }
    return acc
}
```

### 6.4.5 Mutual Recursion

Mutually recursive functions (where A calls B and B calls A) are also subject to the same non-guarantee. A program with mutually recursive functions that would overflow the stack without TCO is not a conforming program under this specification. Rewrite such patterns as state machines or iterative dispatch tables.

### 6.4.6 Future Specification

A future revision of this specification MAY introduce an explicit tail call guarantee, a `tail` call annotation, or a `trampoline` module that implements guaranteed non-stack-growing dispatch. Any such addition will be accompanied by explicit grammar changes and will appear in the release notes for the relevant specification version.
