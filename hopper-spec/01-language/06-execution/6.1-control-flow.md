## Control Flow

### 6.1.1 Overview

Hopper provides the following control flow constructs: `if`/`else`, `while`, `for`, `break`, `continue`, `return`, and `defer`. All constructs are statement-level; there are no expression-level conditionals (no ternary operator). Every construct is syntactically delimited by braces — dangling-else and single-statement bodies are not permitted.

### 6.1.2 Conditional: `if`/`else`

**Syntax:**
```hopper
if (condition) {
    // then-branch
}

if (condition) {
    // then-branch
} else {
    // else-branch
}

if (condition) {
    // then-branch
} else if (other) {
    // else-if branch
} else {
    // else-branch
}
```

The condition expression MUST be of type `bool` or `int`. A `bool` condition proceeds when the value is `true`. An `int` condition proceeds when the value is non-zero (any value other than `0` is treated as true). A `float` or `address` value MUST NOT be used directly as a condition; use an explicit comparison expression instead.

The then-branch and each else-branch are blocks (`{ ... }`). The braces are not optional. An `else if` chain is a syntactic convenience; it is equivalent to a nested `if` inside an `else` block.

The condition expression is evaluated exactly once. If the condition is true, the then-branch executes and all else-branches are skipped. If the condition is false, the then-branch is skipped and the first matching else-branch (if any) executes.

**Example:**
```hopper
int fd = open(path, flags)
if (fd < 0) {
    return fd
} else {
    return read(fd, buf, count)
}
```

### 6.1.3 Loop: `while`

**Syntax:**
```hopper
while (condition) {
    // body
}

while (condition)
invariant expr {
    // body
}
```

The `while` loop evaluates its condition before every iteration, including the first. If the condition is false on entry, the body is never executed. The condition expression follows the same type rules as `if` (§6.1.2): MUST be `bool` or `int`.

#### 6.1.3.1 Invariant Clause

A `while` loop MAY carry one or more `invariant` clauses, each containing an expression. An `invariant` clause is a compile-time assertion that the expression holds at every evaluation of the loop condition — both before the first iteration and after each completed iteration.

```hopper
int i = 0
while (i < n)
invariant i >= 0 {
    i = i + 1
}
```

An invariant expression MUST be of type `bool` or `int`. In non-release builds, the compiler SHOULD emit a runtime check for each invariant at every loop-head evaluation and call `abort()` if the invariant is violated (see §6.6). In release mode, invariant checks MAY be omitted, and the compiler MAY use invariants as optimization assumptions.

A violated invariant in release mode is undefined behavior.

### 6.1.4 Loop: `for`

**Syntax:**
```hopper
for (init; condition; update) {
    // body
}
```

The `for` loop is syntactic sugar for a `while` loop with an explicit initializer and update step. Execution proceeds as follows:

1. The `init` clause executes once before the loop begins.
2. The `condition` expression is evaluated. If false, the loop exits.
3. The body executes.
4. The `update` clause executes.
5. Go to step 2.

**Init clause.** The init clause MUST be one of:
- A variable declaration with initializer: `int i = 0`
- An assignment to an existing variable: `i = 0`
- Empty (the semicolons are still required).

A variable declared in the init clause is scoped to the `for` loop body. It is not visible outside the loop.

**Condition.** The condition is an expression following the same type rules as `if` (§6.1.2). The condition MAY be empty, in which case the loop runs until a `break` or `return` terminates it.

**Update clause.** The update clause MUST be an assignment to an existing variable. It MAY be empty.

**Example:**
```hopper
for (int i = 0; i < count; i = i + 1) {
    buf[i] = 0
}
```

**Example — empty condition (infinite loop):**
```hopper
for (;;) {
    int b = read(fd, buf, 1)
    if (b <= 0) {
        break
    }
}
```

### 6.1.5 `break`

`break` immediately exits the innermost enclosing `while` or `for` loop. Control transfers to the statement immediately following the loop's closing brace.

`break` MUST appear inside a loop body. Using `break` outside a loop is a compile-time error.

`break` does not transfer across function boundaries. It affects only the loop in which it syntactically appears.

**Example:**
```hopper
while (true) {
    int n = read(fd, buf, 64)
    if (n <= 0) {
        break
    }
    process(buf, n)
}
// control resumes here after break
```

### 6.1.6 `continue`

`continue` immediately transfers control to the loop's update step (`for`) or condition check (`while`), skipping the remainder of the current iteration's body.

`continue` MUST appear inside a loop body. Using `continue` outside a loop is a compile-time error.

For a `for` loop, `continue` executes the update clause before re-evaluating the condition. For a `while` loop, `continue` re-evaluates the condition directly.

**Example:**
```hopper
for (int i = 0; i < n; i = i + 1) {
    if (buf[i] == 0) {
        continue          // skip null bytes; update i = i + 1 runs before re-checking
    }
    process(buf[i])
}
```

### 6.1.7 `return`

`return` exits the current function and transfers control to the call site. See §6.3 for the full semantics of return values. A bare `return` (with no expression) is valid only in a procedure (a function with no declared return type).

`return` MAY appear at any point in a function body. Multiple `return` statements are permitted. The compiler MUST ensure that all code paths through a non-void function contain a `return` with a compatible expression (see §6.3.4).

### 6.1.8 `defer`

`defer` schedules a function call for execution when the enclosing function returns, regardless of which `return` statement is reached.

**Syntax:**
```hopper
defer expression
```

The deferred expression MUST be a function call or method call. Deferred calls are executed in last-in, first-out order: the last `defer` statement encountered executes first.

```hopper
function processFile(string path) int {
    int fd = open(path, 0)
    if (fd < 0) {
        return fd
    }
    defer close(fd)         // runs when the function returns, regardless of which return

    int n = read(fd, buf, 1024)
    if (n < 0) {
        return n            // close(fd) executes here
    }
    return n                // close(fd) executes here
}
```

The argument expressions in a deferred call are **re-evaluated at function exit**, not captured at the point the `defer` statement is encountered. The compiler stores the raw AST of the deferred call and calls `genExpr` on it at each `return` site (see `codegenStmt.js` `DeferStmt` case and `emitDeferred` in `codegenExpr.js`). For the common case — passing a variable like `defer close(fd)` where `fd` does not change — the behavior is indistinguishable from eager capture. If an argument expression references a mutable variable that changes between the `defer` site and the `return`, the value at function exit is used. `defer` does not cross function boundaries; it is scoped to the function in which it appears.

### 6.1.9 No `goto`

Hopper does not have a `goto` statement. Unstructured control flow is not permitted. All control flow MUST be expressed through the structured constructs described in this section.
