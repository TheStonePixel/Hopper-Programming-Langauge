## 15.8 Compiler Diagnostics

The Hopper compiler emits structured diagnostic messages to `stderr`. Every diagnostic — whether an error or a warning — is formatted as a bordered block. Diagnostics are never emitted to stdout.

### 15.8.1 Severity Levels

There are two severity levels:

| Severity | Meaning | Exit behavior |
|----------|---------|---------------|
| **Error** | The program cannot be compiled correctly | Compilation aborts with exit code 1 |
| **Warning** | The program is valid but likely wrong or wasteful | Treated as fatal; compilation exits with code 1 |

**Warnings are fatal.** A build that produces any warning exits with code 1, exactly like a build that produces errors. There is no `--allow-warnings` flag. This is a deliberate design choice: a compilation that produces warnings is not a clean compilation.

### 15.8.2 Diagnostic Block Format

Every diagnostic is rendered as a bordered block on stderr:

```
│ Module: hello  File: main.hop  Line: 14
│ Error: Unknown enum variant
│        'Purpl' is not a variant of enum 'Color'
│ Hint: valid variants: Red, Green, Blue
└───────────────────────────────────────────────────────────
```

The left bar `│` and the closing `└───` line share a common color based on the diagnostic kind (see §15.8.5). All blocks are 60 characters wide. Message text wraps at 58 characters; continuation lines are indented to align with the message start.

A successful build emits a single green success block:

```
│ Built: hello  Profile: dev
│ Output: build/hello
└───────────────────────────────────────────────────────────
```

### 15.8.3 Error Types

| Error type | `ErrorType` constant | When emitted |
|------------|---------------------|--------------|
| Undeclared variable | `UndeclaredVariable` | A name is used that was never declared in scope |
| Undeclared function | `UndeclaredFunction` | A call to a function that is not declared anywhere |
| Type mismatch | `TypeError` | Type incompatibility, including assigning to a `const` variable |
| Argument count mismatch | `ArityError` | Wrong number of arguments at a call site |
| Private field access | `EncapsulationError` | Access to a field or method that is not visible |
| Unknown enum variant | `EnumError` | A variant name that does not exist in the enum |
| Constraint violation | `ConstraintError` | A `constrain` range check fails in strict mode |
| Syntax error | `ParseError` | ANTLR lexer or parser error |
| Internal compiler error | `InternalError` | A bug in the compiler itself |

### 15.8.4 Warning Types

| Warning type | `WarnType` constant | When emitted |
|--------------|---------------------|--------------|
| Missing return | `MissingReturn` | A non-void function may reach the end without a `return` |
| Unreachable code | `UnreachableCode` | Statements after a `return`, `break`, or `continue` |
| Unused variable | `UnusedVariable` | A declared local variable is never read |
| Redundant constraint | `TautologicalConstraint` | A `requires`, `ensures`, or `invariant` clause is always true |

### 15.8.5 Diagnostic Colors

Each diagnostic kind has a distinct bar color (requires a color-capable terminal; disabled by `NO_COLOR` environment variable):

| Kind | Color | Example label |
|------|-------|---------------|
| Error | Red | `Error:` |
| Warning | Bright blue | `Warning:` |
| Constraint warning | Orange (256-color 208) | `Warning:` |
| Cascade note | Bright red | `Note:` |
| Parse error | Magenta | `Syntax error:` |
| Success | Green | `Built:` |

**Cascade notes** are secondary errors that result from a previous error. When a variable declaration fails (e.g., the initializer references an undeclared name), the compiler marks the variable as poisoned. Any subsequent error that references the poisoned variable is emitted as a bright-red `Note:` block instead of a red `Error:` block, indicating it is a consequence of the primary failure rather than an independent error.

### 15.8.6 Unused Variable Suppression

Prefix a variable name with `_` to suppress the `UnusedVariable` warning for that variable:

```hopper
function process(int x) int {
    int _scratch = x * 2    // no UnusedVariable warning
    return x
}
```

This convention applies to all declared local variables including loop variables and array declarations. It does NOT apply to function parameters (parameters are never checked for unused status).

### 15.8.7 Statement-Level Error Recovery

The compiler performs per-statement error recovery inside function bodies. When a statement fails (e.g., an undeclared variable in an initializer), the compiler records the error and continues compiling subsequent statements. This allows all errors in a function to be reported in a single compilation pass rather than stopping at the first failure.

When a `VarDecl` statement fails, the declared variable name is added to a poison set. Any subsequent reference to that name produces a cascade `Note:` rather than a fresh `Error:`.

### 15.8.8 Parse Error Routing

ANTLR lexer and parser errors are captured by a custom `ErrorListener` and routed through the same diagnostic formatter as all other errors. Raw ANTLR messages are normalized into readable English before display. Parse errors appear as magenta `Syntax error:` blocks rather than being written as raw text to stderr.

If any parse errors are detected, compilation halts after the parse phase without attempting code generation. No IR is emitted for a file with parse errors.
