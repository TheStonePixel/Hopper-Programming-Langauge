## Compiler Interface

The `hopper` CLI is the sole compiler interface for Hopper programs. All compilation, linking,
and code generation MUST be performed through the CLI. Invoking the underlying compiler
components (`hopperc.js`, `clang`, `ld.lld`) directly is not supported.

### Synopsis

```
hopper <command> [file] [flags]
```

---

### Commands

#### `hopper build`

**Synopsis:** `hopper build [file.hop...] [-o <out>] [--production] [--test]`

Compiles one or more source files into an executable. When no file is specified, the entry
file is read from the `entry` field of `hopper.json` in the current working directory;
if `hopper.json` is absent, `src/main.hop` is used as the default.

The project root is determined by walking up from the entry file's directory until a directory
containing `hopper.json` is found. The binary name is taken from the `name` field of
`hopper.json`; if that field is absent, the entry file's basename (without extension) is used.

If `hopper.json` declares a `board` field naming a known bare-metal target, the build performs
a bare-metal compilation pipeline instead of a hosted build. See §10.8: Cross-compilation.

**Flags:**

| Flag | Description |
|------|-------------|
| `-o <out>` | Write the final executable to `<out>` instead of the default path |
| `--production` | Build with the production profile (`-O2`) |
| `--test` | Build with the test profile (same flags as dev) |
| (none) | Build with the dev profile (`-g -O0`) — the default |

**Profiles:**

| Profile | Clang flags | Output |
|---------|-------------|--------|
| `dev` (default) | `-g -O0 -c -fPIC -Wno-override-module` | `build/<name>` |
| `test` | `-g -O0 -c -fPIC -Wno-override-module` | `build/<name>` |
| `production` | `-O2 -c -fPIC -Wno-override-module` | `build/<name>` |

All profiles link with `clang <objects> -lpthread -o <out>`.

**Output layout:**

```
build/
  ll/        LLVM IR intermediates (.ll per source file)
  o/         object files (.o per source file)
  <name>     final executable
```

**Subcommand — `hopper build module <name>`:**

Creates `modules/<name>/` and `modules/<name>/tests/` in the global module store
(`hopper/modules/`). If the directory already exists, the `tests/` subdirectory is ensured.
This subcommand does not compile anything.

---

#### `hopper run`

**Synopsis:** `hopper run [binary]`

Executes a built binary. When no argument is given, locates the binary produced by the most
recent dev build by reading the `name` field from `hopper.json` and looking for
`build/<name>`. If that binary does not exist, `hopper run` exits with an error and instructs
the user to run `hopper build` first.

When a path argument is given, `hopper run` executes that path directly without any
compilation step.

**Alias — `hopper run test [module]`:** Delegates to the test runner. See `hopper test`.

---

#### `hopper dev`

**Synopsis:** `hopper dev [file.hop...]`

Watch mode. Compiles the given source files with the dev profile, runs the resulting binary,
and then watches the source directories for changes to `.hop` files. When a file changes, the
binary is rebuilt and restarted automatically.

`hopper dev` is implemented using `fs.watch` with a 150 ms debounce. It does not rebuild on
changes to imported modules, only to files in the directories that contain the named source
files. Rebuilds are serialized; if a rebuild is in progress when another change arrives, the
second rebuild runs after the first completes.

---

#### `hopper test`

**Synopsis:** `hopper test [module] [--deps]`

Runs the test suite by delegating to `seed/build_system/tools/test.js`. When no module name
is given, `hopper test` locates the nearest project root from the current working directory
(by walking up until a `hopper.json` is found) and passes it as `--root=<path>`.

When a module name is given, the test runner operates on that specific module. The `--deps`
flag causes the runner to also test each of the module's declared dependencies.

---

#### `hopper install`

**Synopsis:** `hopper install [name...]`

Installs one or more modules into the current project's `modules/` directory, then recursively
installs each module's dependencies into that module's own `modules/` subdirectory. The
resulting directory tree mirrors the dependency graph exactly.

When no name is given, all modules listed in the `dependencies` field of `hopper.json` are
installed.

Modules are resolved from the global module store at `hopper/modules/`. Network installation
from the registry (`github.com/HopperLangauge/<pkg>`) is defined but not yet connected to the
install path; the CLI currently reads only the local stdlib.

After installation, `hopper install` rewrites the `targets` section of `hopper.json` to point
`contract` and `implementation` paths at the locally installed copies under `modules/`.

---

#### `hopper uninstall`

**Synopsis:** `hopper uninstall <name...>`

Removes one or more installed modules from the project's `modules/` directory and deletes the
corresponding entries from the `dependencies` field of `hopper.json`. Operates only on the
current project; does not remove transitive dependencies installed inside `modules/<name>/modules/`.

---

#### `hopper publish`

**Synopsis:** `hopper publish [name]`

Publishes the current project to the Hopper package registry
(`github.com/HopperLangauge/<name>`). The name is taken from the command-line argument if
given, otherwise from the `name` field of `hopper.json`.

Requires the current directory to be a git repository. Creates a public GitHub repository
under the `HopperLangauge` organization and pushes the current `main` branch to it.

`hopper publish` requires the `HOPPER_TOKEN` or `GITHUB_TOKEN` environment variable to be set
to a GitHub personal access token with repository creation rights. If neither variable is set,
the command exits with an error.

---

#### `hopper revoke`

**Synopsis:** `hopper revoke [name]`

Permanently deletes the named package from the registry. The package name must be confirmed
by typing it interactively. This operation is irreversible; any user who has not yet installed
the package will be unable to do so after revocation.

Requires `HOPPER_TOKEN` or `GITHUB_TOKEN`. Exits non-zero if the confirmation does not match
the package name.

---

#### `hopper fix`

**Synopsis:** `hopper fix [path] [--dry-run]`

Validates and auto-repairs a project's structure. See §15.4: Linting.

---

#### `hopper check`

**Synopsis:** `hopper check [file.hop] [--asan] [--tsan] [--ubsan] [--msan]`

Compiles with runtime sanitizers enabled. See §15.4: Linting.

---

#### `hopper format` / `hopper fmt`

**Synopsis:** `hopper format <file.hop...> [--write|-w]`  
**Alias:** `hopper fmt <file.hop...> [--write|-w]`

Reformats source files via the AST-based canonical formatter. See §15.3: Formatting.

---

#### `hopper ir`

**Synopsis:** `hopper ir <file.hop>`

Compiles the given file and prints the generated LLVM IR to standard output. The IR is
produced by `genModule()` in `seed/compiler/src/codegenLLVM.js` and is not written to disk.
The build system respects `hopper.json` target bindings when resolving imports.

This command is intended for compiler developers and for diagnosing code generation issues. It
MUST NOT be used as a compilation step in project scripts.

---

#### `hopper ast`

**Synopsis:** `hopper ast <file.hop>`

Parses the given file, builds the AST, and prints it as formatted JSON to standard output.
The AST is produced by `buildAstFromSource()` in `seed/compiler/src/astBuilder.js`. The build
system resolves `hopper.json` target bindings when building the AST.

This command is intended for tooling authors and for inspecting how the compiler interprets
source files. See §15.6: Reflection.

---

#### `hopper parse`

**Synopsis:** `hopper parse <file.hop>`

Parses the given file using the ANTLR4 grammar and prints the raw parse tree (in ANTLR's
`toStringTree` format) to standard output. Unlike `hopper ast`, this command does not perform
AST construction or name resolution. It is intended for grammar debugging.

See §15.6: Reflection.

---

#### `hopper flash`

**Synopsis:** `hopper flash [--mount <path>] [--dry-run] [--production]`

Writes a compiled bare-metal kernel image to an SD card. Requires `hopper.json` to declare a
`board` field naming a known board configuration (currently: `pi-zero`).

`hopper flash` reads the image path from the board's flash configuration (e.g., `kernel.img`).
If no `--mount` path is given, the SD card is auto-detected by scanning mounted FAT32
filesystems for a known firmware marker file (e.g., `bootcode.bin`). On Linux, `/proc/mounts`
is consulted; common mount locations (`/media`, `/mnt`, `/run/media`) are also searched. On
macOS, `/Volumes` is scanned.

Before writing, the command verifies that required firmware blobs are present on the card. It
also ensures that required lines appear in `config.txt`, appending any missing lines.

After writing, `hopper flash` calls `sync` to flush filesystem buffers.

**Flags:**

| Flag | Description |
|------|-------------|
| `--mount <path>` | Use `<path>` as the SD card mount point instead of auto-detecting |
| `-m <path>` | Alias for `--mount` |
| `--dry-run` | Print what would be written without making any changes |
| `--production` | Use production clang flags (`-O2`) when the image is rebuilt before flashing |

---

#### `hopper debug`

**Synopsis:** `hopper debug`

**Not yet implemented.** Prints "hopper: debug support coming soon" and exits. An integrated
debugger is planned but not available. Users requiring source-level debugging MUST use `gdb`
or `lldb` against dev-profile binaries directly. See §15.5: Debugging.

---

#### `hopper profile`

**Synopsis:** `hopper profile`

**Not yet implemented.** Prints "hopper: profile support coming soon" and exits. Performance
profiling via an integrated tool is planned but not available.

### Legacy Flag Forms

The following flag forms are accepted for backwards compatibility. They behave identically to
the subcommand forms and MUST NOT be used in new scripts.

| Legacy form | Equivalent |
|-------------|------------|
| `hopper -c <file.hop>` | `hopper build <file.hop>` |
| `hopper --compile <file.hop>` | `hopper build <file.hop>` |
| `hopper -r [binary]` | `hopper run [binary]` |
| `hopper --run [binary]` | `hopper run [binary]` |
| `hopper --ir <file.hop>` | `hopper ir <file.hop>` |
| `hopper --ast <file.hop>` | `hopper ast <file.hop>` |
| `hopper --parse <file.hop>` | `hopper parse <file.hop>` |

### Compilation Pipeline (Hosted)

For hosted (non-bare-metal) builds, the compilation pipeline is:

1. **Source loading** — The entry `.hop` file is read. `import` directives are stripped
   from the source before parsing and collected separately.
2. **Binding resolution** — `hopper.json` target bindings are loaded by walking up from the
   source file's directory until a `hopper.json` is found. Each binding maps an interface name
   to interface and implementation file paths.
3. **AST construction** — `buildAstFromSource()` parses the source and all referenced
   implementation files, producing a single merged AST.
4. **Code generation** — `genModule()` emits LLVM IR text for the merged AST.
5. **IR deduplication** — `dedupeDeclarations()` removes duplicate `declare` lines that arise
   from including the same implementation file via multiple bindings.
6. **IR output** — The `.ll` file is written to `build/ll/<name>.ll`.
7. **Object compilation** — `clang <flags> build/ll/<name>.ll -o build/o/<name>.o` compiles
   the IR to a native object file.
8. **Linking** — `clang build/o/*.o -lpthread -o build/<name>` links the final executable.

### Exit Codes

The CLI exits with code `0` on success and code `1` on any error. Errors are written to
standard error. The `hopper revoke` command exits with code `0` if the user aborts by typing
a non-matching confirmation string.
