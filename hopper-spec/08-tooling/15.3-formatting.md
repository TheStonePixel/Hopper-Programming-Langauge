## Formatting

Hopper defines a single canonical source style. The AST-based formatter enforces this style
mechanically. There is no configuration; the canonical style is normative and all conforming
Hopper source files MUST be expressible without loss of meaning by the formatter.

### Commands

**Synopsis:**

```
hopper format <file.hop...> [--write|-w]
hopper fmt    <file.hop...> [--write|-w]
```

`hopper format` and `hopper fmt` are aliases. Both require at least one `.hop` file argument.

**Without `--write` or `-w`:** The formatted output is written to standard output. The source
file is not modified.

**With `--write` or `-w`:** The source file is overwritten with the formatted content in
place. A confirmation line is printed for each file: `formatted: <path>`.

If a named file does not exist, the command exits with an error and code `1` before
processing any files.

### Formatter Architecture

The formatter is implemented in `seed/compiler/src/formatter.js` and exports two functions:

- **`formatAst(ast, originalSource)`** — Produces canonical Hopper source from an already-
  constructed AST. The `originalSource` string is required to recover import lines, which are
  consumed by the AST builder and not stored in the AST. Browser-safe (no filesystem access).
- **`format(source, opts)`** — Builds the AST internally and calls `formatAst`. Node.js only.

The CLI (`hopper fmt`) calls `buildAstFromSource()` with `{ noImports: true }`, which
suppresses resolution of imported modules. The formatter then receives only the declarations
from the file being formatted; it does not attempt to format imported implementation files.

### Import Line Handling

Because the AST builder strips import directives before parsing, import lines are not
represented in the AST. The formatter recovers them by scanning the original source text for
lines that match the pattern `import <name> from <module>`. Recovered import lines are
emitted verbatim as the first section of the formatted output, preserving their original order
and exact text.

The formatter does not validate, sort, or deduplicate import lines. Import lines that are
present in the source will appear in the output unchanged.

### Canonical Style

The following rules define the canonical Hopper source format. The formatter enforces these
rules unconditionally.

**Indentation:** 4 spaces per level. Tab characters MUST NOT appear in formatted output.

**Braces:** Opening brace on the same line as the declaration or control statement (K&R style):

```hopper
function greet(int n) {
    if (n > 0) {
        write(1, msg, len)
    } else {
        write(1, empty, 0)
    }
}
```

**One statement per line.** The formatter does not combine statements onto a single line.

**Top-level separation:** Top-level declarations (functions, classes, interfaces, enums, etc.)
are separated by a single blank line. The formatter joins them with `\n\n` and appends a
trailing newline to the file.

**Else-if chains:** When an `else` block contains exactly one `if` statement, the formatter
collapses it to a same-line `else if`:

```hopper
if (x > 0) {
    ...
} else if (x < 0) {
    ...
} else {
    ...
}
```

**Operator spacing:** Binary operators are surrounded by single spaces:
`a + b`, `x == y`, `i < n`.

**Call argument spacing:** Function and method call arguments are separated by `, ` (comma
followed by one space). No space between the function name and the opening parenthesis.

**For loop:** `for (<init>; <cond>; <update>)` with no trailing semicolon on `<update>`.

**Empty blocks:** An empty block body produces `{\n}` with the closing brace on its own line
at the enclosing indentation level.

### Declaration Forms

The formatter handles all top-level declaration forms recognized by the AST builder:

| Form | Formatter function |
|------|--------------------|
| `import … from …` | Recovered from source text; emitted verbatim |
| `bind` | `fmtBind` |
| `strict` | `fmtStrict` |
| `alias` | `fmtAlias` |
| `const` | `fmtConst` |
| `enum` | `fmtEnum` |
| `bitfield` | `fmtBitfield` |
| `struct` | `fmtStruct` |
| `interface` | `fmtInterface` |
| `class` | `fmtClass` |
| `template` | `fmtTemplate` |
| `function` / `extern function` | `fmtFunction` |
| `entry` | `fmtEntry` |

Built-in and imported declarations (`isBuiltin`, `isImported`) are silently omitted from the
output. `extern function` declarations are emitted as a single-line signature without a body.

### Idempotency

The formatter MUST be idempotent: formatting a file that is already in canonical form produces
output identical to the input. The VS Code extension relies on this property to avoid
unnecessary document edits — if the formatted text equals the original text, the provider
returns an empty `TextEdit` array.

Implementations MUST ensure idempotency is preserved when adding new formatting rules.

### VS Code Format-on-Save

The VS Code extension (`seed/extensions/hopper-vscode/extension.js`) integrates the formatter
as a `DocumentFormattingEditProvider`. When Format Document or Format on Save is triggered:

1. The document's current text is written to a temporary file.
2. `hopper fmt "<tmp>" --write` is executed via `execSync` with a 10-second timeout.
3. The reformatted content is read from the temporary file.
4. The temporary file is deleted.
5. If the content changed, a `TextEdit` replacing the entire document is returned.

Format on save is enabled by default for `.hop` files:

```json
"[hopper]": {
    "editor.defaultFormatter": "hopper.hopper-language",
    "editor.formatOnSave": true
}
```

Users MAY override this in their VS Code settings. The setting `hopper.executablePath` controls
which `hopper` binary the extension invokes; see §15.2.

### Relationship to the Compiler

The formatter invokes the same AST builder used by the compiler (`buildAstFromSource`). A
source file that does not parse successfully cannot be formatted; the formatter will exit
with an error rather than produce partial output.

The formatter does not perform type checking or semantic analysis. It operates solely on the
syntactic structure of the file. Semantic errors that the compiler would reject do not
prevent the formatter from producing output, provided the file parses without error.
