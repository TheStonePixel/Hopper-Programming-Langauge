## Metadata

Hopper projects carry metadata in two files: `hopper.json` (the project manifest, present in
every project) and `<name>.meta.json` (bare-metal build metadata, written by `hopper build`
for bare-metal targets).

---

### `hopper.json` — Project Manifest

Every Hopper project MUST contain a `hopper.json` file at its root. The build system, the
package manager, `hopper fix`, and `hopper flash` all read this file.

The manifest is a JSON object. The fields defined below are the only fields the build system
reads. Additional fields MAY be present and MUST be preserved by tools that rewrite the
manifest (e.g., `hopper install`, `hopper fix`).

#### Common Fields

These fields apply to both executable and library projects:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | The project name. Used as the binary name for executables and as the package identifier in the registry. Characters outside `[a-zA-Z0-9_-]` are not valid in a name produced by `hopper init`; they are replaced with `_`. |
| `version` | string | Yes | Semantic version string, e.g. `"0.1.0"`. |
| `description` | string | No | Human-readable description of the project. |
| `license` | string | No | SPDX license identifier, e.g. `"MIT"`. Default when created by `hopper init --yes`: `"MIT"`. |
| `type` | string | Yes | Either `"executable"` or `"library"`. Determines which fields are required and which `hopper fix` checks apply. |
| `dependencies` | object | No | Maps module names to version strings. Each key is a module that must be installed via `hopper install` before building. `hopper fix` adds missing dependencies here automatically when it can. Example: `{ "linux": "0.1.0", "x86_64": "0.1.0" }` |

#### Executable-Only Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `entry` | string | Yes | Path to the program entry file, relative to the project root. Conventionally `"src/main.hop"`. `hopper build` reads this field when no file argument is given. |
| `targets` | object | No | Target-specific contract-to-implementation bindings. The only currently recognized target name is `"host"`. Each key within a target is an contract name (e.g., `"IO"`, `"FileSystem"`, `"SIMD"`). |

**Target binding fields** (each binding is an object within `targets.<target>.<InterfaceName>`):

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `from` | string | Yes | The module that provides this contract (e.g., `"linux"`, `"x86_64"`). Used by `hopper fix` to determine which modules are covered by target bindings. |
| `contract` | string | Yes | Path to the contract `.hop` file, relative to the project root. After `hopper install`, this path is rewritten to point into `modules/`. |
| `implementation` | string | Yes | Path to the implementation `.hop` file, relative to the project root. After `hopper install`, this path is rewritten to point into `modules/`. |
| `inline` | boolean | No | Reserved; defaults to `false`. |

#### Library-Only Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `main` | string | No | Conventional path to the library's primary source file (e.g., `"src/<name>.hop"`). Currently documentation-only; not read by the build system. |
| `exports` | object | No | Documents the interfaces this library exports. Each key is an contract name; each value has `contract` and `implementation` path fields. The `exports` field is documentation and tooling metadata — the build system does not consume it directly. Consuming projects declare full binding paths in their own `targets` section. `hopper fix` creates stub files for declared but missing `contract` and `implementation` paths. |

#### Bare-Metal Field

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `board` | string | No | Names a known bare-metal board configuration. When present, `hopper build` and `hopper flash` use the bare-metal pipeline. Currently the only recognized value is `"pi-zero"`. |

#### Complete Executable Example

```json
{
    "name": "my-program",
    "version": "0.1.0",
    "description": "Example Hopper program",
    "license": "MIT",
    "type": "executable",
    "entry": "src/main.hop",
    "dependencies": {
        "linux": "0.1.0",
        "x86_64": "0.1.0"
    },
    "targets": {
        "host": {
            "IO": {
                "from": "linux",
                "contract": "modules/linux/interfaces/IO.hop",
                "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
            },
            "FileSystem": {
                "from": "linux",
                "contract": "modules/linux/interfaces/FileSystem.hop",
                "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
            }
        }
    }
}
```

#### Complete Library Example

```json
{
    "name": "my-lib",
    "version": "0.1.0",
    "description": "Example Hopper library",
    "license": "MIT",
    "type": "library",
    "main": "src/my-lib.hop",
    "exports": {
        "MyInterface": {
            "contract": "interfaces/MyInterface.hop",
            "implementation": "src/MyImpl.hop"
        }
    }
}
```

### `.meta.json` — Bare-Metal Build Metadata

When `hopper build` compiles a project targeting a known bare-metal board, it writes a
`.meta.json` file alongside the kernel image in the build directory. The file is named
`<binaryName>.meta.json` (e.g., `build/my-kernel.meta.json`).

This file is read by `hopper flash` to display build information before writing to the SD
card. It is also available for external tooling.

#### Fields

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Binary name (from `hopper.json` `name` field, or entry file basename) |
| `board` | string | Board identifier (e.g., `"pi-zero"`) |
| `outputFile` | string | Filename of the raw image written to the SD card (e.g., `"kernel.img"`) |
| `arch` | string | CPU architecture string (e.g., `"armv6"`) |
| `endian` | string | Byte order: `"little"` or `"big"` |
| `bits` | number | Word size in bits (e.g., `32`) |
| `loadAddress` | string | Hex string of the physical load address (e.g., `"0x8000"`) |
| `entrySymbol` | string | Linker entry symbol name (e.g., `"_start"`) |
| `size` | number | Size of the kernel image in bytes at build time |
| `builtAt` | string | ISO 8601 timestamp of the build (e.g., `"2026-05-23T14:30:00.000Z"`) |

#### Example

```json
{
    "name": "my-kernel",
    "board": "pi-zero",
    "outputFile": "kernel.img",
    "arch": "armv6",
    "endian": "little",
    "bits": 32,
    "loadAddress": "0x8000",
    "entrySymbol": "_start",
    "size": 4096,
    "builtAt": "2026-05-23T14:30:00.000Z"
}
```

#### Notes

- The `builtAt` field is set from `new Date().toISOString()` at link time. It reflects the
  wall-clock time of the build machine, not a reproducible timestamp. Programs that require
  reproducible builds MUST strip this field from the `.meta.json` before comparing artifacts.
- `hopper flash` uses `builtAt` only for display purposes; it does not use it to decide
  whether to flash.
- The `.meta.json` file is not written for hosted (non-bare-metal) builds.
- The schema is defined by the board configuration in `seed/build_system/hopper` (`BOARD_CONFIGS`).
  Fields from the board's `meta` sub-object are merged with the computed fields (`name`,
  `board`, `outputFile`, `size`, `builtAt`). Adding a new board may introduce additional
  fields; consuming tools SHOULD treat unknown fields as opaque.
