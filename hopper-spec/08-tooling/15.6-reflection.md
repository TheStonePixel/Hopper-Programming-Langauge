## Reflection

### Runtime Reflection

Hopper has no runtime reflection. The following capabilities do not exist:

- Runtime type introspection (`typeof`, `instanceof`, RTTI)
- Runtime enumeration of fields, methods, or interfaces
- Dynamic dispatch via reflection
- Runtime creation of types from strings or metadata
- Serialization or deserialization driven by type information

These features are permanently absent from the language design, not planned for a future
version. Programs that require type-driven dispatch MUST encode that dispatch explicitly
using interfaces, enums, or function pointers.

### Compile-Time Introspection Commands

Three CLI commands expose compiler internals for tooling authors, compiler developers, and
users debugging unexpected behavior. These are diagnostic tools; they are not language
features and MUST NOT be invoked from within Hopper source code or project build scripts.

---

#### `hopper ast <file.hop>`

**Synopsis:** `hopper ast <file.hop>`

Parses the given file, resolves `hopper.json` target bindings, constructs the full AST, and
prints it as pretty-printed JSON to standard output.

The AST is produced by `buildAstFromSource()` in `seed/compiler/src/astBuilder.js`. Import
directives in the source file are resolved and the referenced implementation files are
compiled into the AST. The printed AST therefore reflects the merged view that the code
generator will see, not just the top-level file in isolation.

**Typical uses:**

- Verify that a declaration was parsed correctly (correct type, correct parameter names).
- Inspect how an expression was classified (e.g., whether a call was recognized as a
  `MethodCall` or a free `Call`).
- Confirm that an interface binding was resolved and that the implementation's declarations
  appear in the AST.
- Debug the AST builder itself when adding new grammar constructs.

**Output format:** A single JSON object. The top-level keys include `functions`, `classes`,
`interfaces`, `enums`, `consts`, `entry`, and others, each holding an array of AST nodes.
The exact schema is defined by `astBuilder.js` and is not part of the stable specification;
it MAY change across compiler versions.

---

#### `hopper ir <file.hop>`

**Synopsis:** `hopper ir <file.hop>`

Compiles the given file and prints the generated LLVM IR to standard output. The IR is not
written to disk; it is produced transiently by `genModule()` in
`seed/compiler/src/codegenLLVM.js`.

The `hopper.json` target bindings are resolved before generating the IR, so the output
reflects the full merged IR including all bound implementation files.

**Typical uses:**

- Verify that a function was emitted with the correct LLVM signature.
- Inspect the IR for a specific expression to confirm that the intended LLVM instructions are
  being generated (e.g., that a `cast<float>` emits `sitofp`, that an `asm {}` block produces
  the expected inline assembly).
- Confirm that `dedupeDeclarations()` is correctly removing duplicate `declare` lines.
- Diagnose miscompilation by comparing the IR to what a reference LLVM IR would look like.

**Output format:** Valid LLVM IR text, suitable for passing to `llc` or `opt` directly.

---

#### `hopper parse <file.hop>`

**Synopsis:** `hopper parse <file.hop>`

Parses the given file using the ANTLR4 grammar and prints the raw parse tree in ANTLR's
`toStringTree(ruleNames)` format to standard output. No AST construction or name resolution
is performed.

**Typical uses:**

- Distinguish a grammar bug from an AST builder bug: if the parse tree is correct but the AST
  is wrong, the fault is in `astBuilder.js`; if the parse tree is wrong, the fault is in the
  grammar.
- Verify that a new syntactic construct is being recognized by the grammar at all before
  implementing AST construction for it.
- Inspect how the ANTLR parser tokenizes an ambiguous piece of source.

**Output format:** ANTLR `toStringTree` format, which is a nested parenthesized expression.
This format is not JSON and is not machine-readable in a stable way; it is intended for human
inspection only.

---

### Summary

| Command | Level | Resolves imports | Output format |
|---------|-------|-----------------|---------------|
| `hopper parse <file>` | Grammar (ANTLR) | No | ANTLR `toStringTree` |
| `hopper ast <file>` | Semantic (AST) | Yes | JSON |
| `hopper ir <file>` | Code generation (LLVM IR) | Yes | LLVM IR text |

Each command operates at a successively deeper level of the compilation pipeline. Together
they provide full observability into every stage of compilation without requiring access to
the compiler source code.
