## Linting

Hopper provides two tools that perform lint-like checks:

1. **`hopper fix`** — validates and auto-repairs project structure
2. **`hopper check`** — compiles with runtime sanitizers enabled

A dedicated semantic linter (checking type errors, unreachable code, unused variables, etc.)
is **Not yet implemented.**

---

### `hopper fix` — Project Structure Validation

**Synopsis:** `hopper fix [path] [--dry-run|-n]`

`hopper fix` examines a project's directory layout, manifest, and source files and
automatically repairs any structural problems it can correct without user input. Problems that
require a manual decision are reported but not touched.

The project root is determined by searching upward from the current working directory for a
`hopper.json` file. If a `path` argument is given, that directory is used as the starting
point instead.

If `hopper.json` does not exist in the resolved root, `hopper fix` exits with code `1`.  
If `hopper.json` exists but contains invalid JSON, `hopper fix` exits with code `1` and
reports that manual repair is required.

#### `--dry-run` / `-n`

When `--dry-run` or `-n` is passed, `hopper fix` reports what it would change but makes no
modifications to the filesystem or to `hopper.json`. The exit code reflects what would have
happened: `0` if no manual intervention would be needed, `1` if any issue would require
manual resolution.

#### Checks and Auto-repairs

**1. Standard directory structure**

`hopper fix` verifies that the following directories exist:

- `src/`
- `tests/`
- `interfaces/` — for library projects only (where `"type": "library"` in `hopper.json`)

Any missing directory is created automatically.

**2. Entry file for executables**

For executable projects (`"type": "executable"`), `hopper fix` verifies that the file named
by the `entry` field in `hopper.json` exists. If it does not, a stub entry file is created:

```hopper
entry main {
}
```

The parent directory is created if necessary.

**3. Export files for libraries**

For each entry in the `exports` object of `hopper.json`, `hopper fix` verifies that the files
named by `interface` and `implementation` exist. If either is missing, a stub file is created:

- Missing interface file:
  ```hopper
  interface <ExportName> {
  }
  ```
- Missing implementation file:
  ```hopper
  class Standard<ExportName> implements <ExportName> {
      constructor() { }
      destructor() { }
  }
  ```

**4. Undeclared imports**

`hopper fix` scans all `.hop` files in `src/`, `interfaces/`, the file named by `entry`,
and all files named in `exports` for `import X from Y` directives. For each module name `Y`
encountered:

- If `Y` is already declared in `dependencies`, no action is taken.
- If `Y` appears as a `from` field in any `targets` binding, it is considered covered and no
  action is taken.
- If `Y` is in the global module store (`hopper/modules/<Y>/`), the dependency is added to
  `hopper.json` automatically with the version from the module's own `hopper.json`
  (default: `"0.1.0"`).
- If `Y` is not in the global module store, the issue is reported as requiring manual
  resolution (`hopper install <Y>`), and `hopper fix` exits with code `1`.

Self-imports (where `Y` equals the project's own `name`) are ignored.

#### Output Format

`hopper fix` uses ANSI color codes in its output. Each auto-fixed issue is prefixed with a
green checkmark (`✓`). Each issue requiring manual resolution is prefixed with a red cross
(`✗`). Running with `--dry-run` adds a yellow `[dry run]` label to the header.

If no issues are found, the output is:

```
✓  nothing to fix  <project-path>
```

#### Exit Codes

| Condition | Exit code |
|-----------|-----------|
| No issues found, or all issues auto-repaired | `0` |
| Any issue requiring manual resolution | `1` |
| `hopper.json` not found | `1` |
| `hopper.json` contains invalid JSON | `1` |

### `hopper check` — Sanitizer Builds

**Synopsis:** `hopper check [file.hop] [--asan] [--tsan] [--ubsan] [--msan]`

`hopper check` compiles the given source file with runtime sanitizer instrumentation enabled.
It uses the dev profile clang flags (`-g -O0 -c -fPIC -Wno-override-module`) and links the
resulting objects with the specified sanitizers.

When no source file is given, `main.hop` in the current directory is used.

When no sanitizer flag is given, `--asan` is used by default.

Multiple sanitizer flags may be combined. The sanitizers are passed to clang as a
comma-separated list: `clang <objects> -lpthread -fsanitize=<list> -o <out>`.

The output binary is written to `build/<name>.check`. The build system prints the output path
and the sanitizer list on success:

```
check build: build/myapp.check  (sanitizers: address,undefined)
run with: build/myapp.check
```

The output binary MUST be run manually to observe sanitizer output. `hopper check` does not
execute the binary.

#### Sanitizer Flags

| Flag | Clang `-fsanitize` value | Detects |
|------|--------------------------|---------|
| `--asan` | `address` | Out-of-bounds memory accesses, use-after-free, use-after-return, double-free, invalid frees, memory leaks (on supported platforms) |
| `--tsan` | `thread` | Data races between threads |
| `--ubsan` | `undefined` | Undefined behavior: signed integer overflow, null pointer dereference, misaligned pointer access, invalid shift, out-of-bounds array index (for fixed-size arrays), and related classes of UB |
| `--msan` | `memory` | Use of uninitialized memory |

#### Notes on Combining Sanitizers

ASan and TSan are mutually exclusive in most clang versions; combining `--asan --tsan` on the
same binary is undefined behavior at the clang level and SHOULD NOT be used. ASan and UBSan
may be combined safely: `--asan --ubsan`. MSan is incompatible with ASan and TSan.

#### Relationship to `hopper build`

`hopper check` produces a separate output file (`build/<name>.check`) and does not replace or
affect the normal dev or production build artifacts. A sanitizer build and a dev build may
coexist in the same `build/` directory.

### Standalone Semantic Linter

**Not yet implemented.**

No tool currently performs standalone semantic analysis — type checking outside of a full
compilation, dead-code detection, unused variable warnings, or similar checks. Semantic errors
are reported only by the compiler during `hopper build`.
