## 15.9 Test Framework

The Hopper test framework is a lightweight, file-level test runner built into the `hopper` CLI. It compiles each test file, optionally runs it, and compares the outcome against inline directives declared in comments at the top of the file. There is no separate test configuration file — the test file is its own specification.

### 15.9.1 Running Tests

```
hopper test                     run tests for the current project
hopper test --deps              run tests for the current project and its full dependency chain
hopper test <name>              run tests for a named module from the global module store
hopper test <name> --deps       named module and its full dependency chain
```

The runner discovers test files by looking for a `tests/` directory in the project root (or named module root). Every file ending in `.hop` in that directory is a test. Test files are sorted lexicographically before execution.

A project that has no `tests/` directory, or an empty one, produces no test output — it does not fail.

### 15.9.2 Test File Directives

Directives are single-line comments that MUST appear at the top of the file, before any Hopper declarations. Their order relative to each other does not matter. Any directive placed after the first non-comment, non-blank line is not recognized.

#### `// COMPILE_ONLY`

```hopper
// COMPILE_ONLY
// Verify this program compiles without errors or warnings.

function add(int a, int b) int {
    return a + b
}

entry main {
    int _sum = add(1, 2)
}
```

The file MUST compile with exit code 0. The runner does not execute the binary. Use `// COMPILE_ONLY` to verify that legal syntax is accepted, that valid programs produce no diagnostics, and that `_`-prefixed names suppress unused-variable warnings.

#### `// EXPECT_ERROR: <message>`

```hopper
// EXPECT_ERROR: Undeclared variable

entry main {
    int x = notDeclared + 1
}
```

The file MUST fail to compile (exit code ≠ 0). The compiler's stderr+stdout output MUST contain the string after the colon (whitespace-trimmed). The match is a substring match — the message does not need to be the entire error text, only a unique prefix or fragment.

If the file compiles successfully when `// EXPECT_ERROR:` is present, the test fails.

Only one `// EXPECT_ERROR:` directive is recognized per file. To test multiple errors, write separate test files.

#### `// EXPECT: <output-line>`

```hopper
// COMPILE_ONLY is not set — the binary is executed.
// EXPECT: hello
// EXPECT: world

entry main {
    // ... code that writes "hello\nworld\n" to stdout
}
```

Each `// EXPECT:` directive adds one expected output line. Lines are matched in order against the program's stdout, one per line, with trailing whitespace stripped. The test passes if and only if:

- All expected lines are matched in order.
- No unexpected non-empty output lines remain after matching.

Multiple `// EXPECT:` directives may appear in the same file. They are collected in declaration order.

#### `// TEST: <section-name>`

```hopper
// TEST: addition
// EXPECT: 3
// EXPECT: 5

// TEST: subtraction
// EXPECT: 2
// EXPECT: -1

entry main {
    // ... code that writes 3, 5, 2, -1 on separate lines
}
```

`// TEST:` groups subsequent `// EXPECT:` lines into a named section. Named sections appear as labeled subsections in the runner's output. Each section is independent in the output display, but matching is still sequential across all sections combined.

A file may have any number of `// TEST:` sections. A `// TEST:` directive before any `// EXPECT:` lines starts a named section; all preceding `// EXPECT:` directives (if any) belong to the implicit unnamed section.

`// TEST:` is incompatible with `// EXPECT_ERROR:` and `// COMPILE_ONLY` — mixing them produces undefined test behavior.

#### `// XFAIL`

```hopper
// XFAIL
// EXPECT_ERROR: not yet implemented

entry main {
    // ... code exercising an unimplemented feature
}
```

Marks the test as an **expected failure**. An `// XFAIL` test:

- Is still compiled and run (or failed to compile) normally.
- Its result is not counted in the pass/fail total.
- If the test fails (as expected), it is shown with a warning icon in the output.
- If the test unexpectedly passes, it is shown with a question-mark icon to indicate the `// XFAIL` should be removed.

`// XFAIL` is used to document known bugs or unimplemented features without causing the test suite to report failure.

### 15.9.3 Output Format

The runner groups results by project or module. For each group:

```
diagnostics  10 files
────────────────────────────────────────────────────────────
  ✓  clean                    1
  ✓  suppress_unused          1
  ✓  warn_unused_var          1
  ✓  error_enum               1
  ○  xfail_example            (expected failure — not counted)

  10 assertions passed

────────────────────────────────────────────────────────────
✓  10/10 passed
```

Icons:

| Icon | Meaning |
|------|---------|
| `✓`  | Test passed |
| `✗`  | Test failed |
| `○`  | Expected failure (XFAIL) that failed as expected |
| `?`  | Expected failure (XFAIL) that unexpectedly passed |

### 15.9.4 Discovery and Module Context

The runner discovers tests relative to the **current working directory**, not relative to the repository root. Running `hopper test` from `hopper/modules/linux/` finds tests only in `hopper/modules/linux/tests/`. Running it from `hopper/programs/wc/` finds tests only in `hopper/programs/wc/tests/`.

The `--deps` flag causes the runner to also recursively discover tests in each dependency declared in the project's `hopper.json`, looking them up in the global module store at `hopper/modules/`. Circular dependencies are handled by a visited-set — each directory is processed at most once.

### 15.9.5 Compilation Context

Each test file is compiled with the same invocation that `hopper build` uses, but with the test file as the entry point. The compiler receives the file's absolute path, and the output binary is placed in `hopper/build/tests/<group>__<testname>`.

Test files that import modules MUST declare those modules in the project's `hopper.json` and have them installed. The test runner does not do any special module resolution — it relies entirely on the project's existing build configuration.

### 15.9.6 Exit Code

The runner exits with code 0 if all non-XFAIL tests pass, and code 1 if any non-XFAIL test fails. This makes `hopper test` usable in CI pipelines.

Expected failures (`// XFAIL`) never affect the exit code regardless of their outcome.

### 15.9.7 Naming Conventions

The following conventions keep test suites readable:

- `clean.hop` — a fully valid program that should compile without any diagnostics.
- `warn_<name>.hop` — tests that a specific warning is emitted.
- `error_<name>.hop` — tests that a specific compile error is emitted.
- `suppress_<name>.hop` — tests that a suppression mechanism (like `_` prefix) works.
- `xfail_<name>.hop` — a known failure or planned-but-unimplemented feature.

Test files SHOULD use `_`-prefixed variable names for any variables whose values are not checked by `// EXPECT:` lines, to prevent spurious `UnusedVariable` warnings from contaminating tests that use `// COMPILE_ONLY`.
