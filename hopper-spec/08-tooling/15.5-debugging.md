## Debugging

### Dev Build Debug Info

Hopper dev-profile builds include DWARF debug information. The clang flags for the `dev`
and `test` profiles are:

```
-g -O0 -c -fPIC -Wno-override-module
```

`-g` causes clang to embed DWARF into the object files and the final executable. `-O0`
disables all optimization, ensuring that variable values and call frames are visible to a
debugger without having been elided or reordered. Production builds (`--production`) use
`-O2` and omit `-g`, so they do not contain debug info.

### External Debuggers

Users MAY debug Hopper programs using any debugger that supports DWARF debug info:

- **GDB** (GNU Debugger): `gdb build/<name>`
- **LLDB** (LLVM Debugger): `lldb build/<name>`

Because Hopper compiles via LLVM IR, the DWARF information is generated by clang's backend.
Source-level stepping, breakpoints, and variable inspection work to the degree that clang's
DWARF emission supports them for IR-compiled code.

There are no Hopper-specific debugger extensions or helper scripts. Setting breakpoints on
Hopper function names requires knowing the mangled symbol names as they appear in the binary:
methods are compiled as `ClassName_methodName` (e.g., `MyClass_get`); free functions and
`entry` blocks use the name as written.

### Integrated Debugger

**Not yet implemented.**

The `hopper debug` command is a stub:

```
hopper: debug support coming soon
```

It prints this message and exits with code `0`. No debugging session is launched. An
integrated debugger is planned as a future deliverable.

### Runtime Error Detection with Sanitizers

`hopper check` (see §15.4) provides runtime error detection by compiling with LLVM sanitizers
enabled. This is the most productive debugging technique available without a native debugger
integration:

- **`--asan`** (default): Catches out-of-bounds accesses, use-after-free, double-free,
  and (on supported platforms) memory leaks. ASan produces a detailed stack trace when a
  violation is detected.
- **`--ubsan`**: Catches undefined behavior such as signed integer overflow, null pointer
  dereference, and misaligned accesses. UBSan halts with a message identifying the source
  location and the specific UB class triggered.
- **`--asan --ubsan`** (combined): Both sanitizers are active simultaneously and may be used
  together.

Sanitizer builds MUST be run manually after `hopper check` produces the output binary.
`hopper check` does not execute the binary.

### Compiler-Level Debugging

Two commands expose compiler internals for diagnosing miscompilation or unexpected code
generation:

**`hopper ast <file.hop>`** — Dumps the full AST as JSON. Use this to verify how the compiler
interpreted a source file: which declarations were recognized, how expressions were parsed,
and how types were resolved. See §15.6: Reflection.

**`hopper ir <file.hop>`** — Dumps the generated LLVM IR. Use this to verify that the correct
machine operations are being emitted, that function signatures match expectations, and that
the LLVM type system is consistent. The IR is the definitive intermediate representation —
any code generation bug will be visible here.

**`hopper parse <file.hop>`** — Dumps the raw ANTLR4 parse tree. Use this when a syntactic
construct is being rejected or misinterpreted, to distinguish a grammar bug from an AST
builder bug. See §15.6: Reflection.

### Bare-Metal Debugging

For bare-metal targets (e.g., `pi-zero`), traditional debuggers are not available without
dedicated hardware debug probes (JTAG/SWD). No probe support is implemented.

For bare-metal debugging, the recommended approach is to write diagnostic output via the
hardware UART using inline assembly or the board-specific communication contract, and to
use `hopper ir` to inspect generated code before flashing.
