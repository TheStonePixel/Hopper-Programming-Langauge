## Purpose

Hopper exists to solve a specific problem: the gap between what systems software requires and what existing languages provide.

Systems software — operating system kernels, device drivers, embedded firmware, network stacks, real-time controllers — has requirements that most languages cannot satisfy simultaneously:

**Direct hardware access.** The software must be able to read and write specific memory addresses, issue CPU instructions by name, and configure hardware registers without going through an abstraction layer that may not exist on the target platform.

**Predictable resource usage.** Memory allocation, stack depth, and execution time must be knowable from reading the source code. Runtime systems that introduce unpredictable pauses, background threads, or hidden allocations are not usable.

**No mandatory runtime.** The language must be able to produce programs that run on bare metal with no operating system, no heap allocator, and no standard library. A conforming Hopper program may have zero runtime dependencies.

**Hardware portability with source stability.** Programs should be writable in terms of capabilities (I/O, memory, processes) rather than specific hardware implementations. Changing the target hardware should require editing only build configuration, not program source.

**Readable, auditable code.** In safety-critical and security-critical contexts, code must be understandable from inspection alone. Hidden behavior — from implicit conversions, operator overloading, or opaque runtimes — is a liability.

Existing systems languages meet some of these requirements. None meet all of them while also providing a clean, consistent model for hardware abstraction. Hopper is designed around that specific intersection.
