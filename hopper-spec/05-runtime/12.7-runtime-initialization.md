# 12.7 Runtime Initialization

## 12.7.1 Overview

There is no Hopper-specific initialization phase. Programs begin executing at the first
statement in the `entry` block. Everything that must happen before that point is either
handled by `libc` startup or the bare-metal boot stub — neither of which Hopper controls
or extends.

## 12.7.2 Hosted Startup Sequence

```
kernel execve(2)
  → maps ELF segments into process address space
  → kernel jumps to _start  (from libc crt1.o, linked by clang)
    → _start initializes libc internals (errno, locale, stdio, etc.)
    → _start calls @main(argc, argv, envp)
      → Hopper entry block executes
      → entry block returns i64 exit_code
    → _start calls exit(exit_code & 0xff)
  → kernel tears down process
```

The Hopper runtime adds nothing to this sequence. There is no Hopper initializer that
runs between `_start` and `@main`.

## 12.7.3 No Global Constructors

Hopper has no global mutable variables and no global class instances. Therefore there
are no global constructors to run before `@main`. The LLVM `.init_array` / `.ctors`
section is empty in all Hopper-compiled objects.

This is a consequence of the language design: the grammar does not permit `ClassName x`
or `int x = 5` at module (file) scope. State is always local to a function, passed as
a parameter, or stored in a class instance that the `entry` block constructs explicitly.

## 12.7.4 Static String Initialization

String literals are embedded in the binary at compile time. The compiler emits them as
LLVM global constants in the `.rodata` section:

```llvm
@.str.0 = private unnamed_addr constant [6 x i8] c"hello\00"
@.str.1 = private unnamed_addr constant [4 x i8] c"foo\00"
```

These constants are placed by the linker into the read-only data segment. They are
accessible from the moment the kernel maps the ELF image, before `_start` runs. No
initialization code is required.

## 12.7.5 Enum Initialization

Enum values are compile-time integer constants. They are substituted by the compiler
at the use site and generate no runtime data or initialization code.

## 12.7.6 Bare-Metal Startup Sequence

```
power on / reset
  → CPU fetches first instruction from reset vector (board-specific)
  → board firmware (e.g. Raspberry Pi bootcode.bin, start.elf)
    → loads kernel.img at load address (e.g. 0x8000 for Pi Zero)
    → jumps to 0x8000
      → _start  (from boot.s, compiled by hopper build)
        → initializes stack pointer
        → zeros .bss section
        → sets up peripheral registers (board-specific)
        → calls Hopper entry function by name
          → Hopper entry block executes
          → entry block returns (bare-metal programs typically loop forever)
```

The boot stub is board-specific and is maintained in
`seed/build_system/boards/<board-name>/boot.s`. It is not Hopper source code; it is
ARM assembly. Programs MUST NOT rely on any state set by `boot.s` beyond a valid stack
pointer and zeroed `.bss`. All peripheral initialization needed by the program MUST be
done in Hopper source, in the `entry` block or in functions it calls.

## 12.7.7 Implications for Library Design

Because there is no pre-`entry` initialization phase, libraries cannot perform one-time
setup lazily on first import. Libraries that require initialization (e.g. allocating a
global buffer, opening a file, querying the OS) MUST expose an explicit `init` function
that the program calls at the start of its `entry` block. A future version of the language
MAY add a module-level `init` hook, but none exists today.
