# 12.3 Scheduler

## 12.3.1 No Hopper Scheduler

Hopper has no scheduler. There is no coroutine runtime, no fiber system, no green-thread
multiplexer, and no cooperative or preemptive task scheduler built into the language or
its runtime services.

## 12.3.2 Single-Threaded by Default

A Hopper program is single-threaded at startup. The process has exactly one thread of
execution when the `entry` block begins running. This is identical to a standard C
program's startup state.

## 12.3.3 OS Scheduling

The Linux kernel schedules Hopper processes and their threads using its standard scheduling
policy (SCHED_OTHER by default). Hopper has no mechanism to influence scheduling policy
except through the `System` interface in `linux`, which exposes syscalls such as
`sched_setscheduler`, `sched_setparam`, and `setpriority`.

## 12.3.4 Multithreading via OS Threads

Programs MAY create multiple threads by calling the Linux `clone` syscall (or the
`pthread_create` function from `libpthread`) through an `extern function` declaration or
the `Process` interface. The build system links every hosted program against `-lpthread`,
making `libpthread` available without any additional configuration.

However, Hopper provides no synchronization primitives in the language itself. Mutexes,
condition variables, atomic operations, and memory barriers are the program's
responsibility. Programs SHOULD use the `linux/System` and `linux/Process` interfaces or
declare `extern` wrappers to the appropriate `pthread_*` or `syscall` functions.

## 12.3.5 Async / Await

**Not yet implemented.**

Hopper does not currently have `async`/`await` syntax, a future/promise type, or an
event-loop runtime. Asynchronous I/O requires explicit use of `epoll` via the
`linux/Network` interface and manual state management in program code.

## 12.3.6 Coroutines

**Not yet implemented.**

There is no coroutine or generator construct in the language.
