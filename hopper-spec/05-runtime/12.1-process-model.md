# 12.1 Process Model

## 12.1.1 A Hopper Program Is a Linux Process

A hosted Hopper program is a standard Linux ELF executable. The kernel loads it as a
normal process; no Hopper-specific loader or runtime helper is involved.

## 12.1.2 Entry Point

The compiler emits the entry block as an LLVM function returning `i64`:

```
define i64 @main(...)
```

When the `entry` block is named `main`, the generated function is `@main` directly.
When the `entry` block uses any other name (e.g. `entry startup`), the compiler emits
both `@startup` and a thin `@main` stub that calls `@startup` and returns its result:

```llvm
define i64 @main() {
entry:
  %0 = call i64 @startup()
  ret i64 %0
}
```

This ensures that `libc`'s startup code (`_start`) always finds a C-ABI-compatible `@main`
regardless of the name chosen in source.

## 12.1.3 Process Lifecycle

The hosted startup sequence is:

1. The Linux kernel maps the ELF image into the process address space.
2. The kernel transfers control to `_start`, which is provided by `libc` startup objects
   (`crt1.o`).
3. `_start` initializes `libc` internals and calls `@main`.
4. The Hopper `entry` block executes.
5. When the `entry` block returns (explicitly or by falling off the end), `@main` returns
   its `i64` value to `_start`.
6. `_start` passes that value to `exit()`. The process exit code is the low 8 bits of the
   returned `i64`, consistent with POSIX exit-status semantics.

If the `entry` block does not contain a `return` statement, the compiler MUST emit
`ret i64 0`, causing the process to exit with status 0.

## 12.1.4 argc and argv

When the entry block declares parameters, the compiler adapts the C ABI:

```hopper
entry main(int argc, string[] argv) {
    string path = argv[1]
}
```

The generated LLVM signature uses `i32` for `argc` (matching the C calling convention)
and `i8**` for `string[]`. The compiler sign-extends `argc` from `i32` to `i64` before
storing it, so the Hopper program sees a native `int` (i64) throughout its body.

## 12.1.5 No Hopper-Registered At-Exit Handlers

Hopper does not register any `atexit` handlers. Destructors of objects in scope at process
exit are not automatically called (see §12.6). Programs that require cleanup on exit MUST
arrange it explicitly.

## 12.1.6 Bare-Metal Entry

For bare-metal targets (e.g. Pi Zero), `libc` is absent and the C runtime startup is not
used. The build system links a board-specific boot stub (`boot.s`) whose `_start` symbol
initializes the stack and peripheral registers, then calls the Hopper entry function by
name. The linker script places `.text.boot` at the board's load address (e.g. `0x8000`
for Pi Zero). The Hopper entry function is compiled with the same `define i64 @<name>`
signature, but no `@main` stub is emitted and no libc startup is invoked.
