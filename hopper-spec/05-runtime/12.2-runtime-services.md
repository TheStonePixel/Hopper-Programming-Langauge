# 12.2 Runtime Services

## 12.2.1 Scope

"Runtime services" are facilities that a language implementation provides automatically to
every program without requiring the programmer to declare or call them explicitly. This
section enumerates all such services in Hopper ŌĆö the list is intentionally short.

## 12.2.2 Provided Services

### malloc / free (allocate / deallocate)

The compiler emits the following LLVM declarations in every hosted compilation unit:

```llvm
declare i8* @malloc(i64)
declare void @free(i8*)
```

Every `allocate N` expression compiles to a call to `@malloc(N)`, returning an `address`.
Every `deallocate ptr` statement compiles to a call to `@free(ptr)`.

These declarations are omitted for bare-metal targets. Bare-metal programs that need
dynamic allocation MUST supply their own allocator and declare it via `extern function`.

### abort (contract violation)

In non-release builds, the compiler emits:

```llvm
declare void @abort()
```

Every contract-violation site (`requires`, `ensures`, `invariant`, `constrain`) compiles
to a conditional branch: if the condition is false, the program calls `@abort()` followed
by an `unreachable` instruction. `abort()` terminates the process immediately without
unwinding the stack or calling any destructors.

In release builds (`hopper build --production`), `abort` is not declared and no
contract-check IR is emitted. Contract clauses are elided entirely.

## 12.2.3 Services NOT Provided

A conforming implementation MUST NOT provide, and programs MUST NOT depend upon, any of
the following being automatically initialized or available:

- **Reflection** ŌĆö no type-information tables, no `typeof`, no `instanceof`.
- **Virtual dispatch tables** ŌĆö no vtables. Interface conformance is verified at compile
  time; method calls always use the concrete receiver type known at the call site.
- **Exception handling runtime** ŌĆö no `setjmp`/`longjmp` wrappers, no C++ unwind tables,
  no Itanium EH personality functions.
- **Thread-local storage runtime** ŌĆö no `__tls_get_addr` or equivalent.
- **Signal handlers** ŌĆö no default signal handlers installed by Hopper. Programs that
  need signal handling MUST use the `Process` contract from `linux`.
- **Standard I/O initialization** ŌĆö no `stdio` setup. Programs that need buffered I/O
  MUST implement it themselves or use a library module.

## 12.2.4 Relationship to Syscalls

Syscalls are not runtime services. They are hardware-mediated requests to the operating
system kernel. Programs access syscalls through the `linux` contract modules, which
compile to inline assembly. There is no Hopper runtime layer between the program and the
kernel.
