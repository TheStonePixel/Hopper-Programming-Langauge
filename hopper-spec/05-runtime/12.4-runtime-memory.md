# 12.4 Runtime Memory

## 12.4.1 Overview

Memory management in Hopper is entirely manual. There is no garbage collector, no
reference-counting runtime, and no automatic arena. The programmer controls every
allocation and deallocation.

## 12.4.2 Heap Allocation

`allocate N` compiles to `malloc(N)` and returns an `address` (i8*). The returned pointer
is uninitialized. `deallocate ptr` compiles to `free(ptr)`. Double-free, use-after-free,
and heap corruption are program bugs; the language provides no protection beyond what the
OS and sanitizers offer.

For bare-metal targets, `malloc`/`free` are not available. Programs MUST supply a custom
allocator.

## 12.4.3 Stack

Stack memory is managed by the CPU and OS automatically. Local variables, function
parameters, and the IRBuilder's alloca slots are stack-allocated. Stack frames are created
and destroyed by the function prologue and epilogue in the generated machine code.

The default stack size on Linux is 8 MiB per thread, enforced by the kernel's rlimit for
`RLIMIT_STACK`. Programs that require larger stacks MUST set the limit explicitly via
`setrlimit` (available through the `linux/System` contract) or by passing a custom stack
to `clone`/`pthread_create`.

Stack overflow results in a `SIGSEGV` from the kernel. Hopper does not perform stack
checks or grow the stack automatically.

## 12.4.4 Static Data

String literals are embedded in the binary's read-only data segment (`.rodata`) by the
compiler. The compiler emits them as LLVM global constants of the form:

```llvm
@.str.N = private unnamed_addr constant [K x i8] c"...\00"
```

Programs access string literals as `i8*` pointers into `.rodata`. Writes to these pointers
produce undefined behavior.

Class instance data is heap-allocated when a constructor is called. There is no static
class-instance storage.

## 12.4.5 mmap

Programs MAY map memory regions directly using the `mmap` syscall via the `linux/Memory`
contract. This is the correct mechanism for:

- Memory-mapped files.
- Large allocations that bypass the `malloc` heap.
- Shared memory between processes.
- Executable memory for JIT code.

## 12.4.6 No Runtime Arena

There is no built-in arena allocator. Programs that want arena semantics MUST implement
them in Hopper source code, typically as a class that wraps a large `allocate` block and
bump-allocates from it.

## 12.4.7 Sanitizer Support

`hopper check` compiles with LLVM sanitizers (`-fsanitize=address` by default, or
`--tsan`, `--ubsan`, `--msan`). AddressSanitizer instruments every heap and stack access
to detect use-after-free, heap buffer overflows, stack buffer overflows, and
use-after-scope errors at runtime.
