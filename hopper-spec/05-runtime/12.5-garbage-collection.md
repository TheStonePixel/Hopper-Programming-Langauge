# 12.5 Garbage Collection

## 12.5.1 No Garbage Collection

**Not yet implemented.**

Hopper does not have a garbage collector. This is a deliberate design choice, not an
oversight.

## 12.5.2 Rationale

### Predictable Latency

A garbage collector introduces stop-the-world pauses — intervals where the program's
threads are halted while the collector identifies and reclaims unreachable memory. Even
incremental and concurrent collectors introduce unpredictable latency spikes.

Hopper targets systems programming workloads — operating system components, network
servers, embedded firmware, and performance-critical tools — where latency predictability
is essential. A program that must respond within a bounded time window cannot afford
pauses it cannot control.

### No Runtime Complexity

Garbage collectors require significant runtime infrastructure: object headers, root
scanning, write barriers, card tables or remembered sets, and (for moving collectors)
pointer fixup. This infrastructure imposes memory overhead per object, code-size overhead
for write barriers at every pointer store, and CPU overhead for concurrent collection
threads.

Hopper's compiled output contains none of this machinery.

### Compatibility with Bare-Metal

Garbage collectors typically require cooperation from the memory system (mmap, virtual
memory, signal handling). These facilities are unavailable on bare-metal targets. An
optional GC that cannot run on one of Hopper's two supported execution models would be a
second-class feature; Hopper therefore does not include one.

## 12.5.3 Manual Memory Management Discipline

Programs MUST pair every `allocate` with exactly one `deallocate`. The recommended
discipline is:

- Allocate in a constructor, deallocate in the destructor.
- Use `defer deallocate ptr` at the allocation site to guarantee deallocation on all
  exit paths from the enclosing function.
- For long-lived objects, make ownership explicit in the design — only one component is
  responsible for deallocation.

## 12.5.4 Future Consideration

A future version of Hopper MAY introduce an optional, opt-in reference-counting wrapper
or arena type that provides automatic reclamation for programs that do not require
deterministic destruction or bare-metal support. Any such feature would be a library
module, not a language primitive, and would carry no overhead for programs that do not
use it.
