# 12.6 Resource Management

## 12.6.1 Manual Resource Management

Resources — file descriptors, heap memory, memory-mapped regions, mutex handles, socket
descriptors, and any other OS-level object — are managed manually in Hopper. The language
provides no automatic cleanup at scope exit.

This is a known limitation. Programs MUST close, free, or release every resource they
acquire.

## 12.6.2 Constructor / Destructor Pattern

Classes support a constructor and a destructor:

```hopper
class Buffer {
    address data
    int len

    constructor(int n) {
        self.data = allocate n
        self.len  = n
    }

    destructor() {
        deallocate self.data
    }
}
```

The constructor runs when an instance is created. The destructor runs when
`destroy obj` is called explicitly. Destructors are NOT called automatically when
the instance goes out of scope. This is unlike C++ RAII. Programs MUST call
`destroy` (or invoke the destructor explicitly) before the instance variable
leaves scope or is overwritten.

## 12.6.3 defer

The `defer` statement schedules a function call or expression to execute when the enclosing
function returns, regardless of which return path is taken. It is the primary mechanism
for ensuring cleanup in the presence of multiple return paths:

```hopper
function processFile(string path) int {
    int fd = open(path, O_RDONLY, 0)
    if (fd < 0) { return -1 }
    defer close(fd)

    // ... work with fd ...
    return 0
}
```

Deferred calls execute in last-in, first-out order immediately before the function
returns. They run for every `return` statement, including implicit returns added by
the compiler.

`defer` is scoped to the enclosing function, not the enclosing block. There is no
block-scoped defer. Placing a `defer` inside an `if` or `while` body still defers
until the function returns.

## 12.6.4 Known Limitation: No Automatic Scope Destruction

Because destructors are not called at scope exit, the following pattern is unsafe:

```hopper
function risky() {
    Buffer b = Buffer(1024)
    if (someCondition()) {
        return   // b.destructor() is NOT called here
    }
    destroy b
}
```

The correct pattern uses `defer`:

```hopper
function safe() {
    Buffer b = Buffer(1024)
    defer destroy b
    if (someCondition()) {
        return   // defer fires; b.destructor() IS called here
    }
}
```

A future revision of the language MAY add automatic destructor calls at scope exit for
named objects. Until then, `defer destroy` is the required idiom.

## 12.6.5 File Descriptors

File descriptors are integers. The `linux/FileSystem` interface provides `open` and
`close`. Programs MUST call `close` on every file descriptor they open. Leaking file
descriptors causes the process's open-file-descriptor table to fill, eventually causing
`open` to fail with `EMFILE`.

Use `defer close(fd)` immediately after a successful `open` to guarantee closure on all
return paths.

## 12.6.6 Memory-Mapped Regions

Memory mapped via `mmap` (through `linux/Memory`) MUST be unmapped with `munmap` before
the region is no longer needed. Hopper does not track mapped regions. The kernel reclaims
all mappings when the process exits, but relying on process exit for cleanup is poor
practice in long-running programs.
