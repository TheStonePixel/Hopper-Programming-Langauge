## Strings

Hopper has two distinct string concepts: the primitive `string` type built into the compiler,
and a higher-level `String` object class planned for the `ds/` module. This section covers
both.

---

### 13.3.1 The `string` Primitive

`string` is a compiler built-in. It maps directly to `i8*` in LLVM IR — a raw pointer to a
byte sequence expected to be null-terminated. It is identical in representation to `address`;
the distinction is semantic only.

**String literals** are written with double quotes:

```hopper
string greeting = "Hello, world!"
```

String literals are stored in the program's read-only data section. They are immutable.
Writing through a pointer to a string literal is undefined behavior.

**String literals do not carry a length.** The length of a string literal must be computed
by iterating to the null terminator, or known statically by the programmer.

**There is no string concatenation operator.** To concatenate strings the program must:
1. `allocate` a buffer large enough to hold both strings plus the null terminator.
2. Copy the bytes of each string manually using a loop or inline assembly.
3. Place the null terminator at the end.

**There is no `println` or `print` built-in.** Output requires calling `write()` from the
`IO` interface (§13.4) with an explicit file descriptor and byte count.

#### What `string` does NOT provide

The following operations are **not** available without a module or an `extern` declaration:

- `strlen` — string length
- `strcmp` / `strncmp` — string comparison
- `strcpy` / `strncpy` — string copy
- `strcat` / `strncat` — string concatenation
- `strstr` / `strchr` — string search
- `sprintf` / `snprintf` — formatted output into a string

Programs that need these operations MUST either:

1. Implement them in Hopper (preferred), or
2. Declare them with `extern`:

```hopper
extern function strlen(string s) int
extern function strcmp(string a, string b) int
```

Extern declarations to libc are acceptable for diagnostic and prototyping purposes.
They MUST NOT appear in published module code — each module MUST implement its core
operations in Hopper.

#### Casting between `string` and `address`

`string` and `address` are the same LLVM type. The cast module provides explicit conversions:

```hopper
import cast from cast

address raw = cast<address>(myString)   // string → address (bitcast)
string  s   = cast<string>(rawPtr)      // address → string (bitcast; assert null-terminated)
```

#### Reading a single byte from a string

```hopper
string s = "ABC"
address p = cast<address>(s)
byte first = p::value           // 'A' = 65
byte second = (p + 1)::value    // 'B' = 66
```

### 13.3.2 String Module

**Not yet implemented.**

A `String` class providing safe, length-tracked string operations is planned as part of the
`ds/` module (see §13.2). The `ds/string.hop` file will define `template String<byte>` — a
heap-allocated byte array with known length.

A legacy `string` module exists in the repository but is not conforming and MUST NOT be
imported.

Until the `ds/` module is standardized, all string handling in conforming programs MUST use
the `string` primitive with manual memory management.

---

### 13.3.3 String Interoperability with IO

The `write()` function from the `IO` interface (§13.4) requires an `address` buffer and a
byte count. To write a string the program needs a length. The canonical pattern is to compute
the length statically (for literals) or with a custom `strlen`-equivalent loop:

```hopper
// Write a known string literal — length computed statically
function writeLiteral(int fd, string msg, int len) int {
    return write(fd, cast<address>(msg), len)
}
```

Or with a manual length loop:

```hopper
function myStrlen(string s) int {
    int n = 0
    address p = cast<address>(s)
    while (p::value != 0) {
        p = p + 1
        n = n + 1
    }
    return n
}
```
