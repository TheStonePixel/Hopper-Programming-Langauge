## Core Built-ins

The core built-ins are compiler primitives. They are available in every Hopper translation
unit without any import. They are not defined in any module file — the compiler emits them
directly as LLVM IR.

---

### Primitive Types

| Hopper type | LLVM type | Notes |
|-------------|-----------|-------|
| `int` | `i64` | Signed 64-bit integer. Word-size on all supported targets. |
| `byte` | `i8` | Signed 8-bit integer. |
| `char` | `i8` | Alias for `byte`. Used to signal character intent. |
| `bool` | `i1` | Boolean. `true` = 1, `false` = 0. |
| `float` | `double` | 64-bit IEEE 754 double-precision floating point. |
| `address` | `i8*` | Raw untyped pointer. Arithmetic is in byte units. |
| `string` | `i8*` | Null-terminated C string. Identical representation to `address`. |
| `unsigned int` | `i64` | Same bits as `int`; unsigned interpretation for arithmetic and casts. |
| `unsigned byte` | `i8` | Same bits as `byte`; unsigned interpretation. |
| `string[]` | `i8**` | Array of null-terminated strings. Used for `argv`. |

`string` and `address` are the same LLVM type; the distinction is semantic — `string` signals
null-terminated character data and `address` signals a raw memory pointer. The compiler
does not enforce separate type sets for them.

There are no pointer-to-typed-pointer types (no `int*`). Raw memory is always `address`.

---

### Memory Operators

#### `allocate N`

```hopper
address ptr = allocate 128
```

Allocates `N` bytes on the heap. `N` MUST be a compile-time constant integer expression or a
variable of type `int`. Returns `address`. Equivalent to `malloc(N)` in C. No zeroing is
performed — the returned memory is uninitialized.

The program MUST eventually call `deallocate` on the returned address. Failing to do so is a
memory leak. The compiler does not enforce this at compile time.

#### `deallocate ptr`

```hopper
deallocate ptr
```

Frees the memory at `ptr`. `ptr` MUST be an `address` value previously returned by `allocate`.
Calling `deallocate` on a non-heap pointer, a null pointer, or a pointer that has already been
deallocated is undefined behavior.

---

### Address Operators

These operators work directly on `address` values and provide typed access to raw memory.

#### `ptr::value`

Read or write the byte at the address pointed to by `ptr`.

```hopper
address buf = allocate 4
byte b = buf::value          // read byte at buf[0]
buf::value = 65              // write 'A' to buf[0]
```

The type of `ptr::value` is `byte`. For accessing larger values at an address, the program must
cast appropriately or use manual offset arithmetic.

#### `ptr::address`

Read the address stored at the address pointed to by `ptr` (pointer-to-pointer dereference).

```hopper
address slot = buf::address  // read pointer stored at buf
```

#### `ptr::size`

Evaluates to the byte size of the pointed-to type. Because all `address` values are untyped,
this operator is primarily useful for computing offsets in struct-layout arithmetic.

#### Address arithmetic

`address` supports integer addition and subtraction:

```hopper
address next = buf + 1   // advance by 1 byte
address prev = buf - 1   // retreat by 1 byte
```

No bounds checking is performed. Out-of-bounds access is undefined behavior.

### Inline Assembly (`asm {}`)

```hopper
function write(int fd, address buf, int count) int {
    int result
    asm {
        rax = 1
        rdi = fd
        rsi = buf
        rdx = count
        syscall
        result = rax
    }
    return result
}
```

An `asm {}` block may appear inside any function body. Inside the block:

- Register names (`rax`, `rdi`, `rsi`, `rdx`, `r10`, `r8`, `r9`, `xmm0`–`xmm7`, etc.) are
  assignable targets and readable sources.
- Variables from the enclosing scope may appear on the right-hand side of register assignments
  (`rdi = fd`) or on the left-hand side to receive register values (`result = rax`).
- Raw x86-64 instructions may appear as bare statements (`syscall`, `movdqu xmm0, [rdi]`, etc.).
- No `asm {}` blocks are permitted in program source files. Hardware specifics belong in
  `x86_64/` module files.

The compiler emits `asm {}` blocks as LLVM inline assembly. The exact clobber semantics are
determined by which registers the block reads and writes.

### Raw Type Casts

The compiler accepts direct type-name cast expressions:

```hopper
int n     = int (someFloat)
byte b    = byte (someInt)
float f   = float (someInt)
address p = address (someInt)
bool flag = bool (someInt)
```

These expressions perform the LLVM conversion corresponding to the source and destination
types (see the cast table in §13.1.1). They are unguarded — the programmer is responsible for
correctness.

**Prefer `cast<T>(x)` from the `cast/` module** (§13.1.1) over raw cast expressions in
application code. The `cast/` module provides the same conversions with explicit, searchable
syntax and clear documentation of what each overload does.

---

### 13.1.1 The `cast/` Module

The `cast/` module (`cast/src/Cast.hop`) provides template functions that wrap the compiler's
primitive type conversions with named, explicitly-dispatched overloads.

**Import:** No `hopper.json` target binding is required. `cast` is resolved as a module import:

```hopper
import cast from cast
```

**Usage:**

```hopper
int   n = cast<int>(3.9)       // float → int, truncates to 3
float f = cast<float>(7)       // int → float → 7.0
address p = cast<address>(42)  // int → address
byte  b = cast<byte>(300)      // int → byte, truncates to 44 (300 mod 256)
bool  flag = cast<bool>(1)     // int → bool, true
string s = cast<string>(ptr)   // address → string (bitcast; assumes null-terminated)
```

**Supported overloads:**

| `cast<T>` | Accepted source types | Notes |
|-----------|----------------------|-------|
| `cast<int>` | `address`, `byte`, `char`, `float`, `bool`, `unsigned byte` | float→int truncates toward zero |
| `cast<address>` | `int`, `string` | `inttoptr` / `bitcast` |
| `cast<byte>` | `int`, `float`, `bool` | truncates to 8 bits |
| `cast<char>` | `int` | truncates to 8 bits; alias for `cast<byte>` |
| `cast<float>` | `int`, `byte`, `bool` | `sitofp` / `uitofp` |
| `cast<bool>` | `int`, `byte`, `float` | non-zero → `true` |
| `cast<string>` | `address` | `bitcast`; caller asserts null-termination |
| `cast<unsigned int>` | `byte`, `address` | zero-extends byte; `ptrtoint` for address |
| `cast<unsigned byte>` | `int` | truncates to 8 bits with unsigned interpretation |

The full LLVM opcode table for every source/destination pair is in Appendix A.

**Full cast conversion table:**

| From \ To     | `int`      | `byte`     | `float`    | `bool`     | `address`  | `string`  |
|---------------|------------|------------|------------|------------|------------|-----------|
| `int`         | —          | `trunc`    | `sitofp`   | `icmp ne 0`| `inttoptr` | —         |
| `byte`        | `sext`     | —          | `sitofp`   | `icmp ne 0`| —          | —         |
| `float`       | `fptosi`   | `fptosi`   | —          | `fcmp une 0`| —         | —         |

### Built-in Arithmetic Operators

All arithmetic operators are built into the compiler and require no import.

**Integer (`int`, `byte`, `unsigned int`, `unsigned byte`):**
`+`, `-`, `*`, `/`, `%`, `<<`, `>>`, `&`, `|`, `^`, `~`, unary `-`

**Float (`float`):**
`+`, `-`, `*`, `/`, unary `-`

**Comparison (all numeric types):**
`==`, `!=`, `<`, `<=`, `>`, `>=`

**Logical (`bool`):**
`&&`, `||`, `!`

There is no implicit coercion between types. Mixed-type arithmetic (e.g., `int` + `float`)
MUST be written with an explicit cast.

---

### Array Literals and Indexing

Fixed-size stack arrays may be declared with a size literal:

```hopper
int arr[5]
```

Elements are accessed with `[]`:

```hopper
arr[0] = 10
int x = arr[2]
```

No bounds checking is performed at runtime. The size MUST be a compile-time constant.

String indexing via `[]` is not built in — strings are `i8*` pointers; use `address` arithmetic
and `::value` to read individual bytes.
