## Generated Artifacts

The build system writes several categories of files to disk during a build. All are located under the project's `build/` directory unless otherwise noted.

### LLVM IR files (.ll)

**Location:** `build/ll/<basename>.ll`

For every source file compiled in a build, the CLI generates LLVM IR using `genModule(ast, { target })` and writes it to `build/ll/<basename>.ll`, where `<basename>` is the source file's name without its `.hop` extension.

Example: compiling `src/main.hop` and `modules/x86_64/src/LinuxSyscalls.hop` produces:
```
build/ll/main.ll
build/ll/LinuxSyscalls.ll
```

For bare-metal builds, the IR is also passed through `dedupeDeclarations()` before being written, which removes duplicate `declare` lines that arise when multiple source files reference the same external symbol.

### Object files (.o)

**Location:** `build/o/<basename>.o`

Each `.ll` file is compiled to a native object file by invoking `clang` with the flags for the selected build profile. The object file is named after the same basename as the IR file.

Example:
```
build/o/main.o
build/o/LinuxSyscalls.o
```

The `clang` invocation for hosted builds uses `-c -fPIC -Wno-override-module` plus profile-specific optimization flags. For bare-metal builds, it additionally uses `-target arm-none-eabi -mcpu=arm1176jzf-s -mfloat-abi=soft -ffreestanding -nostdlib`.

### Final executable (hosted)

**Location:** `build/<name>`

After all object files are produced, the CLI links them with:
```
clang <obj1> <obj2> ... -lpthread -o build/<name>
```

The binary name is taken from the `"name"` field in `hopper.json`. If that field is absent, the entry file's basename is used. The `-lpthread` flag is always passed; its presence does not require the program to use threads.

### Bare-metal image

**Location:** `build/kernel.img`

For projects with a `"board"` field in `hopper.json`, the final output is a raw binary image rather than a hosted executable. The build process is:

1. Compile all `.hop` files to `.o` files with bare-metal clang flags.
2. Compile the board boot stub (`seed/build_system/boards/pi-zero/boot.s`) to `build/o/boot.o`.
3. Write an auto-generated linker script to `build/link.ld`.
4. Link with `ld.lld` using the linker script: `ld.lld -T build/link.ld boot.o *.o -o build/<name>.elf`.
5. Strip the ELF to a raw binary: `llvm-objcopy -O binary build/<name>.elf build/kernel.img`.

The intermediate ELF file (`build/<name>.elf`) is retained after the build.

The linker script places `.text.boot` (from the boot stub) at `0x8000`, followed by `.text`, `.rodata`, `.data`, and `.bss`. ARM exception index sections are discarded.

Currently defined boards: `pi-zero` (ARMv6, 32-bit, little-endian, load address `0x8000`).

### Board metadata (.meta.json)

**Location:** `build/<name>.meta.json`

Written after every successful bare-metal build. Contains:

| Field | Source |
|---|---|
| `name` | `hopper.json` `"name"` |
| `board` | `hopper.json` `"board"` |
| `outputFile` | Board config `flash.outputFile` (e.g. `"kernel.img"`) |
| `arch` | Board config `meta.arch` (e.g. `"armv6"`) |
| `endian` | Board config `meta.endian` (e.g. `"little"`) |
| `bits` | Board config `meta.bits` (e.g. `32`) |
| `loadAddress` | Board config `meta.loadAddress` (e.g. `"0x8000"`) |
| `entrySymbol` | Board config `meta.entrySymbol` (e.g. `"_start"`) |
| `size` | Byte size of `kernel.img` at build time |
| `builtAt` | ISO 8601 timestamp (`new Date().toISOString()`) |

`hopper flash` reads this file to display build information and validate the image before writing it to an SD card.

### Linker script (link.ld)

**Location:** `build/link.ld`

Auto-generated on every bare-metal build by `generateLinkerScript(cfg)`. Regenerated even if unchanged. Not meaningful for hosted builds.

### Sanitizer check binary (.check)

**Location:** `build/<basename>.check`

`hopper check` produces a sanitizer-instrumented binary. The default sanitizer is AddressSanitizer (`--asan`); `--tsan`, `--ubsan`, and `--msan` are also supported. Multiple sanitizers may be combined. This binary is for manual execution only.

### gitignore requirements

The `build/` directory MUST be listed in the project's `.gitignore`. The build system never writes `.gitignore` automatically; this is the developer's responsibility. All contents of `build/` are derived artifacts and MUST NOT be committed to version control.

The `modules/` directory, which contains installed dependencies, MAY or MAY NOT be committed. Committing `modules/` makes builds reproducible without running `hopper install`; omitting it keeps the repository smaller. Both approaches are valid. If `modules/` is omitted, `hopper install` MUST be run before `hopper build`.
