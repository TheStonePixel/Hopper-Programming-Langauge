## Workspace Visibility

Hopper uses whole-program compilation. There is no module-level namespace, no package prefix, and no explicit visibility qualifier. All top-level declarations from all compiled files are merged into a single flat symbol space for a given build.

### Compilation model

A hosted build compiles exactly the files identified by the build system: the entry file plus all implementation files referenced by active `"targets"` bindings. Each file is compiled individually to an object file (via LLVM IR), and the objects are linked by `clang`. The entry file and all implementation files see each other's declarations because:

1. The entry file's AST is built with `buildAstFromSource`, which receives the full `bindings` map.
2. For each binding, both the contract file and the implementation file are parsed and their top-level declarations (interfaces, classes, free functions) are merged into the entry file's AST before code generation.
3. Free functions from implementation files come into scope without qualification — `import IO from linux` makes the free functions `read`, `write`, etc. callable directly by name.

### No module namespaces

There is no `module` keyword, no `namespace` qualifier, and no dot-prefixed access for names from imported modules. When `import IO from linux` is resolved, the free functions from `LinuxSyscalls.hop` become top-level names in the calling file's scope. Name collisions between implementations are the programmer's responsibility to avoid.

### Interface and implementation visibility

An interface file (e.g. `linux/contracts/IO.hop`) declares function signatures. An implementation file (e.g. `x86_64/src/LinuxSyscalls.hop`) provides both the `interface LinuxSyscalls` declaration and the free function implementations. When the build system resolves an `import IO from linux` binding:

- The interface declaration from `IO.hop` is merged in (providing type information).
- The implementation file `LinuxSyscalls.hop` is compiled as a separate translation unit. Its free functions are linked in and come into scope in the entry file.

Callers in the entry file invoke free functions by their unqualified names. There is no `IO.read(...)` syntax — it is simply `read(...)`.

### Class names are global

Class names must be unique across all files compiled together in a single build. There is no mechanism to qualify a class name with a module prefix. If two compiled files define a class with the same name, the behavior is undefined (duplicate symbol at link time).

### Self-referential method calls vs. free function calls

Inside a class method, an unqualified call such as `read(fd, buf, count)` resolves to the free function `read` from an imported implementation file, not to the method itself. Method calls always require an explicit receiver: `self.read(...)` or `obj.read(...)`. This distinction is by design and means that contract implementations can delegate to imported free functions without ambiguity.

### Entry point visibility

The `entry main` block (or `entry main(int argc, string[] argv)`) is the program's entry point. It is compiled to a function named `main` in LLVM IR. It has access to all declarations from the entry file and all bound implementation files. The entry block is a distinct syntactic construct — it is not a function and cannot be called from other code.

### No cross-project visibility

There is no mechanism for two separate projects (two separate workspace roots) to share declarations at compile time. Dependencies are made visible by installing them into `modules/` and declaring bindings in `"targets"`. Once installed and bound, the dependency's declarations are visible as described above. Without an explicit binding, no declarations from installed modules are visible.

### Whole-program implication

Because all compilation units are linked into a single executable, the entire program is available to the linker at once. There are no shared libraries, no dynamic loading, and no partial compilation. Dead code elimination, if any, is performed by LLVM and `clang` based on the optimization flags for the selected build profile.
