## Artifact Layout

All build outputs are written inside the project's `build/` directory. The layout is determined by the `profileDirs()` function in the CLI, which is called at the start of every build command.

### profileDirs()

`profileDirs(entryFile, profile)` computes the output directories for a given entry file:

1. Walk up from the entry file's directory to find the project root (the nearest ancestor containing `hopper.json`).
2. Set `buildDir = projectRoot/build`.
3. Set `llDir = buildDir/ll`.
4. Set `oDir = buildDir/o`.
5. Create `llDir` and `oDir` with `mkdirSync(..., { recursive: true })` if they do not exist.
6. Return `{ projectRoot, buildDir, llDir, oDir }`.

The `profile` parameter is accepted but does not currently affect the directory structure. All profiles (`dev`, `test`, `production`) write to the same `build/ll/` and `build/o/` locations. The profile affects only the `clang` flags passed during compilation (see §9.6). **Profile-specific subdirectories (`build/dev/`, `build/test/`, `build/production/`) are not yet implemented.**

### Hosted build layout

After `hopper build` on a hosted (non-bare-metal) executable:

```
build/
  <name>          ← final executable, named from hopper.json "name" field
  ll/
    main.ll       ← LLVM IR for the entry file
    LinuxSyscalls.ll   ← LLVM IR for each bound implementation file
    ...
  o/
    main.o        ← object file for the entry file
    LinuxSyscalls.o    ← object file for each bound implementation file
    ...
```

The final executable is placed directly in `build/`, not in a profile subdirectory. Its name comes from the `"name"` field in `hopper.json`. If that field is absent, the executable is named after the entry file's basename (without extension).

LLVM IR and object files are named after the basename (without extension) of their source file. If two source files from different directories share the same basename, the later file's output will overwrite the earlier one. **Filename collision detection is not yet implemented.**

### Bare-metal build layout

After `hopper build` on a project with a `"board"` field in `hopper.json` (e.g. `"board": "pi-zero"`):

```
build/
  kernel.img         ← raw binary image, stripped from the ELF
  <name>.elf         ← linked ELF binary (intermediate, retained for debugging)
  <name>.meta.json   ← board metadata JSON (see below)
  link.ld            ← auto-generated linker script
  ll/
    <file>.ll        ← LLVM IR per source file
    boot.s           ← board boot stub (assembly, compiled by clang directly)
  o/
    <file>.o         ← object files per source file
    boot.o           ← compiled boot stub
```

The output image name is always `kernel.img` for the Pi Zero board (as defined in `BOARD_CONFIGS["pi-zero"].flash.outputFile`). The ELF is retained alongside for inspection. The linker script is regenerated on every build from the board configuration's `textBase`, `linkerEntry`, and section layout.

### .meta.json (bare-metal)

After every bare-metal build, the CLI writes `build/<name>.meta.json` containing:

```json
{
  "name":        "my-project",
  "board":       "pi-zero",
  "outputFile":  "kernel.img",
  "arch":        "armv6",
  "endian":      "little",
  "bits":        32,
  "loadAddress": "0x8000",
  "entrySymbol": "_start",
  "size":        12345,
  "builtAt":     "2026-05-23T00:00:00.000Z"
}
```

This file is consumed by `hopper flash` to display build metadata and verify the image before writing it to an SD card.

### Build profiles and clang flags

The `profile` argument to `clangFlags(profile)` selects the optimization level:

| Profile | Flags |
|---|---|
| `dev` (default) | `-g -O0 -c -fPIC -Wno-override-module` |
| `test` | `-g -O0 -c -fPIC -Wno-override-module` (same as dev) |
| `production` | `-O2 -c -fPIC -Wno-override-module` |

The profile is selected via CLI flags:
- `hopper build` — dev profile (default)
- `hopper build --production` — production profile
- `hopper build --test` — test profile (same flags as dev)

### check build (.check)

`hopper check` produces a sanitizer-instrumented binary named `build/<basename>.check`. This binary is placed in `build/` and is intended for manual execution, not automated test runs. It is not a standard build artifact and does not participate in the normal build/run cycle.

### Test runner output

Test binaries produced by `hopper test` are written to the repository-global test directory at `hopper/build/tests/` (relative to the repository root), not inside the project's own `build/`. They are named `<group>__<testname>` to avoid collisions between test files from different modules.
