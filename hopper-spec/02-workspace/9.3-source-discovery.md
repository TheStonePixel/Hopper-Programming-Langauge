## Source Discovery

The Hopper compiler is file-driven, not directory-driven. The build system never scans a directory for source files to compile. Every file that enters a compilation is either explicitly named (as the entry point) or pulled in through the interface binding system.

### Entry file

For executables, the entry file is specified in `hopper.json` as the `"entry"` field:

```json
{
  "entry": "src/main.hop"
}
```

This path is relative to the project root. When `hopper build` is invoked without arguments, it reads `"entry"` from the `hopper.json` found in the current working directory. If `"entry"` is absent, it defaults to `"src/main.hop"`. If the resolved file does not exist, the build fails with an error.

When `hopper build` is invoked with explicit file arguments (`hopper build src/main.hop`), those files are compiled as-is without reading `"entry"` from `hopper.json`. This form is used by the test runner and for one-off compilations.

For libraries, there is no `"entry"` field. Libraries expose their source through the `"exports"` field in `hopper.json`, which maps interface names to `"interface"` and `"implementation"` file paths. These paths are not compiled standalone — they are compiled when a consuming project binds to them via its `"targets"` section.

### Binding-driven file inclusion

When the compiler encounters an `import X from Y` statement in a source file, it does not scan the filesystem. Instead, the build system calls `loadBindings(sourceFile, target)` to locate the relevant `hopper.json`, then looks up the binding entry for `X`. That entry provides exactly two file paths:

1. `"interface"` — the interface declaration file (`.hop`), which is parsed and its declarations brought into scope.
2. `"implementation"` — the implementation file (`.hop`), which is compiled as a separate translation unit and linked with the entry file.

Both files are resolved as absolute paths at the start of the build. The compiler receives both as part of the `buildAstFromSource` call's `bindings` map.

### hopper.json discovery walk (loadBindings)

For any given source file, the build system finds its binding context by walking up the directory tree from the source file's location:

1. Start at the directory containing the source file.
2. Check whether `hopper.json` exists in that directory.
3. If found, parse it and return the bindings for the requested target (default: `"host"`).
4. If not found, move to the parent directory and repeat.
5. If the filesystem root is reached without finding `hopper.json`, return an empty binding map.

This walk means that implementation files installed inside `modules/` are compiled in the context of the consuming project, not in the context of the module they belong to. The consuming project's bindings take precedence.

Example: `wc/modules/x86_64/src/LinuxSyscalls.hop` is compiled with `wc/hopper.json`'s bindings, not with any `hopper.json` inside `x86_64/`.

### No implicit imports

There are no auto-included prelude files, no implicit standard library, and no magic imports. A source file sees only:
- Its own top-level declarations
- Declarations merged in from files explicitly bound via `import X from Y` statements matched against `hopper.json` targets

### Module import form (legacy, do not use)

The older lowercase module import form (`import io from arch`) bypasses the binding system entirely. It uses a fallback file resolution path inside `buildAstFromSource`, resolving to `modules/<name>/src/<module>.hop` relative to the importing file's project root. This form is present in files under `hopper/nonconform/` and MUST NOT be used in new code. The interface import form with a `"targets"` binding is the only supported form.

### Test file discovery

The test runner (`tools/test.js`) is the only part of the toolchain that scans a directory. It reads all `.hop` files from the `tests/` directory of the project under test and compiles each one individually. Test files are standalone programs — each must be independently compilable. The test runner does not use `"entry"` from `hopper.json`; it passes each test file path directly to the compiler.
