## Project Layout

Every Hopper project — whether executable or library — MUST conform to the following canonical directory structure. `hopper init` creates this structure; build tools depend on it.

```
project/
  hopper.json          ← manifest (required)
  src/                 ← source files (.hop)
  modules/             ← installed dependencies (each is a full project subtree)
  tests/               ← test files (.hop with directive comments)
  interfaces/          ← exported contract contracts
  build/               ← compiler output (MUST be gitignored)
    ll/                ← LLVM IR intermediates (.ll files)
    o/                 ← object files (.o files)
```

For bare-metal targets, additional artifacts appear inside `build/`:

```
  build/
    <name>.elf         ← linked ELF binary (bare-metal only)
    kernel.img         ← raw binary image (bare-metal only)
    <name>.meta.json   ← board metadata (bare-metal only)
    link.ld            ← auto-generated linker script (bare-metal only)
    ll/
    o/
```

### hopper.json

Required. The project manifest. Its presence in a directory identifies that directory as the workspace root. Contains the project name, version, type, entry point, dependency declarations, contract bindings, and optional bare-metal board configuration.

The build system reads this file to determine:
- The binary name (`"name"` field — falls back to the entry file basename if absent)
- The entry file (`"entry"` field for executables; defaults to `src/main.hop` if absent)
- Interface bindings (`"targets"` field) used to resolve `import X from Y` statements
- The board target (`"board"` field) for bare-metal builds

### src/

Contains all source files for this project. For executables, `src/main.hop` is the conventional entry file (overridable by the `"entry"` field in `hopper.json`). For libraries, the convention is `src/<name>.hop` as the primary implementation file.

`hopper init` creates `src/main.hop` (executable) or `src/<name>.hop` (library) as an empty file.

The compiler does not scan `src/` automatically. Files are compiled only when referenced: the entry file is compiled first, and implementation files pulled in via the `"targets"` binding system are compiled alongside it as separate compilation units that are linked together.

### modules/

Contains locally installed dependencies. Each installed dependency is a full project subtree placed here by `hopper install`. The structure mirrors the dependency graph recursively — a module's own dependencies are installed inside its own nested `modules/` subdirectory.

`hopper init` creates this directory with a `.gitkeep` placeholder. The contents are populated exclusively by `hopper install` and MUST NOT be edited manually.

### tests/

Contains test files. Each file is a `.hop` source file that may include directive comments recognized by the test runner:

- `// EXPECT: <line>` — assert that the program prints this exact line to stdout
- `// COMPILE_ONLY` — verify the file compiles without running it
- `// EXPECT_ERROR: <msg>` — assert that compilation fails with a message containing `<msg>`
- `// TEST: <name>` — start a named section grouping subsequent assertions
- `// XFAIL` — the test is expected to fail (shown in output but not counted as a failure)

`hopper init` creates this directory with a `.gitkeep` placeholder.

Tests are discovered and run with `hopper test`. The runner compiles each `.hop` file in `tests/` and compares stdout against the `// EXPECT:` directives in order. Test binaries land in the repository-global test output directory (`hopper/build/tests/` relative to the repository root), not inside the project's own `build/` directory.

### interfaces/

Contains contract contract files (`.hop`) exported by this project. `hopper init` creates this directory only for executables; libraries may create it manually. In practice both executables and library modules use this directory.

Interface files declare the public contract that implementing classes must satisfy. Consuming projects declare bindings to these files in the `"targets"` section of their `hopper.json`.

### build/

Contains all compiler output. Created automatically by `hopper build`; MUST NOT be committed to version control. Projects SHOULD add `build/` to their `.gitignore`.

- `build/ll/` — LLVM IR files (`.ll`), one per compiled source file, named after the source basename
- `build/o/` — Object files (`.o`), produced by `clang` from the IR files
- `build/<name>` — Final linked executable (hosted build), named from the `"name"` field in `hopper.json`
- `build/kernel.img` — Raw stripped binary image (bare-metal build)
- `build/<name>.elf` — Linked ELF binary before stripping (bare-metal build)
- `build/<name>.meta.json` — Board metadata written after every bare-metal build, consumed by `hopper flash`
- `build/link.ld` — Auto-generated linker script (bare-metal build)

The `build/` directory is flat for hosted builds. The `profileDirs()` function in the CLI resolves `build/` relative to the project root found by walking up from the entry file's directory.

### Non-standard directories

The layout above is canonical. The build system does not define or use any other top-level directories (`dist/`, `out/`, `target/`, etc.). Projects MUST NOT introduce alternative output directories.
