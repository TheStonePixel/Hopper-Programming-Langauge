## Workspace Topology

A Hopper workspace is always a single project. There is no monorepo topology, no workspace-level manifest spanning multiple projects, and no shared flat dependency pool. Each project is self-contained; its dependencies are nested physically inside it.

### Nested dependency tree

When `hopper install <name>` runs, it copies the named module from the global module store (`hopper/modules/` inside the repository) into the consuming project's `modules/` directory. It then recurses: for each dependency declared in the installed module's own `hopper.json`, it installs that dependency into the module's own `modules/` subdirectory. This produces a tree that mirrors the dependency graph exactly.

Example: if `wc` depends on `linux` and `x86_64`, and `linux` has no dependencies of its own, the resulting layout is:

```
wc/
  hopper.json
  src/
  modules/
    linux/
      hopper.json
      interfaces/
      src/
      modules/      ← linux's own deps would go here (none in this case)
    x86_64/
      hopper.json
      src/
      modules/      ← x86_64's own deps would go here
```

The dependency graph is represented as a physical tree of directories, not a flat list. If two modules in the tree both depend on the same third module, that third module appears twice in the tree — once under each dependent. The build system deduplicates `declare` statements in the generated LLVM IR at link time; see `dedupeDeclarations()` in the CLI.

### No lockfile

There is no lockfile. Installed versions are recorded only by the presence of the module directory and the version string in `hopper.json`. Installing the same module again is a no-op if the directory already exists (`installModule` returns immediately if `dest` exists).

### Interface binding in the consuming project

The consuming project's `hopper.json` `"targets"` section is the sole location where contract names are bound to concrete file paths. After `hopper install` runs, it rewrites these paths to point at the locally installed copies under `modules/`. This rewriting is performed by `localizeBindings()`.

A binding entry has this shape:

```json
{
  "IO": {
    "from":           "linux",
    "contract":      "modules/linux/contracts/IO.hop",
    "implementation": "modules/x86_64/src/LinuxSyscalls.hop",
    "inline":         true
  }
}
```

Fields:
- `"from"` — the module name that provides this interface. The import statement in source (`import IO from linux`) MUST match this value.
- `"contract"` — path relative to the project root to the interface declaration file (`.hop`).
- `"implementation"` — path relative to the project root to the implementation file (`.hop`).
- `"inline"` — optional boolean. When `true`, the compiler MAY apply more aggressive inlining for the bound implementation. **Not yet implemented as a distinct compilation path — the field is parsed but its effect is reserved.**

All paths in `"targets"` bindings are resolved relative to the project root containing the `hopper.json` file, using `path.resolve(dir, b.contract)` and `path.resolve(dir, b.implementation)`.

### Binding resolution walk

When compiling a source file, the build system calls `loadBindings(sourceFile, target)`. This function walks up from the source file's directory until it finds a `hopper.json`, then extracts the bindings for the given target (default `"host"`). The first `hopper.json` found upward wins. This means that if a module's source file is compiled in the context of a consuming project, the consuming project's bindings apply — not the module's own self-referential bindings.

### Transitive dependencies and the global store

`hopper install` only installs from the global module store at `hopper/modules/` inside the repository. Remote registry installation via GitHub (`fetchTarball`) is defined in the CLI but the GitHub organization (`HopperLangauge`) does not yet host released packages. All currently usable modules are local to the repository. **Remote registry installation is not yet operational.**

### Dependency declaration before installation

A module MUST be declared in `hopper.json` `"dependencies"` before `hopper install` resolves it as a transitive dependency. When `hopper install` is run with no arguments, it reads `"dependencies"` from the current project's `hopper.json` and installs every declared module. When run with explicit names (`hopper install linux x86_64`), it installs those modules and also rewrites the `"targets"` binding paths via `localizeBindings`.

### hopper fix

`hopper fix` scans all source files in `src/` and `interfaces/` for `import X from Y` statements and checks whether each referenced module `Y` is declared in `"dependencies"` or covered by a `"targets"` binding. For any undeclared import whose module exists in the global store, it adds the missing dependency entry to `hopper.json` automatically. For imports whose module is not found in the store, it reports them as requiring manual resolution.

`hopper fix` also creates any missing standard directories (`src/`, `tests/`, `interfaces/` for libraries) and creates stub files for any missing entry points or export files declared in `hopper.json`.
