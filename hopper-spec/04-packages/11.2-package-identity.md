## Package Identity

A package is identified by its `name` field in `hopper.json`. The name is the package's canonical
identifier in the global module store, in the dependency graph, and in registry operations.

---

### Name Format

Package names MUST satisfy all of the following rules:

1. Names MUST consist only of lowercase ASCII letters (`a`–`z`), ASCII digits (`0`–`9`), and
   hyphens (`-`).
2. Names MUST NOT begin or end with a hyphen.
3. Names MUST NOT contain consecutive hyphens.
4. Names MUST NOT be empty.
5. Names SHOULD be descriptive and as short as clarity permits.

The build tool's `hopper init` command sanitizes the user-supplied project name by replacing any
character that is not in `[a-zA-Z0-9_-]` with an underscore before creating the manifest. This
sanitization is a convenience for interactive use; it does not override the naming rules above.
Published packages MUST conform to the rules regardless of how the manifest was created.

Examples of valid names: `linux`, `x86_64`, `cast`, `hello-world`, `sort-lines`.

Examples of invalid names: `MyLib` (uppercase), `sort lines` (space), `-sort` (leading hyphen),
`sort-` (trailing hyphen).

---

### Name Uniqueness

Package names MUST be unique within the global module store. Each package occupies the directory
`hopper/modules/<name>/`. Installing a package whose name already exists in the project's
`modules/` directory is a no-op — the build tool skips installation of that package silently and
leaves the existing copy intact.

There is no namespace mechanism. If two packages share the same name, only one can exist in the
global store at a time. Name conflicts are resolved at the registry level by the package author
claiming the name first.

---

### Name as Directory Key

The package name determines the directory name in every installation location:

- Global store: `hopper/modules/<name>/`
- Project-local: `<project>/modules/<name>/`
- Transitive: `<project>/modules/<dep>/modules/<name>/`

When `hopper install <name>` is run, the build tool looks up `hopper/modules/<name>/` in the
global store. If the directory does not exist, the installation fails with an error.

---

### Identity in Import Statements

The `from` clause of a Hopper import statement names a module by its package name:

```hopper
import IO from linux
import SIMD from x86_64
```

Here `linux` and `x86_64` are package names. They MUST match the `from` field in the
corresponding binding entry in `hopper.json` `targets`. The build tool resolves the binding by
looking up the `from` value in the target's binding map, not by searching the file system.

---

### The `main` Library Field

The `cmdInit` function generates a `main` field (`"main": "src/<name>.hop"`) in library
manifests created interactively. This field is distinct from the executable `entry` field.
Currently, neither the build tool nor the compiler reads `main` from library manifests. It is
present for tooling compatibility and future use. Libraries SHOULD declare `exports` for their
public interfaces instead.
