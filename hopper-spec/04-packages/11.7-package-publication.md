## Package Publication

Package publication pushes a package to the Hopper registry so other projects can install it.

---

### The `hopper publish` Command

```
hopper publish [name]
```

`hopper publish` is a fully implemented command. It performs the following steps:

1. **Read the manifest.** The build tool reads `hopper.json` in the current working directory. If
   no manifest is found, the command aborts. If a `name` argument is supplied on the command line,
   it overrides the `name` field from the manifest.

2. **Require a token.** The command requires a GitHub personal access token in `HOPPER_TOKEN` or
   `GITHUB_TOKEN`. If neither is set, the command aborts with an error.

3. **Require a git repository.** The command verifies that the current directory is inside a git
   repository (`git rev-parse --git-dir`). If it is not, the command aborts.

4. **Create the registry repository.** The command calls the GitHub API:
   ```
   POST /orgs/HopperLangauge/repos
   ```
   with `{ "name": "<package-name>", "private": false }`. If the repository already exists
   (HTTP 422), the command continues — the push in the next step will update the existing
   repository.

5. **Push to the registry.** The command adds a git remote named `registry` pointing at
   `https://github.com/HopperLangauge/<name>.git` and runs `git push registry main`. This pushes
   the current `main` branch to the registry repository.

On success, the command prints the registry URL and the `hopper install` invocation other
developers can use:

```
published: linux@0.1.0  ✓
  https://github.com/HopperLangauge/linux
  hopper install linux
```

---

### The `hopper revoke` Command

```
hopper revoke [name]
```

`hopper revoke` permanently deletes a package from the registry. It performs the following steps:

1. **Read the manifest and resolve the name.** Reads `hopper.json` for the default name, or uses
   the `name` argument if supplied.

2. **Require a token.** Same token requirement as `hopper publish`.

3. **Confirm interactively.** Displays a warning that the deletion is permanent and prompts the
   user to type the exact package name to confirm:
   ```
   This will delete HopperLangauge/<name> from the registry permanently.
   Anyone who has not installed it yet will no longer be able to.

   type the package name to confirm:
   ```
   If the confirmation does not match the package name exactly, the command aborts without making
   any network request.

4. **Delete the registry repository.** Calls the GitHub API:
   ```
   DELETE /repos/HopperLangauge/<name>
   ```
   A successful deletion returns HTTP 204. Any other status is treated as a failure.

Revocation deletes the registry repository. It does not remove the package from the local global
module store, and does not affect any project that has already installed the package.

---

### Publication Requirements

A package SHOULD satisfy the following requirements before publication:

- `hopper.json` MUST be present and valid.
- The `name` field MUST conform to the naming rules in §11.2.
- The `version` field MUST be a valid semver string (§11.3).
- The package MUST build without errors (`hopper build`).
- All transitive dependencies MUST be installable from the global store or registry.

**No automated pre-publication checks are performed by the build tool.** The commands above do
not run `hopper build` or validate the manifest before pushing. The author is responsible for
publishing working packages.

---

### Version Management

Publication does not enforce version uniqueness. Re-running `hopper publish` with the same
version string overwrites the registry repository with the current state of the local repository.
There is no mechanism to publish multiple versions of a package simultaneously — the registry
repository holds one branch (`main`) and the version in `hopper.json` is informational.

Immutable version releases are **not yet implemented**.
