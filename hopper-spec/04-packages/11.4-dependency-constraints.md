## Dependency Constraints

Dependencies are declared in a manifest's `dependencies` field as a JSON object mapping package
name to a version string:

```json
{
  "dependencies": {
    "linux":  "0.1.0",
    "x86_64": "0.1.0",
    "cast":   "0.1.0"
  }
}
```

---

### Current Behavior

The `dependencies` field serves two purposes in the current implementation:

1. **`hopper install` with no arguments** reads `dependencies` to determine which packages to
   install. It installs all listed packages from the global module store into the project's
   `modules/` directory.

2. **`hopper fix`** scans source files for `import` statements, identifies modules that are not
   covered by `dependencies` or `targets` bindings, and adds them to `dependencies` with the
   version string found in that module's own manifest.

The version string in a `dependencies` entry is **not enforced**. The build tool installs whatever
version is available in the global store. If the store has `linux` at `0.1.0` and the manifest
requests `"0.2.0"`, the build tool installs `0.1.0` without warning or error.

**Dependency version constraint enforcement is not yet implemented.**

---

### Transitive Dependencies

When the build tool installs a module, it reads that module's own `dependencies` field and
recursively installs each transitive dependency into the module's own `modules/` subdirectory.
This nesting means each module carries its own dependency tree — deduplication occurs at compile
time, not at install time.

Example: if `core` depends on `arch` and `arch` depends on `x86_64`, installing `core` into
`project/modules/` produces:

```
project/modules/
  core/
    hopper.json
    src/
    interfaces/
    modules/
      arch/
        hopper.json
        src/
        modules/
          x86_64/
            hopper.json
            src/
            interfaces/
```

---

### Modules Declared in `targets` vs. `dependencies`

Modules referenced only through `targets` interface bindings do not strictly require a matching
`dependencies` entry. The build tool installs them when they are named explicitly:

```
hopper install linux x86_64
```

However, packages SHOULD list all modules — including those referenced through `targets` — in
`dependencies`. This ensures that `hopper install` with no arguments installs the complete
dependency set, and that `hopper fix` can audit imports against the declared dependency list.

---

### Undeclared Import Detection

`hopper fix` scans all `.hop` source files in `src/` and `interfaces/` for `import` statements.
For each imported module name that is not covered by `dependencies` or `targets.*.from` entries,
`hopper fix`:

- If the module exists in the global store: adds it to `dependencies` automatically and reports
  the addition.
- If the module does not exist in the global store: reports it as needing manual resolution.

`hopper fix` does not install modules — it only updates `hopper.json`. Run `hopper install` after
`hopper fix` to install newly added dependencies.

---

### Constraint Resolution Algorithm

**Not yet implemented.** A future version of the build tool will implement a constraint resolution
algorithm that:

1. Collects the declared version constraints from all packages in the dependency graph.
2. Selects a single version for each package that satisfies all constraints.
3. Errors if no satisfying version exists (conflict), with a report naming the conflicting
   constraints.
4. Pins the selected versions in a lockfile (see §11.5).

Until then, the single version present in the global store is always used.
