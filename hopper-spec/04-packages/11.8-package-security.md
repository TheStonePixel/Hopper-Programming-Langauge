## Package Security

The current Hopper package system has no supply chain security mechanisms. This section documents
what is absent, why it matters, and what the planned mitigations are.

---

### Current Security Properties

**No content hashing.** The build tool does not compute or verify checksums of installed package
files. A package installed from the global store could be modified on disk without detection.

**No signature verification.** Package sources are not signed. There is no mechanism to verify
that a package installed from the registry was published by the expected author.

**No checksum pinning.** The lockfile that would record expected content hashes does not exist
(see §11.5). Without a lockfile, there is no baseline to compare against.

**No transport-layer pinning.** Packages are distributed as Git repositories over HTTPS. GitHub's
TLS certificate provides transport security. No additional certificate pinning is performed by the
build tool.

**No malware scanning.** Published packages are not scanned for malicious content before or after
publication.

**No access control on publication.** Any holder of a GitHub personal access token with repository
creation permissions in the `HopperLangauge` organization can publish a package. There is no
review gate, no code signing requirement, and no proof-of-origin check.

---

### Install Mechanism and Its Implications

The current install mechanism copies files from the local global module store (`hopper/modules/`)
into the project's `modules/` directory. This is a local file copy — it has no network exposure
and is not subject to registry-level attacks as long as the Hopper repository itself is not
compromised.

When remote installation is implemented (see §11.6), packages will be fetched over HTTPS from
GitHub or the future `thestonepixel.com/hopper` registry. At that point, integrity verification
becomes a prerequisite for safe installation. The `fetchTarball` helper in the build tool handles
redirect following and tarball extraction but performs no hash verification.

---

### Planned Security Measures

The following security features are planned for the package system. All are **not yet
implemented**:

**Content hashing.** Every published package should produce a content hash (SHA-256 of a
canonical tarball) that is recorded in the registry and stored in `hopper.lock`. The build tool
should verify this hash after downloading and before extracting any package.

**Lockfile integrity.** Once the lockfile exists (§11.5), `hopper install` should refuse to
install if the computed hash of a downloaded package does not match the hash recorded in
`hopper.lock`. Mismatches should produce an error that halts the build.

**Signed releases.** Package authors should be able to sign releases with a keypair. The registry
should store the public key associated with each package name. The build tool should verify the
signature before installing.

**Namespace ownership.** The registry should enforce that a package name is owned by a single
verified author. Transfers should require explicit action by the current owner. This prevents
dependency confusion attacks where an attacker publishes a package with the same name as an
internal package.

**Revocation transparency.** When a package is revoked (§11.7), the registry should record the
revocation event and notify downstream consumers. Currently, revocation silently deletes the
repository with no notification to users who have a dependency on the package.

---

### Risk Assessment for Current Usage

The current system is appropriate for a single-organization language toolchain where:

- All packages in the global store are authored and reviewed by the Hopper project.
- Installation is a local file copy from a version-controlled repository.
- The scope of the registry is small and well-known.

As the ecosystem grows and third-party packages enter the registry, the absence of integrity
verification, signature checking, and lockfile support becomes a material risk. Security hardening
of the package system is required before the registry should be treated as broadly trustworthy.
