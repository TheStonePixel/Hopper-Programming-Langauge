## 11.9 The `hopper install` Algorithm

`hopper install` copies modules from the global module store into a project's local `modules/` directory. After installation, each dependency and all of its transitive dependencies are present in a nested tree that mirrors the dependency graph exactly.

### 11.9.1 Invocation Forms

```
hopper install                  install all dependencies declared in hopper.json
hopper install <name>           install a specific module by name
hopper install <a> <b> <c>      install multiple named modules
```

When called with no arguments, the build tool reads `hopper.json` from the current directory and installs every package listed under `"dependencies"`. When called with explicit names, those names are installed directly (the `"dependencies"` field in `hopper.json` is not consulted for the list, but the file must still be present).

### 11.9.2 Global Module Store

The global module store is the directory `hopper/modules/` in the repository root. Every conforming module that ships with Hopper lives here. The store is not a network registry — it is a directory tree checked into the repository. Modules installed from the store are identical copies, not symlinks.

Currently defined modules in the global store:

| Module | Description |
|--------|-------------|
| `linux` | Linux OS contract declarations (`IO`, `FileSystem`, `System`, `Process`, `Socket`, `Network`, `Memory`) |
| `x86_64` | x86-64 syscall implementations and SIMD operations |
| `arch` | Architecture portability shims (legacy) |
| `cast` | Template cast functions for primitive type conversions |

### 11.9.3 Installation Algorithm

For each module name `N` to install, the algorithm proceeds as follows:

1. **Compute destination.** The destination directory is `<projectRoot>/modules/<N>`.

2. **Deduplication check.** If the destination directory already exists, skip this module entirely. This prevents redundant work and handles shared transitive dependencies.

3. **Locate source.** Look up `N` in the global store at `<repoRoot>/hopper/modules/<N>`. If the directory does not exist, print an error and abort.

4. **Copy files.** Create the destination directory. Copy `hopper.json` from the source. Copy the `src/` and `interfaces/` subdirectories if they exist. No other subdirectories are copied — in particular, the source module's own `modules/` directory is not copied; transitive dependencies are resolved recursively in the next step.

5. **Recurse into transitive dependencies.** Read `dependencies` from the newly copied `hopper.json`. For each transitive dependency `D`, call the algorithm recursively with:
   - The dependency name `D`
   - The destination `<projectRoot>/modules/<N>/modules/`

   This nesting is the key invariant: each installed module carries its own `modules/` subtree containing its own dependencies. The folder tree mirrors the dependency graph exactly.

6. **Log.** Print `installed: <N>` at the appropriate indentation depth.

The algorithm produces a tree like this for a project depending on `linux` and `x86_64`:

```
modules/
  linux/
    hopper.json
    interfaces/
      IO.hop
      FileSystem.hop
      ...
    modules/          ← linux's own transitive deps (if any)
  x86_64/
    hopper.json
    src/
      LinuxSyscalls.hop
      SIMD.hop
    interfaces/
      SIMD.hop
    modules/          ← x86_64's own transitive deps (if any)
```

### 11.9.4 Binding Path Rewriting

After all modules are installed, the build tool rewrites the `targets` bindings in the project's `hopper.json` to point at the newly installed local copies.

For each binding that references an installed module (identified by the `"from"` field), the `"contract"` and `"implementation"` path strings are rewritten. Specifically:

- Any path segment `/<moduleName>/` is replaced with `modules/<moduleName>`.

For example, if the project has a binding:

```json
{
  "IO": {
    "from":           "linux",
    "contract":      "hopper/modules/linux/interfaces/IO.hop",
    "implementation": "hopper/modules/x86_64/src/LinuxSyscalls.hop"
  }
}
```

After `hopper install linux x86_64`, it becomes:

```json
{
  "IO": {
    "from":           "linux",
    "contract":      "modules/linux/interfaces/IO.hop",
    "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
  }
}
```

Path rewriting happens in-place: `hopper.json` is updated on disk. This means the project can be compiled immediately after installation without any manual path configuration.

Bindings whose `"from"` field does not match any installed module name are left untouched.

### 11.9.5 Interaction with `hopper build`

`hopper build` does not run `hopper install` automatically. If a dependency is listed in `hopper.json` but not installed (the `modules/<name>` directory is absent), the build fails when the compiler cannot resolve the import.

The intended workflow is:

```
hopper init my-project --yes
# edit hopper.json to add dependencies
hopper install
hopper build
```

### 11.9.6 Uninstallation

```
hopper uninstall <name> [<name>...]
```

Removes the named module directories from `modules/`. Transitive dependencies installed inside the module's own `modules/` tree are removed along with it. The global store is not affected.

### 11.9.7 Conformance Requirements

A conforming implementation of `hopper install` MUST:

- Copy `hopper.json`, `src/`, and `interfaces/` from the global store into the project's local `modules/` directory.
- Recursively install transitive dependencies into each module's own `modules/` subdirectory, not into the project's top-level `modules/` directory.
- Skip any module whose local directory already exists (deduplication).
- Abort with a clear error message if a required module is not present in the global store.
- Rewrite `targets` binding paths to point at the locally installed copies after all modules are installed.
