## Package Registries

The Hopper package registry is the authoritative source for published packages. The build tool
uses the registry when a package is not present in the local global module store.

---

### Current Registry

The current package registry is the GitHub organization **HopperLangauge** at:

```
https://github.com/HopperLangauge
```

Each package is a repository within this organization. The repository name MUST match the package
name declared in the package's `hopper.json`. Publishing a package named `linux` creates or
updates the repository at `https://github.com/HopperLangauge/linux`.

This registry location is hardcoded in the build tool as:

```js
const REGISTRY_ORG = "HopperLangauge";
const REGISTRY_URL = `https://github.com/${REGISTRY_ORG}`;
```

The comment in the source reads: "swap for thestonepixel.com/hopper later." A dedicated registry
at `thestonepixel.com/hopper` is planned. Migrating to that registry will require a build tool
update to change `REGISTRY_URL`.

---

### Install Resolution Order

When `hopper install <name>` is run, the build tool currently resolves packages exclusively from
the local global store (`hopper/modules/<name>/`). It does NOT automatically fetch packages from
GitHub. The GitHub registry is used only by `hopper publish` and `hopper revoke` to manage
registry repositories.

The `fetchTarball` helper function exists in the build tool source for future use in downloading
packages from the registry, but it is not currently invoked during `hopper install`. Remote
installation is **not yet implemented**.

---

### Registry Protocol

**A formal registry protocol is not yet implemented.** The current system uses GitHub as a
distribution mechanism — packages are Git repositories in the `HopperLangauge` organization.
There is no package index, no search API, no tarball hosting separate from GitHub, and no
integrity verification beyond what GitHub provides.

The planned registry at `thestonepixel.com/hopper` is expected to provide:

- A package index with search.
- Versioned tarball downloads.
- Content hashes for integrity verification.
- An HTTP API for publish and revoke operations.

None of these features are implemented in the current build tool. The formal registry protocol
specification is deferred pending the registry implementation.

---

### Private Packages

There is no mechanism for private packages in the current system. All packages published to the
`HopperLangauge` GitHub organization are public. Private package support is **not yet
implemented**.

---

### Registry Credentials

The `hopper publish` and `hopper revoke` commands require a GitHub personal access token with
repository creation and deletion permissions. The token MUST be provided via one of:

- The environment variable `HOPPER_TOKEN`.
- The environment variable `GITHUB_TOKEN`.

The build tool does not store credentials on disk. If neither variable is set, `hopper publish`
and `hopper revoke` abort with an error.

---

### Relationship Between Global Store and Registry

The global module store (`hopper/modules/`) is a collection of packages maintained within the
Hopper repository itself. It is the primary source for `hopper install`. The registry at GitHub
is a publication endpoint — when a package author publishes a package, they push their package
repository to the `HopperLangauge` GitHub organization. Other developers can then obtain the
package by cloning that repository or — once remote install is implemented — by running
`hopper install <name>`.

Currently, the authoritative copy of every standard library package is in `hopper/modules/`. The
GitHub registry is secondary.
