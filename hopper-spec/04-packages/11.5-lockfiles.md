## Lockfiles

**Lockfiles are not yet implemented.**

A lockfile does not exist in the current Hopper package system. The build tool does not generate,
read, or require any lockfile. Every `hopper install` operation installs the version currently
present in the global module store, without recording which version was installed.

---

### What a Lockfile Provides

A lockfile is a generated file that records the exact version of every package — direct and
transitive — that was resolved during a successful `hopper install`. Its purpose is to make builds
reproducible across machines and over time.

Without a lockfile:

- Two developers running `hopper install` at different times may receive different versions of a
  transitive dependency if the global store or registry has been updated between runs.
- A CI build and a developer's local build may produce binaries compiled against different
  interface implementations, even if the top-level `hopper.json` has not changed.
- There is no record of which module versions were in use when a binary was last built.

With a lockfile, every `hopper install` reads the locked versions and installs exactly those
versions. The lockfile is committed to version control. A developer cloning the repository and
running `hopper install` receives an identical dependency tree to every other developer.

---

### Planned Lockfile Design

The lockfile, when implemented, SHOULD:

- Be named `hopper.lock` and reside in the project root alongside `hopper.json`.
- Be a JSON (or similar structured text) file mapping each package name to its resolved version,
  content hash, and source location.
- Be generated automatically by `hopper install` whenever a new resolution is performed.
- Be checked into version control. Its presence in a repository is normative: it asserts that the
  listed versions are correct and verified.
- Be read by `hopper install` when present, bypassing version resolution and installing the pinned
  versions directly.
- Be regenerated by `hopper install --update` (or a similar explicit flag) to pick up new
  versions.

The lockfile MUST cover all transitive dependencies, not only direct dependencies declared in
`hopper.json`.

---

### Current State

Because there is no lockfile and no version constraint enforcement, Hopper builds are not
reproducible in the strict sense. Two builds from the same source tree are identical only if the
global module store has not changed between them. For a language where correctness is verified at
compile time through interface conformance checks, this is an acceptable limitation during early development
— a breaking change in a dependency manifests as a compile error, not a silent runtime regression.
Full reproducibility requires a lockfile and is planned for a future release.
