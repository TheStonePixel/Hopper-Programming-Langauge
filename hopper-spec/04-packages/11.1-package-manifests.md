## Package Manifests

A package manifest is a file named `hopper.json` in the root of a package directory. The manifest
is a JSON object. Every package MUST have a manifest. The build tool reads the manifest to
determine the package type, entry point, dependencies, and contract bindings before invoking the
compiler.

---

### Required Fields

The following fields MUST be present in every manifest:

**`name`** (string) — The package name. MUST conform to the naming rules in §11.2. The build tool
uses this field as the output binary name for executables and as the package's identity in the
module store.

**`version`** (string) — The package version. MUST be a semver string (see §11.3). Example:
`"0.1.0"`.

**`type`** (string) — The package type. MUST be one of: `"executable"`, `"library"`, or
`"program"`. Use `"program"` for bare-metal targets that declare a `board` field.

---

### Optional Metadata Fields

**`description`** (string) — A human-readable description of the package. Displayed in registry
listings and by tooling. SHOULD be present in all published packages.

**`license`** (string) — The SPDX license identifier. Examples: `"MIT"`, `"Apache-2.0"`,
`"GPL-2.0-only"`. SHOULD be present in all published packages.

---

### Executable-Specific Fields

**`entry`** (string) — Path to the `.hop` source file containing the `entry main` block, relative
to the manifest directory. Executables MUST declare this field. Example: `"src/main.hop"`.

**`targets`** (object) — Interface binding map. Executables MUST declare this field. The object's
keys are target names; currently `"host"` is the standard target for Linux hosted builds. Each
target value is a map from binding name to a binding object (see Binding Object below).

```json
{
  "targets": {
    "host": {
      "IO": {
        "from":           "linux",
        "contract":      "modules/linux/interfaces/IO.hop",
        "implementation": "modules/x86_64/src/LinuxSyscalls.hop"
      }
    }
  }
}
```

---

### Library-Specific Fields

**`exports`** (object) — Export map for libraries. The object's keys are contract names; the
values are export objects with `contract` and optionally `implementation` fields (both paths
relative to the manifest directory).

Libraries that provide only contract contracts (like `linux`) MAY omit the `implementation` field
from an export entry:

```json
{
  "exports": {
    "IO": { "contract": "interfaces/IO.hop" }
  }
}
```

Libraries that ship both the contract contract and its implementation declare both:

```json
{
  "exports": {
    "Cast": {
      "contract":      "interfaces/Cast.hop",
      "implementation": "src/Cast.hop"
    }
  }
}
```

The `exports` field is currently informational for consuming projects. The build tool does not
read a library's `exports` to resolve bindings in consuming projects — the consumer declares the
full paths in its own `targets` section. This restriction is expected to be lifted in a future
version of the build system.

---

### Dependency Fields

**`dependencies`** (object) — Map from package name to version string. Declares the packages this
project requires. The build tool reads this field when `hopper install` is run with no arguments.
Example:

```json
{
  "dependencies": {
    "linux":  "0.1.0",
    "x86_64": "0.1.0"
  }
}
```

Modules that are referenced only through `targets` bindings (and not imported by name with an
uppercase binding) do not strictly need a `dependencies` entry, but SHOULD be listed for
documentation and tooling support.

---

### Binding Object

Each entry in a `targets` binding map is a binding object with the following fields:

**`from`** (string, required) — The module name that the import statement names. This MUST match
the module name in `import <Interface> from <module>` in source files.

**`contract`** (string, required) — Path to the `.hop` contract file, relative to the manifest
directory. This file MUST contain an `contract` declaration.

**`implementation`** (string, required) — Path to the `.hop` implementation file, relative to the
manifest directory. This file MUST implement every function declared in the contract.

**`inline`** (boolean, optional) — When `true`, instructs the build system to compile the
implementation with `clang -O2` inlining hints. Default: `false`. Currently: the flag is read by
the build tool and stored in the binding map, but LLVM inlining behavior is governed by the IR
generated by the compiler, not this flag alone.

---

### Bare-Metal Fields

**`board`** (string, optional) — The board name for bare-metal targets. When present, `hopper
build` selects the bare-metal compilation path instead of the hosted path. The board name MUST
match a known entry in the build tool's `BOARD_CONFIGS` table. Currently defined boards:
`"pi-zero"` (Raspberry Pi Zero, ARMv6, 32-bit, loads at `0x8000`).

**`flash`** (object, optional) — Flash configuration for bare-metal programs. Currently accepts a
`mount` field that overrides SD card auto-detection with an explicit mount path.

A complete bare-metal manifest:

```json
{
  "name": "blink",
  "version": "0.1.0",
  "description": "Pi Zero ACT LED + UART blink — bare-metal Hopper demo",
  "license": "Apache-2.0",
  "type": "program",
  "board": "pi-zero",
  "main": "blink.hop",
  "flash": {
    "mount": null
  }
}
```

---

### Complete Executable Example

```json
{
  "name": "wc",
  "version": "0.1.0",
  "description": "word, line, and byte counter",
  "license": "MIT",
  "type": "executable",
  "entry": "src/main.hop",
  "dependencies": {
    "linux": "0.1.0",
    "x86_64": "0.1.0"
  },
  "targets": {
    "host": {
      "IO": {
        "from":           "linux",
        "contract":      "modules/linux/interfaces/IO.hop",
        "implementation": "modules/x86_64/src/LinuxSyscalls.hop",
        "inline":         true
      },
      "FileSystem": {
        "from":           "linux",
        "contract":      "modules/linux/interfaces/FileSystem.hop",
        "implementation": "modules/x86_64/src/LinuxSyscalls.hop",
        "inline":         true
      },
      "SIMD": {
        "from":           "compute",
        "contract":      "modules/compute/interfaces/SIMD.hop",
        "implementation": "modules/x86_64/src/SIMD.hop",
        "inline":         true
      }
    }
  }
}
```

---

### Complete Library Example

```json
{
  "name": "linux",
  "version": "0.1.0",
  "description": "Linux OS interfaces — IO, FileSystem, System, Process, Socket, Network, Memory",
  "license": "Apache-2.0",
  "type": "library",
  "exports": {
    "IO":         { "contract": "interfaces/IO.hop"         },
    "FileSystem": { "contract": "interfaces/FileSystem.hop" },
    "System":     { "contract": "interfaces/System.hop"     },
    "Process":    { "contract": "interfaces/Process.hop"    },
    "Socket":     { "contract": "interfaces/Socket.hop"     },
    "Network":    { "contract": "interfaces/Network.hop"    },
    "Memory":     { "contract": "interfaces/Memory.hop"     }
  }
}
```

---

### Manifest Parsing Rules

The build tool parses `hopper.json` as UTF-8 JSON. A malformed manifest (invalid JSON syntax)
causes the build to abort with an error message identifying the parse failure. Unknown fields are
silently ignored, which permits forward-compatible manifests.

The build tool searches for `hopper.json` by walking up the directory tree from the source file
being compiled. The first `hopper.json` found wins. This allows sub-project builds to use their
own manifest even when nested inside a parent project tree.
