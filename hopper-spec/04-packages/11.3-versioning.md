## Versioning

Every Hopper package carries a version string in its manifest's `version` field. Version strings
use semantic versioning (semver) format: `MAJOR.MINOR.PATCH`, where each component is a
non-negative integer. Example: `"0.1.0"`.

All packages in the current global module store are at version `"0.1.0"`. The build tool defaults
newly initialized packages to `"0.1.0"` unless overridden by the `--version=` flag to
`hopper init`.

---

### Version String Rules

A valid Hopper version string MUST conform to semver 2.0.0:

- Three dot-separated non-negative integers: `MAJOR.MINOR.PATCH`.
- No leading zeros (e.g., `"01.0.0"` is invalid; `"1.0.0"` is correct).
- Pre-release and build metadata suffixes (e.g., `"1.0.0-alpha.1"`) are permitted by semver but
  not currently used or processed by the build tool.

The build tool does not validate the version string format at install time. Invalid version strings
are silently accepted. Tooling such as `hopper fix` reads version strings from module manifests to
fill in dependency entries, and will propagate whatever string is present.

---

### Version Constraint Declarations

The `dependencies` section of a manifest declares versions as strings:

```json
{
  "dependencies": {
    "linux":  "0.1.0",
    "x86_64": "0.1.0"
  }
}
```

This is currently a point-version declaration, not a range constraint. Range operators such as
`^`, `~`, `>=`, and `<` are not parsed by the build tool. The version string in `dependencies` is
documentation for what version was used when the manifest was written, not a constraint that the
build tool enforces.

**Version constraint enforcement is not yet implemented.** The build tool installs whatever version
of a package is present in the global module store, regardless of the version string declared in
`dependencies`. If the store contains `linux` at version `0.1.0` and the manifest requests
`"0.2.0"`, the build tool installs `0.1.0` without warning.

---

### Versioning Semantics (Intent)

The intended semver interpretation, to be enforced once constraint resolution is implemented:

- **PATCH increment** — backward-compatible bug fixes. Existing callers continue to compile and
  run correctly.
- **MINOR increment** — backward-compatible additions. New functions added to an interface;
  existing callers unaffected.
- **MAJOR increment** — breaking changes. Interface functions removed, renamed, or with altered
  signatures. Consuming packages must be updated.

Because Hopper interfaces are checked at compile time, a breaking change produces a compile error
in any consuming package that has not been updated — not a silent runtime mismatch.

---

### No Version Resolution

The current build tool has no version resolution algorithm. There is no dependency solver, no
conflict detection, and no version pinning mechanism. When a project installs a module, it
receives the single version present in the global store. When two modules in the same project
depend on the same transitive dependency at different versions, both receive the same installed
version with no warning.

Formal version resolution is **not yet implemented.**

---

### Relationship to the Registry

When `hopper publish` creates a registry entry, it publishes the current state of the repository
to `github.com/HopperLangauge/<name>`. The published version is whatever `version` string is in
the manifest at the time of publication. The registry does not enforce uniqueness of version
strings — re-publishing the same version string with different content is not prevented. Version
immutability enforcement is **not yet implemented.**
