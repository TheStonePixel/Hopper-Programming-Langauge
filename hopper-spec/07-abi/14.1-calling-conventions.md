## Calling Conventions

Hopper does not specify a calling convention. The Hopper compiler emits standard LLVM IR with no calling-convention annotations, which means LLVM selects the default calling convention for the target triple. For hosted Linux x86-64 that is the x86-64 System V ABI; for bare-metal ARMv6 that is the ARM EABI soft-float convention. This section documents both.

---

### Hosted: x86-64 System V ABI

Hopper's hosted target is Linux x86-64. The calling convention is determined by clang's default for that platform and is the x86-64 System V ABI as defined by the System V AMD64 ABI specification. The Hopper compiler does not annotate IR functions with a calling convention keyword; `ccc` (C calling convention) is the LLVM default and matches System V.

#### Integer and Pointer Argument Passing

Arguments are classified by type. The first six integer or pointer arguments are passed in registers, in order:

| Position | Register |
|---|---|
| 1 | `rdi` |
| 2 | `rsi` |
| 3 | `rdx` |
| 4 | `rcx` |
| 5 | `r8` |
| 6 | `r9` |

Additional integer or pointer arguments are passed on the stack in right-to-left order, pushed before the call.

Hopper maps its integer types to 64-bit values: `int` and `unsigned int` are `i64`; `byte`, `char`, and `unsigned byte` are `i8`; `bool` is `i1`. All of these are passed in the integer registers, zero- or sign-extended to 64 bits as required by the ABI. Pointer types (`address`, `string`, `string[]`) are 64-bit and also use the integer registers.

#### Floating-Point Argument Passing

The first eight floating-point arguments are passed in `xmm0` through `xmm7`. Hopper's `float` maps to `double` (64-bit IEEE 754), which is a floating-point class under System V and uses the XMM registers.

#### Return Values

Integer and pointer return values are placed in `rax`. Floating-point return values are placed in `xmm0`. Functions returning `void` (Hopper procedures with no return type) return nothing; LLVM emits `ret void` and the caller ignores `rax`.

The entry point returns `i64` in `rax`. The C runtime interprets the low 8 bits as the process exit status.

#### Callee-Saved Registers

Under System V, the callee MUST preserve `rbx`, `rbp`, `r12`, `r13`, `r14`, and `r15`. All other registers are caller-saved. Hopper relies on LLVM's register allocator to emit correct save/restore sequences.

#### Register Allocation

Hopper does not perform register allocation. Register assignment is entirely delegated to LLVM's backend. The Hopper compiler uses SSA virtual registers in the IR (`%t0`, `%t1`, ŌĆ”); LLVM maps these to physical registers during compilation.

### Class Instances and the Implicit Self Parameter

Class methods receive the instance pointer as an implicit first argument. The Hopper compiler emits the self parameter explicitly in the IR signature:

```
define i64 @MyClass_method(%class.MyClass* %self, i64 %p0) {
```

Under x86-64 System V, `%self` occupies the first integer register slot (`rdi`). The next declared parameter (`%p0`) occupies the second slot (`rsi`), and so on. Callers MUST pass the instance pointer as the first argument; there is no implicit receiver register.

Constructors and destructors follow the same convention. The constructor receives `self` as its first argument and returns `void`; the object is allocated before the call and the pointer is passed in.

---

### Bare-Metal: ARM EABI Soft-Float

The bare-metal target is ARMv6 (Raspberry Pi Zero, BCM2835 / ARM1176JZF-S). The build system invokes clang with `-target arm-none-eabi -mcpu=arm1176jzf-s -mfloat-abi=soft`. The calling convention is the ARM EABI with software floating-point linkage.

#### ARM EABI Integer Argument Passing

The first four 32-bit word arguments are passed in `r0`ŌĆō`r3`. Additional arguments are passed on the stack. On ARMv6 bare-metal, Hopper's `int` maps to `i32` (not `i64` as on hosted x86-64): the codegen module calls `setWordBits(32)` when the target is `armv6-bare`, which causes `llvmType("int")` to return `i32` instead of `i64`. Pointer types are also 32-bit on this target.

#### Soft-Float

With `-mfloat-abi=soft`, floating-point arguments are passed in integer registers (or on the stack for excess arguments). There are no dedicated FPU registers in the calling convention. LLVM emits calls to soft-float library routines (`__aeabi_dadd`, `__aeabi_dmul`, etc.) for floating-point arithmetic. Hopper's `float` (mapped to `double`) therefore uses software emulation on bare-metal.

#### Return Values

Integer and pointer return values are in `r0`. 64-bit values use `r0`:`r1`. Soft-float doubles are returned in `r0`:`r1`.

#### Callee-Saved Registers

The ARM EABI requires the callee to preserve `r4`ŌĆō`r11`, `r13` (sp), and `r14` (lr) across calls. The stack MUST be 8-byte aligned at all public function entry points. LLVM enforces these constraints.

---

### Variadic Functions

Hopper source functions are never variadic. The `...` syntax in `externParamList` is grammar-legal only in `extern function` declarations. This allows Hopper to call C variadic functions such as `printf`. The Hopper compiler emits the appropriate LLVM variadic call syntax when calling an extern declared with `...`. No Hopper-defined function may be variadic.
