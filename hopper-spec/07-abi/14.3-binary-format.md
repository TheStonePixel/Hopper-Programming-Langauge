## Binary Format

The binary format produced by `hopper build` depends on whether the target is hosted (Linux x86-64) or bare-metal. The two paths are entirely separate: hosted builds produce dynamically linked ELF64 executables via clang's default linker; bare-metal builds produce a raw ARM binary image via `ld.lld` and `llvm-objcopy`.

---

### Hosted: ELF64 on Linux x86-64

#### Output

`hopper build` on a hosted project produces an ELF64 dynamically linked executable, written to `build/<project-name>`. No `-static` flag is passed to clang; the output MUST link against the dynamic linker and libc.

The clang link invocation is:

```
clang <object-files...> -lpthread -o <output>
```

No additional linker flags are specified. Section layout, PHDR placement, and dynamic linking metadata are entirely determined by clang's default linker script for the host platform (typically `ld` or `lld`).

#### Intermediate Files

The build system produces the following intermediate files for each compiled translation unit, stored under `build/ll/` and `build/o/`:

| Directory | File | Contents |
|---|---|---|
| `build/ll/` | `<basename>.ll` | LLVM IR text emitted by the Hopper compiler |
| `build/o/` | `<basename>.o` | ELF64 relocatable object produced by clang |

The final linked executable is placed directly in `build/`.

#### Compilation Flags

Per-file clang invocation:

| Profile | Flags |
|---|---|
| dev (default) | `-g -O0 -c -fPIC -Wno-override-module` |
| production | `-O2 -c -fPIC -Wno-override-module` |
| check (sanitizers) | `-g -O0 -c -fPIC -Wno-override-module` (sanitizer flags added at link time) |

`-fPIC` is passed for all profiles. `-Wno-override-module` suppresses warnings that arise when multiple translation units are compiled separately (each emits its own `target triple` header; the linker sees the combination).

#### Runtime Dependencies

Hosted binaries MUST be run on a Linux x86-64 system with:

- A standard C library (glibc or musl) providing `malloc`, `free`, and `abort`.
- pthreads (`libpthread` or equivalent), linked via `-lpthread`.

No other dynamic libraries are linked by the build system. Programs that need additional C libraries MUST declare them via `extern function` and link them by modifying the build system or using a wrapper.

---

### Bare-Metal: Raw Binary (kernel.img)

#### Output

`hopper build` for a project whose `hopper.json` specifies `"board": "pi-zero"` produces a raw ARM binary file named `kernel.img` in `build/`. The Raspberry Pi Zero GPU loads `kernel.img` from the SD card's FAT partition at address `0x8000` and jumps to the first byte.

The build sequence is:

1. Compile each `.hop` source file to LLVM IR, then to an ELF relocatable object using clang with bare-metal flags.
2. Assemble the board boot stub (`boot.s`) to a separate ELF relocatable object.
3. Link all objects using `ld.lld` with an auto-generated linker script, producing an ELF file (`build/<name>.elf`).
4. Strip the ELF to a raw binary using `llvm-objcopy -O binary`, producing `build/kernel.img`.

#### Bare-Metal Clang Flags

```
-target arm-none-eabi -mcpu=arm1176jzf-s -mfloat-abi=soft -ffreestanding -nostdlib -O2 -c -Wno-override-module
```

`-ffreestanding` suppresses assumptions about the C standard library. `-nostdlib` suppresses implicit linking against libc and the C runtime startup files. No `malloc`, `free`, or `abort` declarations are emitted in the LLVM IR for bare-metal targets (the codegen skips those `declare` lines when `target === "armv6-bare"`).

#### Linker Script

The build system auto-generates a linker script for each bare-metal build. The script is written to `build/link.ld` and passed to `ld.lld -T`. The generated script places sections as follows, with the image loaded at `0x8000`:

```
ENTRY(_start)
SECTIONS
{
    . = 0x8000;
    .text.boot : { KEEP(*(.text.boot)) }   /* boot stub _start, must be first */
    .text       : { *(.text*) }
    .rodata     : { *(.rodata*) }
    .data       : { *(.data*) }
    .bss        : {
        . = ALIGN(4);
        __bss_start = .;
        *(.bss*)
        *(COMMON)
        . = ALIGN(4);
        __bss_end = .;
    }
    /DISCARD/ : {
        *(.ARM.exidx*)
        *(.ARM.extab*)
        *(.note*)
    }
}
```

The `.text.boot` section is always placed first, immediately at `0x8000`, so the GPU's unconditional branch to that address lands on `_start`. ARM exception index and extab sections are discarded; bare-metal Hopper programs do not use C++ exception handling.

#### Boot Stub

The Pi Zero boot stub (`seed/build_system/boards/pi-zero/boot.s`) executes before any Hopper code. It:

1. Zeroes the BSS region using the `__bss_start` and `__bss_end` symbols defined by the linker script.
2. Sets the stack pointer to `0x8000` (growing downward into RAM below the kernel image).
3. Calls `main` via `bl main`.
4. Enters an infinite loop if `main` returns (the `hang` label).

The boot stub is assembled into `.text.boot` and linked first. Its entry symbol `_start` is the linker entry point. No Hopper programmer writes or modifies this stub; the build system links it automatically.

#### Build Metadata

After a successful bare-metal build, the build system writes a `<name>.meta.json` file alongside `kernel.img`. This file records the architecture, endianness, word size, load address, entry symbol, image size, and build timestamp. It is consumed by `hopper flash` and is not part of the deployed binary.

#### Deployment

`hopper flash` copies `kernel.img` to a mounted FAT-formatted SD card containing the Pi Zero GPU firmware blobs (`bootcode.bin` and `start.elf`). It also ensures `config.txt` contains the required clock and LED configuration lines. The SD card is then inserted into the Pi Zero and powered.
