## Platform Compatibility

Hopper currently supports two target platforms. All other platforms are **not yet implemented**. This section documents the two supported targets, their constraints, and what a port to a new platform requires.

---

### Supported Platforms

#### 1. Linux x86-64 (Hosted)

This is the primary development and production target.

| Property | Value |
|---|---|
| OS | Linux (any distribution with glibc or musl) |
| Architecture | x86-64 |
| Mode | Hosted (C runtime, dynamic linking, libc) |
| Compiler | clang (system default) |
| Linker | clang's default (ld or lld) |
| ABI | x86-64 System V |
| Word size | 64-bit |
| `int` LLVM type | `i64` |
| Output | ELF64 dynamically linked executable |
| Entry | `@main` returning `i64` |
| Libraries | libc (`malloc`, `free`, `abort`), pthreads (`-lpthread`) |
| `hopper.json` target key | `host` |

The hosted target is activated when `hopper.json` does not contain a `"board"` field. All development tooling (`hopper dev`, `hopper check`, `hopper test`) operates on the hosted target exclusively.

Programs for the hosted target import OS interfaces from the `linux` module and hardware interfaces from the `x86_64` module. The `hopper.json` targets section maps these imports to concrete implementations in `modules/x86_64/src/LinuxSyscalls.hop`.

#### 2. ARMv6 Bare-Metal (Raspberry Pi Zero)

This is the supported embedded target.

| Property | Value |
|---|---|
| Board | Raspberry Pi Zero (BCM2835, ARM1176JZF-S) |
| Architecture | ARMv6 |
| Mode | Bare-metal (no OS, no libc, no dynamic linker) |
| Compiler | clang with `-target arm-none-eabi` |
| Assembler | clang (for the boot stub) |
| Linker | `ld.lld` with auto-generated linker script |
| Binary converter | `llvm-objcopy -O binary` |
| ABI | ARM EABI soft-float |
| Word size | 32-bit |
| `int` LLVM type | `i32` |
| Output | Raw binary `kernel.img` |
| Load address | `0x8000` |
| Entry | `_start` (boot stub) → `@main` |
| Libraries | None (libc not linked) |
| `hopper.json` field | `"board": "pi-zero"` |

The bare-metal target is activated by setting `"board": "pi-zero"` in `hopper.json`. The build system reads this field and selects the `BOARD_CONFIGS["pi-zero"]` entry, which provides all necessary flags, the boot stub path, and the flash configuration.

The bare-metal codegen path sets `wordBits = 32`, which changes the LLVM type for `int` and `unsigned int` from `i64` to `i32`, and for `address` from `i8*` (64-bit pointer) to `i8*` (32-bit pointer). The ARM EABI linker script is generated by `generateLinkerScript(cfg)` in the build system; it is not read from a file.

The LLVM IR header for bare-metal targets includes explicit `target datalayout` and `target triple` directives:

```llvm
target datalayout = "e-m:e-p:32:32-i64:64-v128:64:128-a:0:32-n32-S64"
target triple = "armv6-none-eabi"
```

These are absent from hosted-target IR, where clang infers the datalayout from the host machine.

### Not Yet Implemented Platforms

The following platforms are not currently supported. They are documented here to indicate the scope of work required for a port.

- **Linux ARM64 (aarch64)** — The logical next hosted target. Would require a new module alongside `x86_64/` with an `arm64/` implementation of `LinuxSyscalls`. No board configuration needed (hosted, standard ELF).
- **Linux RISC-V** — Similar to ARM64 in structure.
- **macOS x86-64 / ARM64** — Would require a Mach-O binary format and different syscall numbers. The `linux` module interfaces would not apply directly; a `darwin` module would be needed.
- **ARMv7 / ARM Cortex-M (bare-metal)** — Would require a new board config entry. The Cortex-M uses Thumb-2 instruction set and a different interrupt vector table layout than ARMv6.
- **RISC-V bare-metal** — Would require a new board config and potentially a RISC-V boot stub.
- **WebAssembly** — Would require a fundamentally different codegen target; LLVM supports wasm32/wasm64 but the ABI and memory model differ substantially.

### Porting to a New Platform

Adding a new target requires changes at two layers: the build system and the module system. The two layers are independent — a new board can share an existing ISA module, and a new OS can share existing board configs.

#### Build System Changes (new board)

Add an entry to `BOARD_CONFIGS` in `seed/build_system/hopper`:

```javascript
"board-name": {
    llvmTarget:    "arch-none-eabi",    // passed to genModule as target name
    clangTarget:   "arch-none-eabi",    // clang -target flag
    mcpu:          "cpu-name",          // clang -mcpu flag
    floatAbi:      "soft",              // "soft", "softfp", or "hard"
    textBase:      0x8000,              // load address (integer)
    linkerEntry:   "_start",            // linker ENTRY() symbol
    bootStub:      path.join(__dirname, "boards", "board-name", "boot.s"),

    meta: {
        arch:        "archname",
        endian:      "little",
        bits:        32,
        loadAddress: "0x8000",
        entrySymbol: "_start",
    },

    flash: {
        method:          "sd-card",
        outputFile:      "kernel.img",
        requiredFirmware: ["bootcode.bin"],
        configLines:     [],
    },
},
```

Provide a `boot.s` assembly file at the `bootStub` path. The stub MUST be placed in `.text.boot`, MUST define a global `_start` symbol, MUST zero BSS between `__bss_start` and `__bss_end`, MUST set up a valid stack pointer, and MUST call `main` (the symbol the Hopper compiler emits for the entry point).

If the new target has a word size different from 64, register the appropriate `llvmTarget` string and add handling in `genModule` (analogous to the `armv6-bare` check that calls `setWordBits(32)`).

#### Module System Changes (new ISA)

Add a hardware implementation module alongside `hopper/modules/x86_64/`. The new module MUST provide at minimum:

- An implementation of the `LinuxSyscalls` interface for the new ISA (if targeting a Linux OS).
- Inline assembly (`asm {}` blocks) or `extern function` stubs for each syscall.

The new module MUST NOT be added to `hopper/nonconform/`; it MUST conform to the standard module layout with a `hopper.json`, `src/`, and `interfaces/` directory.

Programs targeting the new ISA update only their `hopper.json` targets section — changing the `implementation` path from `modules/x86_64/src/LinuxSyscalls.hop` to `modules/newarch/src/LinuxSyscalls.hop`. No program source changes are required.

#### SIMD and ISA-Specific Capabilities

ISA-specific capabilities (SIMD, hardware-specific intrinsics) SHOULD be exposed as a new module (analogous to `x86_64/` providing `SIMD`). Programs that use SIMD already reference `X86SIMD` by its concrete class name; a port that wants to offer equivalent SIMD operations should provide a class with the same contract but different inline assembly. Programs targeting both ISAs conditionally MUST use the build file's `targets` section to select the appropriate implementation.

### Toolchain Requirements

| Target | Required tools |
|---|---|
| Linux x86-64 (hosted) | `clang`, `ld` (or `lld`) |
| ARMv6 bare-metal | `clang` (with ARM cross-compilation support), `ld.lld`, `llvm-objcopy` |
| Any new bare-metal | `clang` (with target support), `ld.lld`, `llvm-objcopy` |

The build system assumes all tools are on `PATH`. No path configuration is exposed in `hopper.json`; if tools are in non-standard locations, the `PATH` environment variable MUST be set before invoking `hopper build`.
