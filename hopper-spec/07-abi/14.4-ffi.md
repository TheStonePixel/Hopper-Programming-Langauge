## FFI: Foreign Function Interface

Hopper's foreign function interface (FFI) is the `extern function` declaration. It maps a Hopper function signature to an external C-linkage symbol, causing the LLVM IR to contain a `declare` directive for that symbol rather than a `define`. The linker resolves the symbol at link time from the platform's C library or any other linked object.

---

### Declaration Syntax

```hopper
extern function name(type param, type param, ...) returnType
extern function name(type param)
```

The grammar rule is:

```
'extern' 'function' Identifier '(' externParamList? ')' type   -- returns a value
'extern' 'function' Identifier '(' externParamList? ')'        -- void (procedure)
```

`externParamList` permits an optional trailing `...` for variadic C functions:

```
param (',' param)* (',' '...')?
'...'
```

The `...` marker is only legal in `extern` declarations. No Hopper-defined function may be variadic.

#### Examples

```hopper
extern function write(int fd, address buf, int count) int
extern function malloc(int size) address
extern function free(address ptr)
extern function printf(string fmt, ...) int
extern function memcpy(address dst, address src, int n) address
```

---

### LLVM IR Emission

An `extern function` declaration emits a single `declare` line in the LLVM IR output:

```llvm
declare i64 @write(i64, i8*, i64)
declare i8* @malloc(i64)
declare void @free(i8*)
declare i32 @printf(i8*, ...)
declare i8* @memcpy(i8*, i8*, i64)
```

The compiler suppresses the `declare` if a concrete definition (`define`) for the same name already exists in the same translation unit. This prevents duplicate-symbol IR when a Hopper source file provides a native implementation of a function that is also declared `extern` (for example, when an implementation module both implements and declares a syscall wrapper).

Duplicate `extern` declarations for the same name within a single compilation unit are also deduplicated: only the first `declare` is emitted.

---

### Type Correspondence

The following table shows how Hopper types map to C types and LLVM IR types when used in `extern` declarations. These mappings are the same as the general Hopper type mappings (§14.5); they are reproduced here because the correspondence to C types is the operative concern for FFI correctness.

| Hopper type | LLVM IR type | C type | Notes |
|---|---|---|---|
| `int` | `i64` | `long` / `int64_t` | **Not** C `int`; see warning below |
| `unsigned int` | `i64` | `unsigned long` / `uint64_t` | **Not** C `unsigned int` |
| `byte` | `i8` | `char` / `int8_t` | |
| `unsigned byte` | `i8` | `unsigned char` / `uint8_t` | |
| `char` | `i8` | `char` | Alias for `byte` |
| `bool` | `i1` | `_Bool` | 1-bit integer |
| `float` | `double` | `double` | **Not** C `float` (32-bit) |
| `address` | `i8*` | `void*` | Raw untyped pointer |
| `string` | `i8*` | `const char*` / `char*` | Null-terminated pointer |
| `string[]` | `i8**` | `char**` | Pointer to pointer (argv) |

#### Critical Warning: `int` Width Mismatch

Hopper's `int` is 64-bit (`i64`) on hosted x86-64. C's `int` is 32-bit (`i32`). When calling a C function that takes or returns a C `int`, a Hopper `extern` declaration using `int` will pass a 64-bit value where the C function expects 32-bit. This is an ABI mismatch that the compiler does not detect or warn about.

To call a C function with `int` parameters correctly, the programmer MUST use explicit knowledge of the C function's actual signature. There is currently no Hopper primitive that maps to C's 32-bit `int`. Workarounds:

- Use `byte` (8-bit) for parameters that fit in a byte.
- For 32-bit C `int` parameters, pass `int` and accept that the upper 32 bits will be present. On x86-64 System V, 32-bit arguments zero-extend to 64 bits in registers, so for many C functions the lower 32 bits are what matters and the call succeeds in practice — but this is not guaranteed and depends on the C function's implementation.
- Declare the parameter as `address` and perform manual bit manipulation if the distinction matters.

**Not yet implemented:** a `c_int` or `i32` type alias that maps to C's native `int`.

#### Note on `float`

Hopper's `float` maps to `double`. To call a C function that takes a C `float` (32-bit), this is a type mismatch. On x86-64, `float` arguments are passed in `xmm` registers and LLVM will emit the correct conversion if the IR type matches. Currently there is no 32-bit float type in Hopper; if you need to call a C function expecting `float`, cast the value and accept the truncation.

---

### Header Imports

Hopper has no `#include` equivalent. There is no automatic import of C headers. All C functions that Hopper code calls MUST be declared with `extern function`. This is by design: the programmer explicitly names and types every external dependency.

---

### Runtime-Declared Externs

The Hopper compiler itself emits `declare` directives for three runtime symbols on hosted targets (not emitted for bare-metal):

```llvm
declare i8* @malloc(i64)
declare void @free(i8*)
declare void @abort()
```

`abort` is emitted only in non-release (debug) builds, where it is used to implement contract failures. In production builds (`--production`), `abort` is suppressed.

These declarations are emitted unconditionally at the top of each hosted translation unit. They do not conflict with user-declared `extern function malloc(...)` declarations because the deduplication pass prevents duplicate `declare` lines.

---

### Inline Assembly and Extern

Programs MUST NOT combine `asm {}` blocks and `extern function` declarations for the same semantic operation. `asm {}` is for direct hardware access in system-level modules; `extern function` is for linking against C library or OS functions. The architectural rule is that inline assembly belongs exclusively in hardware implementation modules (`x86_64/src/LinuxSyscalls.hop`, `x86_64/src/SIMD.hop`); application code MUST NOT contain `asm {}` blocks.
