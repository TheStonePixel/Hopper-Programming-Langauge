## Symbol Mangling

Hopper uses a simple deterministic name mangling scheme for all symbols it emits into LLVM IR. The scheme has five cases: free functions (unoverloaded), overloaded free functions, template functions, class members, and string constants. All mangled names are valid LLVM `@`-prefixed global identifiers.

The mangling rules are implemented in `seed/compiler/src/codegenTypes.js` (`mangledFnName`) and `seed/compiler/src/codegenLLVM.js` (template function name construction).

---

### Free Functions (Unique Name)

A free function whose name does not appear more than once in the translation unit is emitted with its source name, unmodified:

```
@functionName
```

The LLVM IR directive is:

```llvm
define i64 @write(i64 %p0, i8* %p1, i64 %p2) {
```

No prefix, no suffix. The source name IS the symbol name. This means Hopper free functions with unique names have C-linkage-compatible symbols and can be called from C without any name decoration.

---

### Overloaded Free Functions

When two or more free functions share the same name (an overload group), all functions in the group are emitted under mangled names. The mangling scheme is:

```
@name__type1_type2_..._typeN
```

- Double underscore (`__`) separates the base name from the parameter type list.
- Parameter types are joined with single underscores (`_`).
- Each type is the normalized Hopper type name, with non-alphanumeric characters replaced by `_`.

The `mangledFnName` function in `codegenTypes.js`:

```javascript
function mangledFnName(name, params) {
    if (!params || params.length === 0) return `${name}__`;
    const sig = params.map(p => normalizeType(p.type).replace(/[^a-zA-Z0-9]/g, "_")).join("_");
    return `${name}__${sig}`;
}
```

A zero-parameter overloaded function is mangled as `name__` (trailing double underscore, no type components).

#### Examples

```
function write(int fd, address buf, int count) int
  → @write__int_address_int

function write(int fd, string msg) int
  → @write__int_string

function sort()
  → @sort__
```

Call sites perform overload resolution at compile time by matching argument types to the overload group. The resolved mangled name is emitted directly in the call instruction.

---

### Template Functions

Template functions (declared with `template function`) are specialised at each call site. Each specialisation is emitted as a separate LLVM function under a mangled name that encodes both the type parameter and the value parameter types:

```
@__tmpl_name__typeParam__paramType1__paramType2__...__paramTypeN
```

- Leading `__tmpl_` prefix identifies the symbol as a template function specialisation.
- The base name follows immediately.
- Double underscore (`__`) separates the name from the type parameter.
- Double underscore (`__`) separates the type parameter from the parameter type list.
- Parameter types within the list are separated by double underscores (`__`).

The name is constructed in `codegenLLVM.js`:

```javascript
const mangledName = `__tmpl_${tf.name}__${normalizeType(tf.typeParam)}__${paramTypes.join('__')}`;
```

#### Examples

```
template function cast<int>(float src) int
  → @__tmpl_cast__int__float

template function cast<address>(int src) address
  → @__tmpl_cast__address__int

template function cast<byte>(int src) byte
  → @__tmpl_cast__byte__int
```

The `cast<T>(x)` call syntax in Hopper source resolves to a template function lookup; the compiler finds the matching specialisation and emits a call to the appropriate `@__tmpl_cast__...` symbol.

### Class Methods

Class methods are emitted under a name that combines the class name and the method name with a single underscore separator:

```
@ClassName_methodName
```

#### Constructor

```
@ClassName_constructor
```

#### Destructor

```
@ClassName_destructor
```

#### Operators

Operator methods use an encoded operator name appended to the class name. The encoding is handled by `operatorNameSafe`:

```
@ClassName_op_encodedOperator
```

The operator encoding table:

| Hopper operator | Encoded name |
|---|---|
| `+` | `plus` |
| `-` | `minus` |
| `*` | `mul` |
| `/` | `div` |
| `%` | `mod` |
| `==` | `eq` |
| `!=` | `ne` |
| `<` | `lt` |
| `<=` | `le` |
| `>` | `gt` |
| `>=` | `ge` |
| `&` | `band` |
| `\|` | `bor` |
| `^` | `bxor` |
| `<<` | `shl` |
| `>>` | `shr` |
| `[]` | `idx` |
| `[]=` | `setidx` |

#### Examples

```
class Buffer {
    function read(int n) int { ... }
    constructor(int size) { ... }
    destructor() { ... }
    operator [](int i) byte { ... }
    operator []=(int i, byte v) { ... }
}

Emits:
  @Buffer_read
  @Buffer_constructor
  @Buffer_destructor
  @Buffer_op_idx
  @Buffer_op_setidx
```

The LLVM IR definition for a method always takes the instance pointer as its first parameter:

```llvm
define i64 @Buffer_read(%class.Buffer* %self, i64 %p0) {
```

### Entry Point

The Hopper `entry` declaration emits the named function and, if the name is not `main`, also emits a thin `@main` stub that calls it:

```
entry main { ... }
  → defines @main returning i64

entry startup { ... }
  → defines @startup returning i64
  → also defines @main() returning i64, which calls @startup()
```

The `@main` function always returns `i64`. This differs from the C convention of `int main()` returning `i32`. The C runtime's `_start` stub uses the low bits of the return value as the process exit status; in practice `i64` in `rax` is compatible because the Linux `exit_group` syscall takes the low 8 bits regardless of the register width.

When `entry` takes parameters (`entry main(int argc, string[] argv)`), the entry function is defined with those parameters in the IR:

```llvm
define i64 @main(i32 %_param_argc, i8** %_param_argv) {
```

Note that `argc` is declared as `i32` (not `i64`) in the IR signature, matching the C runtime's `main(int argc, char** argv)` convention. The compiler emits a `sext i32 → i64` instruction immediately after the entry block label to widen `argc` to the Hopper `int` width before it is usable as a Hopper variable.

### String Constants

String literal constants are collected during code generation and emitted as private unnamed-addr globals. Each distinct string value gets a unique numbered name:

```
@.str.0 = private unnamed_addr constant [N x i8] c"...\00"
@.str.1 = private unnamed_addr constant [M x i8] c"...\00"
```

The index is a monotonically incrementing counter across the compilation of a single translation unit. String constants are null-terminated; the array length `N` includes the null byte. Duplicate string values are deduplicated: the `addStringConstant` function in the compiler returns the existing name if the same value has already been interned.

---

### Struct and Class Type Names

LLVM named struct types for Hopper classes use the prefix `%class.`:

```
%class.Buffer = type { i8*, i64, i64 }
```

Hopper `struct` types use the prefix `%struct.`:

```
%struct.Stat = type { i64, i64, i64, ... }
```

These are IR-level names only and do not appear in the binary symbol table.

---

### Bind Globals

`bind` directives emit function-pointer globals with a hardware-address-encoded name:

```
@__bind_0x00000004 = global i8* bitcast (void ()* @reset to i8*), section ".hopbind.0x00000004", align 4
```

The global is placed in a named section so the linker can position it at the declared hardware address. The section name encodes the hardware address. These symbols appear in the ELF symbol table as global objects.
